Building the MVP While Keeping the Dream Alive

By chance, I spent part of today refining the MVP roadmap for Piper Morgan, so this reflection on balancing your vision against shipping something feels ultra-timely to me.

July 9

There’s a graveyard somewhere filled with the features we were “definitely going to build in Phase 2.” It’s right next to the cemetery of “we’ll refactor that later” and across from the monument to “temporary solutions.”

But what if — hear me out — what if we actually remembered our dreams while shipping our reality?

The vision reality check

In early July, after nearly six weeks of work, I suggested to my Chief Architect (that’s Claude Opus, for those keeping track) that we review the original Piper Morgan roadmap. You know, the one from when everything was possible and nothing had been built yet.

Reading it was like finding a time capsule from a more optimistic timeline:

  • “AI-powered analytics dashboard integration”
  • “Automatic meeting transcript analysis”
  • “Visual mockup to ticket generation”
  • “Predictive project timeline analysis”
  • “Cross-team knowledge federation”

Meanwhile, in reality, we were still trying to make markdown formatting work consistently.

Documenting your dreams.

Instead of feeling bad about the gap between vision and reality, we did something unusual. We documented the dreams. Not in some dusty “future ideas” file, but in a real, living document: integration-targets.md.

We didn’t just write “analytics integration would be nice.” We wrote things like:

## Analytics Dashboard Integration (Phase 2)

Target Platforms:
- Datadog: API v2, metrics and logs endpoints
- New Relic: GraphQL API, NRQL queries  
- Google Analytics: Reporting API v4

Specific Integration Points:
- Pull performance anomalies → Create investigation tickets
- Weekly metrics summary → Team status reports
- Alert fatigue analysis → Priority recommendations

Authentication Requirements:
- API keys via secure storage
- OAuth2 for GA
- Service account for New Relic

Concrete. Specific. Achievable. Someday.

The art of productive procrastination

Here’s what I learned: documenting future features in detail is productive procrastination. While we were “definitely not building these features,” we:

  1. Researched actual APIs
  2. Identified authentication patterns
  3. Found common integration challenges
  4. Created realistic implementation estimates

When we do get around to this, when this item comes up finally on our agenda, we’ll know where to start.

It’s like meal prepping for features. When Phase 2 arrives, we won’t start from scratch wondering “what did we want to build again?”

The MCP plot twist

Then came the plot twist. Remember MCP (Model Context Protocol)? That new standard from Anthropic that promises to make AI tool integration actually work? One of the first things my Chief of Staff did was help we make a plan to embrace this protocol in our architecture.

Suddenly, half our “Phase 3 dreams” became “possible in Phase 2”:

Before MCP: Build custom integrations for 15 different tools
After MCP: Build one MCP adapter, get 15 integrations

Before MCP: 6 months of API wrangling
After MCP: 6 weeks of standard implementation

The dreams we documented weren’t just dreams anymore. They were a roadmap that suddenly made sense. If anything, this reminded me of the danger of planning too carefully too far in advance, especially when building around rapidly evolving technologies. Better to be ready to evolve than to imagine you can map it all out months in advance and before all the things you inevitably learn when you start trying to do stuff.

The psychology of feature FOMO

Let’s be honest about why we usually don’t document future features:

  1. It feels like admitting defeat — “We can’t build this now”
  2. It creates expectations — “But you said analytics was coming!”
  3. It’s painful — Watching good ideas sit unbuilt
  4. It seems wasteful — Time spent not coding current features

But here’s the flip side:

  1. It’s actually admitting wisdom — “We know what matters most”
  2. It manages expectations — “Here’s the honest roadmap”
  3. It’s motivating — “Look what we’re building toward”
  4. It’s efficient — Future you will thank current you

The three-column reality

We ended up with a beautiful three-column view of Piper Morgan:

Column 1: What’s Built

  • GitHub issue creation ✓
  • Document analysis ✓
  • Basic workflow engine ✓
  • Intent classification ✓

Column 2: What’s Building

  • Workflow persistence ← we were here
  • Jira integration ← we thought this would be next
  • Learning system
  • Web UI polish

Column 3: What’s Documented

  • Analytics integration
  • Meeting intelligence
  • Visual analysis
  • Predictive insights
  • Knowledge federation

Three columns. One product. Zero guilt.

Suddenly I flashed back to a startup whose name I’ll withold. I asked to see the roadmap and they showed me a Google doc. A Google doc with a numbered list in it. With 396 items on the list.

The unexpected benefits

Documenting dreams while building reality had unexpected benefits:

Architecture decisions made sense

When choosing between two implementation approaches, we could ask: “Which one makes Column 3 easier?” Suddenly, technical decisions had long-term context.

Scope creep had boundaries

“Can we add just one more thing to the MVP?”
 “Sure, is it in Column 2?”
 “Well, no, but — “
 “Then it’s documented in Column 3 for Phase 2.”

Scope creep solved by documentation. Who knew?

(Imaginary) investor/stakeholder conversations

“What’s your product vision?”
 hands over integration-targets.md
 “Here’s exactly what we’re building and when.”

Beats hand-waving every time.

The integration targets hall of fame

Some highlights from our documented dreams:

Meeting Transcript Analysis

  • Input: Zoom/Teams recordings
  • Output: Action items, decisions, follow-ups
  • Integration point: Calendar API for automated scheduling
  • Complexity: High
  • Value: Huge
  • Timeline: Phase 2 (Q3 2025) [ROADMAP CHECK: How does this look from September perspective?]

Visual Content Analysis

  • Input: Screenshots, mockups, whiteboard photos
  • Output: User stories, component tickets
  • Integration point: Figma API for design specs
  • Complexity: Medium
  • Value: High
  • Timeline: Phase 2 (Q3 2025)

Predictive Analytics

  • Input: Historical project data
  • Output: Timeline predictions, risk scores
  • Integration point: Jira/GitHub/GitLab history
  • Complexity: Very High
  • Value: Transformative
  • Timeline: Phase 3 (2026)

The discipline of dreams

Here’s the hard part: maintaining the discipline to not build these features. Every week, one of them calls to us:

“Meeting transcripts would be so cool…”

“Visual analysis is basically just another analyzer…”

“We’re already using LLMs, predictive analytics is right there…”

No. Column 2 first. Column 3 waits its turn.

The documentation keeps us honest. It says “we see you, cool feature. Your time will come. But not today.”

For fellow dreamers

If you’re building an MVP while managing a grand vision:

  1. Document dreams concretely — APIs, endpoints, data models
  2. Create clear phases — What’s now, what’s next, what’s someday
  3. Research while you document — Future you needs details
  4. Share the vision — Let others see where you’re going
  5. Revisit quarterly — Dreams change, documentation should too

Where this leaves us

Looking back from September 2025, it’s interesting to see how this three-column approach played out. Some of Column 3 moved faster than expected (Morning Standup integration), while other Column 2 items have taken long (Web UI polish). The MCP integration proved even more transformative than we anticipated, though in different ways than originally planned.

Today, Piper Morgan can create GitHub issues and analyze documents. Not revolutionary. But it does those things well. It can also publish documents, report on GitHub issue status, and check my calendar. It can provide me with what I need for a morning standup.

Tomorrow (actually Q3 2025), it’ll analyze meeting transcripts and integrate with analytics platforms. The foundation we’re building supports those dreams.

Someday (2026?), it’ll predict project timelines and share knowledge across teams. And we’ll build that on top of what comes before.

The dream is documented. The path is clear. The MVP ships first.
