The Demo That Broke (And Why That’s Perfect)

Part two of demo-process flashback weekend. It was nearly a week later when I finally got around to trying to do that demo….

June 23

“Let me just fire up the prototype to show you what we built…”

Famous last words.

The ChromaDB database had other plans. What followed was a debugging session that taught me more about demo infrastructure than any architecture diagram ever could.

The demoable branch

After weeks of building Piper Morgan 1.0’s sophisticated architecture, I wanted to show someone the original prototype. You know, the scrappy POC that started it all. The one that had been working perfectly for weeks.

python pm_agent_poc.py
sqlite3.OperationalError: no such column: collections.topic

Wait, what?

The mystery of the missing column

ChromaDB — our vector database for storing knowledge — had decided to evolve. Without asking. Without warning. Just… evolved.

The detective work began:

  • Our code hadn’t changed
  • The database files were still there
  • Everything SHOULD work

But software doesn’t care about should.

The smoking gun

After digging through configuration files and database schemas, the culprit emerged: unpinned dependencies. Hadn’t we already addressed this? Why did it keep happening?

Our requirements.txt had this innocent line:

chromadb

No version. No constraints. Just “chromadb” with all the trust of a developer who believes in semantic versioning.

ChromaDB 0.4.22 had created our database. ChromaDB 0.5.x was trying to read it. The schema had changed. Game over.

The nuclear option

Sometimes the best debugging is knowing when to stop debugging.

rm -rf ./pm_kb_docs

Delete the incompatible database. Start fresh. Move on with life. It’s not like I’ve got a lot of original data stored up in there yet!

Within minutes, the prototype was running again, creating GitHub issues like nothing had happened. Crisis averted, lesson learned.

The success that looked like failure

With the database fixed, I ran the prototype again:

python pm_agent_poc.py
Running tests...
✓ Created GitHub issue
✓ Checked issue status  
✓ Classified intent with 95% confidence

Then it exited.

“It stopped itself?” I asked.

No. It completed successfully. I had completely forgotten what the initial POC did. The POC was not the prototype! The POC was literally a proof-of-concept. The prototype was a test suite, not an interactive CLI. It did exactly what it was supposed to do — run tests and report results. It had a few dummy documents, a few canned queried, and it ran them, successfully, and then exited.

I had completely forgotten that part!

And without knowing that, success looked exactly like failure.

The philosophy moment

During the debugging, something important crystallized. When asked about the project’s goals, I found myself saying:

“This project is as much a learning opportunity for me as a product build… The goal is to make a serious robust persistent tool for myself. If that succeeds, a natural follow-on goal would be to generalize that.”

Build for yourself first. Solve your own problem. If it works for you, it might work for others.

This isn’t revolutionary. Git started as Linus solving his own source control problem. Docker began as Solomon solving deployment headaches. The best developer tools often start as personal frustration made manifest.

At that point I was really just trying to figure out if I could make anything at all, let alone something useful for myself. But that was when I began thinking this might really be something other people could use someday too.

Demo infrastructure as architecture

You may recall we had worked out our demo strategy a week earlier:

  • Prototype Demo: Show the technical evolution (for developers)
  • Product Demo: Show the user value (for PMs)

Each demo has:

  • Git branch with tagged stable points
  • Locked dependency versions
  • Docker environment
  • One-command startup
  • Clear recovery procedures

Suddenly, “demo infrastructure” wasn’t about presentations. It was about development confidence. Always having a working version. Capturing learning milestones. Speaking to different audiences.

The lessons

  1. Pin your dependencies: chromadb==0.4.22 not chromadb
  2. Demo readiness is development hygiene: If you can’t demo it, is it really working?
  3. Success can look like failure: Always verify expected behavior
  4. Build for yourself first: Your own pain is the best requirements document
  5. Delete and move on: Not every problem needs deep debugging

Once I recalled that the command-line POC and the prototype with the GUI were started with different commands, I was back in the demo business!

The beautiful mundane

By session’s end, we had:

  • A working prototype (again)
  • A demo strategy that was starting to pay off
  • Locked (frozen) requirements files
  • A deeper understanding of why demos matter

The latest ChromaDB crisis led to better architecture decisions than any planning session could have. Sometimes the best insights come from fixing what’s broken.

Next Time: Different audience, different demo

The prototype sits in its pristine state, working again. The 1.0 architecture continues to evolve. But now we have something more: a reliable strategy for showing the work to different audiences.

Technical audience? Here’s how we evolved from POC to platform. Product audience? Here’s how this saves you hours every week. Executive audience? Here’s the vision made tangible.

Same product. Different stories. All true.
