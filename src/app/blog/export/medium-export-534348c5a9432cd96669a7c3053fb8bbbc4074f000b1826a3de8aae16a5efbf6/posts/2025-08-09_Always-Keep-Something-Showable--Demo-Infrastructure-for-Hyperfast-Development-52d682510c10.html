Always Keep Something Showable: Demo Infrastructure for Hyperfast Development

Time for a weekend process flashback, to the third week of this project, when I started to want to demo it my colleagues.

June 14

When you’re building with AI assistance, development moves fast. Really fast. You can go from “working prototype” to “completely broken refactor” in the span of a coffee break. Which is fantastic for velocity, until a friend asks “hey, can you show me what you’ve been working on?”

And you realize your current build is half-demolished, your environment setup is specific to yesterday’s debugging session, and your “production-ready” system won’t even start.

This is the story of when I decided to always keep something showable — and the surprisingly systematic approach that emerged from a simple social need.

The problem with hyperfast development

AI-assisted development creates a particular kind of workflow disruption. You can refactor entire architectures in an afternoon. Add major features between lunch and dinner. Completely redesign your data layer because an AI suggested a better pattern and you thought “why not try it?”

This velocity is intoxicating. But it comes with a social cost: you can never demo your work when people actually ask to see it.

The moment of “can you show me what you’ve built?” invariably arrives when:

  • Your current branch is mid-refactor
  • You just broke something fundamental
  • Your environment is configured for debugging, not demonstrating
  • You remember that thing was working yesterday, but…

The faster you can build, the more likely you are to be caught in an un-showable state.

The “demo branch” insight

The solution turned out to be embarrassingly simple: always maintain a working version you can fall back to instantly.

Not a complex branching strategy. Not elaborate CI/CD. Just: before you start breaking things, tag a commit where everything works and document exactly how to run it.

# Before starting any major changes
git tag demo-stable-$(date +%Y%m%d)
echo "docker-compose up" > START_DEMO.txt
git add . && git commit -m "Demo checkpoint: working prototype"

Now when someone asks to see your work, you can say “give me 30 seconds” instead of “well, it was working yesterday…”

When dependencies attack

The real test of this approach came during a ChromaDB compatibility crisis. I’d been iterating rapidly on the knowledge base functionality, and somewhere along the way, an unpinned dependency had auto-updated and broken the entire storage layer.

Error: no such column: collections.topic

Classic schema incompatibility. The old database was created by ChromaDB 0.4.22, but the current version expected different tables. My current development environment was completely unusable.

But because I had a demo branch with locked dependencies, I could:

  1. Show the working prototype to a colleague who’d scheduled a zoom call to see what I was working on
  2. Systematically debug the issue in a separate branch
  3. Merge the fix back without losing demo capability

The debugging itself was straightforward once I had a stable reference point — remove the incompatible database files, let ChromaDB recreate with the new schema, lock the dependencies properly. But the key insight was having a known working state to compare against.

The systematic approach that emerged

What started as “keep something showable” evolved into a more systematic demo infrastructure strategy:

Two-demo strategy

  • Prototype demo: Technical evolution story for developer/architect audiences
  • Product demo: User value story for PM/product audiences

The product was going to take a while to start approximately the (half-baked) functionality of the prototype, so while I was happy to show what the MVP could already do, demoing the prototype was still a much better way of communicating the vision and showing a few parlor tricks. (“Piper! make me a bug ticket!”).

Also, different audiences care about different aspects. Developers want to see the architectural choices and technical evolution. Product people want to see user workflows and business value.

Environment isolation

  • Git branch strategy with tagged stable points
  • Docker-based self-contained environments
  • One-command startup scripts
  • Locked dependencies for demo stability

Demo maintenance discipline

  • Test demo startup from clean environment regularly
  • Document exact working configuration
  • Update demo branches when adding significant value
  • Treat demo-ready state as a deliverable, not overhead

The meta-learning about building for yourself

The most interesting discovery wasn’t really technical. It was philosophical. The pressure to “always keep something showable” forced better development practices overall.

When you know you need to maintain a working demo, you:

  • Break changes into smaller, more logical commits
  • Document your environment setup better
  • Test your “getting started” experience regularly
  • Think more carefully about when to introduce breaking changes

Demo-ready discipline creates better development hygiene.

The social dimension of building

There’s something profound about building tools for yourself first. Not because of some grand philosophy, but because of the immediate feedback loop. When your own daily workflows depend on the tool working, you catch usability issues immediately.

But that “build for yourself” approach only works if you can actually show other people what you’re building. The social validation — “this is cool, how does it work?” — provides essential perspective on whether your personal tool has broader potential.

Demo infrastructure isn’t just about presentations. It’s about maintaining the bridge between personal tool and potential product.

The practical implementation

The actual technical setup turned out to be simpler than the conceptual breakthrough:

Before major changes:

# Create stable checkpoint
git tag demo-stable-$(date +%Y%m%d)
pip freeze > requirements-demo-exact.txt  
echo "Current demo startup: docker-compose up" > DEMO_NOTES.txt

When someone wants to see our work:

# Switch to stable demo
git checkout demo-stable-20250614
pip install -r requirements-demo-exact.txt
cat DEMO_NOTES.txt  # Remember how to start it

After completing changes:

# Update demo if worthwhile
git tag demo-stable-$(date +%Y%m%d)  
# Or keep previous stable version

No complex branching. No elaborate automation. Just: always have a way to quickly show working functionality.

The compound benefits

Maintaining demo-ready builds created unexpected advantages beyond social showability:

Development confidence: Always having a working version to fall back to made experimentation less risky.

Learning documentation: Each demo represents a captured milestone of what you’d actually accomplished.

Stakeholder communication: Different audiences could see appropriate technical depth without overwhelming detail.

Integration reality-checking: If you can’t demo it end-to-end, it probably doesn’t actually work yet.

The broader pattern

This approach scales beyond individual projects. Any team doing rapid iteration — whether with AI assistance or not — faces the same showability problem.

The faster you can build, the more important it becomes to maintain stable reference points. Not for elaborate deployment pipelines, but for the simple human need to share your work when opportunities arise.

The best development velocity is meaningless if you can never show what you’ve built.
