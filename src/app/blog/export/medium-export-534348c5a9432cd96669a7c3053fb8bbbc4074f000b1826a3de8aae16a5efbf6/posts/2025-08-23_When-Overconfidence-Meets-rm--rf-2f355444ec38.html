When Overconfidence Meets rm -rf

It’s the weekend and time for another flashback post more about process or insight than the daily building narrative. At this point, we are exactly one month into the project:

June 27

“I’ve been confidently declaring ‘this will fix it!’ over and over, and we’re back to the same 404.”

That was my moment of humbling self-awareness, typed in despair to my robotic chat partner, about two hours into what should have been a five-minute GitHub Pages deployment. What started as “just deploy the docs” had somehow escalated into accidentally deleting my entire Git repository while chasing a problem that had a one-line solution.

This is the story of how overconfidence, impatience, and a poorly-timed rm -rf .* command taught me everything I needed to know about systematic debugging versus "confidence-driven development."

Famous last words

The task was simple enough: deploy our cleaned-up documentation to GitHub Pages so the team could easily browse our architecture decisions. We’d organized everything in the /docs folder, committed it to main, and configured GitHub Pages to serve from there.

How hard could that be?

I must say that so far the most harrowing experiences I have had doing this have come when I have let my guard down, started to relax, and just “gone for it.”

The docs existed, the settings looked right, and GitHub was showing successful builds. But clicking on the URL gave a 404 error. No big deal — probably just a caching issue or a simple configuration tweak.

Famous last words, as I seem to constantly say.

The confidence cascade

What followed was a masterclass in how overconfidence can turn simple problems into complex disasters. Each “solution” I tried was accompanied by declarations like “this will definitely fix it” or “obviously this is the issue.” The only thing more frustrating than a chatbot confidently guessing wrong over and over is the way it declares “I found the issue!” each time (wrong) or declares its solution “Perfect!” before checking or waiting to hear my opinion.

Attempt 1: “It’s probably a Jekyll issue. Let me add a .nojekyll file.”
Result: Still 404.

Attempt 2: “Must be a branch configuration problem. Let me check the gh-pages branch.”
Result: Still 404, but now with branch confusion.

Attempt 3: “I’ll just clean up the gh-pages branch and push fresh content.”
Result: Git conflicts and more confusion.

By the way, every time I say “I decided” or “I tried” here, I should mention this was happening in collaboration with Claude as my debugging partner. When I say “we” people start to look at me funny, but even though I am responsible for all the substantive content and ideas built into Piper Morgan, I know a lot less about debugging than Claude does (but a lot more than I did a month ago).

The nuclear option: when rm -rf goes wrong

After about an hour of fighting with branch configurations and file synchronization, I took the advice of Claude to try the “nuclear option” and clean up all the hidden files that might be causing issues.

rm -rf .*

I know this one. This is a classic unix joke. You get a newb to try out the cool new command and it deletes *.* or recursively deletes a root folder and everything in it. It’s like when you watch yourself locking your keys in the car (when cars used to let you do that). It happens in slow motion and you start to mouth nooooooooo but it’s too late.

“Wait… did I just delete my entire .git directory?”

Yep. Sure did. Along with .gitignore, .github/, and every other dotfile in the repository. Two hours of debugging, and I'd just nuked my local repo while trying to clean up "unnecessary" files.

The most dangerous commands are the ones that feel like they’re solving everything.

The recovery: git clone as time machine

The good news about accidentally deleting .git is that it's completely recoverable if you have a remote repository. The bad news is that it forces you to acknowledge how far down the wrong path you've gone.

# The walk of shame
git clone git@github.com:mediajunkie/piper-morgan-product.git
cd piper-morgan-product

Starting fresh gave me a moment to recalibrate. Instead of continuing to chase complex solutions, maybe I should figure out what was actually wrong.

The systematic debugging breakthrough

With a clean repository and a humbled attitude, I started over with basic first principles and three simple steps:

  1. Create a simple test file to verify GitHub Pages was actually working
  2. Check if the problem was with Jekyll processing or file structure
  3. Work from simple to complex, not the other way around

Created a basic index.html:

<h1>Test Page</h1>
<p>If you can see this, GitHub Pages is working.</p>

Boom. Worked perfectly. So GitHub Pages was fine — the problem was with how my documentation was being processed.

The Jekyll revelation

Here’s where systematic debugging paid off. The test HTML file worked, but the README.md wasn’t being converted to the expected landing page. That suggested Jekyll processing was the issue.

The problem: I’d disabled Jekyll with .nojekyll because I thought it was causing problems

The reality: Jekyll was exactly what I needed to convert README.md into a proper index page

Removed the .nojekyll file, and suddenly everything worked. The markdown documentation was properly converted to HTML, properly styled, and properly accessible.

One file deletion fixed what two hours of branch manipulation couldn’t.

This is kind of like when you miss you exit on the highway so you decide “I’ll just drive down this road and see if it ever reconnects to a highway” because it’s late and you don’t feel like backtracking.

The overconfidence anti-pattern

Looking back, the session was a perfect example of overconfidence.

Symptom identification: Jump to conclusions about root causes without testing assumptions

Solution confidence: Declare each attempt “will definitely fix it” instead of acknowledging uncertainty

Escalating complexity: When simple solutions don’t work, assume complex problems require complex solutions

Destructive debugging: Use increasingly aggressive tactics when frustrated

I swear, this happened a lot in the first few weeks. I’m not doing this anymore, or at least not on this scale (foibles are fractal, I have found).

The most wasteful debugging happens when you’re too confident to admit you don’t understand the problem.

Recognizing when you’re fighting the platform

Another insight from this debugging marathon was recognizing when you’re fighting the platform instead of working with it.

Platform fighting: Disable Jekyll, manage multiple branches, complex build configurations

Platform cooperation: Use default Jekyll processing, single branch deployment, standard markdown structure

GitHub Pages has defaults for good reasons. Jekyll processes markdown for good reasons. Sometimes the platform knows better than you do.

This is another risk of bot-assisted development: I get lazy and don’t take the time to educate myself about the methods we are adopting. Then I am at the mercy of a pray-and-spray debugging approach if we don’t have a strict methodology in place and I don’t have the chops to recognize a bad idea when I hear it.

The “this will fix it” warning sign

The most valuable lesson was recognizing “this will definitely fix it” as a warning sign rather than a sign of progress.

Confident declarations: Usually indicate guesswork disguised as analysis

Uncertain exploration: Usually indicate systematic understanding of the problem space

When you hear yourself (or your LLM) making confident predictions about complex problems, that’s often when you should slow down and test your assumptions more carefully.

The destructive command lesson

The rm -rf .* disaster taught me about the danger of destructive commands when frustrated:

  • High-stakes commands require extra verification, especially with wildcards
  • The worst time to use irreversible operations is when you are frustrated and trying to expedite thingds
  • Clear intentions can save you ( “Delete specific file” not “delete everything that might be related”)
# Dangerous: Wildcard deletion when frustrated
rm -rf .*

# Safer: Specific file targeting
rm .nojekyll

The most dangerous commands are the ones that feel like shortcuts when you’re impatient.

The fresh start wisdom

Sometimes the fastest way forward is to start over completely. The git clone wasn’t just recovery from my mistake — it was an opportunity to approach the problem with fresh perspective.

A continuing broken state accumulates assumptions and workarounds, whereas a fresh start forces you to identify what actually matters.

When you’ve been debugging for hours, sometimes the best debugging tool is git clone.

The systematic vs. shotgun debugging

This session perfectly illustrated the difference between systematic and shotgun debugging approaches:

Shotgun debugging: Try multiple solutions quickly, hoping one works

Systematic debugging: Test one variable at a time, understand each result

Systematic debugging feels slower initially but prevents you from solving the wrong problem with increasingly complex solutions.

I’m writing this nearly two months further along on this project and I can report that wild frantic debugging sessions are a thing of the past. (Bugs are not.) Most debugging I do now is around edge cases or when testing end-to-end experiences and finding the rough spots. Verifications, testing and tracking have tamed the wild bughunts. But we’re talking about the day I nuked my local repository by mistake.

The documentation irony

There’s some irony in spending 2.5 hours debugging documentation deployment for a project that’s turning out to be all about systematic development practices. But maybe that’s exactly the kind of humbling experience that makes you better at recognizing when your process has gone off the rails.

The best process insights seem to come directly from watching your own process fail!

The meta-learning about AI collaboration

One interesting aspect of this debugging session was how it affected AI collaboration. When we felt confident (and were wrong), Claude led me down unproductive paths. When I pointed out the uncertainty and asked for systematic approaches, Claude provided much better guidance.

Overconfidence in human or LLM: AI follows bad assumptions

Uncertain but systematic human: AI provides better analytical support

Your collaboration quality with AI systems often reflects your own debugging discipline.

The platform defaults vindication

The final lesson was about trusting platform defaults until you have good reasons not to:

GitHub Pages defaults: Main branch, /docs folder, Jekyll enabled

Accumulated customizations from disjointed chat sessions: gh-pages branch, .nojekyll file, manual file management

The defaults worked perfectly. My poorly documented customizations created all the problems.

Sometimes the most sophisticated solution is accepting that someone else has already solved your problem correctly.

When this happened

This debugging adventure happened right in the middle of our late June architectural recovery sequence. While we were battle-testing complex domain logic and systematic development approaches, I was demonstrating how even simple deployment tasks can spiral out of control when you abandon systematic thinking.

Good processes work at every scale — from architectural decisions to documentation deployment.
