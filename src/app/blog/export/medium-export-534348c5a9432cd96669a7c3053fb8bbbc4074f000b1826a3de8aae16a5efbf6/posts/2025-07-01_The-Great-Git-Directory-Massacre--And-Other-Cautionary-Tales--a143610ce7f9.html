The Great Git Directory Massacre (And Other Cautionary Tales)

June 27, 2025

So there I was at 10 PM, my tireless chat companion confidently declaring “this will fix it!” for the seventh time while debugging a GitHub Pages 404 error. Two hours later, I’d learned three valuable lessons:

  1. rm -rf .* deletes your entire .git directory (whoops)
  2. Sometimes platform defaults exist for good reasons
  3. The most dangerous developer is one who’s sure they know what’s wrong

But first let me back up a bit.

The Setup: A Simple Documentation Deployment

After weeks of building Piper Morgan’s file analysis features, I decided to finally get the documentation site working. GitHub Pages. How hard could it be? I’d done this dozens of times.

Without spoiling anything, let me just note that the more sophisticated I have become over the years about technology, the more catastrophically I have been able to competely b0rk data and systems.

The docs were clean, organized in /docs, with a proper README.md. GitHub Pages was configured for the main branch. Everything should have just worked.

404 error.

“No problem,” I thought. “Probably just needs a push to trigger the build.”

The Descent: Fighting the Platform

What followed was a masterclass in what not to do when debugging.

First, I decided GitHub wasn’t finding my files properly. Added an index.html. Still 404.

Then I decided Jekyll was the problem. Added .nojekyll to disable it. Still 404.

By the way, every time I say “I decided” it really means either Claude or my Cursor Agent proposed a diagnosis and I said “Sure, OK, I guess.” Occasionally typing something despairing like “but this worked before!”

Clearly the issue was the gh-pages branch confusing things. Started trying to sync branches, move files around, reset various states…

At some point, frustrated with all the hidden files cluttering my directory listing, I ran what seemed like a reasonable cleanup command:

rm -rf .*

The terminal went quiet. Too quiet.

Then I tried to check git status.

fatal: not a git repository

My writing bot suggests this is a good place for me to ADD PERSONAL ANECDOTE FROM ANY TIME I’VE ACCIDENTALLY DESTROYED IMPORTANT DATA so well… I lost the only video I had of the my dad telling the story of how he met my mom a few weeks before her died. No joke. OK, that got morbid. Let’s distract ourselves with geekery!

The Recovery: Sometimes Fresh Starts Are Best

After the obligatory moment of staring at the terminal in disbelief, I did what any reasonable developer would do:

  1. Laughed at myself
  2. Re-cloned the repository
  3. Started over with a simpler approach

This time, instead of fighting the platform, I decided to work with it. Created a simple test index.html. It displayed! Progress.

That’s when it hit me: Jekyll wasn’t the problem. Jekyll was the solution. GitHub Pages uses Jekyll to convert README.md files into proper HTML index pages. My .nojekyll file was preventing the exact processing I needed. Oh, and github keeps its grubby hands off of . files. (They often contain private info you don’t want out there on your public repository.)

And code-brilliant chatbots aren’t all knowing. They are just very well read but not always current and not always correct in their assumptions. At least once if not twice Claude (Opus 4!) told me to remove the remote .nojekyll file with a command that would never work. I deleted it manually through the web UI and the docs appeared perfectly.

Total time to fix after starting fresh: 10 minutes. Total time wasted fighting the platform: 2 hours.

The Patterns: What This Teaches Us

1. Destructive Commands Require Extra Paranoia

rm -rf .* seems reasonable until you remember that . includes .git, .env, and every other hidden file you actually need.

Better approaches:

  • Use specific filenames: rm -f .nojekyll .DS_Store
  • Use git clean: git clean -fdx (respects .gitignore)
  • When in doubt, ls -la .* first to see what you're about to delete

Measure eight times, cut maybe.

2. Platform Defaults Usually Have Good Reasons

Jekyll processing on GitHub Pages isn’t a quirk — it’s the core feature. It’s what makes markdown documentation sites “just work.” Fighting it because you assume it’s the problem is one of those Chesterton’s Fence things. Don’t remove a guardrail without understanding why “they” put it there in teh first place.

When a platform behavior seems wrong, ask:

  • What problem is this solving that I don’t see?
  • Am I trying to do something unusual?
  • Have I actually verified this is the issue?

3. Confidence Theatre vs. Actual Debugging

I kept declaring “this will fix it!” with complete confidence. The AI assistant (bless its helpful heart) never questioned my certainty. But confidence without understanding is just noise.

Real debugging looks like:

  • “Let me verify what’s actually happening”
  • “I think X might be the issue, let’s test that hypothesis”
  • “I don’t understand why this isn’t working, let me simplify”

Not:

  • “This is definitely the problem” (narrator: it wasn’t)
  • “One more fix and we’re done” (seven fixes later…)
  • “The platform is broken” (the platform was fine)

The Meta-Lesson: Admitting Uncertainty

The session log my bot wrote afterward included this gem:

“I’ve been confidently declaring ‘this will fix it!’ over and over, and we’re back to the same 404.”
That moment of recognition — that my guide had been performatively confident rather than systematically debugging — was worth more than fixing the actual issue.

When you’re stuck, the most powerful phrase isn’t “I know what’s wrong.” It’s “I don’t understand what’s happening, so let me start with something simple that works.”

The Happy Ending: Embracing Simplicity

The final, working configuration:

  • Source: main branch, /docs folder
  • Jekyll: Enabled (no .nojekyll file)
  • Index: README.md (processed by Jekyll)
  • URL: Working perfectly

No complex branch management. No fighting platform defaults. No clever workarounds. Just using GitHub Pages as designed.

Takeaways for Your Next Debugging Adventure

  1. Before running destructive commands, actually look at what they’ll affect
  2. When platform defaults seem wrong, question your assumptions first
  3. When you hear yourself saying “this will definitely fix it” for the third time, stop and reassess
  4. Fresh starts aren’t giving up — sometimes they’re the fastest path forward
  5. Document your failures — they’re often more instructive than successes

I would love to say tht this kind of mistake gets less common with experience, but I honestly don’t think that’s true. I do think you can get better at spotting certain familiar (anti-)patterns before they get too far ou of hand, and I am a great believer in iterating on process. They way to excellence lies through all these problems, or as we used to say at one of my startups: “The probem is the path.”

The documentation site works now. It took destroying a git repository to get there, but hey, sometimes you need to burn things down to see clearly.

Just maybe use git clean next time instead of rm -rf .*.
