Refining AI Chat Continuity for Complex Projects

The next thing I asked my Chief of Staff to help with was improving how I managed continuity between chats when one filled up.

July 3

When building complex software with AI assistance, one of the most underappreciated challenges is maintaining context across chat sessions. Today I refactored my entire documentation system for Piper Morgan to solve this problem — not because I love documentation, but because I was tired of explaining the same debugging context every morning.

The 800-word monster

AI chats have context limits, and I was hitting them hard. When you’re deep in debugging a workflow persistence issue caused by legacy code contamination, the last thing you want is to spend 20 minutes explaining the situation to a fresh chat session.

My original approach involved increasingly lengthy “continuation prompts” that tried to pack all context into a single message. The most recent one had grown to over 800 words — and was still missing critical details about the persistence layer architecture.

The layered documentation solution

Instead of cramming everything into prompts, I created a hierarchical documentation system that lets AI contexts bootstrap quickly:

Minimal project instructions

The core identity and approach in under 100 words, pointing to other documents for details. Think “executive summary” rather than “complete briefing.”

Chat protocols (how we work)

  • Session management rules that actually get followed
  • File handling conventions that prevent the “which version?” problem
  • Multi-device workflow support for when you’re coding on different machines

And what I called at the time the BREAK protocol for continuity across human interruptions. Any time I got tired or burnt out I’d write “BREAK” and it would trigger an update to the session log and handoff prompt and a recap of what we had just finished and what we were planning to do next. (This protocol we abandoned after it evolved into the much more thorough GitHub tracking system we use now.)

Architectural checklist (what to verify)

  • Grep commands for pattern discovery (because you forget the syntax)
  • Layer separation verification steps
  • Common antipatterns catalog (learned the hard way)
  • Debugging approaches that work systematically

Standardized templates

  • Handoff document: Captures session state, decisions, and next actions
  • Continuity prompt: Lean 300-word template that references other docs instead of containing them

The key insight about reference chains

The continuity prompt doesn’t need to contain everything — it just needs to point to where everything lives. The new template is 60% shorter while providing better context through its reference chain:

Continuity Prompt → Session Log & Handoff Doc → Protocols & Checklist → Core Documentation

This architectural approach means AI contexts can drill down to the level of detail they need without drowning in information they don’t.

What this solved in practice

This refactoring addresses several pain points I was hitting repeatedly:

  1. Context loss: Structured handoffs preserve technical decisions between sessions
  2. Architectural drift: Checklists catch violations before they compound into major rewrites
  3. Session startup: From 20+ minutes of explanation to under 5 minutes of reference checking
  4. Pattern recognition: Concatenated session archive enables learning from past debugging approaches

Real example from the PM-011 debugging

During the workflow persistence debugging, we discovered that proof-of-concept code from June 3 was mixed with MVP code from June 7, causing subtle persistence failures. This kind of discovery now gets captured systematically in:

  • The session log (what happened and why)
  • The handoff document (how to continue the work)
  • The architectural checklist (how to prevent similar mixing in the future)

Before this system, insights like “POC code mixed with MVP code causes persistence failures” would get lost between sessions. Now they become institutional knowledge.

The self-improving feedback loop

Perhaps most importantly, this system creates a feedback loop that gets better over time. Every debugging session contributes to pattern recognition, which improves the checklists, which prevents future issues. The documentation becomes smarter as you use it.

The templates and protocols are now part of Piper Morgan’s core documentation, but the approach works for any complex project using AI assistance. The key insight is recognizing that context management isn’t overhead — it’s a core engineering practice, like version control or testing.

What I learned about AI collaboration

Working with AI on complex projects requires different documentation habits than solo development or traditional team work. You need systems that can bootstrap AI context quickly while preserving the nuanced understanding that builds up over weeks of debugging.

The old approach — cramming everything into longer and longer prompts — doesn’t scale. The new approach — creating reference architectures that AI can navigate — does.

It’s not about making AI remember things (it can’t). It’s about making knowledge discoverable and contextual information structured enough that fresh AI sessions can get productive quickly.
