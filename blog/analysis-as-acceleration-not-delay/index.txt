1b:T2a24,<figure><img alt="A robot prepares to enter a maze of tangled pipes while a human beside it holds a map showing a clear, direct path through." src="https://cdn-images-1.medium.com/max/1024/1*97nOC0Joz6BTvlpehAvdLg.png" /><figcaption>“Stick to the plan”</figcaption></figure><p><em>December 8, 2025</em></p><p>The setup wizard needed refactoring. The main function ran 267 lines. Four separate API key sections each duplicated roughly 100 lines of nearly identical code. Not elegant, but functional, and we’d been focused on more pressing problems.</p><p>The real problem was that it had an old-school command line interface (CLI) and the terminal would not allow pasting of API keys, which required either typing them perfectly (nearly impossible, at least for me) or using a complicated work around to manually add them to a file. Not only did I want to save people the ridiculous typing but I also wanted to store people’s credentials in their keychain and not in the clear in any statics files in the project.</p><p>Monday morning, with CRUD operations finally working and six UI polish issues resolved, we had time. Not deadline pressure, not crisis urgency, just clear space to do something right instead of just getting it done.</p><p>The temptation was obvious: open the file, start refactoring, figure it out as we go. We knew the problem. We knew roughly what needed to happen. Why not just dive in?</p><p>Instead, we stopped and analyzed for forty-five minutes.</p><h3>The planning session</h3><p>Forty-five minutes doesn’t sound like much. But when you’re eager to start coding, when the problem seems clear, when you just want to <em>make progress</em>, it feels like forever.</p><p>We opened the file and documented what we saw. Not fixing anything yet, just understanding:</p><ul><li>Main wizard function: 267 lines, doing everything</li><li>Four API key collection sections: OpenAI, Anthropic, Gemini, GitHub</li><li>Each section: ~100 lines of nearly identical logic</li><li>Pattern repeated four times: keychain check → environment variable check → manual entry → validation</li><li>Total duplication: roughly 400 lines saying almost the same thing four different ways</li></ul><p>Then we asked: what would “good” look like?</p><p>A single helper function that handled API key collection generically. Pass in the provider name, pass in whether it’s required, pass in any special validation rules. One function replaces four duplicate implementations.</p><p>We (verbally) sketched the interface. Thought through some edge cases. Identified what could be unified and what needed to stay provider-specific. Designed the solution before writing any code.</p><p>Forty-five minutes. Then we started implementing.</p><h3>The implementation</h3><p>Two hours later, the refactoring was complete.</p><p>The results:</p><ul><li>API key collection: 82% reduction (400 duplicate lines → 148-line helper)</li><li>Main wizard function: 71% reduction (267 lines → 76-line orchestrator)</li><li>All functions under 50 lines except justified complex helpers</li><li>Zero duplicate code blocks over 10 lines</li></ul><p>But more important than the metrics: no surprises, no backtracking, no “wait, this doesn’t work” moments halfway through. The implementation went smoothly because we’d answered all the design questions upfront.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/713/1*xe_3ILJLJH7JCBII-ogU-g.png" /><figcaption>A big improvement over the command-line wizard, trust me!</figcaption></figure><p>The analysis meant we knew exactly what we were building. The helper function’s signature was clear. The edge cases were identified. The migration path was obvious. We didn’t discover problems during implementation because we’d already thought them through.</p><h3>Why this is hard</h3><p>Analysis feels like delay. It <em>looks</em> like delay. You’re not writing code. You’re not shipping features. You’re just… thinking. Documenting. Sketching interfaces.</p><p>And there’s always a voice saying: “We could be making progress right now. We could be halfway done already. This is just planning paralysis disguised as rigor.”</p><p>That voice gets louder when:</p><ul><li>The problem seems straightforward</li><li>You’re confident you understand the solution</li><li>You’re eager to see results</li><li>Someone (even just future-you) is waiting</li></ul><p>But here’s what that voice misses: the time you spend analyzing isn’t extra time on top of implementation. It’s time you either spend thinking carefully upfront, or spend thrashing, backtracking, and debugging later.</p><p>The forty-five minutes we spent analyzing Issue #439 wasn’t forty-five minutes <em>added</em> to a two-hour implementation. It was forty-five minutes that prevented those two hours from becoming six hours of implementation + debugging + refactoring the refactoring.</p><h3>The alternative timeline</h3><p>We can imagine the other path. Start refactoring immediately. Pull out the first duplicated section into a function. Seems to work. Pull out the second section. Wait, this one’s slightly different. Modify the function. Third section — another edge case. Modify again. Fourth section — now the function’s getting messy.</p><p>Realize the original design doesn’t handle all the cases cleanly. Refactor the helper function. Update all the call sites. Discover that GitHub API keys have different validation rules. Add a parameter. Discover that optional vs. required keys need different handling. Add another parameter.</p><p>Three hours in, the helper function has seven parameters, three of which are optional, and nobody (including you) is entirely sure which combination is valid. The code works, but it’s fragile. The next person (probably you in three months) will look at it and wonder what you were thinking.</p><p>That’s the thrashing tax. The cost of figuring things out during implementation instead of before it.</p><h3>What analysis buys you</h3><p>The forty-five minutes of upfront analysis bought us:</p><p><strong>Clarity</strong>: We knew exactly what we were building and why <strong>Confidence</strong>: No second-guessing during implementation <strong>Speed</strong>: Two hours of smooth work instead of six hours of thrashing<strong>Quality</strong>: Clean interfaces, no parameter soup, obvious intent <strong>Documentation</strong>: The analysis itself became the design doc</p><p>But the biggest thing it bought us was <em>freedom</em>. Freedom to implement without constantly stopping to make design decisions. Freedom to focus on getting the code right instead of figuring out what “right” even means.</p><p>When you know where you’re going, the journey is faster. When you’re figuring out the destination while you’re traveling, every step requires navigation decisions.</p><h3>The timing question</h3><p>“But when do you know you’ve analyzed enough?”</p><p>Good question. Too much analysis is real — you can absolutely overthink things. But here’s a useful heuristic:</p><p>You’ve analyzed enough when you can describe the solution without writing code. When you can sketch the interfaces. When you’ve identified the edge cases. When you know what success looks like.</p><p>For Issue #439, that was forty-five minutes. For the S2 encryption sprint preparation, it was five hours creating a comprehensive review package. For simple bug fixes, it might be five minutes of reading the code to understand the problem.</p><p>The analysis time scales with complexity and stakes. A four-line bug fix doesn’t need a design document. A cryptographic implementation that handles sensitive user data needs careful architectural review.</p><p>But the principle holds: understand the problem before you solve it. Design the solution before you implement it. Think through the edge cases before you encounter them in production.</p><h3>The productivity paradox</h3><p>This creates an interesting productivity paradox. The day we spent forty-five minutes analyzing and two hours implementing, we closed one GitHub issue. The day before, diving straight into implementation, we closed six issues.</p><p>If you’re measuring productivity by issues closed per day, analysis looks inefficient.</p><p>But if you’re measuring by technical debt avoided, complexity managed, and future maintenance burden reduced, that forty-five minutes was one of the most productive investments of the week.</p><p>We didn’t just fix a problem. We fixed it in a way that makes the codebase simpler, more maintainable, and easier to understand. The next time someone needs to add an API key provider, they’ll follow the pattern we established. The cognitive complexity we removed doesn’t come back.</p><h3>When to skip the analysis</h3><p>There are times to skip detailed analysis:</p><p><strong>Small, well-understood problems</strong>: If you’ve solved this exact thing before, you probably don’t need to re-analyze it.</p><p><strong>Throwaway code</strong>: Prototypes and experiments benefit from exploration. Analysis is for code you expect to keep.</p><p><strong>Time-critical bugs</strong>: If production is down, fix it first, analyze later. (But do analyze later — don’t let the emergency fix become permanent infrastructure.)</p><p><strong>Learning exercises</strong>: Sometimes the fastest way to understand a problem is to try solving it and see what breaks.</p><p>But for production code that you expect to maintain, for complex problems with multiple edge cases, for refactorings that touch many files — analysis accelerates rather than delays.</p><p>The setup wizard refactoring proved it. Forty-five minutes of thinking. Two hours of implementing. Zero hours of debugging, refactoring the refactor, or explaining to future-you why the code is the way it is.</p><p>That’s not delay. That’s acceleration.</p><p><em>Next on Building Piper Morgan, we resume the build narrative with “Pattern Sweep 2.0” from February 3 and 4.</em></p><p><em>How do you balance analysis and action in your own work? Not only did I want to save people the ridiculous typing but I also wanted to store people’s credentials in their keychain and not in the clear in any statics files in the project.</p><p name="247b" id="247b" class="graf graf--p graf-after--p">Monday morning, with CRUD operations finally working and six UI polish issues resolved, we had time. Not deadline pressure, not crisis urgency, just clear space to do something right instead of just getting it done.</p><p name="97d0" id="97d0" class="graf graf--p graf-after--p">The temptation was obvious: open the file, start refactoring, figure it out as we go. We knew the problem. We knew roughly what needed to happen. Why not just dive in?</p><p name="23b0" id="23b0" class="graf graf--p graf-after--p">Instead, we stopped and analyzed for forty-five minutes.</p><h3 name="e51b" id="e51b" class="graf graf--h3 graf-after--p">The planning session</h3><p name="c91f" id="c91f" class="graf graf--p graf-after--h3">Forty-five minutes doesn’t sound like much. But when you’re eager to start coding, when the problem seems clear, when you just want to <em class="markup--em markup--p-em">make progress</em>, it feels like forever.</p><p name="ab5b" id="ab5b" class="graf graf--p graf-after--p">We opened the file and documented what we saw. Not fixing anything yet, just understanding:</p><ul class="postList"><li name="db5c" id="db5c" class="graf graf--li graf-after--p">Main wizard function: 267 lines, doing everything</li><li name="6901" id="6901" class="graf graf--li graf-after--li">Four API key collection sections: OpenAI, Anthropic, Gemini, GitHub</li><li name="b24e" id="b24e" class="graf graf--li graf-after--li">Each section: ~100 lines of nearly identical logic</li><li name="6cec" id="6cec" class="graf graf--li graf-after--li">Pattern repeated four times: keychain check → environment variable check → manual entry → validation</li><li name="0adf" id="0adf" class="graf graf--li graf-after--li">Total duplication: roughly 400 lines saying almost the same thing four different ways</li></ul><p name="5e19" id="5e19" class="graf graf--p graf-after--li">Then we asked: what would “good” look like?</p><p name="317c" id="317c" class="graf graf--p graf-after--p">A single helper function that handled API key collection generically. Pass in the provider name, pass in whether it’s required, pass in any special validation rules. One function replaces four duplicate implementations.</p><p name="cf21" id="cf21" class="graf graf--p graf-after--p">We (verbally) sketched the interface. Thought through some edge cases. Identified what could be unified and what needed to stay provider-specific. Designed the solution before writing any code.</p><p name="787e" id="787e" class="graf graf--p graf-after--p">Forty-five minutes. Then we started implementing.</p><h3 name="32e0" id="32e0" class="graf graf--h3 graf-after--p">The implementation</h3><p name="4335" id="4335" class="graf graf--p graf-after--h3">Two hours later, the refactoring was complete.</p><p name="b25c" id="b25c" class="graf graf--p graf-after--p">The results:</p><ul class="postList"><li name="3d9d" id="3d9d" class="graf graf--li graf-after--p">API key collection: 82% reduction (400 duplicate lines → 148-line helper)</li><li name="ea17" id="ea17" class="graf graf--li graf-after--li">Main wizard function: 71% reduction (267 lines → 76-line orchestrator)</li><li name="5384" id="5384" class="graf graf--li graf-after--li">All functions under 50 lines except justified complex helpers</li><li name="dca9" id="dca9" class="graf graf--li graf-after--li">Zero duplicate code blocks over 10 lines</li></ul><p name="8cb9" id="8cb9" class="graf graf--p graf-after--li">But more important than the metrics: no surprises, no backtracking, no “wait, this doesn’t work” moments halfway through. The implementation went smoothly because we’d answered all the design questions upfront.</p><figure name="68e5" id="68e5" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*xe_3ILJLJH7JCBII-ogU-g.png" data-width="713" data-height="710" src="https://cdn-images-1.medium.com/max/800/1*xe_3ILJLJH7JCBII-ogU-g.png"><figcaption class="imageCaption">A big improvement over the command-line wizard, trust me!</figcaption></figure><p name="68b3" id="68b3" class="graf graf--p graf-after--figure">The analysis meant we knew exactly what we were building. The helper function’s signature was clear. The edge cases were identified. The migration path was obvious. We didn’t discover problems during implementation because we’d already thought them through.</p><h3 name="58eb" id="58eb" class="graf graf--h3 graf-after--p">Why this is hard</h3><p name="66bf" id="66bf" class="graf graf--p graf-after--h3">Analysis feels like delay. It <em class="markup--em markup--p-em">looks</em> like delay. You’re not writing code. You’re not shipping features. You’re just… thinking. Documenting. Sketching interfaces.</p><p name="9d4d" id="9d4d" class="graf graf--p graf-after--p">And there’s always a voice saying: “We could be making progress right now. We could be halfway done already. This is just planning paralysis disguised as rigor.”</p><p name="a4cf" id="a4cf" class="graf graf--p graf-after--p">That voice gets louder when:</p><ul class="postList"><li name="9319" id="9319" class="graf graf--li graf-after--p">The problem seems straightforward</li><li name="366c" id="366c" class="graf graf--li graf-after--li">You’re confident you understand the solution</li><li name="a79a" id="a79a" class="graf graf--li graf-after--li">You’re eager to see results</li><li name="b0a7" id="b0a7" class="graf graf--li graf-after--li">Someone (even just future-you) is waiting</li></ul><p name="6693" id="6693" class="graf graf--p graf-after--li">But here’s what that voice misses: the time you spend analyzing isn’t extra time on top of implementation. It’s time you either spend thinking carefully upfront, or spend thrashing, backtracking, and debugging later.</p><p name="d013" id="d013" class="graf graf--p graf-after--p">The forty-five minutes we spent analyzing Issue #439 wasn’t forty-five minutes <em class="markup--em markup--p-em">added</em> to a two-hour implementation. It was forty-five minutes that prevented those two hours from becoming six hours of implementation + debugging + refactoring the refactoring.</p><h3 name="b36b" id="b36b" class="graf graf--h3 graf-after--p">The alternative timeline</h3><p name="feb4" id="feb4" class="graf graf--p graf-after--h3">We can imagine the other path. Start refactoring immediately. Pull out the first duplicated section into a function. Seems to work. Pull out the second section. Wait, this one’s slightly different. Modify the function. Third section — another edge case. Modify again. Fourth section — now the function’s getting messy.</p><p name="0ab8" id="0ab8" class="graf graf--p graf-after--p">Realize the original design doesn’t handle all the cases cleanly. Refactor the helper function. Update all the call sites. Discover that GitHub API keys have different validation rules. Add a parameter. Discover that optional vs. required keys need different handling. Add another parameter.</p><p name="83f7" id="83f7" class="graf graf--p graf-after--p">Three hours in, the helper function has seven parameters, three of which are optional, and nobody (including you) is entirely sure which combination is valid. The code works, but it’s fragile. The next person (probably you in three months) will look at it and wonder what you were thinking.</p><p name="2643" id="2643" class="graf graf--p graf-after--p">That’s the thrashing tax. The cost of figuring things out during implementation instead of before it.</p><h3 name="0e15" id="0e15" class="graf graf--h3 graf-after--p">What analysis buys you</h3><p name="3af2" id="3af2" class="graf graf--p graf-after--h3">The forty-five minutes of upfront analysis bought us:</p><p name="ddbf" id="ddbf" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Clarity</strong>: We knew exactly what we were building and why <strong class="markup--strong markup--p-strong">Confidence</strong>: No second-guessing during implementation <strong class="markup--strong markup--p-strong">Speed</strong>: Two hours of smooth work instead of six hours of thrashing<strong class="markup--strong markup--p-strong">Quality</strong>: Clean interfaces, no parameter soup, obvious intent <strong class="markup--strong markup--p-strong">Documentation</strong>: The analysis itself became the design doc</p><p name="4b79" id="4b79" class="graf graf--p graf-after--p">But the biggest thing it bought us was <em class="markup--em markup--p-em">freedom</em>. Freedom to implement without constantly stopping to make design decisions. Freedom to focus on getting the code right instead of figuring out what “right” even means.</p><p name="a9d5" id="a9d5" class="graf graf--p graf-after--p">When you know where you’re going, the journey is faster. When you’re figuring out the destination while you’re traveling, every step requires navigation decisions.</p><h3 name="edfe" id="edfe" class="graf graf--h3 graf-after--p">The timing question</h3><p name="f0b2" id="f0b2" class="graf graf--p graf--startsWithDoubleQuote graf-after--h3">“But when do you know you’ve analyzed enough?”</p><p name="3202" id="3202" class="graf graf--p graf-after--p">Good question. Too much analysis is real — you can absolutely overthink things. But here’s a useful heuristic:</p><p name="4b23" id="4b23" class="graf graf--p graf-after--p">You’ve analyzed enough when you can describe the solution without writing code. When you can sketch the interfaces. When you’ve identified the edge cases. When you know what success looks like.</p><p name="8e9c" id="8e9c" class="graf graf--p graf-after--p">For Issue #439, that was forty-five minutes. For the S2 encryption sprint preparation, it was five hours creating a comprehensive review package. For simple bug fixes, it might be five minutes of reading the code to understand the problem.</p><p name="66dc" id="66dc" class="graf graf--p graf-after--p">The analysis time scales with complexity and stakes. A four-line bug fix doesn’t need a design document. A cryptographic implementation that handles sensitive user data needs careful architectural review.</p><p name="d45d" id="d45d" class="graf graf--p graf-after--p">But the principle holds: understand the problem before you solve it. Design the solution before you implement it. Think through the edge cases before you encounter them in production.</p><h3 name="7854" id="7854" class="graf graf--h3 graf-after--p">The productivity paradox</h3><p name="6512" id="6512" class="graf graf--p graf-after--h3">This creates an interesting productivity paradox. The day we spent forty-five minutes analyzing and two hours implementing, we closed one GitHub issue. The day before, diving straight into implementation, we closed six issues.</p><p name="6d8d" id="6d8d" class="graf graf--p graf-after--p">If you’re measuring productivity by issues closed per day, analysis looks inefficient.</p><p name="8e85" id="8e85" class="graf graf--p graf-after--p">But if you’re measuring by technical debt avoided, complexity managed, and future maintenance burden reduced, that forty-five minutes was one of the most productive investments of the week.</p><p name="732b" id="732b" class="graf graf--p graf-after--p">We didn’t just fix a problem. We fixed it in a way that makes the codebase simpler, more maintainable, and easier to understand. The next time someone needs to add an API key provider, they’ll follow the pattern we established. The cognitive complexity we removed doesn’t come back.</p><h3 name="1a05" id="1a05" class="graf graf--h3 graf-after--p">When to skip the analysis</h3><p name="97f9" id="97f9" class="graf graf--p graf-after--h3">There are times to skip detailed analysis:</p><p name="651d" id="651d" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Small, well-understood problems</strong>: If you’ve solved this exact thing before, you probably don’t need to re-analyze it.</p><p name="86b1" id="86b1" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Throwaway code</strong>: Prototypes and experiments benefit from exploration. Analysis is for code you expect to keep.</p><p name="2dfb" id="2dfb" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Time-critical bugs</strong>: If production is down, fix it first, analyze later. (But do analyze later — don’t let the emergency fix become permanent infrastructure.)</p><p name="d283" id="d283" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Learning exercises</strong>: Sometimes the fastest way to understand a problem is to try solving it and see what breaks.</p><p name="eb8f" id="eb8f" class="graf graf--p graf-after--p">But for production code that you expect to maintain, for complex problems with multiple edge cases, for refactorings that touch many files — analysis accelerates rather than delays.</p><p name="eae2" id="eae2" class="graf graf--p graf-after--p">The setup wizard refactoring proved it. Forty-five minutes of thinking. Two hours of implementing. Zero hours of debugging, refactoring the refactor, or explaining to future-you why the code is the way it is.</p><p name="d28c" id="d28c" class="graf graf--p graf-after--p graf--trailing">That’s not delay.
