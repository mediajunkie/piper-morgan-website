I’d been fighting it since feeling a pang in my sinus on Friday and it was established by Saturday, at which point I had started doing shorter sessions and lighter work on Piper Morham.</p><p>But Monday was different. I couldn’t focus. Couldn’t coordinate. Could barely follow what was happening in the session logs. So I did something I’d not done since starting this project nine month ago: I let go.</p><h3>The cathedral without the architect</h3><p>Monday produced exactly one agent session — a brief Documentation run to synthesize the previous day’s omnibus. I published a blog post that had been drafted earlier. Corresponded with Ted Nadeau about some GitHub questions. That was it.</p><p>Tuesday, I was still sick. But something strange happened.</p><p>The Documentation agent started a routine task: auditing ADR links to make sure all the cross-references still worked. Ted Nadeau had mentioned a broken link in passing, and this was the follow-up.</p><p>The audit found the broken links. Seven of them, across three files. Easy fixes. But then I noticed something in the agent’s report.</p><p>The dev/ directory was nearly empty.</p><h3>2,155 files</h3><p>The numbers didn’t make sense at first. We expected around 2,463 files in dev/. The audit found 308.</p><p>Eighty-seven percent of the files were gone.</p><p>Session logs — the daily record of everything we’d built — had dropped from 650 to 52. Gameplans — the detailed plans that guide every implementation — were completely gone. Not a single one remained.</p><p>Foggy as I was, I knew this was not good.</p><p>The investigation began immediately. Was this a fresh clone that hadn’t pulled the files? No — git reflog showed continuous history. Had someone deleted them? The evidence pointed to something more mundane and more preventable: a destructive git command, probably git checkout . or git restore ., run sometime around February 7th or 8th.</p><p>The dev/ directory was gitignored. It held working files, not committed code. When the restore command ran, git happily overwrote everything with… nothing. The files had never been committed, so there was nothing to restore them from.</p><p>Except there was.</p><h3>Git archaeology</h3><p>Git remembers everything — even files that were later gitignored. Every file that had ever been committed, even once, still existed somewhere in the repository’s history. The recovery command was ugly but effective:</p><pre>git log --all --full-history --diff-filter=A --name-only --format=&#39;%H&#39; -- &#39;dev/*&#39; | \<br>  while read hash; do git checkout $hash -- dev/ 2&gt;/dev/null; done</pre><p>Translation: find every commit that ever added a file to dev/, and restore that file from that commit.</p><p>The result: 2,781 files recovered. More than we’d lost, because the archaeology surfaced files from months ago that had been deleted intentionally but were now restored as a side effect.</p><p>The afternoon became a cleanup operation. Sort the recovered files. Delete the ones we didn’t need. Compress the archives. Move things to their proper homes. By evening, dev/ had gone from 5.1 GB of chaos to 1.2 GB of organized structure.</p><p>The institutional memory was intact.</p><h3>The December bug</h3><p>Wednesday brought a different kind of discovery.</p><p>Dominique Derosena — a new alpha tester referred by my 18F colleague Magdaline Derosena — was setting up Piper on Windows. Fresh machine, following the alpha tester instructions. He ran the setup script.</p><p>It failed immediately.</p><p>Not after a few steps. Not with a subtle error. Immediately. The script printed its opening message and then… stopped.</p><p>The bug was in the batch file structure. Helper subroutines used exit /b to return to the caller. But in Windows batch files, exit /b inside a subroutine doesn&#39;t return — it exits the entire script. The correct command is goto :eof.</p><p>This bug had existed since December 2025. Two months. Through Ted Nadeau’s extensive Windows testing — fourteen issues filed and fixed. Through multiple releases. Through the entire alpha program.</p><p>I wonder why Ted hand’t run into it?</p><p>The answer was in his issue reports. Ted’s fourteen issues focused on requirements.txt conflicts, missing migrations, documentation gaps. All real problems. But nothing about the setup script failing on first run.</p><p>He must have hit the failure, shrugged, and switched to manual setup, or kust stopped at that point. Then filed issues about everything else he encountered. Maybe, the script failure was so immediate, so total, that it didn’t even register as a bug worth reporting — just an obstacle to work around.</p><p>Pairing with Dominique, as he set things up, coming in fresh, we caught it and logged the issue.</p><h3>Systematic prevention</h3><p>The Windows bug led to a broader question: why wasn’t this caught automatically?</p><p>The answer was uncomfortable. We had no Windows CI. Our continuous integration tested on Linux and macOS. Windows was… assumed. If a batch file was syntactically valid, we figured it would work.</p><p>By Thursday afternoon, we had:</p><ul><li>A fixed setup script (restructured, goto :eof replacing exit /b)</li><li>A Windows CI workflow (runs on every change to batch files, PowerShell scripts, or requirements.txt)</li><li>A Windows smoke test script (for alpha testers to validate their setup manually)</li></ul><p>The gap that had hidden for two months was now systematically covered.</p><h3>What the flu revealed</h3><p>I was sick for most of this week. Genuinely, frustratingly, can’t-think-straight sick.</p><p>And the cathedral kept building itself.</p><p>Not autonomously — agents still needed prompts to start, guidance when stuck, decisions when options diverged. But the infrastructure we’d built over months proved itself. The omnibus logs captured what happened even when I couldn’t follow along in real time. The audit protocols surfaced the file loss before it became permanent. The methodology — investigate first, fix second — led to systematic prevention rather than one-off patches.</p><p>By Friday, I was still recovering, but was able to make the rounds of all eight leadership agents for the weekly review — Chief of Staff, Architect, CIO, CXO, PPM, HoSR, Communications Chief, and the Lead Developer’s notes. They synthesized seven days of work across twenty-two sessions. Their conclusion matched what I’d been feeling all week: the infrastructure held.</p><p>The files that nearly vanished taught me something about institutional memory. The December bug taught me something about fresh eyes. And the flu taught me something about cathedrals: the good ones don’t need their architects present every day. They’re built to stand.</p><p><em>Next on Building Piper Morgan: The Assembly Assumption, in which we find features that are built correctly but do not compose properly.</em></p><p><em>Have you ever had to step back from something you were building and watch it continue without you? Not a single one remained.</p><p name="729c" id="729c" class="graf graf--p graf-after--p">Foggy as I was, I knew this was not good.</p><p name="572b" id="572b" class="graf graf--p graf-after--p">The investigation began immediately. Was this a fresh clone that hadn’t pulled the files? No — git reflog showed continuous history. Had someone deleted them? The evidence pointed to something more mundane and more preventable: a destructive git command, probably <code class="markup--code markup--p-code">git checkout .</code> or <code class="markup--code markup--p-code">git restore .</code>, run sometime around February 7th or 8th.</p><p name="6092" id="6092" class="graf graf--p graf-after--p">The dev/ directory was gitignored. It held working files, not committed code. When the restore command ran, git happily overwrote everything with… nothing. The files had never been committed, so there was nothing to restore them from.</p><p name="dc9d" id="dc9d" class="graf graf--p graf-after--p">Except there was.</p><h3 name="8a8f" id="8a8f" class="graf graf--h3 graf-after--p">Git archaeology</h3><p name="8a2a" id="8a2a" class="graf graf--p graf-after--h3">Git remembers everything — even files that were later gitignored. Every file that had ever been committed, even once, still existed somewhere in the repository’s history. The recovery command was ugly but effective:</p><pre data-code-block-mode="1" spellcheck="false" data-code-block-lang="bash" name="57ce" id="57ce" class="graf graf--pre graf-after--p graf--preV2"><span class="pre--content">git <span class="hljs-built_in">log</span> --all --full-history --diff-filter=A --name-only --format=<span class="hljs-string">&#x27;%H&#x27;</span> -- <span class="hljs-string">&#x27;dev/*&#x27;</span> | \<br />  <span class="hljs-keyword">while</span> <span class="hljs-built_in">read</span> <span class="hljs-built_in">hash</span>; <span class="hljs-keyword">do</span> git checkout <span class="hljs-variable">$hash</span> -- dev/ 2&gt;/dev/null; <span class="hljs-keyword">done</span></span></pre><p name="894c" id="894c" class="graf graf--p graf-after--pre">Translation: find every commit that ever added a file to dev/, and restore that file from that commit.</p><p name="b210" id="b210" class="graf graf--p graf-after--p">The result: 2,781 files recovered. More than we’d lost, because the archaeology surfaced files from months ago that had been deleted intentionally but were now restored as a side effect.</p><p name="f99f" id="f99f" class="graf graf--p graf-after--p">The afternoon became a cleanup operation. Sort the recovered files. Delete the ones we didn’t need. Compress the archives. Move things to their proper homes. By evening, dev/ had gone from 5.1 GB of chaos to 1.2 GB of organized structure.</p><p name="1b12" id="1b12" class="graf graf--p graf-after--p">The institutional memory was intact.</p><h3 name="3d75" id="3d75" class="graf graf--h3 graf-after--p">The December bug</h3><p name="e43d" id="e43d" class="graf graf--p graf-after--h3">Wednesday brought a different kind of discovery.</p><p name="658b" id="658b" class="graf graf--p graf-after--p">Dominique Derosena — a new alpha tester referred by my 18F colleague Magdaline Derosena — was setting up Piper on Windows. Fresh machine, following the alpha tester instructions. He ran the setup script.</p><p name="2003" id="2003" class="graf graf--p graf-after--p">It failed immediately.</p><p name="7d7b" id="7d7b" class="graf graf--p graf-after--p">Not after a few steps. Not with a subtle error. Immediately. The script printed its opening message and then… stopped.</p><p name="56b2" id="56b2" class="graf graf--p graf-after--p">The bug was in the batch file structure. Helper subroutines used <code class="markup--code markup--p-code">exit /b</code> to return to the caller. But in Windows batch files, <code class="markup--code markup--p-code">exit /b</code> inside a subroutine doesn&#39;t return — it exits the entire script. The correct command is <code class="markup--code markup--p-code">goto :eof</code>.</p><p name="878c" id="878c" class="graf graf--p graf-after--p">This bug had existed since December 2025. Two months. Through Ted Nadeau’s extensive Windows testing — fourteen issues filed and fixed. Through multiple releases. Through the entire alpha program.</p><p name="38c5" id="38c5" class="graf graf--p graf-after--p">I wonder why Ted hand’t run into it?</p><p name="d0c9" id="d0c9" class="graf graf--p graf-after--p">The answer was in his issue reports. Ted’s fourteen issues focused on requirements.txt conflicts, missing migrations, documentation gaps. All real problems. But nothing about the setup script failing on first run.</p><p name="1547" id="1547" class="graf graf--p graf-after--p">He must have hit the failure, shrugged, and switched to manual setup, or kust stopped at that point. Then filed issues about everything else he encountered. Maybe, the script failure was so immediate, so total, that it didn’t even register as a bug worth reporting — just an obstacle to work around.</p><p name="f720" id="f720" class="graf graf--p graf-after--p">Pairing with Dominique, as he set things up, coming in fresh, we caught it and logged the issue.</p><h3 name="1796" id="1796" class="graf graf--h3 graf-after--p">Systematic prevention</h3><p name="ec34" id="ec34" class="graf graf--p graf-after--h3">The Windows bug led to a broader question: why wasn’t this caught automatically?</p><p name="6027" id="6027" class="graf graf--p graf-after--p">The answer was uncomfortable. We had no Windows CI. Our continuous integration tested on Linux and macOS. Windows was… assumed. If a batch file was syntactically valid, we figured it would work.</p><p name="162b" id="162b" class="graf graf--p graf-after--p">By Thursday afternoon, we had:</p><ul class="postList"><li name="eb86" id="eb86" class="graf graf--li graf-after--p">A fixed setup script (restructured, <code class="markup--code markup--li-code">goto :eof</code> replacing <code class="markup--code markup--li-code">exit /b</code>)</li><li name="692c" id="692c" class="graf graf--li graf-after--li">A Windows CI workflow (runs on every change to batch files, PowerShell scripts, or requirements.txt)</li><li name="a2ad" id="a2ad" class="graf graf--li graf-after--li">A Windows smoke test script (for alpha testers to validate their setup manually)</li></ul><p name="acd6" id="acd6" class="graf graf--p graf-after--li">The gap that had hidden for two months was now systematically covered.</p><h3 name="8220" id="8220" class="graf graf--h3 graf-after--p">What the flu revealed</h3><p name="ec99" id="ec99" class="graf graf--p graf-after--h3">I was sick for most of this week. Genuinely, frustratingly, can’t-think-straight sick.</p><p name="5b33" id="5b33" class="graf graf--p graf-after--p">And the cathedral kept building itself.</p><p name="423b" id="423b" class="graf graf--p graf-after--p">Not autonomously — agents still needed prompts to start, guidance when stuck, decisions when options diverged. But the infrastructure we’d built over months proved itself. The omnibus logs captured what happened even when I couldn’t follow along in real time. The audit protocols surfaced the file loss before it became permanent. The methodology — investigate first, fix second — led to systematic prevention rather than one-off patches.</p><p name="060f" id="060f" class="graf graf--p graf-after--p">By Friday, I was still recovering, but was able to make the rounds of all eight leadership agents for the weekly review — Chief of Staff, Architect, CIO, CXO, PPM, HoSR, Communications Chief, and the Lead Developer’s notes. They synthesized seven days of work across twenty-two sessions. Their conclusion matched what I’d been feeling all week: the infrastructure held.</p><p name="ec80" id="ec80" class="graf graf--p graf-after--p graf--trailing">The files that nearly vanished taught me something about institutional memory. The December bug taught me something about fresh eyes. And the flu taught me something about cathedrals: the good ones don’t need their architects present every day.
