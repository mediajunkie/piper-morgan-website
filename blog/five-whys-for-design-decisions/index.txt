1a:T1c88,<p><em>December 20, 2025</em></p>
<p>Five Whys is a debugging technique. You ask "why did this break?" and then ask "why?" to each answer until you reach a root cause. It's standard practice for fixing bugs.</p>
<p>On December 20th, we discovered it works for something else entirely: <strong>figuring out why a feature doesn't exist.</strong></p>
<h1>The failing test</h1>
<p>A user asked Piper Morgan: "What services do you offer?"</p>
<p>The response was a generic answer. Not a menu of capabilities. Not a helpful overview. Just... words.</p>
<p>A second user asked: "Help me set up my projects."</p>
<p>The system interpreted this as a status request and returned their current work—not setup guidance.</p>
<p>These weren't bugs in the traditional sense. Nothing crashed. No errors in the logs. The system responded. It just responded wrong.</p>
<h1>The investigation</h1>
<p>We applied Five Whys to the first failure:</p>
<p><em><em>Why did "What services do you offer?" get a generic response?</em></em></p>
<p>→ Because no pattern matched it to the IDENTITY category (capability discovery).</p>
<p><em><em>Why did no pattern match?</em></em></p>
<p>→ Because our IDENTITY patterns didn't include the word "services."</p>
<p><em><em>Why didn't they include "services"?</em></em></p>
<p>→ Because when we designed the patterns, we thought about "capabilities" and "features," not "services."</p>
<p><em><em>Why did we think that way?</em></em></p>
<p>→ Because we designed from the system's perspective (what it can do) not the user's perspective (what they might ask).</p>
<p><em><em>Why did we design from the system's perspective?</em></em></p>
<p>→ Because <strong>the system is command-oriented, not discovery-oriented.</strong></p>
<p>That fifth answer wasn't a bug fix. It was an architectural diagnosis.</p>
<h1>The bigger gap</h1>
<p>The second failure—"help me set up my projects" being interpreted as a status request—had a similar root cause chain:</p>
<p><em><em>Why did it return status instead of setup guidance?</em></em></p>
<p>→ Because "my projects" matched the STATUS pattern before reaching the setup logic.</p>
<p><em><em>Why did STATUS match first?</em></em></p>
<p>→ Because our pattern matching is greedy—first match wins.</p>
<p><em><em>Why is it greedy?</em></em></p>
<p>→ Because we optimized for speed, not for intent disambiguation.</p>
<p><em><em>Why didn't we disambiguate?</em></em></p>
<p>→ Because we assumed users would use command-like language.</p>
<p><em><em>Why did we assume that?</em></em></p>
<p>→ Because <strong>we built for power users giving commands, not new users exploring.</strong></p>
<p>Same root cause. Two different symptoms. The architecture was oriented toward people who already knew what they wanted, not people trying to figure out what was possible.</p>
<h1>The reframe</h1>
<p>Here's what made this investigation different from normal debugging:</p>
<p>Engineering Five Whys: "Why is this broken?" → Fix the bug at its root cause.</p>
<p>Design Five Whys: "Why doesn't this work the way users expect?" → Discover the architectural assumption that needs rethinking.</p>
<p>There is a key difference. When debugging code, we assume the system is right but that something went wrong. The second question assumes the user is right and the system has a gap.</p>
<p>Both are valid. But they lead to different places. Bug fixing repairs functional gaps. Design questioning leads to architectural insight.</p>
<h1>What we found</h1>
<p>The December 20th investigation revealed four distinct gaps:</p>
<ol><li><strong>Pattern gaps</strong>: Missing vocabulary in intent classification</li><li><strong>Priority gaps</strong>: Pattern matching order didn't reflect user mental models</li><li><strong>Capability gaps</strong>: No API for "what can this system do?"</li><li><strong>Test gaps</strong>: No coverage for discovery scenarios</li></ol>
<p>A traditional bug fix would have added the word "services" to the IDENTITY patterns. Done. Ship it.</p>
<p>But Five Whys revealed that adding one word wouldn't fix the architectural issue. Users would find other words we hadn't anticipated. The system would keep failing in new ways.</p>
<p>The real fix was to shift from command-oriented to discovery-oriented design—a much bigger change that addressed the root cause rather than the symptoms.</p>
<h1>The technique</h1>
<p>Five Whys for design decisions works like this:</p>
<ol><li><strong>Start with user behavior, not system behavior.</strong> "The user asked X and expected Y" not "the system returned Z."</li><li><strong>Ask why the expectation wasn't met.</strong> Not "why did it break" but "why didn't it work the way they expected?"</li><li><strong>Keep asking until you hit an assumption.</strong> The root cause of a design gap is usually an assumption you made during initial design. Find it.</li><li><strong>Name the assumption explicitly.</strong> "We assumed users would..." or "We designed for..." Make the implicit explicit.</li><li><strong>Evaluate the assumption.</strong> Was it correct? Is it still correct? Should it change?</li></ol>
<h1>When to use it</h1>
<p>Five Whys for design works best when:</p>
<ul><li>Users report "it doesn't do what I expected" (not crashes or errors)</li><li>Multiple users report similar confusion</li><li>You've fixed the same area multiple times</li><li>The feature technically works but doesn't feel right</li></ul>
<p>It works less well when:</p>
<ul><li>There's an actual bug (crash, error, wrong data)</li><li>The issue is performance, not behavior</li><li>You already know the architectural gap and just need to fix it</li></ul>
<h1>Today's meta-insight</h1>
<p>The session on December 20th started as a bug investigation and ended as an architectural review. We created a new epic (Discovery Orientation), added 31 new tests, and redesigned how the system handles capability questions.</p>
<p>All from asking "why?" five times.</p>
<p>The technique isn't magic. It's just systematic. But applying a debugging technique to design questions—treating user confusion as a symptom of architectural assumptions—turned a patch job into a strategic improvement.</p>

<p>For the curious, here is the diagnostic trace that grounded the investigation:</p>
<table><thead><tr><th>Message</th><th>Pre-classifier match</th><th>LLM category</th><th>Handler</th><th>Root cause</th></tr></thead><tbody><tr><td>"What services do you offer?"</td><td>None (NO MATCH)</td><td>QUERY / CONVERSATION</td><td>Generic</td><td>IDENTITY patterns lack "services" — LLM doesn't clarify</td></tr><tr><td>"Help me setup my projects"</td><td>STATUS (FALSE POSITIVE)</td><td>STATUS</td><td><code>_handle_status_query</code></td><td>"my projects" pattern matches without checking "setup" verb</td></tr></tbody></table>
<p>Two messages, two completely different failure modes, same architectural root.</p>
