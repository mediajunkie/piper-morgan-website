My second thought: OK what did I do now, and how bad is it? My third thought: coffee first, then crisis.</p><h3>The alert</h3><p>GCP’s suspension notice was alarming but vague. “Hijacked resources” could mean anything from cryptocurrency mining to spam operations to something worse. The project was locked. Services were down. And I had no idea what had happened.</p><p>I spun up a Security Operations agent at 5:38 AM. First task: figure out what was affected.</p><p>The answer came back quickly: Gemini API. Google’s AI service. We used it as a backup for certain operations — not our primary path, but integrated into the system. Someone had been using our API key.</p><h3>The bleeding</h3><p>By 5:48 AM, we’d confirmed the scope. The Gemini key was compromised. Usage was spiking in ways we hadn’t caused. Someone else was running requests on our dime.</p><p>At 6:04 AM, we deleted the key via AI Studio. The bleeding stopped. Whatever they were doing, they couldn’t do it anymore.</p><p>But that raised the obvious question: how did they get the key in the first place? We did an investigation</p><h3>The five whys</h3><ol><li>Why was the key compromised? Because it was exposed in a committed file.</li><li>Why was it in a committed file? Because it appeared in dev/server-startup.log.</li><li>Why was the key in a log file? Because httpx logs full URLs, and our Gemini calls included the key as a query parameter.</li><li>Why was that log file committed? Because dev/ wasn&#39;t in our .gitignore.</li><li>Why wasn’t it caught before now? Because we had no automated secret scanning on the repository.</li></ol><p>The root cause wasn’t a hack. It wasn’t sophisticated. It was a logging library doing exactly what logging libraries do — recording URLs — and us not realizing that those URLs can contain secrets.</p><h3>The timeline</h3><p>We traced the leak to October 16, 2025. Three months of exposure.</p><p>Here’s what likely happened: Someone — probably an automated scanner — found the key in our public repository. They started using it for their own Gemini API calls. Google detected the anomalous usage pattern and suspended the project.</p><p>In a way, Google’s suspension was a favor. It forced us to discover a vulnerability we didn’t know existed.</p><h3>The fix</h3><p>One-layer fixes don’t prevent recurrence. We implemented five.</p><p>The immediate fix was adding dev/ to .gitignore. No more development logs in version control.</p><p>For detection, we deployed a URL redaction filter to all HTTP loggers. Any URL containing sensitive patterns gets redacted before logging.</p><p>For scanning, we ran git secrets --scan-history across the entire repository history, which found and flagged the October commit.</p><p>For prevention, we enabled GitHub’s secret scanning on the repository. Future commits with exposed secrets will be blocked.</p><p>For storage, we migrated all API keys from Apple Notes (yes, really) to macOS Keychain with proper access controls.</p><p>Each layer addresses a different point of failure. Together, they make the same mistake much harder to repeat.</p><p>We also filed an appeal and detailed all the measures we have taken. No word yet….</p><h3>The lesson</h3><p>The October Ghost wasn’t a sophisticated attack. It was a combination of ordinary behaviors: a library logging URLs (normal), URLs containing API keys (unfortunately common), development files getting committed (easy mistake), no scanning for secrets (oversight), and credentials stored in notes (convenient but insecure).</p><p>None of these individually is unusual. Together, they create a three-month window where anyone scanning public GitHub repositories could find and use your API key.</p><p>The fix wasn’t heroic. It was systematic. Five layers, each addressing one link in the chain. Any one of them would have prevented or detected the leak.</p><h3>The meta-pattern</h3><p>What struck me afterward was how the same methodology we use for debugging code worked for debugging security. Five Whys traced the root cause. Defense in depth guided the multi-layer fix. Verification confirmed the scope. Documentation captures the learning.</p><p>Crisis response isn’t different from regular development. It’s just faster and higher stakes. The same systematic approaches apply.</p><h3>The cost</h3><p>We lost a few hours to the incident. I expect the project suspension will be lifted for an innocent mistake now well mitigated. I can’t be the only goofball who has made this mistake!</p><p>No user data was affected — Gemini doesn’t handle our user data. The financial cost was some unauthorized API usage within our free quota but of course costing Google something.</p><p>But we gained something: a security posture we should have had from day one. It forced us to discover a vulnerability we didn’t know existed.</p><h3 name="1db6" id="1db6" class="graf graf--h3 graf-after--p">The fix</h3><p name="856a" id="856a" class="graf graf--p graf-after--h3">One-layer fixes don’t prevent recurrence. We implemented five.</p><p name="ead8" id="ead8" class="graf graf--p graf-after--p">The immediate fix was adding <code class="markup--code markup--p-code">dev/</code> to <code class="markup--code markup--p-code">.gitignore</code>. No more development logs in version control.</p><p name="36e6" id="36e6" class="graf graf--p graf-after--p">For detection, we deployed a URL redaction filter to all HTTP loggers. Any URL containing sensitive patterns gets redacted before logging.</p><p name="2f0d" id="2f0d" class="graf graf--p graf-after--p">For scanning, we ran <code class="markup--code markup--p-code">git secrets --scan-history</code> across the entire repository history, which found and flagged the October commit.</p><p name="2c6f" id="2c6f" class="graf graf--p graf-after--p">For prevention, we enabled GitHub’s secret scanning on the repository. Future commits with exposed secrets will be blocked.</p><p name="479a" id="479a" class="graf graf--p graf-after--p">For storage, we migrated all API keys from Apple Notes (yes, really) to macOS Keychain with proper access controls.</p><p name="7577" id="7577" class="graf graf--p graf-after--p">Each layer addresses a different point of failure. Together, they make the same mistake much harder to repeat.</p><p name="e241" id="e241" class="graf graf--p graf-after--p">We also filed an appeal and detailed all the measures we have taken. No word yet….</p><h3 name="1218" id="1218" class="graf graf--h3 graf-after--p">The lesson</h3><p name="c577" id="c577" class="graf graf--p graf-after--h3">The October Ghost wasn’t a sophisticated attack. It was a combination of ordinary behaviors: a library logging URLs (normal), URLs containing API keys (unfortunately common), development files getting committed (easy mistake), no scanning for secrets (oversight), and credentials stored in notes (convenient but insecure).</p><p name="bbdc" id="bbdc" class="graf graf--p graf-after--p">None of these individually is unusual. Together, they create a three-month window where anyone scanning public GitHub repositories could find and use your API key.</p><p name="a521" id="a521" class="graf graf--p graf-after--p">The fix wasn’t heroic. It was systematic. Five layers, each addressing one link in the chain. Any one of them would have prevented or detected the leak.</p><h3 name="3076" id="3076" class="graf graf--h3 graf-after--p">The meta-pattern</h3><p name="f07e" id="f07e" class="graf graf--p graf-after--h3">What struck me afterward was how the same methodology we use for debugging code worked for debugging security. Five Whys traced the root cause. Defense in depth guided the multi-layer fix. Verification confirmed the scope. Documentation captures the learning.</p><p name="2a6c" id="2a6c" class="graf graf--p graf-after--p">Crisis response isn’t different from regular development. It’s just faster and higher stakes. The same systematic approaches apply.</p><h3 name="aff1" id="aff1" class="graf graf--h3 graf-after--p">The cost</h3><p name="eab9" id="eab9" class="graf graf--p graf-after--h3">We lost a few hours to the incident. I expect the project suspension will be lifted for an innocent mistake now well mitigated. I can’t be the only goofball who has made this mistake!</p><p name="e1b2" id="e1b2" class="graf graf--p graf-after--p">No user data was affected — Gemini doesn’t handle our user data.
