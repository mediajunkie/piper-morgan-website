Should have been straightforward.</p><p>“All 19 tests fail,” the Lead Developer reported back. “Not just one.”</p><h3>The thread that wouldn’t stop</h3><p>The test failure traced to a missing fixture — TEST_USER_ID wasn’t being set, so the config service couldn’t isolate properly. Four minutes to fix. But that fix revealed something else.</p><p>While investigating a History Sidebar issue — a new feature from the MUX sprints that wasn’t behaving as expected — the Lead Developer found something worse: cross-user session bleed. Under certain conditions, User A could see data that belonged to User B. Not passwords or credentials — conversation metadata. But still. A user isolation bug in what’s supposed to be a multi-tenant system.</p><p>The morning had started with one failing test. By afternoon, we’d found five interconnected bugs. What are we trying to say? How does the website relate to the actual product?</p><p>The CXO and Communications Director spent the day on it. Applied to Piper, it surfaced something I’d been circling for months:</p><p><em>PM tools assume work is items in lists. But PM work is actually relationships between concerns at different scales.</em></p><p>That’s the gap. Jira gives you tickets. Asana gives you tasks. Linear gives you issues. They’re all list managers with different paint jobs. None of them model what a PM actually does: hold relationships between a user need and a feature, between a strategic goal and a quarterly milestone, between today’s bug and last month’s architectural decision.</p><p>By Sunday evening, we had a complete seven-page site draft. Monday, implementation began — five phases, executed cleanly. By the end of the week, the bones of a real website existed.</p><h3>Two kinds of work</h3><p>What strikes me about this stretch is the contrast.</p><p>Friday was reactive. One question led to another, each answer revealing a new problem. The discipline was following the thread — not stopping when the first fix worked, not declaring victory until we understood the full shape of what was broken.</p><p>Sunday was proactive. We stepped back from the code entirely and asked: what are we building, and who is it for? The discipline was different — not following threads but weaving them together, finding the unifying insight that makes everything else cohere.</p><p>Both are necessary. You can’t build a cathedral if you don’t follow the threads that reveal structural problems. Applied to Piper, it surfaced something I’d been circling for months:</p><p name="9e40" id="9e40" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">PM tools assume work is items in lists. But PM work is actually relationships between concerns at different scales.</em></p><p name="1a91" id="1a91" class="graf graf--p graf-after--p">That’s the gap. Jira gives you tickets. Asana gives you tasks. Linear gives you issues. They’re all list managers with different paint jobs. None of them model what a PM actually does: hold relationships between a user need and a feature, between a strategic goal and a quarterly milestone, between today’s bug and last month’s architectural decision.</p><p name="3f27" id="3f27" class="graf graf--p graf-after--p">By Sunday evening, we had a complete seven-page site draft. Monday, implementation began — five phases, executed cleanly. By the end of the week, the bones of a real website existed.</p><h3 name="237f" id="237f" class="graf graf--h3 graf-after--p">Two kinds of work</h3><p name="ead9" id="ead9" class="graf graf--p graf-after--h3">What strikes me about this stretch is the contrast.</p><p name="c25b" id="c25b" class="graf graf--p graf-after--p">Friday was reactive. One question led to another, each answer revealing a new problem. The discipline was following the thread — not stopping when the first fix worked, not declaring victory until we understood the full shape of what was broken.</p><p name="6cbc" id="6cbc" class="graf graf--p graf-after--p">Sunday was proactive. We stepped back from the code entirely and asked: what are we building, and who is it for? The discipline was different — not following threads but weaving them together, finding the unifying insight that makes everything else cohere.</p><p name="f244" id="f244" class="graf graf--p graf-after--p graf--trailing">Both are necessary.
