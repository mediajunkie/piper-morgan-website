1b:T1b29,<p><em>March 15–16</em></p>
<p>Saturday afternoon. The tests said the floor was in place. The Lead Developer had built the conversational floor the evening before, the LLM fallback that was supposed to catch every query the handlers couldn't answer. Twenty-three tests passing. I'd seen the screenshot. It clearly worked. Time to test it.
</p>
<p>I typed seven messages into Piper. The kind of thing a normal user would say. "What should I focus on today?" "Tell me about AI agents." Simple, conversational, the exact queries the floor was built to handle.
</p>
<p>Six of the seven came back as template boilerplate. Canned responses. Not the LLM engaging thoughtfully with my question, just the same pre-written handler output I'd been staring at for weeks. Some of it even clunkier.
</p>
<p>So what was the problem. Turns out there was a floor down there, but the user never hits it.
</p>
<p>I want to be honest here. I started worrying that maybe I've just wasted a lot of time on something that I may never figure out how to get working.
</p>
<p>But I'm not a quitter, so I kept digging.
</p>
<h2>The routing was inverted</h2>
<p>I told the Lead Developer: "Investigate the architecture and docbase, write a report."
</p>
<p>The report came back. The routing architecture was still backwards from the design intent. Our own design documents — PDR-002, ADR-039 — described a system where the conversational floor was the default and structured handlers were enhancements. What we'd actually built was the opposite: canonical handlers were the default path, and the floor was a last resort that almost nothing reached.
</p>
<p>Every query hit a handler first. Most handlers had something to say — even if what they had to say was a template response that didn't actually engage with the question. The floor existed, it worked, it was tested, and it was unreachable.
</p>
<p>Three stubs were literally labeled "implementation pending" — and they were catching real user queries and returning nothing useful instead of letting the floor handle them. The Lead Developer replaced all three with floor routing, removed a wrong todo fallback, and cleaned up a stale debug print. Small fixes, but each one had been silently eating queries.
</p>
<h2>Phase 1</h2>
<p>The first fix was scoped and shipped the same day. GUIDANCE intents — the broad category covering "help me think about this" queries — now route to the conversational floor with assembled context. A new context assembler gathers calendar events, projects, and priorities as structured facts and passes them to the LLM. Nineteen new tests. Merged to main.
</p>
<p>I tested again. "What should I focus on today?" This time it hit the PRIORITY handler — a different category entirely, not GUIDANCE. The routing was more tangled than one category could fix.
</p>
<p>The query that did reach the floor — a question about AI agents — came back with Piper parroting its own system prompt. Instead of engaging with my question, the LLM read its own instructions aloud. "I'm here to help you with product management tasks..."
</p>
<p>The Lead Developer diagnosed both issues. PRIORITY was a separate classification category — Phase 1 only covered GUIDANCE. And the floor prompt was written in descriptive mode ("I'm a PM assistant that helps with...") instead of directive mode ("Respond directly to what the user said"). The LLM did exactly what descriptive prompts tell it to do: describe itself.
</p>
<p>The prompt was rewritten. The parroting stopped.
</p>
<h2>"Floor works" ≠ "users experience the floor"</h2>
<p>This is the pattern that kept repeating. Mechanical correctness doesn't guarantee user experience. The floor passed every test. The routing worked as coded. The handlers returned responses. Everything was green.
</p>
<p>And a real person typing a real question got template boilerplate six times out of seven.
</p>
<p>As I review and edit this draft for publication today, I will admit I am feeling that trepidation that builds up before manual testing. I've got user acceptance testing scheduled with my CXO to see if the M1 sprint has landed and I'm kind of afraid to check.
</p>
<h2>The contract gap</h2>
<p>The next day brought more testing and a deeper diagnosis. The leadership team converged again — Architect, PPM, CXO — this time on the structural relationship between classification and handling.
</p>
<p>The Architect refined the core question: "Does this intent require an operation the LLM cannot perform?" If no — if it's a conversational question, not an action — the floor should handle it. Handlers are for side effects: creating todos, closing issues, checking calendars. The Action Gate criterion was born.
</p>
<p>The CXO added a voice rule: "Never say 'I can't.'" If Piper can't perform the specific action requested, it should still engage conversationally with the topic. The floor makes this possible — there's always a response, always an engagement.
</p>
<p>The PPM synthesized: the classifier's accuracy matters for <em>actions</em>, not for <em>conversations</em>. If a query gets misclassified but both categories route to the floor with context, the user doesn't notice. Classifier accuracy only matters when the wrong classification would trigger the wrong side effect.
</p>
<p>We were getting pretty good at these roundtables!
</p>
<p>The Lead Developer ran a Five Whys analysis that afternoon, exposing a systemic contract gap between the classification layer and the handling layer. The classifier was making fine-grained distinctions, but the handlers on the receiving end didn't honor those distinctions — they returned generic responses regardless of the classified intent. Precision in, mush out.
</p>
<p>Nine issues closed by end of day. The architecture was starting to face the right direction.
</p>
<h2>The lesson</h2>
<p>Two days of testing. Two days of discovering that a system that "worked" — by every automated measure — didn't work at all for the person it was built for.
</p>
<p>The floor existed. The tests passed. The routing was mechanically correct. But the architecture was inverted from the design intent, the prompts were written in the wrong mode, the handlers were swallowing queries they should have forwarded, and the contract between classifier and handler was broken.
</p>
<p>None of this was visible from the test suite. All of it was visible within thirty seconds of typing a question.
</p>
<p>I love these smart little talking computer critters, but it's funny how it often it comes down to an actual human (usually me) trying to do some real thing, to find out if what we've made is really even anything at all.
</p>
