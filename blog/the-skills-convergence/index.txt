1b:T31cd,<figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*qfRLZk8hDUU7wpCB2UwHVQ.png" /><figcaption>“These all fit together!”</figcaption></figure><p><em>November 15</em></p><p>Sometimes you discover you’ve been working the hard way. Not because you made bad decisions, but because you didn’t have all the information yet. That’s what happened on Saturday morning when I brought a new idea to my Chief of Staff. We had the Chief Architect analyze token usage patterns and we found something remarkable: we could reduce our most expensive operations by 98.7%.</p><p>But this isn’t a story about optimization. It’s about what happens when efficiency reveals itself as strategy, when three separate problems converge into one solution, and when an economic model emerges that funds its own transformation.</p><h3>The expensive way to process documents</h3><p>Saturday November 15 started at 5:39 AM with my Chief Architect investigating a question: how efficiently was Piper using Model Context Protocol (MCP)?</p><p>A few days earlier, Anthropic had published research showing something counterintuitive. When LLMs call tools through MCP, they consume tokens for every step: input, output, reasoning. But when they write code that uses those tools and then execute that code? Token savings of 98.7%. The difference is fundamental: orchestration happens in the model’s context (expensive) versus orchestration happens in code execution (cheap).</p><p>My Chief Architect looked at Piper’s actual usage patterns. We had five MCP servers active: GitHub (heavy usage), Slack (medium), Calendar (light), Notion (heavy), Serena (medium). The token-heavy pattern was immediately visible: document processing consumed ~150,000 tokens per operation.</p><p>Here’s why. When a user asks “summarize my documents from last week,” the model orchestrates a chain:</p><ol><li>Call Notion MCP to list documents (3K tokens)</li><li>Parse results, identify relevant docs (2K tokens)</li><li>Call Notion MCP to fetch each document (10K tokens × 5 docs)</li><li>Call GitHub MCP to check related issues (5K tokens)</li><li>Synthesize everything into summary (8K tokens)</li></ol><p>Total: ~65K tokens just for the tool calls. Add reasoning overhead and context management, and you’re at 150K tokens for a single document query.</p><p>[PLACEHOLDER: When you first saw these numbers, what was your reaction? Did this surprise you, or does this connect to past experiences with inefficient architectures that only revealed themselves through measurement?]</p><p>The expensive part isn’t any individual step — it’s that every step flows through the model’s context window. Each tool’s output becomes the next tool’s input, all mediated by the LLM. The model is doing secretarial work instead of thinking work.</p><h3>Three tiers of tool usage</h3><p>At 5:50 AM, the Chief Architect proposed a three-tier strategy:</p><p><strong>Tier 1: Direct MCP</strong> — Keep for simple, one-shot operations under 1,000 tokens. Checking a calendar, posting a Slack message, creating a GitHub issue. These are fine. The orchestration overhead is minimal.</p><p><strong>Tier 2: Skills MCP</strong> — NEW approach for common multi-step patterns. Instead of the model orchestrating tool calls, we encapsulate entire workflows as executable skills. The model writes code once, we execute it repeatedly.</p><p><strong>Tier 3: Code Execution</strong> — For complex, custom workflows that don’t fit predefined patterns. The model writes fresh code, we execute it.</p><p>The key insight: LLMs are better at writing code than calling tools. Their training data is full of code. They’ve seen millions of examples of “fetch data, process it, return results.” They’re excellent at generating Python or JavaScript that accomplishes multi-step workflows.</p><p>By 6:00 AM, the Skills MCP architecture was designed:</p><ul><li><strong>SkillsMCP meta-server</strong> provides executable skills to agents</li><li><strong>Skills encapsulate workflows</strong>: StandupToIssuesSkill, DocumentAnalysisSkill, BatchProcessingSkill</li><li><strong>Data stays in execution environment</strong>: No massive outputs flowing back to model</li><li><strong>Only summaries return</strong>: Model sees ~50 tokens of result, not 50K tokens of intermediate data</li></ul><p>The projected savings were dramatic: 90–98% token reduction on common patterns.</p><h3>When everything connects</h3><p>At 7:30 AM, something interesting happened. The Chief Architect was reviewing yesterday’s UX audit findings — 350 pages documenting 68 gaps across six user journeys.</p><p>Journey 6 (Document Management) had scored 2 out of 10. The crisis was clear: users couldn’t effectively retrieve or manage documents they’d uploaded. The system stored documents but provided no good way to find them again.</p><p>The Chief Architect looked at Journey 6 and saw: DocumentAnalysisSkill.</p><p>The document management problem? Solvable with a skill that encapsulates document retrieval, filtering, and summarization.</p><p>The token efficiency problem? Solvable with Skills MCP architecture.</p><p>The UX improvement funding problem? Solvable with the resources saved from token efficiency.</p><p>Three initiatives that had seemed separate — UX transformation (my researcher estimated it as requiring $130K worth of UX work in the traditional market), Skills MCP implementation, Learning System completion — were actually one integrated transformation.</p><p>By 7:40 AM, the Chief Architect had modeled the economics:</p><p><strong>Current state</strong>: Estimated future document processing costs ~$4,440 per month in tokens (150K tokens per query × 200 queries/day × $0.015/1K tokens)</p><p><strong>With Skills MCP</strong>: Same operations cost ~$60 per month (token usage drops to 2K per query — just the model writing the skill execution code)</p><p><strong>Potential monthly savings</strong>: $4,380</p><p><strong>Potential three-month savings</strong>: $13,140</p><p><strong>Thirteen-week transformation budget</strong>: $130K total</p><p>The math was elegant: by week 4, Skills MCP would generate enough savings to fully fund UX improvements over three months. By week 13, the transformation would have paid for itself and established a sustainable economic model. (Even without the “funny money” that drove this logic, the tradeoffs are similar!)</p><h3>The virtuous cycle</h3><p>The Chief Architect mapped the convergence:</p><p><strong>Efficiency Stream</strong> (Skills MCP):</p><ul><li>Measure, prototype, validate</li><li>Build core skills library</li><li>Migrate agents to Skills</li><li>Dynamic skill generation from learning patterns</li></ul><p><strong>Experience Stream</strong> (UX Transformation):</p><ul><li>Design system + quick fixes (Journey scores 4.0 → 5.5)</li><li>Core improvements (5.5 → 6.5)</li><li>Document management overhaul (6.5 → 7.5, Journey 6 jumps 2→8)</li><li>Advanced features + accessibility (7.5 → 7.8)</li></ul><p><strong>Intelligence Stream</strong> (Learning System):</p><ul><li>User interactions generate patterns</li><li>Patterns inform skill creation</li><li>Skills make operations more efficient</li><li>Efficiency enables more sophisticated features</li></ul><p>Each stream strengthens the others:</p><ul><li>Better UX → More users → More interaction patterns</li><li>More patterns → Better skills → Lower costs</li><li>Lower costs → Fund more UX improvements → Better experience</li></ul><p>This wasn’t just “make things faster.” It was “build an economic engine that funds its own evolution while improving user experience and creating competitive moats.”</p><h3>From discovery to execution</h3><p>The roadmap laid out clear milestones:</p><p><strong>Week 1</strong>: Measure everything. Add token counting to all MCP operations. Establish baseline costs. Identify the most expensive patterns.</p><p><strong>Weeks 2–3</strong>: Build ONE prototype skill (probably DocumentAnalysisSkill, since it solves the worst UX problem). Validate the savings. These cover 80% of our token-heavy operations.</p><p><strong>Weeks 7–9</strong>: Migrate agents to use SkillsMCP. Update prompts, add skill selection logic, train agents to prefer skills over raw MCP calls.</p><p><strong>Weeks 10–13</strong>: Advanced patterns. Dynamic skill generation from learning system. Skills that write themselves based on observed usage patterns.</p><p>By February 2026 (MVP target), the transformation would be complete:</p><ul><li>Journey scores averaging 7.8/10 (up from 4.0/10)</li><li>Token costs reduced by 95%+ on common operations</li><li>Economic model established and validated</li><li>Learning system generating new skills automatically</li></ul><p>Again, even if the dates change (most of these time estimates tend to be based on human teams) the larger patterns hold.</p><p>The parallel work that Saturday reinforced the plan’s viability. The UX transformation was already proving its value.</p><h3>What working the hard way teaches you</h3><p>We’d been using MCP correctly according to its design. The working system generated the evidence that pointed toward the optimal system.</p><p>That’s the cathedral-building payoff: patience in process creates possibilities that rushing misses. Not because you made bad decisions, but because you didn’t have all the information yet. That’s what happened on Saturday morning when I brought a new idea to my Chief of Staff. We had the Chief Architect analyze token usage patterns and we found something remarkable: we could reduce our most expensive operations by 98.7%.</p><p name="4de9" id="4de9" class="graf graf--p graf-after--p">But this isn’t a story about optimization. It’s about what happens when efficiency reveals itself as strategy, when three separate problems converge into one solution, and when an economic model emerges that funds its own transformation.</p><h3 name="3745" id="3745" class="graf graf--h3 graf-after--p">The expensive way to process documents</h3><p name="b57f" id="b57f" class="graf graf--p graf-after--h3">Saturday November 15 started at 5:39 AM with my Chief Architect investigating a question: how efficiently was Piper using Model Context Protocol (MCP)?</p><p name="76da" id="76da" class="graf graf--p graf-after--p">A few days earlier, Anthropic had published research showing something counterintuitive. When LLMs call tools through MCP, they consume tokens for every step: input, output, reasoning. But when they write code that uses those tools and then execute that code? Token savings of 98.7%. The difference is fundamental: orchestration happens in the model’s context (expensive) versus orchestration happens in code execution (cheap).</p><p name="7624" id="7624" class="graf graf--p graf-after--p">My Chief Architect looked at Piper’s actual usage patterns. We had five MCP servers active: GitHub (heavy usage), Slack (medium), Calendar (light), Notion (heavy), Serena (medium). The token-heavy pattern was immediately visible: document processing consumed ~150,000 tokens per operation.</p><p name="96f4" id="96f4" class="graf graf--p graf-after--p">Here’s why. When a user asks “summarize my documents from last week,” the model orchestrates a chain:</p><ol class="postList"><li name="6efa" id="6efa" class="graf graf--li graf-after--p">Call Notion MCP to list documents (3K tokens)</li><li name="e249" id="e249" class="graf graf--li graf-after--li">Parse results, identify relevant docs (2K tokens)</li><li name="1f53" id="1f53" class="graf graf--li graf-after--li">Call Notion MCP to fetch each document (10K tokens × 5 docs)</li><li name="1c87" id="1c87" class="graf graf--li graf-after--li">Call GitHub MCP to check related issues (5K tokens)</li><li name="8134" id="8134" class="graf graf--li graf-after--li">Synthesize everything into summary (8K tokens)</li></ol><p name="95dc" id="95dc" class="graf graf--p graf-after--li">Total: ~65K tokens just for the tool calls. Add reasoning overhead and context management, and you’re at 150K tokens for a single document query.</p><p name="1158" id="1158" class="graf graf--p graf-after--p">[PLACEHOLDER: When you first saw these numbers, what was your reaction? Did this surprise you, or does this connect to past experiences with inefficient architectures that only revealed themselves through measurement?]</p><p name="c84f" id="c84f" class="graf graf--p graf-after--p">The expensive part isn’t any individual step — it’s that every step flows through the model’s context window. Each tool’s output becomes the next tool’s input, all mediated by the LLM. The model is doing secretarial work instead of thinking work.</p><h3 name="22c1" id="22c1" class="graf graf--h3 graf-after--p">Three tiers of tool usage</h3><p name="72fc" id="72fc" class="graf graf--p graf-after--h3">At 5:50 AM, the Chief Architect proposed a three-tier strategy:</p><p name="7f66" id="7f66" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Tier 1: Direct MCP</strong> — Keep for simple, one-shot operations under 1,000 tokens. Checking a calendar, posting a Slack message, creating a GitHub issue. These are fine. The orchestration overhead is minimal.</p><p name="a5c0" id="a5c0" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Tier 2: Skills MCP</strong> — NEW approach for common multi-step patterns. Instead of the model orchestrating tool calls, we encapsulate entire workflows as executable skills. The model writes code once, we execute it repeatedly.</p><p name="45c0" id="45c0" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Tier 3: Code Execution</strong> — For complex, custom workflows that don’t fit predefined patterns. The model writes fresh code, we execute it.</p><p name="11ce" id="11ce" class="graf graf--p graf-after--p">The key insight: LLMs are better at writing code than calling tools. Their training data is full of code. They’ve seen millions of examples of “fetch data, process it, return results.” They’re excellent at generating Python or JavaScript that accomplishes multi-step workflows.</p><p name="ab27" id="ab27" class="graf graf--p graf-after--p">By 6:00 AM, the Skills MCP architecture was designed:</p><ul class="postList"><li name="3df7" id="3df7" class="graf graf--li graf-after--p"><strong class="markup--strong markup--li-strong">SkillsMCP meta-server</strong> provides executable skills to agents</li><li name="b7ed" id="b7ed" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Skills encapsulate workflows</strong>: StandupToIssuesSkill, DocumentAnalysisSkill, BatchProcessingSkill</li><li name="6549" id="6549" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Data stays in execution environment</strong>: No massive outputs flowing back to model</li><li name="1ca0" id="1ca0" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Only summaries return</strong>: Model sees ~50 tokens of result, not 50K tokens of intermediate data</li></ul><p name="f10a" id="f10a" class="graf graf--p graf-after--li">The projected savings were dramatic: 90–98% token reduction on common patterns.</p><h3 name="11c5" id="11c5" class="graf graf--h3 graf-after--p">When everything connects</h3><p name="9ba2" id="9ba2" class="graf graf--p graf-after--h3">At 7:30 AM, something interesting happened. The Chief Architect was reviewing yesterday’s UX audit findings — 350 pages documenting 68 gaps across six user journeys.</p><p name="350f" id="350f" class="graf graf--p graf-after--p">Journey 6 (Document Management) had scored 2 out of 10. The crisis was clear: users couldn’t effectively retrieve or manage documents they’d uploaded. The system stored documents but provided no good way to find them again.</p><p name="c5e2" id="c5e2" class="graf graf--p graf-after--p">The Chief Architect looked at Journey 6 and saw: DocumentAnalysisSkill.</p><p name="ee39" id="ee39" class="graf graf--p graf-after--p">The document management problem? Solvable with a skill that encapsulates document retrieval, filtering, and summarization.</p><p name="b180" id="b180" class="graf graf--p graf-after--p">The token efficiency problem? Solvable with Skills MCP architecture.</p><p name="bce8" id="bce8" class="graf graf--p graf-after--p">The UX improvement funding problem? Solvable with the resources saved from token efficiency.</p><p name="8b75" id="8b75" class="graf graf--p graf-after--p">Three initiatives that had seemed separate — UX transformation (my researcher estimated it as requiring $130K worth of UX work in the traditional market), Skills MCP implementation, Learning System completion — were actually one integrated transformation.</p><p name="7071" id="7071" class="graf graf--p graf-after--p">By 7:40 AM, the Chief Architect had modeled the economics:</p><p name="0103" id="0103" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Current state</strong>: Estimated future document processing costs ~$4,440 per month in tokens (150K tokens per query × 200 queries/day × $0.015/1K tokens)</p><p name="241e" id="241e" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">With Skills MCP</strong>: Same operations cost ~$60 per month (token usage drops to 2K per query — just the model writing the skill execution code)</p><p name="3075" id="3075" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Potential monthly savings</strong>: $4,380</p><p name="ea3d" id="ea3d" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Potential three-month savings</strong>: $13,140</p><p name="d882" id="d882" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Thirteen-week transformation budget</strong>: $130K total</p><p name="9919" id="9919" class="graf graf--p graf-after--p">The math was elegant: by week 4, Skills MCP would generate enough savings to fully fund UX improvements over three months.
