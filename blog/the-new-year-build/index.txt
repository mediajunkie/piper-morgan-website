Yes, this is how I ring in the new year (lolsob).</p><p>Then I pulled up the Sprint A12 backlog: 13 issues assigned, including a P0 bug (#489) that was supposedly crashing user sessions with 422 errors. I had an inkling that a lot of the recent alpha testing and fixing may have already addressed a few of these, so I asked the Lead Developer to investigate.</p><h3>The stale backlog</h3><p>The investigation took minutes. Issue #489, the P0 bug? Already fixed. December 22, commit 859197ba. The fix was live on both main and production branches.</p><p>Issue #486, a test failure? Also fixed. December 27, commit 1afc5e26.</p><p>Two of three “bugs” weren’t bugs anymore. They were stale tracking. The fixes had shipped, but nobody had closed the issues.</p><p>This is a pattern worth naming. Work happens, fixes deploy, issues stay open. The backlog becomes a fossil record of past problems rather than a map of current ones. We’d been carrying phantom bugs.</p><h3>The health dashboard</h3><p>With the phantom bugs cleared, we moved to real work: Issue #530, the Integration Health Check Dashboard.</p><p>This was foundation work for Alpha Setup. Before users could configure their integrations, they needed to see what was working and what wasn’t. Before <em>we</em> could test the setup wizards, we needed visibility into connection status.</p><p>The implementation came in three phases:</p><p><strong>Phase 1: Dashboard UI</strong> (576 lines). Overall status display with health icons. Individual cards for Notion, Slack, GitHub, Calendar. Status dots per integration. Test buttons with loading states. A “Test All Connections” button for bulk testing.</p><p><strong>Phase 2: Health Check API</strong> (453 lines). Three endpoints: overall health, test single integration, test all integrations. Pydantic models for type safety. Response formatting that humans could actually read.</p><p><strong>Phase 3: Error Guidance System</strong>. The part that mattered most. Specific fix suggestions for common errors. “Notion: Invalid API key” with instructions. “Slack: Token expired” with refresh steps. “Calendar: Auth failed, MCP not running” with the actual solution.</p><p>By 3 PM on New Year’s Eve, the dashboard was complete. Ready for commit.</p><h3>The OAuth sprint</h3><p>New Year’s Day brought the OAuth integrations. Slack first, then Calendar. It’s been bugging me that alpha testers have to type in ginormous API keys. Yes, I made them a nice GUI to do it with. Yes, I store the keys in their keychain and not in the clear in their environment, yay! But still, OAuth is just plain easier. We use it all the time now without even noticing.</p><p>Time to get into the details:</p><p>Slack followed the pattern we’d established: TDD scaffolding (10 tests), OAuth endpoints (3), UI integration, CLI update, integration testing (22 tests total). Five phases, clean execution.</p><p>Calendar was different. Google Calendar uses file-based OAuth by default — credentials.json and token.json sitting on disk. Desktop pattern. Not web-friendly.</p><p>We made a decision: implement full web OAuth instead, matching the Slack pattern. Standard redirect flow. User-friendly for web setup.</p><p>That required me to create Google Cloud OAuth credentials first — a PM prerequisite before implementation could start. Once the environment variables were set, the implementation followed the same five-phase structure as Slack: 12 tests, OAuth handler service, backend endpoints, frontend UI, adapter integration.</p><p>By the end of New Year’s Day: both OAuth flows complete, 34 tests passing, ready for end-to-end browser testing.</p><h3>The methodology enhancement</h3><p>Parallel to the OAuth work, we updated the omnibus log methodology. The Document Manager had synthesized seven days of logs (Dec 25–31), and the synthesis revealed a gap: we were compressing HIGH-COMPLEXITY days the same way we compressed STANDARD days.</p><p>The enhancement added specific guidance: space allocation (250 timeline + 280 executive summary for HIGH-COMPLEXITY vs. 60 + 200 for STANDARD), compression ratios (20–30% for HIGH-COMPLEXITY, more aggressive for STANDARD), red flags (compression under 30%, missing workstreams, collapsed events).</p><p>We also “reinflated” a few overly compressed recent complex days.</p><h3>The shipping news</h3><p>To wrap up the day, my Chief of Staff helped me pull together and publish <a href="https://www.linkedin.com/pulse/weekly-ship-023-capability-sprint-christian-crumlish-02ttc">Weekly Ship #023</a> that evening: “The capability sprint.” Coverage from December 19–25. We use it all the time now without even noticing.</p><p name="c85b" id="c85b" class="graf graf--p graf-after--p">Time to get into the details:</p><p name="37ef" id="37ef" class="graf graf--p graf-after--p">Slack followed the pattern we’d established: TDD scaffolding (10 tests), OAuth endpoints (3), UI integration, CLI update, integration testing (22 tests total). Five phases, clean execution.</p><p name="b118" id="b118" class="graf graf--p graf-after--p">Calendar was different. Google Calendar uses file-based OAuth by default — <code class="markup--code markup--p-code">credentials.json</code> and <code class="markup--code markup--p-code">token.json</code> sitting on disk. Desktop pattern. Not web-friendly.</p><p name="12fb" id="12fb" class="graf graf--p graf-after--p">We made a decision: implement full web OAuth instead, matching the Slack pattern. Standard redirect flow. User-friendly for web setup.</p><p name="db5e" id="db5e" class="graf graf--p graf-after--p">That required me to create Google Cloud OAuth credentials first — a PM prerequisite before implementation could start. Once the environment variables were set, the implementation followed the same five-phase structure as Slack: 12 tests, OAuth handler service, backend endpoints, frontend UI, adapter integration.</p><p name="b4df" id="b4df" class="graf graf--p graf-after--p">By the end of New Year’s Day: both OAuth flows complete, 34 tests passing, ready for end-to-end browser testing.</p><h3 name="1da2" id="1da2" class="graf graf--h3 graf-after--p">The methodology enhancement</h3><p name="5a5b" id="5a5b" class="graf graf--p graf-after--h3">Parallel to the OAuth work, we updated the omnibus log methodology. The Document Manager had synthesized seven days of logs (Dec 25–31), and the synthesis revealed a gap: we were compressing HIGH-COMPLEXITY days the same way we compressed STANDARD days.</p><p name="2b12" id="2b12" class="graf graf--p graf-after--p">The enhancement added specific guidance: space allocation (250 timeline + 280 executive summary for HIGH-COMPLEXITY vs. 60 + 200 for STANDARD), compression ratios (20–30% for HIGH-COMPLEXITY, more aggressive for STANDARD), red flags (compression under 30%, missing workstreams, collapsed events).</p><p name="3bf6" id="3bf6" class="graf graf--p graf-after--p">We also “reinflated” a few overly compressed recent complex days.</p><h3 name="129a" id="129a" class="graf graf--h3 graf-after--p">The shipping news</h3><p name="565c" id="565c" class="graf graf--p graf-after--h3 graf--trailing">To wrap up the day, my Chief of Staff helped me pull together and publish <a href="https://www.linkedin.com/pulse/weekly-ship-023-capability-sprint-christian-crumlish-02ttc" data-href="https://www.linkedin.com/pulse/weekly-ship-023-capability-sprint-christian-crumlish-02ttc" class="markup--anchor markup--p-anchor" rel="noopener" target="_blank">Weekly Ship #023</a> that evening: “The capability sprint.” Coverage from December 19–25.
