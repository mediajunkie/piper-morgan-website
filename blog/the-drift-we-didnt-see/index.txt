But fixing one comparison revealed the next problem.</p><h3>The widening gyre</h3><p>Once we started looking for timezone drift, we found it everywhere: 239 utcnow() calls needed updating. Each investigation revealed adjacent issues. The scope kept expanding. We ran a systematic audit. The answer: 47 database columns using timestamp without time zone when they should have been timestamptz. Dozens of code paths using datetime.utcnow() when they should have been using timezone-aware functions. The drift had been there since the beginning, invisible because it was everywhere.</p><h3>The schema migration</h3><p>The fix required touching the database schema itself. This always scares me a bit, but I remind myself, we’re still in alpha! We have no real users to inconvenience.</p><p>We updated the 47 columns. Every timestamp without time zone converted to timestamp with time zone. Every stored datetime now explicitly UTC.</p><p>The migration ran cleanly — PostgreSQL handles the conversion gracefully when the data is already effectively UTC. But the migration was only half the fix. The code still needed to use UTC-aware operations.</p><p>We created datetime_utils.py—a utility module with functions for timezone-aware datetime handling: utc_now() for the current time in UTC, ensure_utc() to normalize incoming datetimes, is_timezone_aware() to validate. Then we updated every call site. 80+ files. Every datetime.utcnow() examined. Every comparison audited.</p><h3>What drift looks like</h3><p>Timezone drift didn’t announce itself. There are no error messages, no stack traces, no logs saying “WARNING: comparing apples to oranges.” The code runs. The comparisons complete. The results are wrong. Sometime in hidden ways and in other cases more blatantly. I’d get shown tomorrow’s calendar because it’s already tomorrow in the UK, for example!</p><p>This is the insidious nature of infrastructure drift. The system keeps working. The tests pass — because the tests were written with the same assumptions as the code. The bugs hide in the gap between “runs without errors” and “produces correct results.”</p><p>We only caught it because warnings appeared during an unrelated debugging session. If I hadn’t noticed them scrolling by in the terminal, or hadn’t asked about them, we might have run for months with subtle datetime mismatches, gradually accumulating invisible issues.</p><h3>The cascade pattern</h3><p>Each investigation revealed adjacent issues. That’s not a bug in the methodology — it’s a feature. Pulling one thread reveals others. The alternative is fixing bugs one at a time, never seeing the systemic pattern, patching symptoms while the root cause persists.</p><p>The work also led to discovering 3 pre-existing test failures that were “not related to this issue but that we then fixed afterward. The investigation that led to a net improvement in the codebase and not just a remediation.</p><p>The todo debugging led to noticing warnings led to the schema audit led to the 47-column migration. Each step was discovery, not scope creep. We weren’t adding work — we were finding work that was always there, hidden by its own consistency.</p><p>One day of investigation. 80+ files modified. 15 new tests for the datetime utilities. One comprehensive schema migration. The drift we didn’t see is now the infrastructure we can trust.</p><h3>What we couldn’t have known</h3><p>The timestamps were set up this way at the beginning. Naive datetimes are the Python default. timestamp without time zone is a valid PostgreSQL choice. Each individual decision was reasonable. The drift accumulated silently, one reasonable decision at a time.</p><p>This is why infrastructure audits matter. Not because the original decisions were wrong, but because small choices compound. A timezone here, a naive datetime there — each one fine in isolation, together creating systemic drift.</p><p>We didn’t know to look for timezone problems until warnings appeared during unrelated work. Now we have explicit UTC handling, schema enforcement, and utility functions that make the right choice the easy choice. The next developer won’t have to discover this the hard way.</p><h3>The test that passes</h3><p>One day, 47 columns, 80 files. That’s the math of infrastructure drift: a small symptom reveals a systemic problem. The fix is never as contained as the symptom suggests.</p><p>But now the drift is visible. Now it’s fixed. Now the timestamps mean what they say. And one more piece of Piper Morgan is factual, reliable, and stable.</p><p><em>Next up on Building Piper Morgan we flash back to November 23 for an insight article, “Investigation as Investment” about a bug classification system and how investigation time prevents wrong turns.</em></p><p><em>Have you experienced infrastructure drift — problems that hid because they were too consistent to notice? The system keeps working. The tests pass — because the tests were written with the same assumptions as the code. The bugs hide in the gap between “runs without errors” and “produces correct results.”</p><p name="aa07" id="aa07" class="graf graf--p graf-after--p">We only caught it because warnings appeared during an unrelated debugging session. If I hadn’t noticed them scrolling by in the terminal, or hadn’t asked about them, we might have run for months with subtle datetime mismatches, gradually accumulating invisible issues.</p><h3 name="36a2" id="36a2" class="graf graf--h3 graf-after--p">The cascade pattern</h3><p name="eaca" id="eaca" class="graf graf--p graf-after--h3">Each investigation revealed adjacent issues. That’s not a bug in the methodology — it’s a feature. Pulling one thread reveals others. The alternative is fixing bugs one at a time, never seeing the systemic pattern, patching symptoms while the root cause persists.</p><p name="9758" id="9758" class="graf graf--p graf-after--p">The work also led to discovering 3 pre-existing test failures that were “not related to this issue but that we then fixed afterward. The investigation that led to a net improvement in the codebase and not just a remediation.</p><p name="e3a7" id="e3a7" class="graf graf--p graf-after--p">The todo debugging led to noticing warnings led to the schema audit led to the 47-column migration. Each step was discovery, not scope creep. We weren’t adding work — we were finding work that was always there, hidden by its own consistency.</p><p name="9adb" id="9adb" class="graf graf--p graf-after--p">One day of investigation. 80+ files modified. 15 new tests for the datetime utilities. One comprehensive schema migration. The drift we didn’t see is now the infrastructure we can trust.</p><h3 name="9400" id="9400" class="graf graf--h3 graf-after--p">What we couldn’t have known</h3><p name="daec" id="daec" class="graf graf--p graf-after--h3">The timestamps were set up this way at the beginning. Naive datetimes are the Python default. <code class="markup--code markup--p-code">timestamp without time zone</code> is a valid PostgreSQL choice. Each individual decision was reasonable. The drift accumulated silently, one reasonable decision at a time.</p><p name="aadc" id="aadc" class="graf graf--p graf-after--p">This is why infrastructure audits matter. Not because the original decisions were wrong, but because small choices compound. A timezone here, a naive datetime there — each one fine in isolation, together creating systemic drift.</p><p name="210a" id="210a" class="graf graf--p graf-after--p">We didn’t know to look for timezone problems until warnings appeared during unrelated work. Now we have explicit UTC handling, schema enforcement, and utility functions that make the right choice the easy choice. The next developer won’t have to discover this the hard way.</p><h3 name="2a65" id="2a65" class="graf graf--h3 graf-after--p">The test that passes</h3><p name="d7aa" id="d7aa" class="graf graf--p graf-after--h3">One day, 47 columns, 80 files. That’s the math of infrastructure drift: a small symptom reveals a systemic problem. The fix is never as contained as the symptom suggests.</p><p name="a25d" id="a25d" class="graf graf--p graf-after--p graf--trailing">But now the drift is visible. Now it’s fixed.
