1a:T1f48,<p><em>February 25 – May 12, 2026</em></p>
<p>Ever since I gave my agents mailboxes, I've had to remind them manually to check their inboxes for new memos from other agents. I try to remember to nudge them at session start and whenever I notice that they have new mail, including after compaction.</p>
<p>Context compaction is what happens when a conversation with Claude gets too long. The system summarizes earlier content to make room for new work. It's necessary, but it has a cost — details get lost. Things that were discussed earlier might not survive the summary.</p>
<p>The mailbox rule existed because of this cost. Important information that arrived during a session might vanish during compaction. If you don't check the mailbox after compaction, you might miss something critical that was sent to you earlier.</p>
<p>It was a good rule. We agreed it was important. And my agents kept forgetting to follow it.</p>
<h1>The compliance problem</h1>
<p>Rules that depend on memory have a structural weakness: memory fails. Humans forget. AI agents lose context to compaction. The mechanism differs but the outcome is the same — a rule you agreed to follow doesn't fire.</p>
<p>It's not about discipline or caring. My Lead Developer agent cared deeply about checking the mailbox. After a long implementation session, context compacts, and the agent is picking up where it left off with fresh context — the mailbox check often just doesn't happen. The rule exists. The intention exists. The behavior doesn't.</p>
<p>We tried reminders. We added it to the session protocols. We documented it prominently. Compliance improved, but it never reached reliability.</p>
<p>The rule was right. The enforcement mechanism was wrong.</p>
<h1>Gadzooks! enter Claude's hooks...</h1>
<p>So now we've built a small shell script — the session-start hook — that runs automatically before every agent session begins. Four checks, executed without anyone having to remember:</p>
<p><ol><li><strong>Session log continuity</strong> — find today's log if one exists, so the agent can resume rather than restart</li><li><strong>Mailbox check</strong> — count unread messages across all role inboxes</li><li><strong>Briefing freshness</strong> — warn if the project's current-state briefing is more than seven days stale</li><li><strong>Role identity</strong> — remind the agent which role they're playing in this session</li></ol></p>
<p>The checks that kept getting forgotten now happen automatically. Not because anyone remembers to do them — because they're infrastructure.</p>
<p>The mailbox rule didn't change. What changed was its enforcement layer. It moved from protocol (something you're supposed to do) to infrastructure (something that happens).</p>
<h1>Protocol vs infrastructure</h1>
<p>This distinction matters more than it might seem.</p>
<p><strong>Protocols</strong> are agreements about behavior. They're documented, discussed, agreed upon. They depend on people remembering and choosing to follow them. They fail when attention lapses, when context shifts, when the pressure of immediate work crowds out the discipline of process.</p>
<p><strong>Infrastructure</strong> is environment that shapes behavior. It doesn't ask for compliance — it creates conditions where the right thing happens automatically. You don't decide to check the mailbox; the system surfaces unread items before you can proceed.</p>
<p>Protocols require ongoing effort. Infrastructure requires upfront investment but then runs on its own.</p>
<h1>The graduation pattern</h1>
<p>Looking back at how Piper Morgan's methodology evolved, I see a pattern:</p>
<p><strong>Stage 1: Discovery.</strong> We notice a problem. Context gets lost after compaction. Important memos get missed. Something keeps going wrong.</p>
<p><strong>Stage 2: Protocol.</strong> We create a rule to address it. "Always check mailbox after compaction." The rule works when followed.</p>
<p><strong>Stage 3: Failure.</strong> The rule isn't followed reliably. Not because people disagree with it, but because protocol compliance is inherently fragile.</p>
<p><strong>Stage 4: Infrastructure.</strong> We encode the rule into the environment. The check happens automatically. Compliance becomes structural rather than behavioral.</p>
<p>This is methodology graduating to infrastructure. The insight stays the same — checking the mailbox matters. The enforcement mechanism matures from "remember to do this" to "this happens."</p>
<h2>What else could graduate?</h2>
<p>Once you see this pattern, you start noticing other protocols waiting for infrastructure:</p>
<p><strong>"Always run tests before committing"</strong> — This is a protocol for most teams. For teams with pre-commit hooks, it's infrastructure. The graduation is a shell script.</p>
<p><strong>"Document architectural decisions"</strong> — Protocol: remember to write Architectural Decision Records (ADRs). Infrastructure: templates that prompt for decisions, continuous integration (CI) checks that flag undocumented changes.</p>
<p><strong>"Review code before merging"</strong> — Protocol: ask someone to review. Infrastructure: branch protection rules that require approval.</p>
<p><strong>"Check for breaking changes"</strong> — Protocol: think carefully about compatibility. Infrastructure: automated compatibility testing in CI.</p>
<p>Each of these starts as something people agree to do. Each can become something the environment ensures happens.</p>
<h1>The cost of graduation</h1>
<p>Infrastructure isn't free. The session-start hook took time to implement. It requires maintenance. It adds complexity to the development environment.</p>
<p>Not every protocol should graduate. Some rules are context-dependent in ways that resist automation. Some are too rare to justify the investment. Some need human judgment that can't be encoded.</p>
<p>The question isn't "can this become infrastructure?" — almost anything can, with enough effort. The question is "should this become infrastructure?" The answer depends on:</p>
<ul><li>How often does the protocol need to be followed?</li><li>How reliably do people follow it without enforcement?</li><li>How severe are the consequences of non-compliance?</li><li>How complex would the infrastructure be to build and maintain?</li></ul>
<p>For the mailbox check: frequently needed, unreliably followed, consequences were missed information and duplicated work, infrastructure was a simple shell script. Easy graduation.</p>
<p>For something like "make good architectural decisions" — frequently needed, but what would infrastructure even look like? Some things remain protocols because they require judgment that can't be automated.</p>
<h1>Implications for your methodology</h1>
<p>Building Piper Morgan has taught me to think about methodology in two layers:</p>
<p><strong>The insight layer</strong>: What practices make development better? What should we do?</p>
<p><strong>The enforcement layer</strong>: How do we ensure the practices actually happen? Protocol or infrastructure?</p>
<p>Early in a project, everything is protocol. You're discovering what works, documenting agreements, building shared understanding. Protocols are fast to create and easy to change.</p>
<p>As practices stabilize, the best ones graduate. The things that matter most, that need to happen reliably, that keep getting forgotten — those become infrastructure. The insight stays the same. The enforcement matures.</p>
<p>The session-start hook isn't a big feature. But it represents something important: methodology that doesn't depend on memory. Rules that enforce themselves. A system that does the right thing automatically, not because anyone decided to do it, but because the environment makes it happen.</p>
<p>That's the graduation worth pursuing.</p>
