1b:T2e35,<p><em>February 28</em></p><figure><img alt="Four distinct nonhuman AI beings — a ghostlike form, glowing neural blob, floating machine-brain, and dolphin-like creature — join one by one in song as a human conductor guides them, their voices merging into a single structured form above." src="https://cdn-images-1.medium.com/max/1024/1*zcWnMTMffFS4S6BdgS0MRA.png" /><figcaption>“Pure harmony!”</figcaption></figure><p>The conversation lifecycle spec was stuck. It wasn’t blocked by technical complexity or missing requirements. It was blocked by the kind of problem that happens when a specification touches multiple domains at once: nobody owned the whole thing, so nobody finished it.</p><p>Then we tried something different. Four agents, each with a distinct perspective, worked the spec in sequence. By evening, a specification that had lingered undone was ready for implementation.</p><h3>The spec that needed too many things</h3><p>Issue #858 — “Conversation Lifecycle Specification” — had been open since early February. The concept was straightforward: define how conversations begin, persist, and end. When does a conversation start? What happens when a user returns after being away? How do we handle the graceful ending of a conversation versus abandonment?</p><p>Simple questions. But the answers touched everything:</p><p><strong>Backend architecture</strong>: How is conversation state stored? What triggers lifecycle transitions? How does the system know when a user has returned versus started fresh?</p><p><strong>User experience</strong>: What does the user see at each stage? How do we surface conversation history? What happens when they click on an old conversation?</p><p><strong>Product strategy</strong>: Should conversations be discrete sessions or persistent threads? How does this interact with the “colleague” model — colleagues remember context but don’t remember everything?</p><p><strong>Implementation constraints</strong>: What does the existing codebase support? What would require new infrastructure? What’s feasible in the current sprint versus later?</p><p>The spec needed backend depth and UX sensitivity and product vision and implementation awareness. It needed someone who understood all four domains well enough to make tradeoffs between them.</p><p>We didn’t have that person. We had four agents who each understood one domain deeply.</p><h3>The pipeline</h3><p>Rather than trying to find one agent who could do everything, we built a pipeline. Each agent added what they were best at, in sequence:</p><p><strong>First: Research.</strong> The Docs agent conducted deep research across 14 documents and 22 GitHub issues. Not writing the spec — mapping the territory. What had we already decided? What was assumed but unstated? What constraints existed in the codebase that nobody had written down?</p><p>The research surfaced things nobody remembered: early decisions about conversation threading, abandoned approaches, implicit assumptions in how the existing code handled sessions. It also flagged an important compatibility question — how would this spec interact with Ted Nadeau’s MultiChat PRD, which assumed multi-entity conversations that our current design didn’t fully support?</p><p><strong>Second: Review and direction.</strong> I reviewed the research report and asked for a specific additional pass: check everything against Ted’s PRD and flag incompatibilities. The research agent did a second round, producing a compatibility assessment.</p><p><strong>Third: UX guidance.</strong> The CXO took the research and translated it into user experience terms. What states should be visible to users? (Fewer than exist internally — users don’t need to see technical states.) What should the sidebar show? (The “entity surface” framing — conversations in context of what they’re about, not a pure archive.) Where should conversation boundaries fall? (Calendar days as natural soft-close points.)</p><p>The CXO’s memo didn’t touch backend architecture at all. It was pure UX: what users see, what they can do, how it should feel. But it gave the specification a user-centered frame that the research alone couldn’t provide.</p><p><strong>Fourth: Structure and alignment.</strong> The PPM reviewed the CXO memo and added product structure. A formal section breakdown for the spec. Explicit decision points that needed PM sign-off. An “anti-flattening checklist” to ensure implementation wouldn’t simplify away the nuances.</p><p>The PPM also contributed a specific feature insight: “continue yesterday” as an affordance. When a user returns after being away, should Piper offer to continue where they left off? The research had flagged conversation resumption as a question. The CXO had addressed how it should feel. The PPM made it concrete: yes, this is a feature, and here’s how to spec it.</p><h3>What each voice added</h3><p>Looking back at the pipeline, each agent contributed something the others couldn’t:</p><p><strong>Research</strong> provided the foundation — you can’t make good decisions without knowing what’s already been decided, what constraints exist, what prior art exists. But research alone doesn’t make decisions. It maps options.</p><p><strong>PM direction</strong> (me, in this case) provided focus. The research report was comprehensive but needed prioritization. The compatibility pass was assigned because I knew it mattered for Ted’s work. Direction without research would be guessing; research without direction would be sprawling.</p><p><strong>UX guidance</strong> provided the user lens. The CXO wasn’t constrained by backend possibilities or product roadmaps. They asked the pure question: what should users experience? That freedom produced insights (calendar-day boundaries, entity surface framing) that wouldn’t emerge from more constrained perspectives.</p><p><strong>Product structure</strong> provided implementability. The PPM took UX ideals and made them buildable — section breakdowns, decision points, checklist items. Vision without structure remains vision; structure made it a spec.</p><p>None of these perspectives was complete. Each needed the others to produce something useful. The pipeline worked because it let each voice contribute its strength without requiring any voice to have all strengths.</p><h3>The same day</h3><p>The most surprising thing wasn’t that the pipeline worked — it was how fast it worked.</p><p>The research started in the morning. The PM review happened around midday. The CXO memo was complete by mid-afternoon. The PPM synthesis was done by evening.</p><p>A spec that had been stuck for weeks was ready for implementation in one day.</p><p>Part of this was coordination. Having a clear handoff chain meant nobody waited for unclear dependencies. Research knew to produce a report for PM review. PM knew to direct it to CXO. CXO knew to produce guidance for PPM. Each step had a clear input and clear output.</p><p>Part of it was scoping. Each agent did one thing, not everything. The CXO wasn’t trying to also do backend architecture. The PPM wasn’t trying to also do user research. Narrow scope meant faster completion meant faster handoff.</p><p>Part of it was trust. Each agent trusted that the previous step was done well. The CXO didn’t re-research; they built on research. The PPM didn’t re-do UX; they structured it. Trust in handoffs eliminated redundant work.</p><h3>Multi-agent coordination as operating model</h3><p>We’ve been running a multi-agent setup since the beginning of Piper Morgan. Different specialized agents — Lead Developer, Chief Architect, CXO, PPM, and others — each handling their domain. But for most tasks, that means one agent does most of the work, with occasional handoffs for review or input.</p><p>The conversation lifecycle spec was different. It required genuine synthesis — multiple perspectives contributing to a single artifact, in sequence, each building on what came before.</p><p>This is harder than it sounds. Multi-agent systems can easily become multi-agent chaos: unclear ownership, redundant work, gaps where everyone assumed someone else was handling it. The stuck spec was stuck precisely because it sat in the intersection of domains, owned by everyone and therefore no one.</p><p>The pipeline approach solved this by making the sequence explicit. Not “multiple agents collaborate” but “agent A produces X, which goes to agent B who produces Y, which goes to agent C.” Clear inputs. Clear outputs. Clear handoffs.</p><p>I think of it as an operating model question, not a technical one. How do you structure work so that specialized perspectives combine into coherent outcomes? The answer isn’t “get a generalist who can do everything.” The answer is “design handoffs that preserve each perspective’s contribution while building toward synthesis.”</p><h3>What the spec became</h3><p>By the end of the day, issue #858 had:</p><ul><li>A research foundation covering 14 documents and 22 issues</li><li>A compatibility assessment against external requirements</li><li>A UX guidance memo defining user-visible states and interaction patterns</li><li>A product structure document with sections, decisions, and anti-flattening checklist</li><li>Approval from CXO, PPM, and (after review) the Chief Architect</li></ul><p>The Lead Developer could now implement with clarity. Not just “build conversation lifecycle” but specific decisions about state visibility, sidebar behavior, session boundaries, and resumption affordances. The spec answered the questions that would have come up during implementation, before implementation started.</p><p>Was it a perfect spec? Probably not. Implementation would surface edge cases nobody anticipated. But it was a complete spec — complete enough to build against, complete enough to revise from.</p><h3>The operating model lesson</h3><p>The conversation lifecycle spec could have been written by one person who understood all four domains. Such people exist. They’re rare and expensive and bottlenecks.</p><p>The pipeline approach distributed the work across four agents who each understood one domain. It took the same total effort (maybe less, since each agent worked in their strength) and completed in one day instead of weeks.</p><p>This is the operating model question that Piper Morgan keeps teaching me: how do you get coherent outcomes from distributed expertise? Not by finding unicorns who have all the expertise. By designing systems where specialized expertise combines.</p><p>Multi-agent coordination isn’t just a technical architecture. It’s an answer to a fundamental question about how work gets done. The conversation lifecycle spec was a small example. But the pattern scales: research → direction → domain expertise → structure → implementation. Each step adds value. Each handoff preserves it.</p><p>Four voices. One spec. Done in a day.</p><p><em>Next on Building Piper Morgan, The Gate Closes, as we finally track down all the loose ends needed for conversational glue.</em></p><p><em>Have you ever unstuck work by changing the structure of collaboration rather than adding more effort?</em></p><img src="https://medium.com/_/stat?event=post.clientViewed&referrerSource=full_rss&postId=168e71571f6b" width="1" height="1" alt=""><hr><p><a href="https://medium.com/building-piper-morgan/four-voices-one-spec-168e71571f6b">Four Voices, One Spec</a> was originally published in <a href="https://medium.com/building-piper-morgan">Building Piper Morgan</a> on Medium, where people are continuing the conversation by highlighting and responding to this story.</p>1c:T3c63,<section name="386e" class="section section--body section--first"><div class="section-divider"><hr class="section-divider"></div><div class="section-content"><div class="section-inner sectionLayout--insetColumn"><h3 name="aeb1" id="aeb1" class="graf graf--h3 graf--leading graf--title">Four Voices, One Spec</h3><p name="ef77" id="ef77" class="graf graf--p graf-after--h3"><em class="markup--em markup--p-em">February 28</em></p><figure name="9738" id="9738" class="graf graf--figure graf--startsWithDoubleQuote graf-after--p"><img class="graf-image" data-image-id="1*zcWnMTMffFS4S6BdgS0MRA.png" data-width="1536" data-height="1024" alt="Four distinct nonhuman AI beings — a ghostlike form, glowing neural blob, floating machine-brain, and dolphin-like creature — join one by one in song as a human conductor guides them, their voices merging into a single structured form above." src="https://cdn-images-1.medium.com/max/800/1*zcWnMTMffFS4S6BdgS0MRA.png"><figcaption class="imageCaption">“Pure harmony!”</figcaption></figure><p name="8993" id="8993" class="graf graf--p graf--hasDropCapModel graf--hasDropCap graf-after--figure"><span class="graf-dropCap">T</span>he conversation lifecycle spec was stuck. It was blocked by the kind of problem that happens when a specification touches multiple domains at once: nobody owned the whole thing, so nobody finished it.</p><p name="e498" id="e498" class="graf graf--p graf-after--p">Then we tried something different. Four agents, each with a distinct perspective, worked the spec in sequence. By evening, a specification that had lingered undone was ready for implementation.</p><h3 name="c805" id="c805" class="graf graf--h3 graf-after--p">The spec that needed too many things</h3><p name="967a" id="967a" class="graf graf--p graf-after--h3">Issue #858 — “Conversation Lifecycle Specification” — had been open since early February. The concept was straightforward: define how conversations begin, persist, and end. When does a conversation start? What happens when a user returns after being away? How do we handle the graceful ending of a conversation versus abandonment?</p><p name="257c" id="257c" class="graf graf--p graf-after--p">Simple questions. But the answers touched everything:</p><p name="7123" id="7123" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Backend architecture</strong>: How is conversation state stored? What triggers lifecycle transitions? How does the system know when a user has returned versus started fresh?</p><p name="6a83" id="6a83" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">User experience</strong>: What does the user see at each stage? How do we surface conversation history? What happens when they click on an old conversation?</p><p name="6064" id="6064" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Product strategy</strong>: Should conversations be discrete sessions or persistent threads? How does this interact with the “colleague” model — colleagues remember context but don’t remember everything?</p><p name="b0c8" id="b0c8" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Implementation constraints</strong>: What does the existing codebase support? What would require new infrastructure? What’s feasible in the current sprint versus later?</p><p name="33be" id="33be" class="graf graf--p graf-after--p">The spec needed backend depth and UX sensitivity and product vision and implementation awareness. It needed someone who understood all four domains well enough to make tradeoffs between them.</p><p name="60f7" id="60f7" class="graf graf--p graf-after--p">We didn’t have that person. We had four agents who each understood one domain deeply.</p><h3 name="2273" id="2273" class="graf graf--h3 graf-after--p">The pipeline</h3><p name="177d" id="177d" class="graf graf--p graf-after--h3">Rather than trying to find one agent who could do everything, we built a pipeline. Each agent added what they were best at, in sequence:</p><p name="dd12" id="dd12" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">First: Research.</strong> The Docs agent conducted deep research across 14 documents and 22 GitHub issues. Not writing the spec — mapping the territory. What had we already decided? What was assumed but unstated? What constraints existed in the codebase that nobody had written down?</p><p name="2d85" id="2d85" class="graf graf--p graf-after--p">The research surfaced things nobody remembered: early decisions about conversation threading, abandoned approaches, implicit assumptions in how the existing code handled sessions. It also flagged an important compatibility question — how would this spec interact with Ted Nadeau’s MultiChat PRD, which assumed multi-entity conversations that our current design didn’t fully support?</p><p name="e839" id="e839" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Second: Review and direction.</strong> I reviewed the research report and asked for a specific additional pass: check everything against Ted’s PRD and flag incompatibilities. The research agent did a second round, producing a compatibility assessment.</p><p name="df5c" id="df5c" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Third: UX guidance.</strong> The CXO took the research and translated it into user experience terms. What states should be visible to users? (Fewer than exist internally — users don’t need to see technical states.) What should the sidebar show? (The “entity surface” framing — conversations in context of what they’re about, not a pure archive.) Where should conversation boundaries fall? (Calendar days as natural soft-close points.)</p><p name="51b4" id="51b4" class="graf graf--p graf-after--p">The CXO’s memo didn’t touch backend architecture at all. It was pure UX: what users see, what they can do, how it should feel. But it gave the specification a user-centered frame that the research alone couldn’t provide.</p><p name="fc81" id="fc81" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Fourth: Structure and alignment.</strong> The PPM reviewed the CXO memo and added product structure. A formal section breakdown for the spec. Explicit decision points that needed PM sign-off. An “anti-flattening checklist” to ensure implementation wouldn’t simplify away the nuances.</p><p name="b7bc" id="b7bc" class="graf graf--p graf-after--p">The PPM also contributed a specific feature insight: “continue yesterday” as an affordance. When a user returns after being away, should Piper offer to continue where they left off? The research had flagged conversation resumption as a question. The CXO had addressed how it should feel. The PPM made it concrete: yes, this is a feature, and here’s how to spec it.</p><h3 name="78bc" id="78bc" class="graf graf--h3 graf-after--p">What each voice added</h3><p name="bb43" id="bb43" class="graf graf--p graf-after--h3">Looking back at the pipeline, each agent contributed something the others couldn’t:</p><p name="a7f8" id="a7f8" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Research</strong> provided the foundation — you can’t make good decisions without knowing what’s already been decided, what constraints exist, what prior art exists. But research alone doesn’t make decisions. It maps options.</p><p name="ae58" id="ae58" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">PM direction</strong> (me, in this case) provided focus. The research report was comprehensive but needed prioritization. The compatibility pass was assigned because I knew it mattered for Ted’s work. Direction without research would be guessing; research without direction would be sprawling.</p><p name="e6f3" id="e6f3" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">UX guidance</strong> provided the user lens. The CXO wasn’t constrained by backend possibilities or product roadmaps. They asked the pure question: what should users experience? That freedom produced insights (calendar-day boundaries, entity surface framing) that wouldn’t emerge from more constrained perspectives.</p><p name="919c" id="919c" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Product structure</strong> provided implementability. The PPM took UX ideals and made them buildable — section breakdowns, decision points, checklist items. Vision without structure remains vision; structure made it a spec.</p><p name="7a36" id="7a36" class="graf graf--p graf-after--p">None of these perspectives was complete. Each needed the others to produce something useful. The pipeline worked because it let each voice contribute its strength without requiring any voice to have all strengths.</p><h3 name="9928" id="9928" class="graf graf--h3 graf-after--p">The same day</h3><p name="a23f" id="a23f" class="graf graf--p graf-after--h3">The most surprising thing wasn’t that the pipeline worked — it was how fast it worked.</p><p name="a607" id="a607" class="graf graf--p graf-after--p">The research started in the morning. The PM review happened around midday. The CXO memo was complete by mid-afternoon. The PPM synthesis was done by evening.</p><p name="20d6" id="20d6" class="graf graf--p graf-after--p">A spec that had been stuck for weeks was ready for implementation in one day.</p><p name="d350" id="d350" class="graf graf--p graf-after--p">Part of this was coordination. Having a clear handoff chain meant nobody waited for unclear dependencies. Research knew to produce a report for PM review. PM knew to direct it to CXO. CXO knew to produce guidance for PPM. Each step had a clear input and clear output.</p><p name="b58d" id="b58d" class="graf graf--p graf-after--p">Part of it was scoping. Each agent did one thing, not everything. The CXO wasn’t trying to also do backend architecture. The PPM wasn’t trying to also do user research. Narrow scope meant faster completion meant faster handoff.</p><p name="6c34" id="6c34" class="graf graf--p graf-after--p">Part of it was trust. Each agent trusted that the previous step was done well. The CXO didn’t re-research; they built on research. The PPM didn’t re-do UX; they structured it. Trust in handoffs eliminated redundant work.</p><h3 name="2680" id="2680" class="graf graf--h3 graf-after--p">Multi-agent coordination as operating model</h3><p name="ecc4" id="ecc4" class="graf graf--p graf-after--h3">We’ve been running a multi-agent setup since the beginning of Piper Morgan. Different specialized agents — Lead Developer, Chief Architect, CXO, PPM, and others — each handling their domain. But for most tasks, that means one agent does most of the work, with occasional handoffs for review or input.</p><p name="395d" id="395d" class="graf graf--p graf-after--p">The conversation lifecycle spec was different. It required genuine synthesis — multiple perspectives contributing to a single artifact, in sequence, each building on what came before.</p><p name="2cf6" id="2cf6" class="graf graf--p graf-after--p">This is harder than it sounds. Multi-agent systems can easily become multi-agent chaos: unclear ownership, redundant work, gaps where everyone assumed someone else was handling it. The stuck spec was stuck precisely because it sat in the intersection of domains, owned by everyone and therefore no one.</p><p name="d93e" id="d93e" class="graf graf--p graf-after--p">The pipeline approach solved this by making the sequence explicit. Not “multiple agents collaborate” but “agent A produces X, which goes to agent B who produces Y, which goes to agent C.” Clear inputs. Clear outputs. Clear handoffs.</p><p name="98ec" id="98ec" class="graf graf--p graf-after--p">I think of it as an operating model question, not a technical one. How do you structure work so that specialized perspectives combine into coherent outcomes? The answer isn’t “get a generalist who can do everything.” The answer is “design handoffs that preserve each perspective’s contribution while building toward synthesis.”</p><h3 name="eb90" id="eb90" class="graf graf--h3 graf-after--p">What the spec became</h3><p name="cb74" id="cb74" class="graf graf--p graf-after--h3">By the end of the day, issue #858 had:</p><ul class="postList"><li name="6f79" id="6f79" class="graf graf--li graf-after--p">A research foundation covering 14 documents and 22 issues</li><li name="50e5" id="50e5" class="graf graf--li graf-after--li">A compatibility assessment against external requirements</li><li name="b124" id="b124" class="graf graf--li graf-after--li">A UX guidance memo defining user-visible states and interaction patterns</li><li name="21e7" id="21e7" class="graf graf--li graf-after--li">A product structure document with sections, decisions, and anti-flattening checklist</li><li name="8c18" id="8c18" class="graf graf--li graf-after--li">Approval from CXO, PPM, and (after review) the Chief Architect</li></ul><p name="6edb" id="6edb" class="graf graf--p graf-after--li">The Lead Developer could now implement with clarity. Not just “build conversation lifecycle” but specific decisions about state visibility, sidebar behavior, session boundaries, and resumption affordances. The spec answered the questions that would have come up during implementation, before implementation started.</p><p name="dfe1" id="dfe1" class="graf graf--p graf-after--p">Was it a perfect spec? Probably not. Implementation would surface edge cases nobody anticipated. But it was a complete spec — complete enough to build against, complete enough to revise from.</p><h3 name="ca47" id="ca47" class="graf graf--h3 graf-after--p">The operating model lesson</h3><p name="9045" id="9045" class="graf graf--p graf-after--h3">The conversation lifecycle spec could have been written by one person who understood all four domains. Such people exist. They’re rare and expensive and bottlenecks.</p><p name="5357" id="5357" class="graf graf--p graf-after--p">The pipeline approach distributed the work across four agents who each understood one domain. It took the same total effort (maybe less, since each agent worked in their strength) and completed in one day instead of weeks.</p><p name="e266" id="e266" class="graf graf--p graf-after--p">This is the operating model question that Piper Morgan keeps teaching me: how do you get coherent outcomes from distributed expertise? Not by finding unicorns who have all the expertise. By designing systems where specialized expertise combines.</p><p name="b99c" id="b99c" class="graf graf--p graf-after--p">Multi-agent coordination isn’t just a technical architecture. It’s an answer to a fundamental question about how work gets done. The conversation lifecycle spec was a small example. But the pattern scales: research → direction → domain expertise → structure → implementation. Each step adds value. Each handoff preserves it.</p><p name="b1d2" id="b1d2" class="graf graf--p graf-after--p graf--trailing">Four voices. One spec.
