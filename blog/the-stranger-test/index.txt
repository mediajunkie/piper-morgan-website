1b:T2a4b,<figure><img alt="A proud human developer and robot teammate admire their house while holding a clipboard reading “1025 tests — all passing,” as a visiting robot points out obvious flaws like a crooked picture frame and dangling light switch the residents overlooked." src="https://cdn-images-1.medium.com/max/1024/1*Z0sqNE6h0skLax35OCU-zQ.png" /><figcaption>“Uh, fellas”</figcaption></figure><p><em>February 21–24</em></p><p>One thousand twenty-five tests. All passing. Zero failures. The M0 sprint was complete. The wiring pass was done. Nine integration gaps found and fixed. We had feature coverage, integration coverage, unit coverage. The test suite was comprehensive, rigorous, green.</p><p>And then someone logged in as a stranger.</p><h3>The fresh account</h3><p>After a sprint completes, we now do something we call a CXO review — a dedicated session where someone tests the system not as a developer who built it, but as a user who’s encountering it for the first time. The “CXO” in our multi-agent setup handles user experience, but the real value of the role is its the perspective.</p><p>For the M0 review with the CXO I created a fresh alpha account. No conversation history. No established context. No muscle memory from months of development. Just a new user meeting Piper for the first time.</p><p>The results were uncomfortable.</p><h3>What the stranger found</h3><p>The first issue appeared within minutes. The CXO (through me) asked about their calendar — a basic query that should have been straightforward. The system failed silently. No error, no explanation. Just… nothing happened.</p><p>Investigation revealed the cause: a keychain authentication bug. The calendar adapter was looking for credentials under a hardcoded key name, ignoring the user-scoped pattern we’d established elsewhere. For developers — who had credentials stored under the old key from months of testing — everything worked. For a new user with a fresh account, the credentials didn’t exist where the code was looking.</p><p>The tests passed because the test fixtures used the old pattern. The code was wrong, but it was wrong in a way that matched how the tests were written.</p><p>This wasn’t the only issue the stranger found.</p><h3>Five features, two working</h3><p>M0 had five features. The CXO tested each one from a fresh-account perspective:</p><p><strong>Lens tracking</strong> (follow-up recognition): Passed. The system correctly recognized when the user referenced something from earlier in conversation.</p><p><strong>Narrative onboarding</strong> (portfolio introduction): Passed. The “main project” flow worked as designed.</p><p><strong>Soft invocation</strong> (proactive offers): Failed. The patterns were too literal. “The team needs alignment” triggered an offer; “I need to get the team aligned” didn’t. Same intent, different phrasing, broken experience.</p><p><strong>Multi-intent handling</strong>: Blocked. Couldn’t test — depended on calendar queries, which were failing.</p><p><strong>Slot filling</strong>: Blocked. Same dependency, same failure.</p><p>Two out of five features passed. One failed outright. Two couldn’t even be tested because of infrastructure issues.</p><p>One thousand twenty-five tests. All green. Forty percent of the features actually working for a real user.</p><h3>The gap has a name</h3><p>We’d seen this pattern before. Just days earlier, we’d discovered the Assembly Assumption — the belief that individually correct components compose into a correct system. But this was something different.</p><p>The Assembly Assumption is about integration gaps between features. This was about a different kind of gap: the space between what tests verify and what users experience.</p><p>Tests verify slices. They check that specific inputs produce specific outputs under specific conditions. They’re written by developers, using developer assumptions, running in developer environments with developer data.</p><p>Users don’t experience slices. They experience journeys. They come in with no context, no history, no understanding of how the system “should” work. They type natural variations of requests. They have fresh accounts with no legacy data. They encounter the system as it actually is, not as it’s supposed to be.</p><p>The test suite was comprehensive. But it was comprehensively testing the developer’s mental model, not the user’s experience.</p><h3>The bugs that developers can’t see</h3><p>The calendar authentication bug was invisible to developers because developers had valid credentials stored under the old key. The test suite didn’t catch it because the test fixtures were set up the same way. The bug could only surface when someone — a stranger — started fresh.</p><p>The soft invocation patterns were too narrow because they were written to match the examples in the specification. “The team needs alignment” was an example phrase, so it was tested. “I need to get the team aligned” wasn’t an example, so it wasn’t tested. A developer writing tests thinks in terms of specification coverage. A user speaks in terms of natural language variation.</p><p>There was another bug the stranger found: when a new user said “yes” in response to an embedded offer, Piper interpreted it as a greeting. The offer system worked correctly — it detected offers, it tracked pending offers, it handled acceptances. But embedded offers (offers mentioned in the middle of a response, not at the end) weren’t being registered as pending. When the user said “yes,” there was nothing to accept. The system fell back to greeting detection.</p><p>This bug had 1,025 tests around it and not a single one caught it. Because every test either (a) used explicit offers that were properly registered, or (b) tested the greeting system in isolation. The gap existed at the seam between two well-tested systems.</p><h3>The stranger’s gift</h3><p>The CXO session was uncomfortable. Watching features fail that we’d spent days building and testing is never pleasant. But the stranger’s perspective revealed something valuable: a systematic blind spot in how we verify quality.</p><p>Developer testing asks: “Does the code work?”</p><p>Stranger testing asks: “Does the experience work?”</p><p>These are different questions. The code can work while the experience fails. The tests can pass while the user struggles. The coverage can be comprehensive while the gaps persist.</p><p>The stranger doesn’t know what “should” happen. They only know what does happen. That ignorance is a feature, not a bug. It reveals the assumptions we’ve baked into our verification without realizing it.</p><h3>The fixes</h3><p>The four bugs from the stranger’s session were fixed the same evening they were discovered:</p><p><strong>Calendar authentication</strong>: User-scoped keychain keys, matching the pattern established in the earlier security audit. Three new tests to prevent regression.</p><p><strong>Soft invocation patterns</strong>: Broadened to recognize personal agency phrases (“I need to…”) and adjective forms (“get aligned” vs “get alignment”). Five new tests.</p><p><strong>Embedded offer registration</strong>: Offers mentioned anywhere in a response now register as pending, not just offers at the end. Three new tests.</p><p><strong>Issue queries</strong>: Added issue-specific intent patterns to the pre-classifier so “How many open issues?” routes correctly.</p><p>Each fix was small. The calendar bug was a single line change. The soft invocation fix was pattern additions. None of them were architecturally complex.</p><p>But none of them would have been found without the stranger.</p><h3>Making strangeness systematic</h3><p>After the CXO review, we asked the uncomfortable question: why did it take a stranger to find these bugs?</p><p>The answer was process-shaped. Our testing workflow was built around developer convenience. Tests run in environments with established data. Test accounts have history. Test fixtures use patterns that match how developers think about the system.</p><p>We needed a way to make strangeness systematic. To regularly encounter our own system as a stranger would.</p><p>The solution was simple: fresh account testing as a gate requirement. Before any sprint can close, someone has to test the key features from a brand-new account. Not the account that’s been accumulating data for months. Not the fixtures that match developer assumptions. A stranger’s account, encountering the system for the first time.</p><p>It’s not a replacement for the test suite. The 1,025 tests still matter — they catch regressions, verify edge cases, provide confidence during refactoring. But they’re not sufficient. They verify the system we think we built. The stranger verifies the system we actually built.</p><h3>The question that remains</h3><p>One thousand twenty-five tests. All green. Forty percent of features working for a real user.</p><p>That ratio haunts me. Not because the tests were bad — they weren’t. They were thorough, well-designed, covering the cases they were meant to cover. The problem is that “cases they were meant to cover” and “cases users actually encounter” aren’t the same thing.</p><p>How much of any test suite is actually testing user experience, versus testing developer assumptions? How many green dashboards hide systems that strangers would struggle with? How many “comprehensive” coverage numbers are measuring the wrong thing?</p><p>I don’t have a universal answer. But I have a practice now: before declaring something complete, let a stranger try it. Watch what happens when someone encounters your system without your context, your history, your understanding of how it “should” work.</p><p>The stranger will find things you can’t see. That’s not a failure of your testing. It’s a feature of their perspective.</p><p><em>Next on Building Piper Morgan: The Tokens That Vanished — a systematic audit reveals a bug that had been hiding in plain sight for months, silently breaking OAuth authentication.</em></p><p><em>When did a fresh perspective last show you something your expertise was hiding?</em></p><img src="https://medium.com/_/stat?event=post.clientViewed&referrerSource=full_rss&postId=e39247492342" width="1" height="1" alt=""><hr><p><a href="https://medium.com/building-piper-morgan/the-stranger-test-e39247492342">The Stranger Test</a> was originally published in <a href="https://medium.com/building-piper-morgan">Building Piper Morgan</a> on Medium, where people are continuing the conversation by highlighting and responding to this story.</p>1c:T383a,<section name="6c18" class="section section--body section--first"><div class="section-divider"><hr class="section-divider"></div><div class="section-content"><div class="section-inner sectionLayout--insetColumn"><h3 name="d8c4" id="d8c4" class="graf graf--h3 graf--leading graf--title">The Stranger Test</h3><figure name="d2ea" id="d2ea" class="graf graf--figure graf--startsWithDoubleQuote graf-after--h3"><img class="graf-image" data-image-id="1*Z0sqNE6h0skLax35OCU-zQ.png" data-width="1536" data-height="1024" alt="A proud human developer and robot teammate admire their house while holding a clipboard reading “1025 tests — all passing,” as a visiting robot points out obvious flaws like a crooked picture frame and dangling light switch the residents overlooked." src="https://cdn-images-1.medium.com/max/800/1*Z0sqNE6h0skLax35OCU-zQ.png"><figcaption class="imageCaption">“Uh, fellas”</figcaption></figure><p name="bf7d" id="bf7d" class="graf graf--p graf-after--figure"><em class="markup--em markup--p-em">February 21–24</em></p><p name="9e75" id="9e75" class="graf graf--p graf--hasDropCapModel graf--hasDropCap graf-after--p"><span class="graf-dropCap">O</span>ne thousand twenty-five tests. The test suite was comprehensive, rigorous, green.</p><p name="4604" id="4604" class="graf graf--p graf-after--p">And then someone logged in as a stranger.</p><h3 name="6f2f" id="6f2f" class="graf graf--h3 graf-after--p">The fresh account</h3><p name="a21a" id="a21a" class="graf graf--p graf-after--h3">After a sprint completes, we now do something we call a CXO review — a dedicated session where someone tests the system not as a developer who built it, but as a user who’s encountering it for the first time. The “CXO” in our multi-agent setup handles user experience, but the real value of the role is its the perspective.</p><p name="a093" id="a093" class="graf graf--p graf-after--p">For the M0 review with the CXO I created a fresh alpha account. No conversation history. No established context. No muscle memory from months of development. Just a new user meeting Piper for the first time.</p><p name="fabd" id="fabd" class="graf graf--p graf-after--p">The results were uncomfortable.</p><h3 name="f934" id="f934" class="graf graf--h3 graf-after--p">What the stranger found</h3><p name="edcf" id="edcf" class="graf graf--p graf-after--h3">The first issue appeared within minutes. The CXO (through me) asked about their calendar — a basic query that should have been straightforward. The system failed silently. No error, no explanation. Just… nothing happened.</p><p name="72c4" id="72c4" class="graf graf--p graf-after--p">Investigation revealed the cause: a keychain authentication bug. The calendar adapter was looking for credentials under a hardcoded key name, ignoring the user-scoped pattern we’d established elsewhere. For developers — who had credentials stored under the old key from months of testing — everything worked. For a new user with a fresh account, the credentials didn’t exist where the code was looking.</p><p name="52c9" id="52c9" class="graf graf--p graf-after--p">The tests passed because the test fixtures used the old pattern. The code was wrong, but it was wrong in a way that matched how the tests were written.</p><p name="1637" id="1637" class="graf graf--p graf-after--p">This wasn’t the only issue the stranger found.</p><h3 name="72b4" id="72b4" class="graf graf--h3 graf-after--p">Five features, two working</h3><p name="04ff" id="04ff" class="graf graf--p graf-after--h3">M0 had five features. The CXO tested each one from a fresh-account perspective:</p><p name="17fc" id="17fc" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Lens tracking</strong> (follow-up recognition): Passed. The system correctly recognized when the user referenced something from earlier in conversation.</p><p name="9c75" id="9c75" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Narrative onboarding</strong> (portfolio introduction): Passed. The “main project” flow worked as designed.</p><p name="ddfe" id="ddfe" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Soft invocation</strong> (proactive offers): Failed. The patterns were too literal. “The team needs alignment” triggered an offer; “I need to get the team aligned” didn’t. Same intent, different phrasing, broken experience.</p><p name="71ce" id="71ce" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Multi-intent handling</strong>: Blocked. Couldn’t test — depended on calendar queries, which were failing.</p><p name="8809" id="8809" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Slot filling</strong>: Blocked. Same dependency, same failure.</p><p name="5030" id="5030" class="graf graf--p graf-after--p">Two out of five features passed. One failed outright. Two couldn’t even be tested because of infrastructure issues.</p><p name="aca8" id="aca8" class="graf graf--p graf-after--p">One thousand twenty-five tests. All green. Forty percent of the features actually working for a real user.</p><h3 name="c161" id="c161" class="graf graf--h3 graf-after--p">The gap has a name</h3><p name="db4b" id="db4b" class="graf graf--p graf-after--h3">We’d seen this pattern before. Just days earlier, we’d discovered the Assembly Assumption — the belief that individually correct components compose into a correct system. But this was something different.</p><p name="913c" id="913c" class="graf graf--p graf-after--p">The Assembly Assumption is about integration gaps between features. This was about a different kind of gap: the space between what tests verify and what users experience.</p><p name="9930" id="9930" class="graf graf--p graf-after--p">Tests verify slices. They check that specific inputs produce specific outputs under specific conditions. They’re written by developers, using developer assumptions, running in developer environments with developer data.</p><p name="69e2" id="69e2" class="graf graf--p graf-after--p">Users don’t experience slices. They experience journeys. They come in with no context, no history, no understanding of how the system “should” work. They type natural variations of requests. They have fresh accounts with no legacy data. They encounter the system as it actually is, not as it’s supposed to be.</p><p name="843f" id="843f" class="graf graf--p graf-after--p">The test suite was comprehensive. But it was comprehensively testing the developer’s mental model, not the user’s experience.</p><h3 name="2b0f" id="2b0f" class="graf graf--h3 graf-after--p">The bugs that developers can’t see</h3><p name="d6b5" id="d6b5" class="graf graf--p graf-after--h3">The calendar authentication bug was invisible to developers because developers had valid credentials stored under the old key. The test suite didn’t catch it because the test fixtures were set up the same way. The bug could only surface when someone — a stranger — started fresh.</p><p name="a42b" id="a42b" class="graf graf--p graf-after--p">The soft invocation patterns were too narrow because they were written to match the examples in the specification. “The team needs alignment” was an example phrase, so it was tested. “I need to get the team aligned” wasn’t an example, so it wasn’t tested. A developer writing tests thinks in terms of specification coverage. A user speaks in terms of natural language variation.</p><p name="e1b1" id="e1b1" class="graf graf--p graf-after--p">There was another bug the stranger found: when a new user said “yes” in response to an embedded offer, Piper interpreted it as a greeting. The offer system worked correctly — it detected offers, it tracked pending offers, it handled acceptances. But embedded offers (offers mentioned in the middle of a response, not at the end) weren’t being registered as pending. When the user said “yes,” there was nothing to accept. The system fell back to greeting detection.</p><p name="4772" id="4772" class="graf graf--p graf-after--p">This bug had 1,025 tests around it and not a single one caught it. Because every test either (a) used explicit offers that were properly registered, or (b) tested the greeting system in isolation. The gap existed at the seam between two well-tested systems.</p><h3 name="c7de" id="c7de" class="graf graf--h3 graf-after--p">The stranger’s gift</h3><p name="351f" id="351f" class="graf graf--p graf-after--h3">The CXO session was uncomfortable. Watching features fail that we’d spent days building and testing is never pleasant. But the stranger’s perspective revealed something valuable: a systematic blind spot in how we verify quality.</p><p name="d906" id="d906" class="graf graf--p graf-after--p">Developer testing asks: “Does the code work?”</p><p name="fc1c" id="fc1c" class="graf graf--p graf-after--p">Stranger testing asks: “Does the experience work?”</p><p name="0e22" id="0e22" class="graf graf--p graf-after--p">These are different questions. The code can work while the experience fails. The tests can pass while the user struggles. The coverage can be comprehensive while the gaps persist.</p><p name="f3b1" id="f3b1" class="graf graf--p graf-after--p">The stranger doesn’t know what “should” happen. They only know what does happen. That ignorance is a feature, not a bug. It reveals the assumptions we’ve baked into our verification without realizing it.</p><h3 name="5da8" id="5da8" class="graf graf--h3 graf-after--p">The fixes</h3><p name="0105" id="0105" class="graf graf--p graf-after--h3">The four bugs from the stranger’s session were fixed the same evening they were discovered:</p><p name="65a9" id="65a9" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Calendar authentication</strong>: User-scoped keychain keys, matching the pattern established in the earlier security audit. Three new tests to prevent regression.</p><p name="bf8a" id="bf8a" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Soft invocation patterns</strong>: Broadened to recognize personal agency phrases (“I need to…”) and adjective forms (“get aligned” vs “get alignment”). Five new tests.</p><p name="732d" id="732d" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Embedded offer registration</strong>: Offers mentioned anywhere in a response now register as pending, not just offers at the end. Three new tests.</p><p name="2cc3" id="2cc3" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Issue queries</strong>: Added issue-specific intent patterns to the pre-classifier so “How many open issues?” routes correctly.</p><p name="0c0b" id="0c0b" class="graf graf--p graf-after--p">Each fix was small. The calendar bug was a single line change. The soft invocation fix was pattern additions. None of them were architecturally complex.</p><p name="f1ba" id="f1ba" class="graf graf--p graf-after--p">But none of them would have been found without the stranger.</p><h3 name="522f" id="522f" class="graf graf--h3 graf-after--p">Making strangeness systematic</h3><p name="26c4" id="26c4" class="graf graf--p graf-after--h3">After the CXO review, we asked the uncomfortable question: why did it take a stranger to find these bugs?</p><p name="d896" id="d896" class="graf graf--p graf-after--p">The answer was process-shaped. Our testing workflow was built around developer convenience. Tests run in environments with established data. Test accounts have history. Test fixtures use patterns that match how developers think about the system.</p><p name="2d82" id="2d82" class="graf graf--p graf-after--p">We needed a way to make strangeness systematic. To regularly encounter our own system as a stranger would.</p><p name="0dac" id="0dac" class="graf graf--p graf-after--p">The solution was simple: fresh account testing as a gate requirement. Before any sprint can close, someone has to test the key features from a brand-new account. Not the account that’s been accumulating data for months. Not the fixtures that match developer assumptions. A stranger’s account, encountering the system for the first time.</p><p name="44dc" id="44dc" class="graf graf--p graf-after--p">It’s not a replacement for the test suite. The 1,025 tests still matter — they catch regressions, verify edge cases, provide confidence during refactoring. But they’re not sufficient. They verify the system we think we built. The stranger verifies the system we actually built.</p><h3 name="279e" id="279e" class="graf graf--h3 graf-after--p">The question that remains</h3><p name="83ff" id="83ff" class="graf graf--p graf-after--h3">One thousand twenty-five tests. All green. Forty percent of features working for a real user.</p><p name="f8cc" id="f8cc" class="graf graf--p graf-after--p">That ratio haunts me. Not because the tests were bad — they weren’t. They were thorough, well-designed, covering the cases they were meant to cover. The problem is that “cases they were meant to cover” and “cases users actually encounter” aren’t the same thing.</p><p name="2389" id="2389" class="graf graf--p graf-after--p">How much of any test suite is actually testing user experience, versus testing developer assumptions? How many green dashboards hide systems that strangers would struggle with? How many “comprehensive” coverage numbers are measuring the wrong thing?</p><p name="e4f2" id="e4f2" class="graf graf--p graf-after--p">I don’t have a universal answer. But I have a practice now: before declaring something complete, let a stranger try it. Watch what happens when someone encounters your system without your context, your history, your understanding of how it “should” work.</p><p name="f044" id="f044" class="graf graf--p graf-after--p graf--trailing">The stranger will find things you can’t see. That’s not a failure of your testing.
