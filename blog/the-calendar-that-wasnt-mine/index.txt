This is a user without my credentials seeing my calendar! This is a major security issue for any product intend to serve multiple users.</p><h3 name="ba10" id="ba10" class="graf graf--h3 graf-after--p">The leak</h3><p name="651d" id="651d" class="graf graf--p graf-after--h3">Multi-tenancy is supposed to mean isolation. User A’s data stays with User A. User B’s data stays with User B. The walls between accounts are supposed to be absolute.</p><p name="9feb" id="9feb" class="graf graf--p graf-after--p">The walls weren’t there. Or rather — they were there for some things and not others. User accounts existed. Login worked correctly. The database had <code class="markup--code markup--p-code">owner_id</code> columns on most tables. It looked multi-tenant. It felt multi-tenant.</p><p name="40c1" id="40c1" class="graf graf--p graf-after--p">But the calendar integration stored OAuth tokens globally. When I connected my calendar, the system saved my Google credentials. When the next user came along, the system retrieved… my credentials. Because they were the only credentials there. No user scoping. No isolation. Just a global token bucket.</p><p name="aa34" id="aa34" class="graf graf--p graf-after--p">One user’s calendar. Showing in another user’s account. The leak was complete.</p><h3 name="2f05" id="2f05" class="graf graf--h3 graf-after--p">The archaeology</h3><p name="7e06" id="7e06" class="graf graf--p graf-after--h3">This wasn’t a new bug. This was an old assumption, finally visible.</p><p name="806d" id="806d" class="graf graf--p graf-after--p">Back in October 2025, we’d added multi-user support. The commit looked comprehensive — user accounts, authentication, <code class="markup--code markup--p-code">owner_id</code> foreign keys on the main tables. The PR passed review. The tests passed. Multi-user: done.</p><p name="92a6" id="92a6" class="graf graf--p graf-after--p">Except the integration layer never got updated. It must have been aj oversight in the plan that got past me. Calendar tokens, Slack credentials, GitHub connections — all the external service authentication still used global storage. The October commit updated the core, but the edges remained single-tenant.</p><p name="248f" id="248f" class="graf graf--p graf-after--p">Eight months of development on top of that foundation. Features built, bugs fixed, patterns established — all assuming the multi-user support was complete. It wasn’t. The calendar leak proved it.</p><h3 name="0ba9" id="0ba9" class="graf graf--h3 graf-after--p">The scope of the problem</h3><p name="eecb" id="eecb" class="graf graf--p graf-after--h3">Once we knew what to look for, we found it everywhere.</p><p name="678e" id="678e" class="graf graf--p graf-after--p">Calendar tokens: global. Every external integration stored credentials without user context.</p><p name="8cba" id="8cba" class="graf graf--p graf-after--p">This is why I audit things when I find a problem. It’s rarely a one-off!</p><p name="fad3" id="fad3" class="graf graf--p graf-after--p">And it wasn’t just tokens. The projects table had a unique constraint on <code class="markup--code markup--p-code">name</code> — globally unique, not per-user unique. Two users couldn&#39;t both have a project called &quot;Q1 Planning.&quot; The constraint would reject the second one. Data isolation failure of a different kind.</p><p name="2d07" id="2d07" class="graf graf--p graf-after--p">The API key storage used a global prefix. The config services didn’t require user context. The OAuth callbacks didn’t verify which user initiated the flow.</p><p name="57aa" id="57aa" class="graf graf--p graf-after--p">We’d built multi-user authentication on top of single-user infrastructure. The login was isolated. Everything behind it wasn’t.</p><h3 name="2651" id="2651" class="graf graf--h3 graf-after--p">The fix that couldn’t be incremental</h3><p name="a826" id="a826" class="graf graf--p graf-after--h3">My Lead Developer proposed a phased approach: update each integration one at a time, make <code class="markup--code markup--p-code">user_id</code> optional at first, gradually require it everywhere. Safe. Incremental. Low risk.</p><p name="9cf7" id="9cf7" class="graf graf--p graf-after--p">This is the type of advice you get from LLMs trained on best practices, but every time I get this type of advice I remind them that we do not have any real users yet. We are not in production. We are in alpha. There is no value to a phased approach and much risk of not foinishing.</p><p name="e546" id="e546" class="graf graf--p graf-after--p">My Chief Architect agreed: make <code class="markup--code markup--p-code">user_id</code> required from the start.</p><p name="55d5" id="55d5" class="graf graf--p graf-after--p">The reasoning: optional parameters create hiding places. If <code class="markup--code markup--p-code">user_id</code> is optional, code paths that don&#39;t provide it will continue to work. They shouldn&#39;t work — they&#39;re the exact paths that leak data — but they will. The migration &quot;completes&quot; with gaps still hidden.</p><p name="614b" id="614b" class="graf graf--p graf-after--p">Required parameters create a forcing function. Every code path that lacks user context fails immediately. The failures tell you exactly where to look. No hiding. No gradual drift. No “we’ll get to that later.”</p><p name="6860" id="6860" class="graf graf--p graf-after--p">We made <code class="markup--code markup--p-code">user_id</code> required from Phase 1.</p><h3 name="2cc5" id="2cc5" class="graf graf--h3 graf-after--p">Nine phases</h3><p name="255a" id="255a" class="graf graf--p graf-after--h3">The fix touched everything.</p><p name="fc62" id="fc62" class="graf graf--p graf-after--p">Phase 1: Repositories require <code class="markup--code markup--p-code">owner_id</code>. No more optional user context. Every database operation scoped to a specific user.</p><p name="87c3" id="87c3" class="graf graf--p graf-after--p">Phase 2: OAuth investigation. Map every token storage and retrieval path. Find every place credentials flow without user context.</p><p name="a550" id="a550" class="graf graf--p graf-after--p">Phase 3: RequestContext enforcement. Every authenticated request carries user identity. Middleware validates it. No anonymous operations on protected resources.</p><p name="d648" id="d648" class="graf graf--p graf-after--p">Phase 4: Repository isolation tests. Eighteen new tests verifying that User A cannot access User B’s data through any repository method.</p><p name="e71e" id="e71e" class="graf graf--p graf-after--p">Phase 5: OAuth state redesign.
