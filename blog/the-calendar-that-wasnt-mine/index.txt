1b:T231e,<figure><img alt="Cutaway apartment building where separate apartments share a single basement bin of calendars, causing one resident to see someone else’s schedule." src="https://cdn-images-1.medium.com/max/1024/1*kSean4p6wljAch1kLDeC7g.png" /><figcaption>“Mail for all the tenants delivered!”</figcaption></figure><p><em>January 28, 2026</em></p><p>Tuesday afternoon. I’m testing the alpha release with a fresh account. I am in hurry to test certain things so I give my API key for an LLM and skip connecting my Notion, Slack, and Google Calendar to this account.</p><p>Then I greet Piper and they run down my appointments for the day! This is a user without my credentials seeing my calendar! This is a major security issue for any product intend to serve multiple users.</p><h3>The leak</h3><p>Multi-tenancy is supposed to mean isolation. User A’s data stays with User A. User B’s data stays with User B. The walls between accounts are supposed to be absolute.</p><p>The walls weren’t there. Or rather — they were there for some things and not others. User accounts existed. Login worked correctly. The database had owner_id columns on most tables. It looked multi-tenant. It felt multi-tenant.</p><p>But the calendar integration stored OAuth tokens globally. When I connected my calendar, the system saved my Google credentials. When the next user came along, the system retrieved… my credentials. Because they were the only credentials there. No user scoping. No isolation. Just a global token bucket.</p><p>One user’s calendar. Showing in another user’s account. The leak was complete.</p><h3>The archaeology</h3><p>This wasn’t a new bug. This was an old assumption, finally visible.</p><p>Back in October 2025, we’d added multi-user support. The commit looked comprehensive — user accounts, authentication, owner_id foreign keys on the main tables. The PR passed review. The tests passed. Multi-user: done.</p><p>Except the integration layer never got updated. It must have been aj oversight in the plan that got past me. Calendar tokens, Slack credentials, GitHub connections — all the external service authentication still used global storage. The October commit updated the core, but the edges remained single-tenant.</p><p>Eight months of development on top of that foundation. Features built, bugs fixed, patterns established — all assuming the multi-user support was complete. It wasn’t. The calendar leak proved it.</p><h3>The scope of the problem</h3><p>Once we knew what to look for, we found it everywhere.</p><p>Calendar tokens: global. Slack tokens: global. GitHub tokens: global. Notion tokens: global. Every external integration stored credentials without user context.</p><p>This is why I audit things when I find a problem. It’s rarely a one-off!</p><p>And it wasn’t just tokens. The projects table had a unique constraint on name — globally unique, not per-user unique. Two users couldn&#39;t both have a project called &quot;Q1 Planning.&quot; The constraint would reject the second one. Data isolation failure of a different kind.</p><p>The API key storage used a global prefix. The config services didn’t require user context. The OAuth callbacks didn’t verify which user initiated the flow.</p><p>We’d built multi-user authentication on top of single-user infrastructure. The login was isolated. Everything behind it wasn’t.</p><h3>The fix that couldn’t be incremental</h3><p>My Lead Developer proposed a phased approach: update each integration one at a time, make user_id optional at first, gradually require it everywhere. Safe. Incremental. Low risk.</p><p>This is the type of advice you get from LLMs trained on best practices, but every time I get this type of advice I remind them that we do not have any real users yet. We are not in production. We are in alpha. There is no value to a phased approach and much risk of not foinishing.</p><p>My Chief Architect agreed: make user_id required from the start.</p><p>The reasoning: optional parameters create hiding places. If user_id is optional, code paths that don&#39;t provide it will continue to work. They shouldn&#39;t work — they&#39;re the exact paths that leak data — but they will. The migration &quot;completes&quot; with gaps still hidden.</p><p>Required parameters create a forcing function. Every code path that lacks user context fails immediately. The failures tell you exactly where to look. No hiding. No gradual drift. No “we’ll get to that later.”</p><p>We made user_id required from Phase 1.</p><h3>Nine phases</h3><p>The fix touched everything.</p><p>Phase 1: Repositories require owner_id. No more optional user context. Every database operation scoped to a specific user.</p><p>Phase 2: OAuth investigation. Map every token storage and retrieval path. Find every place credentials flow without user context.</p><p>Phase 3: RequestContext enforcement. Every authenticated request carries user identity. Middleware validates it. No anonymous operations on protected resources.</p><p>Phase 4: Repository isolation tests. Eighteen new tests verifying that User A cannot access User B’s data through any repository method.</p><p>Phase 5: OAuth state redesign. The OAuth callback now embeds user_id in the state parameter. When Google redirects back, we know which user initiated the flow.</p><p>Phase 6: Credential storage separation. IntegrationConfigService for app-level credentials. UserTokenService for user-specific tokens. Clear separation, clear ownership.</p><p>Phase 7: Config service signatures. Every method that touches user data requires user_id as a parameter. No more inferring. No more globals.</p><p>Phase 8: Singleton manager refactor. The conversation managers, the onboarding managers — all the stateful services now explicitly track which user they’re serving.</p><p>Phase 9: Workspace activation. The workspace_id concept that had been designed but dormant, finally wired in as the future path for team isolation.</p><p>94 new tests. ADR-058 documenting every decision. One day of execution.</p><h3>What the forcing function revealed</h3><p>The Architect and I were right. Making user_id required immediately surfaced gaps we wouldn&#39;t have found otherwise.</p><ul><li>A webhook handler that processed Slack events without checking which user’s Slack connection triggered them.</li><li>An OAuth callback that stored tokens before verifying the user session.</li><li>A config lookup that fell back to global defaults when user-specific values weren’t found.</li></ul><p>Each failure was a discovery. Each discovery was a fix. By the end of the day, the system was genuinely isolated — not “mostly isolated” or “isolated except for edge cases.” Every code path either handled user context explicitly or had been deliberately designed for system-level operations.</p><h3>The October lesson</h3><p>The October 2025 commit wasn’t wrong. Adding user accounts, authentication, and owner_id columns was necessary work. The mistake was assuming it was complete.</p><p>Multi-tenancy isn’t a feature you add. It’s a property that must hold everywhere. One leaky integration undermines the entire model. One global token bucket means the walls don’t exist.</p><p>The October commit added the structure. The January fix ensured the property. Eight months between them — eight months where the system looked multi-tenant but wasn’t.</p><h3>The calendar now</h3><p>I tested again after the fix. Fresh account, again skipped connecting a calendar. New bug: now Piper told me I have no events coming up (when really it should admit it doesn’t know my schedule). We opened up two new issues: one for failing gracefully now, and a later plan for Piper to offer to attach your calendar if it isn’t added yet (but not to keep offering if you refuse).</p><p>The walls exist now. 94 tests verify they hold. ADR-058 documents why they’re built this way. The forcing function ensured we found every gap.</p><p>One day of aggressive fixing. Eight months of hidden assumptions surfaced. User A’s calendar stays with User A.</p><p>That’s what multi-tenancy actually means.</p><p><em>Next on Building Piper Morgan: The Drift We Didn’t See, where a timezone issue turned out to be everywhere.</em></p><p><em>Have you discovered that “multi-user support” wasn’t as complete as you thought? The leak was complete.</p><h3 name="2f05" id="2f05" class="graf graf--h3 graf-after--p">The archaeology</h3><p name="7e06" id="7e06" class="graf graf--p graf-after--h3">This wasn’t a new bug. This was an old assumption, finally visible.</p><p name="806d" id="806d" class="graf graf--p graf-after--p">Back in October 2025, we’d added multi-user support. The commit looked comprehensive — user accounts, authentication, <code class="markup--code markup--p-code">owner_id</code> foreign keys on the main tables. The PR passed review. The tests passed. Multi-user: done.</p><p name="92a6" id="92a6" class="graf graf--p graf-after--p">Except the integration layer never got updated. It must have been aj oversight in the plan that got past me. Calendar tokens, Slack credentials, GitHub connections — all the external service authentication still used global storage. The October commit updated the core, but the edges remained single-tenant.</p><p name="248f" id="248f" class="graf graf--p graf-after--p">Eight months of development on top of that foundation. Features built, bugs fixed, patterns established — all assuming the multi-user support was complete. It wasn’t. The calendar leak proved it.</p><h3 name="0ba9" id="0ba9" class="graf graf--h3 graf-after--p">The scope of the problem</h3><p name="eecb" id="eecb" class="graf graf--p graf-after--h3">Once we knew what to look for, we found it everywhere.</p><p name="678e" id="678e" class="graf graf--p graf-after--p">Calendar tokens: global. Slack tokens: global. GitHub tokens: global. Notion tokens: global. Every external integration stored credentials without user context.</p><p name="8cba" id="8cba" class="graf graf--p graf-after--p">This is why I audit things when I find a problem. It’s rarely a one-off!</p><p name="fad3" id="fad3" class="graf graf--p graf-after--p">And it wasn’t just tokens. The projects table had a unique constraint on <code class="markup--code markup--p-code">name</code> — globally unique, not per-user unique. Two users couldn&#39;t both have a project called &quot;Q1 Planning.&quot; The constraint would reject the second one. Data isolation failure of a different kind.</p><p name="2d07" id="2d07" class="graf graf--p graf-after--p">The API key storage used a global prefix. The config services didn’t require user context. The OAuth callbacks didn’t verify which user initiated the flow.</p><p name="57aa" id="57aa" class="graf graf--p graf-after--p">We’d built multi-user authentication on top of single-user infrastructure. The login was isolated. Everything behind it wasn’t.</p><h3 name="2651" id="2651" class="graf graf--h3 graf-after--p">The fix that couldn’t be incremental</h3><p name="a826" id="a826" class="graf graf--p graf-after--h3">My Lead Developer proposed a phased approach: update each integration one at a time, make <code class="markup--code markup--p-code">user_id</code> optional at first, gradually require it everywhere. Safe. Incremental. Low risk.</p><p name="9cf7" id="9cf7" class="graf graf--p graf-after--p">This is the type of advice you get from LLMs trained on best practices, but every time I get this type of advice I remind them that we do not have any real users yet. We are not in production. We are in alpha. There is no value to a phased approach and much risk of not foinishing.</p><p name="e546" id="e546" class="graf graf--p graf-after--p">My Chief Architect agreed: make <code class="markup--code markup--p-code">user_id</code> required from the start.</p><p name="55d5" id="55d5" class="graf graf--p graf-after--p">The reasoning: optional parameters create hiding places. If <code class="markup--code markup--p-code">user_id</code> is optional, code paths that don&#39;t provide it will continue to work. They shouldn&#39;t work — they&#39;re the exact paths that leak data — but they will. The migration &quot;completes&quot; with gaps still hidden.</p><p name="614b" id="614b" class="graf graf--p graf-after--p">Required parameters create a forcing function. Every code path that lacks user context fails immediately. The failures tell you exactly where to look. No hiding. No gradual drift. No “we’ll get to that later.”</p><p name="6860" id="6860" class="graf graf--p graf-after--p">We made <code class="markup--code markup--p-code">user_id</code> required from Phase 1.</p><h3 name="2cc5" id="2cc5" class="graf graf--h3 graf-after--p">Nine phases</h3><p name="255a" id="255a" class="graf graf--p graf-after--h3">The fix touched everything.</p><p name="fc62" id="fc62" class="graf graf--p graf-after--p">Phase 1: Repositories require <code class="markup--code markup--p-code">owner_id</code>. No more optional user context. Every database operation scoped to a specific user.</p><p name="87c3" id="87c3" class="graf graf--p graf-after--p">Phase 2: OAuth investigation. Map every token storage and retrieval path. Find every place credentials flow without user context.</p><p name="a550" id="a550" class="graf graf--p graf-after--p">Phase 3: RequestContext enforcement. Every authenticated request carries user identity. Middleware validates it. No anonymous operations on protected resources.</p><p name="d648" id="d648" class="graf graf--p graf-after--p">Phase 4: Repository isolation tests. Eighteen new tests verifying that User A cannot access User B’s data through any repository method.</p><p name="e71e" id="e71e" class="graf graf--p graf-after--p">Phase 5: OAuth state redesign. The OAuth callback now embeds <code class="markup--code markup--p-code">user_id</code> in the state parameter. When Google redirects back, we know which user initiated the flow.</p><p name="1f16" id="1f16" class="graf graf--p graf-after--p">Phase 6: Credential storage separation. <code class="markup--code markup--p-code">IntegrationConfigService</code> for app-level credentials. <code class="markup--code markup--p-code">UserTokenService</code> for user-specific tokens. Clear separation, clear ownership.</p><p name="6cb7" id="6cb7" class="graf graf--p graf-after--p">Phase 7: Config service signatures. Every method that touches user data requires <code class="markup--code markup--p-code">user_id</code> as a parameter. No more inferring. No more globals.</p><p name="0a99" id="0a99" class="graf graf--p graf-after--p">Phase 8: Singleton manager refactor. The conversation managers, the onboarding managers — all the stateful services now explicitly track which user they’re serving.</p><p name="d1a2" id="d1a2" class="graf graf--p graf-after--p">Phase 9: Workspace activation. The <code class="markup--code markup--p-code">workspace_id</code> concept that had been designed but dormant, finally wired in as the future path for team isolation.</p><p name="2660" id="2660" class="graf graf--p graf-after--p">94 new tests. ADR-058 documenting every decision. One day of execution.</p><h3 name="5695" id="5695" class="graf graf--h3 graf-after--p">What the forcing function revealed</h3><p name="e0c9" id="e0c9" class="graf graf--p graf-after--h3">The Architect and I were right. Making <code class="markup--code markup--p-code">user_id</code> required immediately surfaced gaps we wouldn&#39;t have found otherwise.</p><ul class="postList"><li name="5dc9" id="5dc9" class="graf graf--li graf-after--p">A webhook handler that processed Slack events without checking which user’s Slack connection triggered them.</li><li name="ddaf" id="ddaf" class="graf graf--li graf-after--li">An OAuth callback that stored tokens before verifying the user session.</li><li name="e555" id="e555" class="graf graf--li graf-after--li">A config lookup that fell back to global defaults when user-specific values weren’t found.</li></ul><p name="149e" id="149e" class="graf graf--p graf-after--li">Each failure was a discovery. Each discovery was a fix. By the end of the day, the system was genuinely isolated — not “mostly isolated” or “isolated except for edge cases.” Every code path either handled user context explicitly or had been deliberately designed for system-level operations.</p><h3 name="9b17" id="9b17" class="graf graf--h3 graf-after--p">The October lesson</h3><p name="d66e" id="d66e" class="graf graf--p graf-after--h3">The October 2025 commit wasn’t wrong. Adding user accounts, authentication, and <code class="markup--code markup--p-code">owner_id</code> columns was necessary work. The mistake was assuming it was complete.</p><p name="615c" id="615c" class="graf graf--p graf-after--p">Multi-tenancy isn’t a feature you add. It’s a property that must hold everywhere. One leaky integration undermines the entire model. One global token bucket means the walls don’t exist.</p><p name="5251" id="5251" class="graf graf--p graf-after--p">The October commit added the structure. The January fix ensured the property. Eight months between them — eight months where the system looked multi-tenant but wasn’t.</p><h3 name="0b4c" id="0b4c" class="graf graf--h3 graf-after--p">The calendar now</h3><p name="2c96" id="2c96" class="graf graf--p graf-after--h3">I tested again after the fix. Fresh account, again skipped connecting a calendar. New bug: now Piper told me I have no events coming up (when really it should admit it doesn’t know my schedule). We opened up two new issues: one for failing gracefully now, and a later plan for Piper to offer to attach your calendar if it isn’t added yet (but not to keep offering if you refuse).</p><p name="1fa7" id="1fa7" class="graf graf--p graf-after--p">The walls exist now. 94 tests verify they hold. Eight months of hidden assumptions surfaced.
