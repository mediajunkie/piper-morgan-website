1b:T1f35,<p><em>January 19, 2026</em></p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*QhIxK-lYaGJqYBkWH9FitQ.png" /><figcaption>“Hello, world!”</figcaption></figure><p>After eight months of building an AI assistant, we finally wrote down what it actually thinks about. Not the data model. Not the schema. The <em>grammar</em> — the fundamental sentence structure that describes how Piper Morgan understands the world.</p><p>It’s five words: <strong>Entities experience Moments in Places.</strong></p><h3>Why grammar, not schema</h3><p>A database schema tells you what fields exist. A data model tells you how tables relate. But neither tells you what the system <em>cares about</em>.</p><p>Grammar is different. Grammar describes how meaning gets constructed. Subject-verb-object isn’t just syntax — it’s a claim about how the world works: there are actors, there are actions, there are things acted upon.</p><p>Our grammar makes a similar claim:</p><p><strong>Entities</strong> are actors with identity and agency. Not just data objects. Not just records in a database. Things that can perceive, remember, and anticipate.</p><p><strong>Experience</strong> is the verb — perception plus memory plus anticipation. Not “have” or “contain” or “reference.” Experience implies awareness, not storage.</p><p><strong>Moments</strong> are bounded significant occurrences. Not timestamps. Not events. Moments have dramatic unities — a beginning, a middle, an end. They have weight. Some moments matter more than others.</p><p><strong>Places</strong> are contexts with atmosphere. Not containers. Not folders. Places have character. A standup meeting feels different from a strategy session, even if they’re both “meetings.”</p><h3>What this changes</h3><p>The grammar isn’t just philosophy. It shapes implementation decisions.</p><p><strong>Without the grammar</strong>, you might build:</p><ul><li>A task list (entities as rows, attributes as columns)</li><li>An event log (moments as timestamps, places as tags)</li><li>A folder structure (places as containers, entities as files)</li></ul><p><strong>With the grammar</strong>, you build differently:</p><ul><li>Entities that perceive their environment through lenses</li><li>Moments that have dramatic structure, not just start/end times</li><li>Places that carry atmosphere and influence how entities behave within them</li></ul><p>The difference is subtle in code but significant in experience. A task list doesn’t notice when you’re overwhelmed. A system that understands “experience” might.</p><h3>The eight lenses</h3><p>How do entities perceive their world? Through lenses — distinct ways of making sense of the same information.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/640/1*XFy8K-rQY3RVW27E20Sltg.png" /></figure><p>The same project looks different through each lens. Through Temporal, it’s a timeline. Through Hierarchy, it’s a tree. Through Flow, it’s a kanban board. Through Causal, it’s a dependency graph.</p><p>An AI assistant that only has one lens gives one kind of answer. An assistant that can shift lenses gives the answer that matches how you’re thinking.</p><h3>The implementation test</h3><p>On January 19th, we turned this grammar into code. (We first derived it from fat-marker sketches back in November, but we had a lot of other things to fix first).</p><p>The test wasn’t whether we could write the classes — of course we could. The test was whether the implementation preserved the philosophy.</p><p>Three questions guided verification:</p><p><strong>1. Can entities perceive, or do they just store?</strong></p><p>If an entity is just a data container, we’ve flattened the model. The implementation needs perception methods — ways for entities to sense their environment, not just hold data.</p><p><strong>2. Do moments have weight, or just timestamps?</strong></p><p>If moments are just events with start/end times, we’ve lost the dramatic structure. The implementation needs significance markers — ways to distinguish moments that matter from moments that don’t.</p><p><strong>3. Do places have atmosphere, or are they just containers?</strong></p><p>If places are just folders or categories, we’ve lost context. The implementation needs atmosphere properties — characteristics that influence how entities behave within them.</p><p>These aren’t tests you write in pytest. They’re design reviews. Does the code preserve the grammar, or does it flatten the grammar into something simpler but less meaningful?</p><h3>The risk of flattening</h3><p>Every implementation decision carries flattening risk. It’s always easier to build a simpler model: entities as database rows, moments as timestamps, places as string tags, experience as CRUD operations.</p><p>The flattened version works. It passes tests. It ships faster.</p><p>But it doesn’t <em>feel</em> different from every other PM tool. The grammar is what makes Piper Morgan distinct. Flatten the grammar and you have a sophisticated to-do list. Preserve the grammar and you have something that understands how work actually feels.</p><h3>Why it took eight months</h3><p>We didn’t write this grammar in month one. We discovered it gradually: months 1–2 were building features with no coherent model; months 3–4 were noticing patterns and starting to name things; months 5–6 brought “fat markers” sketching sessions and visual thinking; months 7–8 saw the grammar crystallize and implementation begin.</p><figure><img alt="sketch of a container (“one place”) with goals or outocomes, the user and Piper, other people, policy, in side of a process and taking place ovrer a period of time" src="https://cdn-images-1.medium.com/max/424/1*sGuYDqLCyvxaEShVnRTkYg.png" /><figcaption>The sentence grew out of this sketch trying to describe what a Moment would contain</figcaption></figure><p>The delay wasn’t wasted time. We couldn’t have written this grammar in June. We didn’t know enough yet. The grammar emerged from building, not from planning.</p><p>This is the paradox of conceptual work: you need to build to understand, but you need to understand to build well. The grammar came late because it could only come late. Now that we have it, everything else builds on it.</p><h3>The sentence as compass</h3><p>“Entities experience Moments in Places” is now our design compass. When we’re unsure about a feature, we ask: does this fit the grammar?</p><ul><li>“Should we add a calendar view?” → Does it help entities perceive moments?</li><li>“Should we add team collaboration?” → Does it enrich how entities experience places together?</li><li>“Should we add analytics?” → Does it give entities new lenses for perception?</li></ul><p>Features that fit the grammar belong. Features that require a different grammar probably don’t.</p><p>It’s not a rigid rule. Grammars evolve. But having an explicit grammar means we can notice when we’re changing it, rather than drifting unconsciously.</p><p><em>Next on Building Piper Morgan: The Calendar That Wasn’t Mine, when alpha testing revealed that our multi-tenancy implementation wasn’t as complete as we thought.</em></p><p><em>Have you ever discovered that naming something changed how you built it? A standup meeting feels different from a strategy session, even if they’re both “meetings.”</p><h3 name="05f5" id="05f5" class="graf graf--h3 graf-after--p">What this changes</h3><p name="9f05" id="9f05" class="graf graf--p graf-after--h3">The grammar isn’t just philosophy. It shapes implementation decisions.</p><p name="8cf4" id="8cf4" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Without the grammar</strong>, you might build:</p><ul class="postList"><li name="1877" id="1877" class="graf graf--li graf-after--p">A task list (entities as rows, attributes as columns)</li><li name="b07b" id="b07b" class="graf graf--li graf-after--li">An event log (moments as timestamps, places as tags)</li><li name="54ce" id="54ce" class="graf graf--li graf-after--li">A folder structure (places as containers, entities as files)</li></ul><p name="cb3f" id="cb3f" class="graf graf--p graf-after--li"><strong class="markup--strong markup--p-strong">With the grammar</strong>, you build differently:</p><ul class="postList"><li name="f9df" id="f9df" class="graf graf--li graf-after--p">Entities that perceive their environment through lenses</li><li name="50b3" id="50b3" class="graf graf--li graf-after--li">Moments that have dramatic structure, not just start/end times</li><li name="85e1" id="85e1" class="graf graf--li graf-after--li">Places that carry atmosphere and influence how entities behave within them</li></ul><p name="8387" id="8387" class="graf graf--p graf-after--li">The difference is subtle in code but significant in experience. A task list doesn’t notice when you’re overwhelmed. A system that understands “experience” might.</p><h3 name="f566" id="f566" class="graf graf--h3 graf-after--p">The eight lenses</h3><p name="6db6" id="6db6" class="graf graf--p graf-after--h3">How do entities perceive their world? Through lenses — distinct ways of making sense of the same information.</p><figure name="8e30" id="8e30" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*XFy8K-rQY3RVW27E20Sltg.png" data-width="640" data-height="382" src="https://cdn-images-1.medium.com/max/800/1*XFy8K-rQY3RVW27E20Sltg.png"></figure><p name="5ba2" id="5ba2" class="graf graf--p graf-after--figure">The same project looks different through each lens. Through Temporal, it’s a timeline. Through Hierarchy, it’s a tree. Through Flow, it’s a kanban board. Through Causal, it’s a dependency graph.</p><p name="39b6" id="39b6" class="graf graf--p graf-after--p">An AI assistant that only has one lens gives one kind of answer. An assistant that can shift lenses gives the answer that matches how you’re thinking.</p><h3 name="fd0b" id="fd0b" class="graf graf--h3 graf-after--p">The implementation test</h3><p name="8e21" id="8e21" class="graf graf--p graf-after--h3">On January 19th, we turned this grammar into code. (We first derived it from fat-marker sketches back in November, but we had a lot of other things to fix first).</p><p name="cc7b" id="cc7b" class="graf graf--p graf-after--p">The test wasn’t whether we could write the classes — of course we could. The test was whether the implementation preserved the philosophy.</p><p name="37ab" id="37ab" class="graf graf--p graf-after--p">Three questions guided verification:</p><p name="3099" id="3099" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">1. Can entities perceive, or do they just store?</strong></p><p name="6cb7" id="6cb7" class="graf graf--p graf-after--p">If an entity is just a data container, we’ve flattened the model. The implementation needs perception methods — ways for entities to sense their environment, not just hold data.</p><p name="1658" id="1658" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">2. Do moments have weight, or just timestamps?</strong></p><p name="704b" id="704b" class="graf graf--p graf-after--p">If moments are just events with start/end times, we’ve lost the dramatic structure. The implementation needs significance markers — ways to distinguish moments that matter from moments that don’t.</p><p name="aadc" id="aadc" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">3. Do places have atmosphere, or are they just containers?</strong></p><p name="39ac" id="39ac" class="graf graf--p graf-after--p">If places are just folders or categories, we’ve lost context. The implementation needs atmosphere properties — characteristics that influence how entities behave within them.</p><p name="d436" id="d436" class="graf graf--p graf-after--p">These aren’t tests you write in pytest. They’re design reviews. Does the code preserve the grammar, or does it flatten the grammar into something simpler but less meaningful?</p><h3 name="64cf" id="64cf" class="graf graf--h3 graf-after--p">The risk of flattening</h3><p name="fcd3" id="fcd3" class="graf graf--p graf-after--h3">Every implementation decision carries flattening risk. It’s always easier to build a simpler model: entities as database rows, moments as timestamps, places as string tags, experience as CRUD operations.</p><p name="7f8f" id="7f8f" class="graf graf--p graf-after--p">The flattened version works. It passes tests. It ships faster.</p><p name="a06a" id="a06a" class="graf graf--p graf-after--p">But it doesn’t <em class="markup--em markup--p-em">feel</em> different from every other PM tool. The grammar is what makes Piper Morgan distinct. Flatten the grammar and you have a sophisticated to-do list. Preserve the grammar and you have something that understands how work actually feels.</p><h3 name="77f0" id="77f0" class="graf graf--h3 graf-after--p">Why it took eight months</h3><p name="00b6" id="00b6" class="graf graf--p graf-after--h3">We didn’t write this grammar in month one. We discovered it gradually: months 1–2 were building features with no coherent model; months 3–4 were noticing patterns and starting to name things; months 5–6 brought “fat markers” sketching sessions and visual thinking; months 7–8 saw the grammar crystallize and implementation begin.</p><figure name="36d2" id="36d2" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*sGuYDqLCyvxaEShVnRTkYg.png" data-width="424" data-height="395" alt="sketch of a container (“one place”) with goals or outocomes, the user and Piper, other people, policy, in side of a process and taking place ovrer a period of time" src="https://cdn-images-1.medium.com/max/800/1*sGuYDqLCyvxaEShVnRTkYg.png"><figcaption class="imageCaption">The sentence grew out of this sketch trying to describe what a Moment would contain</figcaption></figure><p name="8cd8" id="8cd8" class="graf graf--p graf-after--figure">The delay wasn’t wasted time. We couldn’t have written this grammar in June. We didn’t know enough yet. The grammar emerged from building, not from planning.</p><p name="fee6" id="fee6" class="graf graf--p graf-after--p">This is the paradox of conceptual work: you need to build to understand, but you need to understand to build well. The grammar came late because it could only come late. Now that we have it, everything else builds on it.</p><h3 name="431c" id="431c" class="graf graf--h3 graf-after--p">The sentence as compass</h3><p name="4290" id="4290" class="graf graf--p graf--startsWithDoubleQuote graf-after--h3">“Entities experience Moments in Places” is now our design compass. When we’re unsure about a feature, we ask: does this fit the grammar?</p><ul class="postList"><li name="746f" id="746f" class="graf graf--li graf--startsWithDoubleQuote graf-after--p">“Should we add a calendar view?” → Does it help entities perceive moments?</li><li name="1ba5" id="1ba5" class="graf graf--li graf--startsWithDoubleQuote graf-after--li">“Should we add team collaboration?” → Does it enrich how entities experience places together?</li><li name="d6a0" id="d6a0" class="graf graf--li graf--startsWithDoubleQuote graf-after--li">“Should we add analytics?” → Does it give entities new lenses for perception?</li></ul><p name="378e" id="378e" class="graf graf--p graf-after--li">Features that fit the grammar belong. Features that require a different grammar probably don’t.</p><p name="277c" id="277c" class="graf graf--p graf-after--p graf--trailing">It’s not a rigid rule. Grammars evolve.
