The framework’s dependency injection design exists for good reasons — security, performance, clarity.</p><p name="4b94" id="4b94" class="graf graf--p graf-after--p">Fighting the framework to catch 2 additional error types (that happen rarely — auth errors only occur during session expiration or token issues) doesn’t justify violating architectural patterns.</p><h4 name="cb5d" id="cb5d" class="graf graf--h4 graf-after--p"><strong class="markup--strong markup--h4-strong">Cost-Benefit Analysis</strong></h4><p name="b3c9" id="b3c9" class="graf graf--p graf-after--h4">4–6 hours (Option A) or 8–12 hours (Option B) to improve UX for &lt;5% of errors. In alpha testing with 5–10 trusted users. Who can handle technical auth errors without confusion.</p><p name="d1cd" id="d1cd" class="graf graf--p graf-after--p">That effort could instead: Polish other features. Fix actual bugs. Improve core functionality. Add value.</p><h4 name="b6b1" id="b6b1" class="graf graf--h4 graf-after--p"><strong class="markup--strong markup--h4-strong">Alpha Testing Context</strong></h4><p name="02bc" id="02bc" class="graf graf--p graf-after--h4">We’re not shipping to public internet. We’re alpha testing with trusted users who understand the system is in development. Technical auth errors aren’t catastrophic for alpha.</p><p name="a300" id="a300" class="graf graf--p graf-after--p">Production may require better auth error UX. But production is not alpha. Alpha requirement is “functional enough to test core value.” Mission accomplished at 4/6.</p><h4 name="cf70" id="cf70" class="graf graf--h4 graf-after--p"><strong class="markup--strong markup--h4-strong">The 80% Pattern Inverted</strong></h4><p name="56ae" id="56ae" class="graf graf--p graf-after--h4">Usually we fight the 80% pattern — stopping at “good enough” instead of truly complete. Here we’re doing the opposite: Recognizing that 67% (4/6) represents complete work given architectural constraints.</p><p name="83b4" id="83b4" class="graf graf--p graf-after--p">Not “we got tired and stopped.” But “we hit architectural limit and accepted it intelligently.”</p><h3 name="855a" id="855a" class="graf graf--h3 graf-after--p">What Monday’s mixed results reveal</h3><p name="76a4" id="76a4" class="graf graf--p graf-after--h3">Let me be explicit about Monday’s outcomes:</p><p name="dac3" id="dac3" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Issue #284</strong>: ✅ Mapping layer created, all tests passing</p><p name="ae71" id="ae71" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Issue #285</strong>: ✅ Todo system wired, API + chat working</p><p name="9daf" id="9daf" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Issue #283</strong>: ✅ 4/6 error types humanized, architectural limit accepted</p><p name="0fb7" id="0fb7" class="graf graf--p graf-after--p">The framing matters. If “complete” means “achieved everything specified without limitation,” Monday is 2/3 success.</p><p name="d22d" id="d22d" class="graf graf--p graf-after--p">But if “complete” means “resolved all issues to architecturally appropriate state,” Monday is 3/3 success.</p><p name="64e4" id="64e4" class="graf graf--p graf-after--p">I’m going with the second framing. Because Issue #283 isn’t incomplete. It’s complete to the extent architecture allows without violating framework patterns.</p><h3 name="9866" id="9866" class="graf graf--h3 graf-after--p">The archaeological discovery pattern continues</h3><p name="d3b2" id="d3b2" class="graf graf--p graf-after--h3">Monday marks the third consecutive day of archaeological discoveries:</p><p name="854e" id="854e" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Saturday</strong>: DocumentService 75% complete → wired in 2 hours</p><p name="9b98" id="9b98" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Sunday</strong>: Todo infrastructure 75% complete → wired in 4–6 hours</p><p name="2f8f" id="2f8f" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Monday</strong>: ActionHumanizer + EnhancedErrorMiddleware complete → wired immediately</p><p name="afe7" id="afe7" class="graf graf--p graf-after--p">The pattern is now established methodology, at least around these parts, in this stage, of this project:</p><ol class="postList"><li name="8bd1" id="8bd1" class="graf graf--li graf-after--p"><strong class="markup--strong markup--li-strong">Phase −1 investigation first</strong> (not optional, required)</li><li name="449f" id="449f" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Assume infrastructure might exist</strong> (check before building)</li><li name="8792" id="8792" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Discover what’s there</strong> (archaeological approach)</li><li name="228f" id="228f" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Assess completion state</strong> (75%? 90%? 10%?)</li><li name="95b5" id="95b5" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Wire vs rebuild decision</strong> (usually wire is faster)</li></ol><p name="1f28" id="1f28" class="graf graf--p graf-after--li">Monday proves the pattern works even when findings are mixed. We found ActionHumanizer (partially helpful). We found EnhancedErrorMiddleware (fully helpful, needed wiring). We found the architectural limit (FastAPI dependency injection phase).</p><p name="5bcf" id="5bcf" class="graf graf--p graf-after--p">All valuable discoveries. All inform the solution. None would have been found without Phase -1 investigation.</p><h3 name="a3db" id="a3db" class="graf graf--h3 graf-after--p">The critical system bug discovery</h3><p name="bda3" id="bda3" class="graf graf--p graf-after--h3">Monday also brought unexpected discovery: Critical system bug in Claude interface.</p><p name="1347" id="1347" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">The problem</strong>: False “Human:” responses being generated.</p><ul class="postList"><li name="b02f" id="b02f" class="graf graf--li graf-after--p">Contain fabricated details</li><li name="4156" id="4156" class="graf graf--li graf-after--li">Not written by PM</li><li name="bdbe" id="bdbe" class="graf graf--li graf-after--li">Not recognized by Claude</li><li name="5450" id="5450" class="graf graf--li graf-after--li">Source “unknown”</li></ul><p name="f10b" id="f10b" class="graf graf--p graf-after--li"><strong class="markup--strong markup--p-strong">Impact</strong>: “Expensive service pollution” with made-up content.</p><p name="4071" id="4071" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Action</strong>: Reported to Anthropic.</p><p name="2451" id="2451" class="graf graf--p graf-after--p">Not related to Piper Morgan code. But impacting development process. If responses falsely attributed to human contain incorrect guidance, agents might implement wrong solutions.</p><p name="7772" id="7772" class="graf graf--p graf-after--p">Critical enough to report. But not blocking Monday’s work. Development continued while escalation processed.</p><h3 name="195d" id="195d" class="graf graf--h3 graf-after--p">What acceptance of limits teaches</h3><p name="8a77" id="8a77" class="graf graf--p graf-after--h3">Monday’s architectural decision — accepting 4/6 as complete — reveals something about mature engineering:</p><p name="ab04" id="ab04" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Not every problem requires solution.</strong></p><p name="84ca" id="84ca" class="graf graf--p graf-after--p">Some problems have solutions that cost more than the problem’s impact. Some limitations are features of the architecture you’re using. Some gaps are acceptable given context.</p><p name="b302" id="b302" class="graf graf--p graf-after--p">The discipline isn’t “solve every problem.” It’s “solve problems worth solving given constraints and context.”</p><p name="2ce3" id="2ce3" class="graf graf--p graf-after--p">FastAPI’s dependency injection design is good architecture. It protects security. It ensures fast failures. It maintains clarity. That design creates limitation: Dependency errors can’t be caught by normal exception handlers.</p><p name="c726" id="c726" class="graf graf--p graf-after--p">We could work around that limitation. Break the pattern. Achieve 6/6 error humanization. At cost of violating framework architecture and spending 8–12 hours.</p><p name="6be7" id="6be7" class="graf graf--p graf-after--p">Or we could accept 4/6 represents complete work given architectural reality. Spend those 8–12 hours elsewhere. Deliver more value.</p><p name="c1f9" id="c1f9" class="graf graf--p graf-after--p">Monday chose acceptance. Not compromise. Not giving up. But intelligent assessment that fighting architecture for marginal gain isn’t worth cost.</p><p name="5230" id="5230" class="graf graf--p graf-after--p">That’s mature engineering. That’s product thinking. That’s systematic methodology applied to decision-making, not just execution.</p><h3 name="e406" id="e406" class="graf graf--h3 graf-after--p">The foundation work preview</h3><p name="26a7" id="26a7" class="graf graf--p graf-after--h3">Monday’s two completed issues (action mapping + todo system) set up Tuesday’s work.</p><p name="8be7" id="8be7" class="graf graf--p graf-after--p">Issue #285 looked simple Sunday: “Wire todo system, infrastructure exists, 4–6 hours.”</p><p name="c29b" id="c29b" class="graf graf--p graf-after--p">Monday wires the surface layer. Chat handlers. Web routes. Basic CRUD.</p><p name="c98a" id="c98a" class="graf graf--p graf-after--p">But Monday also reveals deeper architectural question: The domain model foundation. How TodoItem relates to universal Item base class. Polymorphic inheritance patterns.</p><p name="69ee" id="69ee" class="graf graf--p graf-after--p">Tuesday will spend 16.5 hours on that foundation. Not because Monday’s work was wrong. But because Monday’s work revealed architectural improvements worth making.</p><p name="718b" id="718b" class="graf graf--p graf-after--p">That’s the difference between reactive fixing and systematic building. Systematic asks: “While we’re here, what’s the right architecture?”</p><p name="afae" id="afae" class="graf graf--p graf-after--p graf--trailing">Monday completes what it set out to complete.
