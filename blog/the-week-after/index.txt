Just seven agents working in parallel on what comes next.</p><h3>Consolidation as Strategy</h3><p>The <a href="https://www.linkedin.com/pulse/weekly-ship-020-convergence-christian-crumlish-bamcc/?trackingId=e%2B84s7veToSoVdexXthDHA%3D%3D">Weekly Ship</a> went out first thing early on Friday. Twenty-two open beads — small tracked tasks that accumulate during sprint work — sat in the system waiting for attention. Not bugs exactly, not features exactly, just the organizational debris that builds up when you’re moving fast. The kind of thing that feels low-priority until you realize you’re spending mental energy remembering them all.</p><p>The solution wasn’t to grind through twenty-two items. Four GitHub epics emerged from the chaos: SEC-RBAC (security track, role-based access control epic) work, infrastructure improvements, Slack TDD gaps, and test debt. The backlog went from “things we should probably do sometime” to “here are four clear next steps when we’re ready.”</p><p>That consolidation took a little time (those productive wee hours before I focus on my day-job priorities). Seven agents, four subagents, zero conflicts. CRUD operations were failing again — different symptoms than Thursday’s integration gaps, but same result: users couldn’t create todos, couldn’t upload files, couldn’t add projects. The mobile testing revealed something broken deeper in the stack.</p><p>Twenty-four hours later, we’d discovered <strong><em>six</em></strong> root causes. Each one seemed like <em>the</em> problem until fixing it revealed the next layer. This is where the habit of asking the LLMs to look again, to dig deeper, to think categorically, to notice antipatterns, to find systematic issues really pays off:</p><p>Wrong repository type in dependency injection. Method name mismatches in the Projects routes. BaseRepository expecting kwargs but receiving domain objects. Silent database errors in Files that swallowed exceptions and lied about success. Every CRUD operation failed at the moment it tried to touch the database, but all the code before that point—including all the unit tests with mocks—passed perfectly.</p><p>Schema/model type drift. It explained everything. The previous five “fixes” had been addressing symptoms of this single root cause. We changed five models from String to postgresql.UUID(as_uuid=False), and suddenly all four entity pages worked.</p><p>Why had this escaped detection so long? Unit tests with mocks bypass database type checking. The migrations changed the schema but nobody updated the models. PostgreSQL is strict about types in ways that auto-casting ORMs hide. We’d been working with a foundational mismatch that only manifested at SQL execution time with a real database.</p><p>Integration testing isn’t about running more tests. It’s about running tests that touch the actual integration points — real databases, real type systems, real constraints. The tests pass when they mock away the very things that break in production.</p><h3>Velocity Through Systematic Work</h3><p>Monday morning, with CRUD operations finally working, we moved to cleanup. From “make it work” to “make it right.”</p><p>The crisis teaches you what’s broken. The week after teaches you how to build sustainably.</p><p><em>Next in series: The Milestone Pause, as I take stock and figure out what needs attention next, and overall I slow my pace.</em></p><p><em>Have you ever had a period of consolidation and cleanup after a heavy-duty late night or weekend push? Tell me about it.</em></p><img src="https://medium.com/_/stat?event=post.clientViewed&referrerSource=full_rss&postId=500d1e637647" width="1" height="1" alt=""><hr><p><a href="https://medium.com/building-piper-morgan/the-week-after-500d1e637647">The Week After</a> was originally published in <a href="https://medium.com/building-piper-morgan">Building Piper Morgan</a> on Medium, where people are continuing the conversation by highlighting and responding to this story.</p>1c:T3ba7,<section name="b321" class="section section--body section--first"><div class="section-divider"><hr class="section-divider"></div><div class="section-content"><div class="section-inner sectionLayout--insetColumn"><h3 name="6fe7" id="6fe7" class="graf graf--h3 graf--leading graf--title">The Week After</h3><figure name="e3f8" id="e3f8" class="graf graf--figure graf--startsWithDoubleQuote graf-after--h3"><img class="graf-image" data-image-id="1*A8l07trcU0uaVNktD0Hbpg.png" data-width="1536" data-height="1024" data-is-featured="true" alt="A robot installs a newfangled donut machine" src="https://cdn-images-1.medium.com/max/800/1*A8l07trcU0uaVNktD0Hbpg.png"><figcaption class="imageCaption">“This will be so much better!”</figcaption></figure><p name="eb2f" id="eb2f" class="graf graf--p graf-after--figure"><em class="markup--em markup--p-em">December 5–9, 2025</em></p><p name="611e" id="611e" class="graf graf--p graf--hasDropCapModel graf--hasDropCap graf-after--p"><span class="graf-dropCap">T</span>he crisis resolved Thursday night at 10:19 PM. By the time I closed our laptops, all four entity pages worked for actual users. Friday morning brought something different. Not the crisis or urgency of the past few days. Just seven agents working in parallel on what comes next.</p><h3 name="61ad" id="61ad" class="graf graf--h3 graf-after--p">Consolidation as Strategy</h3><p name="22a6" id="22a6" class="graf graf--p graf-after--h3">The <a href="https://www.linkedin.com/pulse/weekly-ship-020-convergence-christian-crumlish-bamcc/?trackingId=e%2B84s7veToSoVdexXthDHA%3D%3D" data-href="https://www.linkedin.com/pulse/weekly-ship-020-convergence-christian-crumlish-bamcc/?trackingId=e%2B84s7veToSoVdexXthDHA%3D%3D" class="markup--anchor markup--p-anchor" rel="noopener" target="_blank">Weekly Ship</a> went out first thing early on Friday. We’d spent Thursday night documenting the integration marathon, and now it was time to publish what we’d learned.</p><p name="afe9" id="afe9" class="graf graf--p graf-after--p">Then came the backlog. Twenty-two open beads — small tracked tasks that accumulate during sprint work — sat in the system waiting for attention. Not bugs exactly, not features exactly, just the organizational debris that builds up when you’re moving fast. The kind of thing that feels low-priority until you realize you’re spending mental energy remembering them all.</p><p name="4adb" id="4adb" class="graf graf--p graf-after--p">The solution wasn’t to grind through twenty-two items. It was to recognize patterns and consolidate. Four GitHub epics emerged from the chaos: SEC-RBAC (security track, role-based access control epic) work, infrastructure improvements, Slack TDD gaps, and test debt. Twenty-two scattered items became four coherent initiatives. The backlog went from “things we should probably do sometime” to “here are four clear next steps when we’re ready.”</p><p name="eebb" id="eebb" class="graf graf--p graf-after--p">That consolidation took a little time (those productive wee hours before I focus on my day-job priorities). But it freed up cognitive space that had been holding all those individual items. Sometimes the most productive work is deciding what <em class="markup--em markup--p-em">not</em> to do right now, and putting everything else into a structure that doesn’t require constant remembering.</p><p name="0ca1" id="0ca1" class="graf graf--p graf-after--p">Meanwhile, the mobile track continued is on hold until I have time to get Xcode setup properly on my Mac, because the Expo app I had hoped to protoype with is throwing some some of version mismatch error.</p><p name="1464" id="1464" class="graf graf--p graf-after--p">By lunchtime Friday, four parallel tracks had completed their work without a single coordination issue. Seven agents, four subagents, zero conflicts. Not because we’d built elaborate coordination systems, but because each track had clear boundaries and everyone stayed in their lane.</p><h3 name="d7f6" id="d7f6" class="graf graf--h3 graf-after--p">The Database Layer</h3><p name="ad5e" id="ad5e" class="graf graf--p graf-after--h3">Sunday morning brought a different kind of problem. CRUD operations were failing again — different symptoms than Thursday’s integration gaps, but same result: users couldn’t create todos, couldn’t upload files, couldn’t add projects. The mobile testing revealed something broken deeper in the stack.</p><p name="fb2a" id="fb2a" class="graf graf--p graf-after--p">Twenty-four hours later, we’d discovered <strong class="markup--strong markup--p-strong"><em class="markup--em markup--p-em">six</em></strong> root causes. Each one seemed like <em class="markup--em markup--p-em">the</em> problem until fixing it revealed the next layer. This is where the habit of asking the LLMs to look again, to dig deeper, to think categorically, to notice antipatterns, to find systematic issues really pays off:</p><p name="a216" id="a216" class="graf graf--p graf-after--p">Wrong repository type in dependency injection. Method name mismatches in the Projects routes. BaseRepository expecting kwargs but receiving domain objects. Silent database errors in Files that swallowed exceptions and lied about success. Missing eager loading causing “detached instance” errors.</p><p name="69f6" id="69f6" class="graf graf--p graf-after--p">But the breakthrough came at 6:48 AM when I finally understood what had been hiding under all five symptom fixes.</p><p name="c325" id="c325" class="graf graf--p graf-after--p">The database schema defined <code class="markup--code markup--p-code">owner_id</code> columns as UUID type. The SQLAlchemy models defined them as String. PostgreSQL, being strict about types, rejected operations with a type mismatch error at SQL execution time. Every CRUD operation failed at the moment it tried to touch the database, but all the code before that point—including all the unit tests with mocks—passed perfectly.</p><p name="056f" id="056f" class="graf graf--p graf-after--p">Schema/model type drift. It explained everything. The previous five “fixes” had been addressing symptoms of this single root cause. We changed five models from <code class="markup--code markup--p-code">String</code> to <code class="markup--code markup--p-code">postgresql.UUID(as_uuid=False)</code>, and suddenly all four entity pages worked.</p><p name="ae7e" id="ae7e" class="graf graf--p graf-after--p">Why had this escaped detection so long? Unit tests with mocks bypass database type checking. The migrations changed the schema but nobody updated the models. PostgreSQL is strict about types in ways that auto-casting ORMs hide. We’d been working with a foundational mismatch that only manifested at SQL execution time with a real database.</p><p name="ff27" id="ff27" class="graf graf--p graf-after--p">Integration testing isn’t about running more tests. It’s about running tests that touch the actual integration points — real databases, real type systems, real constraints. The tests pass when they mock away the very things that break in production.</p><h3 name="949f" id="949f" class="graf graf--h3 graf-after--p">Velocity Through Systematic Work</h3><p name="c96f" id="c96f" class="graf graf--p graf-after--h3">Monday morning, with CRUD operations finally working, we moved to cleanup. Six open issues, ranging from hamburger menu breakpoints to dialog styling problems. The kind of UI polish work that accumulates during alpha testing, partly because I have to use it myself and these half-assed vibe-coded UIs we’ve spun up drive me uts.</p><p name="913e" id="913e" class="graf graf--p graf-after--p">We fixed all six in a few hours. But more interesting was what happened next.</p><p name="800f" id="800f" class="graf graf--p graf-after--p">Issue #439 had been sitting in the backlog for weeks: refactor the setup wizard. The main function was 267 lines. Four separate API key sections each duplicated ~100 lines of nearly identical code. This wasn’t elegant, but it worked, and we’d been focused on more pressing problems.</p><p name="7403" id="7403" class="graf graf--p graf-after--p">Except now we had time to do it right. We extracted a single helper function that handled API key collection generically — keychain check, environment variable check, manual entry, validation. One 148-line function replaced ~400 lines of duplication.</p><p name="a91d" id="a91d" class="graf graf--p graf-after--p">The results: 82% reduction in the API key collection code, 71% reduction in the main wizard function. From 267 lines down to 76 lines for the orchestrator. No functions over 50 lines except justified complex helpers. Zero duplicate code blocks over 10 lines.</p><p name="781b" id="781b" class="graf graf--p graf-after--p">The refactoring took about two hours of focused work. But we’d done something important beforehand: we’d created a plan. Forty-five minutes analyzing the problem, identifying the patterns, designing the solution. That analysis meant the implementation went smoothly, with no surprises and no need to backtrack.</p><p name="591f" id="591f" class="graf graf--p graf-after--p">Analysis as valuable work — not delay, but acceleration. The hour spent planning saved hours of implementation thrashing.</p><h3 name="736c" id="736c" class="graf graf--h3 graf-after--p">Two Epics in One Day</h3><p name="2d34" id="2d34" class="graf graf--p graf-after--h3">Tuesday was different. Not debugging, not cleanup, but completion.</p><p name="a0e5" id="a0e5" class="graf graf--p graf-after--p">The T2 Sprint — test infrastructure improvements — had been building for weeks. We needed smoke tests, the fast subset that could run in CI/CD gates. The goal: mark ~25% of the test suite as smoke tests, target 5 seconds or less execution time.</p><p name="dedf" id="dedf" class="graf graf--p graf-after--p">By noon, we’d profiled 705 tests, identified 656 fast candidates, marked 602 tests as smoke, and validated the suite. Execution time: 2–3 seconds, 40–60% faster than the target. Phantom test rate: less than 1%, excellent test hygiene. Six GitHub issues closed.</p><p name="ff79" id="ff79" class="graf graf--p graf-after--p">The morning’s work was complete, comprehensive, and production-ready.</p><p name="3493" id="3493" class="graf graf--p graf-after--p">Then came the afternoon. The S2 Sprint — encryption at rest — needed preparation before implementation. Sensitive user data required proper cryptographic protection. But this wasn’t the kind of work you just start coding. It needed careful design, compliance mapping, architectural review.</p><p name="3d46" id="3d46" class="graf graf--p graf-after--p">We created a comprehensive review package for Ted Nadeau, our cryptographic advisor. Thirteen specific questions about the architectural approach. Five Whys analysis for every decision. GDPR and SOC2 compliance mapping. A 42-hour implementation gameplan broken into six phases with daily breakdown. Four GitHub issue templates for deferred S3 work.</p><p name="b04e" id="b04e" class="graf graf--p graf-after--p">By evening, the S2 preparation was complete. Not implemented, but ready <em class="markup--em markup--p-em">to</em> implement — with all the architectural thinking done, all the questions identified, all the compliance requirements mapped.</p><p name="94fc" id="94fc" class="graf graf--p graf-after--p">Two epics in one day. Not through heroic effort or cutting corners, but through systematic work. The T2 Sprint moved fast because the infrastructure already existed and the scope was clear. The S2 Sprint prep moved fast because we focused on the right question: “What decisions need to be made before we write any code?”</p><p name="30bd" id="30bd" class="graf graf--p graf-after--p">Preparatory work as valuable work. The five hours creating that crypto review package and implementation gameplan would save <em class="markup--em markup--p-em">days</em> when Code agents actually implemented the encryption. Every question answered upfront is a question that doesn’t derail implementation later.</p><h3 name="e8c7" id="e8c7" class="graf graf--h3 graf-after--p">What Systematic Looks Like</h3><p name="1ba2" id="1ba2" class="graf graf--p graf-after--h3">The week after the crisis looks different than the crisis itself. Thursday night’s integration marathon had dramatic tension — users blocked, bugs layered, debugging until past 10 PM. The following week had none of that drama but equal value.</p><p name="decf" id="decf" class="graf graf--p graf-after--p">Consolidation work that freed up cognitive space. Database debugging that found root causes instead of applying symptom fixes. Refactoring that eliminated technical debt. Analysis that accelerated future implementation. Epic completion that finished entire initiatives in single days.</p><p name="a48a" id="a48a" class="graf graf--p graf-after--p">This is what systematic building looks like. Not reacting to crises, but working through backlogs methodically. Not rushing fixes, but investigating thoroughly. Not just shipping code, but eliminating duplication. Not just starting work, but preparing properly.</p><p name="8da9" id="8da9" class="graf graf--p graf-after--p">The rhythm matters as much as the outcomes. Friday’s seven-agent orchestration showed we could coordinate at scale without coordination overhead. Monday’s six-hour refactoring session showed we could do careful work without deadline pressure. Tuesday’s two-epic day showed that preparatory work creates velocity, not drag.</p><p name="f3ce" id="f3ce" class="graf graf--p graf-after--p">By Tuesday evening, we had:</p><ul class="postList"><li name="0acf" id="0acf" class="graf graf--li graf-after--p">Clean backlog (22 items → 4 clear epics)</li><li name="6597" id="6597" class="graf graf--li graf-after--li">Working CRUD operations (six-layer root cause resolved)</li><li name="f501" id="f501" class="graf graf--li graf-after--li">Drastically simpler codebase (71–82% reduction in key areas)</li><li name="4a97" id="4a97" class="graf graf--li graf-after--li">Complete test infrastructure (T2 Sprint done)</li><li name="8499" id="8499" class="graf graf--li graf-after--li">Ready encryption implementation (S2 Sprint prep done)</li><li name="42c7" id="42c7" class="graf graf--li graf-after--li">Zero open P0 or P1 bugs</li></ul><p name="138f" id="138f" class="graf graf--p graf-after--li">None of this made for dramatic storytelling in the moment. There was no 10 PM breakthrough, no crisis resolution, no “it finally works” catharsis. Just five days of systematic execution, each building on the previous day’s work.</p><p name="7d93" id="7d93" class="graf graf--p graf-after--p">But that’s the point. After you resolve the crisis and fix the integration gaps, the work doesn’t stop — it changes character. From reactive to systematic. From urgent to deliberate. From “make it work” to “make it right.”</p><p name="0dfa" id="0dfa" class="graf graf--p graf-after--p graf--trailing">The crisis teaches you what’s broken.
