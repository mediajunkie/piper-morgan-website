1b:T308d,<figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*KvRoORTxLvltb-fKXtAh6g.png" /><figcaption>“Something’s not connected right”</figcaption></figure><p><em>February 14–20</em></p><p>The M0 sprint was supposed to take two to three weeks. We finished it in three days. We bragged about that in <a href="https://www.linkedin.com/pulse/weekly-ship-031-assembly-required-christian-crumlish-bivfc/">Weekly Ship #031</a>, with a tiny bit of prescient hedging.</p><p>Five features. Over five hundred tests. A Lead Developer marathon that started before dawn on February 16 and didn’t stop until February 18. By Wednesday evening, every feature was complete, every test was green, and the sprint gate had evidence posted for all five capabilities.</p><p>I should have been celebrating. Instead, I was staring at a list of nine things that didn’t work.</p><h3>The sprint that wasn’t supposed to be fast</h3><p>M0 — “Conversational Glue” — was our most ambitious sprint yet. Five interconnected features that would transform Piper from a capable but mechanical assistant into something that felt more like a colleague:</p><ul><li><strong>Narrative onboarding</strong>: Recognizing when someone mentions their “main project” and responding naturally</li><li><strong>Lens tracking</strong>: Following up on things mentioned earlier in conversation</li><li><strong>Slot filling</strong>: Gathering information through dialogue rather than forms</li><li><strong>Multi-intent handling</strong>: Decomposing complex requests into manageable pieces</li><li><strong>Soft invocation</strong>: Recognizing implied needs and offering to help</li></ul><p>Each feature had its own issue, its own gameplan, its own acceptance criteria. Each was implemented with rigorous test coverage. Each passed.</p><p>The sprint estimate of 13–22 days came from the scope. Five features, each touching multiple layers of the system, each requiring careful integration with existing capabilities. The three-day completion came from something else entirely — a Lead Developer session that ran twelve hours straight, methodically working through each feature with the kind of focus that only happens when everything else falls away.</p><p>By Wednesday evening, I had a complete sprint. Five hundred thirty-three new tests, all green. Sprint gate evidence posted. Ready for review.</p><h3>The first hint</h3><p>The trouble started almost immediately.</p><p>After closing the last feature (#767, soft invocation detection), the Lead Developer did what we always do: a quick sanity check to make sure everything hangs together. The check revealed something uncomfortable.</p><p>Each feature worked perfectly in isolation. But when you tried to use them together — the way an actual user would — they didn’t connect.</p><p>Soft invocation detection worked beautifully in its own tests. It recognized phrases like “I need to schedule that meeting” and correctly identified them as implied workflow needs. But the detected invocations never appeared in actual responses. The feature existed in a bubble, producing outputs that nothing consumed.</p><p>Lens tracking worked too. It could identify when a user referenced something from earlier in the conversation. But it wasn’t wired into the response pipeline. The lens was extracted, logged, and then… nothing.</p><p>The pattern repeated across all five features. Each one complete. Each one tested. Each one disconnected from the others.</p><h3>Nine gaps</h3><p>A deeper investigation revealed the full scope. Nine integration gaps, ranging from trivial (a missing two-line wire-up) to substantial (an entire subsystem with 124 tests and zero consumers):</p><p>The soft invocation detector produced offers that were never appended to responses. The lens extraction system tracked context that was never used. The offer accept/decline cycle had no way to complete — users could trigger an offer but couldn’t respond to it. The slot filling framework, with its elegant state machine and comprehensive test coverage, was entirely orphaned. Nothing in the system ever called it.</p><p>Some gaps were architectural. The trust stage was hardcoded to “BUILDING” because nobody had wired in the actual trust computation. Some were subtle. The lens stack had push and pop methods that were designed, documented, tested — and never invoked.</p><p>Each gap was small in isolation. A few lines here, a missing call there. But together they meant that M0, despite being “complete,” didn’t actually work.</p><h3>The five whys</h3><p>We use a technique called “five whys” to find root causes. You ask why something happened, then why <em>that</em> happened, and keep going until you hit something structural.</p><p><strong>Why didn’t the features integrate?</strong> Because they were implemented as independent units.</p><p><strong>Why were they implemented independently?</strong> Because each gameplan was scoped per-issue, with acceptance criteria that could be met in isolation.</p><p><strong>Why were the gameplans scoped that way?</strong> Because the issues were written as independent capabilities, not user journeys.</p><p><strong>Why were the issues written that way?</strong> Because the epic was decomposed into horizontal capability slices, not vertical user experience slices.</p><p><strong>Why horizontal slices?</strong> Because that’s how we think about features. Each one a capability. Each one testable. Each one… separate.</p><p>The root cause wasn’t a bug. It wasn’t a missed requirement. It was an assumption baked into how we planned the work.</p><p>We assumed that individually correct components would compose into a correct system.</p><p>They didn’t.</p><h3>The assembly assumption</h3><p>This pattern has a name now: the Assembly Assumption. It’s the belief — usually implicit, rarely examined — that if you build each piece correctly, the whole will work correctly too.</p><p>The assumption feels reasonable. It’s how we think about most engineering. Build the engine, build the transmission, build the wheels, connect them together, and you have a car. Each component has a specification. Meet the specification, and composition follows.</p><p>But software doesn’t work that way. The connections between components aren’t physical interfaces with obvious shapes. They’re data flows, implicit contracts, shared state. Feature A produces something. Feature B consumes something. If A produces eight fields and B needs ten, both features “work” — A produces a valid output, B handles its inputs gracefully. The gap only appears when they’re composed.</p><p>Test coverage doesn’t catch this. Each feature’s tests verify the feature. The boundaries between features — the seams where data flows from one to another — exist in the spaces between specifications. They’re systematically excluded from what we verify.</p><h3>The wiring pass</h3><p>We fixed all nine gaps in a single session. Most were trivial once identified — the kind of thing where you look at the code and say “oh, obviously that needs to connect to that.” A few required more thought. One (the unified formality system) needed architectural review and got deferred.</p><p>But fixing the gaps wasn’t the real lesson. The lesson was that we needed a new step in our process.</p><p>After a sprint completes — after all the features are implemented and all the tests are green — we now do a “wiring pass.” It’s a dedicated session whose sole purpose is verifying composition. Not debugging (you don’t know what’s broken yet). Discovery (you’re finding what was never specified).</p><p>The wiring pass has a simple protocol:</p><p><strong>Map the seams.</strong> Before touching code, draw the data flow across all features. Where does Feature A’s output become Feature B’s input? This map often reveals gaps before you look at any code.</p><p><strong>Write integration scenarios.</strong> Not integration tests — scenarios. User journeys that cross feature boundaries. “User mentions their main project, then asks a multi-part question about it, and Piper responds with appropriate formality.” These scenarios are the specification for composition that was missing from the original planning.</p><p><strong>Walk the code path.</strong> For each scenario, trace the actual execution through the composed system. Read the code. Verify that data flows correctly across boundaries. This is where implicit contract violations surface.</p><p><strong>Fix and test.</strong> For each gap discovered, create a targeted integration test, then fix the gap. The test serves as the specification that was missing.</p><p>The wiring pass isn’t glamorous. It doesn’t produce new features or impressive test counts. But it’s the difference between a sprint that’s complete and a sprint that works.</p><h3>The humbling part</h3><p>I’ve been making software for a long time. I’ve written about methodology, about verification, about the importance of systematic approaches. I should have seen this coming.</p><p>But I didn’t. I watched a three-day sprint produce five hundred tests and felt satisfied. The tests were green. The features were complete. The methodology was working.</p><p>Except it wasn’t working. It was verifying slices while missing the whole.</p><p>The Assembly Assumption is particularly insidious because it hides behind good practices. Test-driven development? We did that. Comprehensive coverage? We had it. Rigorous acceptance criteria? Every feature met them. The assumption doesn’t contradict good practice — it lives in the gaps between good practices.</p><p>Agile methodologies actually make this worse. The emphasis on small, independent stories encourages horizontal decomposition. Each story is a unit. Each unit is testable. The planning framework itself creates the blind spot.</p><p>I don’t have a perfect solution. The wiring pass helps, but it’s reactive — you still build the features separately and then discover the gaps. A better approach would catch composition issues during planning, before implementation begins. We’re working on that.</p><p>For now, the wiring pass is a checkpoint. A reminder that individually correct components don’t guarantee a correct composition. A scheduled moment to ask: does this actually work together?</p><h3>What remained</h3><p>By the end of February 18, M0 was actually complete. Not just tested-and-green complete, but wired-together-and-working complete. The wiring pass added 79 more tests — integration tests that verified the seams we’d been missing.</p><p>The sprint that was supposed to take three weeks finished in three days, plus one more day to discover that “finished” wasn’t the same as “working.”</p><p>It was a good reminder. The methodology isn’t done evolving. The patterns we trust can have blind spots. The assumptions we don’t examine are the ones that bite us.</p><p>The Assembly Assumption is documented now, as Pattern-062 in our pattern library. It joins a growing collection of things we’ve learned the hard way — not from theory, but from building something and watching it fail to compose.</p><p>Next time, we’ll plan differently. We’ll think about vertical slices, not just horizontal capabilities. We’ll schedule the wiring pass before the sprint gate, not after.</p><p>And we’ll remember: individually correct is not the same as collectively working.</p><p><em>Next on Building Piper Morgan: The Stranger Test — when someone logs in with a fresh account and no developer muscle memory, they find things a thousand tests couldn’t see.</em></p><p><em>Have you ever watched a system pass all its tests and still not work? What was hiding in the gaps?</em></p><img src="https://medium.com/_/stat?event=post.clientViewed&referrerSource=full_rss&postId=70db03f45991" width="1" height="1" alt=""><hr><p><a href="https://medium.com/building-piper-morgan/the-assembly-assumption-70db03f45991">The Assembly Assumption</a> was originally published in <a href="https://medium.com/building-piper-morgan">Building Piper Morgan</a> on Medium, where people are continuing the conversation by highlighting and responding to this story.</p>1c:T41d5,<section name="f07d" class="section section--body section--first"><div class="section-divider"><hr class="section-divider"></div><div class="section-content"><div class="section-inner sectionLayout--insetColumn"><h3 name="f163" id="f163" class="graf graf--h3 graf--leading graf--title">The Assembly Assumption</h3><figure name="e727" id="e727" class="graf graf--figure graf--startsWithDoubleQuote graf-after--h3"><img class="graf-image" data-image-id="1*KvRoORTxLvltb-fKXtAh6g.png" data-width="1536" data-height="1024" src="https://cdn-images-1.medium.com/max/800/1*KvRoORTxLvltb-fKXtAh6g.png"><figcaption class="imageCaption">“Something’s not connected right”</figcaption></figure><p name="2e9f" id="2e9f" class="graf graf--p graf-after--figure"><em class="markup--em markup--p-em">February 14–20</em></p><p name="5541" id="5541" class="graf graf--p graf--hasDropCapModel graf--hasDropCap graf-after--p"><span class="graf-dropCap">T</span>he M0 sprint was supposed to take two to three weeks. We finished it in three days. We bragged about that in <a href="https://www.linkedin.com/pulse/weekly-ship-031-assembly-required-christian-crumlish-bivfc/" data-href="https://www.linkedin.com/pulse/weekly-ship-031-assembly-required-christian-crumlish-bivfc/" class="markup--anchor markup--p-anchor" rel="noopener" target="_blank">Weekly Ship #031</a>, with a tiny bit of prescient hedging.</p><p name="5d5e" id="5d5e" class="graf graf--p graf-after--p">Five features. Over five hundred tests. A Lead Developer marathon that started before dawn on February 16 and didn’t stop until February 18. By Wednesday evening, every feature was complete, every test was green, and the sprint gate had evidence posted for all five capabilities.</p><p name="b470" id="b470" class="graf graf--p graf-after--p">I should have been celebrating. Instead, I was staring at a list of nine things that didn’t work.</p><h3 name="23c8" id="23c8" class="graf graf--h3 graf-after--p">The sprint that wasn’t supposed to be fast</h3><p name="a15a" id="a15a" class="graf graf--p graf-after--h3">M0 — “Conversational Glue” — was our most ambitious sprint yet. Five interconnected features that would transform Piper from a capable but mechanical assistant into something that felt more like a colleague:</p><ul class="postList"><li name="1dbd" id="1dbd" class="graf graf--li graf-after--p"><strong class="markup--strong markup--li-strong">Narrative onboarding</strong>: Recognizing when someone mentions their “main project” and responding naturally</li><li name="3f28" id="3f28" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Lens tracking</strong>: Following up on things mentioned earlier in conversation</li><li name="883a" id="883a" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Slot filling</strong>: Gathering information through dialogue rather than forms</li><li name="6dfc" id="6dfc" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Multi-intent handling</strong>: Decomposing complex requests into manageable pieces</li><li name="a574" id="a574" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Soft invocation</strong>: Recognizing implied needs and offering to help</li></ul><p name="f6fa" id="f6fa" class="graf graf--p graf-after--li">Each feature had its own issue, its own gameplan, its own acceptance criteria. Each was implemented with rigorous test coverage. Each passed.</p><p name="1a35" id="1a35" class="graf graf--p graf-after--p">The sprint estimate of 13–22 days came from the scope. Five features, each touching multiple layers of the system, each requiring careful integration with existing capabilities. The three-day completion came from something else entirely — a Lead Developer session that ran twelve hours straight, methodically working through each feature with the kind of focus that only happens when everything else falls away.</p><p name="9a41" id="9a41" class="graf graf--p graf-after--p">By Wednesday evening, I had a complete sprint. Five hundred thirty-three new tests, all green. Sprint gate evidence posted. Ready for review.</p><h3 name="d942" id="d942" class="graf graf--h3 graf-after--p">The first hint</h3><p name="f230" id="f230" class="graf graf--p graf-after--h3">The trouble started almost immediately.</p><p name="754c" id="754c" class="graf graf--p graf-after--p">After closing the last feature (#767, soft invocation detection), the Lead Developer did what we always do: a quick sanity check to make sure everything hangs together. The check revealed something uncomfortable.</p><p name="68b5" id="68b5" class="graf graf--p graf-after--p">Each feature worked perfectly in isolation. But when you tried to use them together — the way an actual user would — they didn’t connect.</p><p name="033e" id="033e" class="graf graf--p graf-after--p">Soft invocation detection worked beautifully in its own tests. It recognized phrases like “I need to schedule that meeting” and correctly identified them as implied workflow needs. But the detected invocations never appeared in actual responses. The feature existed in a bubble, producing outputs that nothing consumed.</p><p name="73fd" id="73fd" class="graf graf--p graf-after--p">Lens tracking worked too. It could identify when a user referenced something from earlier in the conversation. But it wasn’t wired into the response pipeline. The lens was extracted, logged, and then… nothing.</p><p name="b88a" id="b88a" class="graf graf--p graf-after--p">The pattern repeated across all five features. Each one complete. Each one tested. Each one disconnected from the others.</p><h3 name="8715" id="8715" class="graf graf--h3 graf-after--p">Nine gaps</h3><p name="487e" id="487e" class="graf graf--p graf-after--h3">A deeper investigation revealed the full scope. Nine integration gaps, ranging from trivial (a missing two-line wire-up) to substantial (an entire subsystem with 124 tests and zero consumers):</p><p name="336f" id="336f" class="graf graf--p graf-after--p">The soft invocation detector produced offers that were never appended to responses. The lens extraction system tracked context that was never used. The offer accept/decline cycle had no way to complete — users could trigger an offer but couldn’t respond to it. The slot filling framework, with its elegant state machine and comprehensive test coverage, was entirely orphaned. Nothing in the system ever called it.</p><p name="e37d" id="e37d" class="graf graf--p graf-after--p">Some gaps were architectural. The trust stage was hardcoded to “BUILDING” because nobody had wired in the actual trust computation. Some were subtle. The lens stack had push and pop methods that were designed, documented, tested — and never invoked.</p><p name="d1b9" id="d1b9" class="graf graf--p graf-after--p">Each gap was small in isolation. A few lines here, a missing call there. But together they meant that M0, despite being “complete,” didn’t actually work.</p><h3 name="5c9b" id="5c9b" class="graf graf--h3 graf-after--p">The five whys</h3><p name="83db" id="83db" class="graf graf--p graf-after--h3">We use a technique called “five whys” to find root causes. You ask why something happened, then why <em class="markup--em markup--p-em">that</em> happened, and keep going until you hit something structural.</p><p name="a4cb" id="a4cb" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why didn’t the features integrate?</strong> Because they were implemented as independent units.</p><p name="15f1" id="15f1" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why were they implemented independently?</strong> Because each gameplan was scoped per-issue, with acceptance criteria that could be met in isolation.</p><p name="97ca" id="97ca" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why were the gameplans scoped that way?</strong> Because the issues were written as independent capabilities, not user journeys.</p><p name="8edf" id="8edf" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why were the issues written that way?</strong> Because the epic was decomposed into horizontal capability slices, not vertical user experience slices.</p><p name="67a6" id="67a6" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why horizontal slices?</strong> Because that’s how we think about features. Each one a capability. Each one testable. Each one… separate.</p><p name="13c3" id="13c3" class="graf graf--p graf-after--p">The root cause wasn’t a bug. It wasn’t a missed requirement. It was an assumption baked into how we planned the work.</p><p name="7c11" id="7c11" class="graf graf--p graf-after--p">We assumed that individually correct components would compose into a correct system.</p><p name="c35b" id="c35b" class="graf graf--p graf-after--p">They didn’t.</p><h3 name="87e5" id="87e5" class="graf graf--h3 graf-after--p">The assembly assumption</h3><p name="2fb2" id="2fb2" class="graf graf--p graf-after--h3">This pattern has a name now: the Assembly Assumption. It’s the belief — usually implicit, rarely examined — that if you build each piece correctly, the whole will work correctly too.</p><p name="0795" id="0795" class="graf graf--p graf-after--p">The assumption feels reasonable. It’s how we think about most engineering. Build the engine, build the transmission, build the wheels, connect them together, and you have a car. Each component has a specification. Meet the specification, and composition follows.</p><p name="5a28" id="5a28" class="graf graf--p graf-after--p">But software doesn’t work that way. The connections between components aren’t physical interfaces with obvious shapes. They’re data flows, implicit contracts, shared state. Feature A produces something. Feature B consumes something. If A produces eight fields and B needs ten, both features “work” — A produces a valid output, B handles its inputs gracefully. The gap only appears when they’re composed.</p><p name="5c8b" id="5c8b" class="graf graf--p graf-after--p">Test coverage doesn’t catch this. Each feature’s tests verify the feature. The boundaries between features — the seams where data flows from one to another — exist in the spaces between specifications. They’re systematically excluded from what we verify.</p><h3 name="5ae5" id="5ae5" class="graf graf--h3 graf-after--p">The wiring pass</h3><p name="d392" id="d392" class="graf graf--p graf-after--h3">We fixed all nine gaps in a single session. Most were trivial once identified — the kind of thing where you look at the code and say “oh, obviously that needs to connect to that.” A few required more thought. One (the unified formality system) needed architectural review and got deferred.</p><p name="de77" id="de77" class="graf graf--p graf-after--p">But fixing the gaps wasn’t the real lesson. The lesson was that we needed a new step in our process.</p><p name="d6d2" id="d6d2" class="graf graf--p graf-after--p">After a sprint completes — after all the features are implemented and all the tests are green — we now do a “wiring pass.” It’s a dedicated session whose sole purpose is verifying composition. Not debugging (you don’t know what’s broken yet). Discovery (you’re finding what was never specified).</p><p name="e7ed" id="e7ed" class="graf graf--p graf-after--p">The wiring pass has a simple protocol:</p><p name="1518" id="1518" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Map the seams.</strong> Before touching code, draw the data flow across all features. Where does Feature A’s output become Feature B’s input? This map often reveals gaps before you look at any code.</p><p name="dedd" id="dedd" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Write integration scenarios.</strong> Not integration tests — scenarios. User journeys that cross feature boundaries. “User mentions their main project, then asks a multi-part question about it, and Piper responds with appropriate formality.” These scenarios are the specification for composition that was missing from the original planning.</p><p name="bf90" id="bf90" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Walk the code path.</strong> For each scenario, trace the actual execution through the composed system. Read the code. Verify that data flows correctly across boundaries. This is where implicit contract violations surface.</p><p name="c1e1" id="c1e1" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Fix and test.</strong> For each gap discovered, create a targeted integration test, then fix the gap. The test serves as the specification that was missing.</p><p name="c80a" id="c80a" class="graf graf--p graf-after--p">The wiring pass isn’t glamorous. It doesn’t produce new features or impressive test counts. But it’s the difference between a sprint that’s complete and a sprint that works.</p><h3 name="662d" id="662d" class="graf graf--h3 graf-after--p">The humbling part</h3><p name="191e" id="191e" class="graf graf--p graf-after--h3">I’ve been making software for a long time. I’ve written about methodology, about verification, about the importance of systematic approaches. I should have seen this coming.</p><p name="0a89" id="0a89" class="graf graf--p graf-after--p">But I didn’t. I watched a three-day sprint produce five hundred tests and felt satisfied. The tests were green. The features were complete. The methodology was working.</p><p name="859c" id="859c" class="graf graf--p graf-after--p">Except it wasn’t working. It was verifying slices while missing the whole.</p><p name="cb48" id="cb48" class="graf graf--p graf-after--p">The Assembly Assumption is particularly insidious because it hides behind good practices. Test-driven development? We did that. Comprehensive coverage? We had it. Rigorous acceptance criteria? Every feature met them. The assumption doesn’t contradict good practice — it lives in the gaps between good practices.</p><p name="b853" id="b853" class="graf graf--p graf-after--p">Agile methodologies actually make this worse. The emphasis on small, independent stories encourages horizontal decomposition. Each story is a unit. Each unit is testable. The planning framework itself creates the blind spot.</p><p name="1ccb" id="1ccb" class="graf graf--p graf-after--p">I don’t have a perfect solution. The wiring pass helps, but it’s reactive — you still build the features separately and then discover the gaps. A better approach would catch composition issues during planning, before implementation begins. We’re working on that.</p><p name="1920" id="1920" class="graf graf--p graf-after--p">For now, the wiring pass is a checkpoint. A reminder that individually correct components don’t guarantee a correct composition. A scheduled moment to ask: does this actually work together?</p><h3 name="168e" id="168e" class="graf graf--h3 graf-after--p">What remained</h3><p name="7edc" id="7edc" class="graf graf--p graf-after--h3">By the end of February 18, M0 was actually complete. Not just tested-and-green complete, but wired-together-and-working complete. The wiring pass added 79 more tests — integration tests that verified the seams we’d been missing.</p><p name="b955" id="b955" class="graf graf--p graf-after--p">The sprint that was supposed to take three weeks finished in three days, plus one more day to discover that “finished” wasn’t the same as “working.”</p><p name="fdee" id="fdee" class="graf graf--p graf-after--p">It was a good reminder. The methodology isn’t done evolving. The patterns we trust can have blind spots. The assumptions we don’t examine are the ones that bite us.</p><p name="9c4a" id="9c4a" class="graf graf--p graf-after--p">The Assembly Assumption is documented now, as Pattern-062 in our pattern library. It joins a growing collection of things we’ve learned the hard way — not from theory, but from building something and watching it fail to compose.</p><p name="f742" id="f742" class="graf graf--p graf-after--p">Next time, we’ll plan differently. We’ll think about vertical slices, not just horizontal capabilities. We’ll schedule the wiring pass before the sprint gate, not after.</p><p name="eab1" id="eab1" class="graf graf--p graf-after--p graf--trailing">And we’ll remember: individually correct is not the same as collectively working.</p></div></div>f:["$","$L1a",null,{"post":{"title":"The Assembly Assumption","excerpt":"“Something’s not connected right”February 14–20The M0 sprint was supposed to take two to three weeks.
