1b:T2cf8,<figure><img alt="A robot child proudly shows off a jigsaw puzzle while its parents (a person and a robot) point out that there is still one piece missing." src="https://cdn-images-1.medium.com/max/1024/1*ra8fet8sGHSSm2QyMMpXpg.png" /><figcaption>“Almost!”</figcaption></figure><p><em>December 4</em></p><p>Thursday evening, debugging why list creation failed. Three separate bugs, all presenting as the same symptom. Each one followed the same pattern: the infrastructure existed, the integration didn’t.</p><p>Three weeks earlier, we’d formalized the completion matrix — explicit criteria for what “done” means. And yet here we were, debugging 75% complete work. The discipline didn’t prevent these bugs. It just made them recognizable.</p><h3>The overall pattern</h3><p>The first bug: API contract defined, Pydantic models created, but endpoints still expected query parameters instead of JSON bodies. The model was there. It just wasn’t being used.</p><p>The second bug: Dependency injection pattern implemented, async generators created, but the actual dependency functions still expected middleware that was never written. The pattern existed. The connection didn’t.</p><p>The third bug: CSS design tokens defined with all our colors and spacing, but five templates never linked the stylesheet. The system was there. The templates didn’t reference it.</p><p>Three instances of the same anti-pattern: 75% complete. Scaffolded but not finished. “It’s mostly there” which means it doesn’t work.</p><h3>The signature</h3><p>75% complete work has a signature. The hard part is done — architecture designed, infrastructure built, patterns established. The easy part is skipped — wiring things together, updating call sites, linking stylesheets. It looks done because code exists, files are present, structure is visible. It doesn’t work because the connections aren’t made.</p><p>The pattern is seductive because the hard intellectual work is complete. The API contract is thoughtfully designed. The dependency injection pattern is architecturally sound. The design token system is well-structured. All the difficult decisions are made.</p><p>What’s left is mechanical — update the endpoints to use the models, change the functions to use the generators, add &lt;link&gt; tags to the templates. No deep thinking required. Just execution.</p><p>And that’s exactly why it gets skipped. The interesting work is done. What remains feels like tedious cleanup. Easy to defer, easy to forget, easy to assume someone else will handle.</p><h3>Why it’s invisible to the builder</h3><p>The person who builds infrastructure knows what’s complete and what isn’t. They’re intimately familiar with what works and what needs wiring. They test the pieces they built. Those pieces work.</p><p>But they don’t necessarily test the integration. They know the API models exist, so they assume the endpoints use them. They know the DI pattern is implemented, so they assume the functions reference it. They know the tokens are defined, so they assume the templates link them.</p><p>The assumptions are reasonable. The work looks done from the builder’s perspective. All the files exist. All the patterns are in place. It should work.</p><p>Then a user tries it. Or another developer tries to use the infrastructure. And it doesn’t work. Not because the infrastructure is bad, but because it’s not connected to the places that need it.</p><h3>The three instances</h3><p>For the API contract bug, we’d designed thoughtful Pydantic models — CreateListRequest with name and optional description fields. Clean, validated, well-structured. The models existed in the codebase. The endpoints just didn’t use them. They still expected query parameters. When the frontend sent JSON (which is what Pydantic models expect), the backend returned 422 Unprocessable Entity. The contract existed. The wiring didn’t.</p><p>For the dependency injection bug, we’d implemented the async generator pattern — get_db()yielding sessions with proper async context management. Clean session lifecycle. The pattern existed. The dependency functions still expected request.state.db, waiting for middleware to set it. The middleware was never created. The pattern existed. The connection didn’t.</p><p>For the CSS design tokens bug, we’d created a comprehensive token system — --color-primary, --spacing-md, dozens more. Professional design system, properly structured. Five templates just never included the stylesheet link. The tokens existed. The templates didn’t reference them. Buttons and forms rendered with no styling.</p><h3>Why testing doesn’t catch it</h3><p>Unit tests passed. We tested that the Pydantic models validated correctly. We tested that the async generators managed sessions properly. We tested that the CSS tokens defined the right values.</p><p>All those tests passed because they tested the infrastructure in isolation. They didn’t test integration — whether the endpoints actually used the models, whether the functions actually used the generators, whether the templates actually linked the tokens.</p><p>Integration tests should catch this. But integration tests require thinking about how pieces connect, not just whether pieces work. You have to test the wiring, not just the components.</p><p>The 75% complete pattern hides in that gap between unit tests (components work) and integration tests (components connect).</p><p>Taking a moment to plan and work through all the integration points that must succceed for a workflow to deliver can help ensure that integration testing is comprehensive and tests the right things.</p><h3>The scaffolding trap</h3><p>The pattern often starts with good architectural thinking. “We should use Pydantic models for API contracts.” “We should have a proper DI pattern.” “We should have a design token system.” All correct.</p><p>The first implementation creates the infrastructure. Models defined, patterns established, systems built. This feels like progress. It is progress — the hard architectural work is done.</p><p>But then the implementation stops at scaffolding. The infrastructure exists but isn’t fully integrated. Maybe time pressure happened. Maybe the builder moved to something else. Maybe they assumed someone else would handle the wiring.</p><p>The result: scaffolding that looks done but doesn’t function. The appearance of completion without the reality of integration.</p><h3>How to detect 75% completion</h3><p>Look for these signals.</p><p><strong>Files exist but aren’t imported</strong> — utility created but never used, helper functions defined but not called, libraries added but not integrated.</p><p><strong>Patterns defined but not applied</strong> — architecture documented, examples shown, but actual code doesn’t follow the pattern.</p><p><strong>Configuration exists but isn’t referenced </strong>— settings defined, environment variables listed, but the code reads from different sources.</p><p><strong>Tests pass but features don’t work </strong>— unit tests green, integration tests absent, users report failures.</p><p>The consistent signal: existence without integration. The thing is there but not connected.</p><h3>How to prevent it</h3><p>Make integration explicit. Don’t consider infrastructure complete until it’s integrated. The API models aren’t done when they’re defined — they’re done when the endpoints use them.</p><p>Test integration, not just components. Write tests that verify connections, not just correctness. Does the endpoint actually use the model? Does the function actually use the generator? Does the template actually link the stylesheet?</p><p>Complete before moving on. Finish integration before starting new infrastructure. Don’t leave wiring for later — later often means never.</p><p>Checklists for follow-through. After building infrastructure, check: What needs to reference this? Have all those references been updated? Have I tested the integration?</p><p>Manual testing and eyeballing the work yourself are also critical. It’s almost a guarantee that if I don’t look at it and just accept that the tests we ran are passing, that I will miss a gap.</p><h3>Today’s lesson</h3><p>Three bugs, same pattern, same evening. Each one revealed infrastructure that existed but wasn’t integrated. Fixing them wasn’t intellectually difficult — the architecture was sound. But finding them required patient investigation because they all looked complete.</p><p>The API models existed. The DI pattern existed. The design tokens existed. They just weren’t connected to the places that needed them. The scaffolding was there. The wiring wasn’t.</p><p>This is the anti-pattern of sophisticated development: we build elegant infrastructure and forget the mundane work of wiring it together. We solve the interesting architectural problems and skip the boring integration work.</p><p>I think we can chalk up this AI behavior, by the way, to the phenomenon I’ve taken to calling, “I learned it from <em>you</em>, Dad!” Why don’t LLMs always finish the job? Yes, it’s because they have no larger context and are just following limited orders, etc., but it’s also because they are trained on a semantic corpus generated by human beings who themselves… don’t always finish the job.</p><p>I mean just look at my kitchen. Stuff in the sink, cabinet doors not fully closed, crumbs on the bread board. Guilty as charged!</p><h3>The completion discipline, revisited</h3><p>Infrastructure isn’t complete when it exists. It’s complete when it’s integrated.</p><p>Models aren’t complete when they’re defined. They’re complete when endpoints use them.</p><p>Patterns aren’t complete when they’re implemented. They’re complete when code follows them.</p><p>Systems aren’t complete when they’re built. They’re complete when templates reference them.</p><p>The hard work is architecture. The discipline is follow-through. Both matter. The 75% complete pattern happens when we do the hard work and skip the discipline.</p><p>This day taught us to value both. Not just building elegant infrastructure, but completing the mundane work of wiring it together. Not just defining patterns, but ensuring they’re applied. Not just creating systems, but verifying they’re referenced.</p><p>The satisfaction isn’t in building the infrastructure. It’s in seeing the user create a list, successfully, because all three layers of infrastructure are actually connected.</p><p><em>Next on Building Piper Morgan, we resume the building narrative with “The MVP Rubric,” covering work done January 26, in which we resolved a disagreement between my UX and Product agents to come up with a better answer.</em></p><p><em>When have you seen the 75% complete pattern? It just made them recognizable.</p><h3 name="c1de" id="c1de" class="graf graf--h3 graf-after--p">The overall pattern</h3><p name="5ee8" id="5ee8" class="graf graf--p graf-after--h3">The first bug: API contract defined, Pydantic models created, but endpoints still expected query parameters instead of JSON bodies. The model was there. It just wasn’t being used.</p><p name="d9d8" id="d9d8" class="graf graf--p graf-after--p">The second bug: Dependency injection pattern implemented, async generators created, but the actual dependency functions still expected middleware that was never written. The pattern existed. The connection didn’t.</p><p name="355f" id="355f" class="graf graf--p graf-after--p">The third bug: CSS design tokens defined with all our colors and spacing, but five templates never linked the stylesheet. The system was there. The templates didn’t reference it.</p><p name="50fc" id="50fc" class="graf graf--p graf-after--p">Three instances of the same anti-pattern: 75% complete. Scaffolded but not finished. “It’s mostly there” which means it doesn’t work.</p><h3 name="d71a" id="d71a" class="graf graf--h3 graf-after--p">The signature</h3><p name="e87f" id="e87f" class="graf graf--p graf-after--h3">75% complete work has a signature. The hard part is done — architecture designed, infrastructure built, patterns established. The easy part is skipped — wiring things together, updating call sites, linking stylesheets. It looks done because code exists, files are present, structure is visible. It doesn’t work because the connections aren’t made.</p><p name="94d2" id="94d2" class="graf graf--p graf-after--p">The pattern is seductive because the hard intellectual work is complete. The API contract is thoughtfully designed. The dependency injection pattern is architecturally sound. The design token system is well-structured. All the difficult decisions are made.</p><p name="0c76" id="0c76" class="graf graf--p graf-after--p">What’s left is mechanical — update the endpoints to use the models, change the functions to use the generators, add <code class="markup--code markup--p-code">&lt;link&gt;</code> tags to the templates. No deep thinking required. Just execution.</p><p name="4444" id="4444" class="graf graf--p graf-after--p">And that’s exactly why it gets skipped. The interesting work is done. What remains feels like tedious cleanup. Easy to defer, easy to forget, easy to assume someone else will handle.</p><h3 name="eed4" id="eed4" class="graf graf--h3 graf-after--p">Why it’s invisible to the builder</h3><p name="98a7" id="98a7" class="graf graf--p graf-after--h3">The person who builds infrastructure knows what’s complete and what isn’t. They’re intimately familiar with what works and what needs wiring. They test the pieces they built. Those pieces work.</p><p name="518c" id="518c" class="graf graf--p graf-after--p">But they don’t necessarily test the integration. They know the API models exist, so they assume the endpoints use them. They know the DI pattern is implemented, so they assume the functions reference it. They know the tokens are defined, so they assume the templates link them.</p><p name="8f4c" id="8f4c" class="graf graf--p graf-after--p">The assumptions are reasonable. The work looks done from the builder’s perspective. All the files exist. All the patterns are in place. It should work.</p><p name="ef28" id="ef28" class="graf graf--p graf-after--p">Then a user tries it. Or another developer tries to use the infrastructure. And it doesn’t work. Not because the infrastructure is bad, but because it’s not connected to the places that need it.</p><h3 name="2565" id="2565" class="graf graf--h3 graf-after--p">The three instances</h3><p name="83ec" id="83ec" class="graf graf--p graf-after--h3">For the API contract bug, we’d designed thoughtful Pydantic models — <code class="markup--code markup--p-code">CreateListRequest</code> with name and optional description fields. Clean, validated, well-structured. The models existed in the codebase. The endpoints just didn’t use them. They still expected query parameters. When the frontend sent JSON (which is what Pydantic models expect), the backend returned 422 Unprocessable Entity. The contract existed. The wiring didn’t.</p><p name="c15b" id="c15b" class="graf graf--p graf-after--p">For the dependency injection bug, we’d implemented the async generator pattern — <code class="markup--code markup--p-code">get_db()</code>yielding sessions with proper async context management. Clean session lifecycle. The pattern existed. The dependency functions still expected <code class="markup--code markup--p-code">request.state.db</code>, waiting for middleware to set it. The middleware was never created. The pattern existed. The connection didn’t.</p><p name="4978" id="4978" class="graf graf--p graf-after--p">For the CSS design tokens bug, we’d created a comprehensive token system — <code class="markup--code markup--p-code">--color-primary</code>, <code class="markup--code markup--p-code">--spacing-md</code>, dozens more. Professional design system, properly structured. Five templates just never included the stylesheet link. The tokens existed. The templates didn’t reference them. Buttons and forms rendered with no styling.</p><h3 name="0e6b" id="0e6b" class="graf graf--h3 graf-after--p">Why testing doesn’t catch it</h3><p name="1cde" id="1cde" class="graf graf--p graf-after--h3">Unit tests passed. We tested that the Pydantic models validated correctly. We tested that the async generators managed sessions properly. We tested that the CSS tokens defined the right values.</p><p name="3766" id="3766" class="graf graf--p graf-after--p">All those tests passed because they tested the infrastructure in isolation. They didn’t test integration — whether the endpoints actually used the models, whether the functions actually used the generators, whether the templates actually linked the tokens.</p><p name="f6a1" id="f6a1" class="graf graf--p graf-after--p">Integration tests should catch this. But integration tests require thinking about how pieces connect, not just whether pieces work. You have to test the wiring, not just the components.</p><p name="6fbd" id="6fbd" class="graf graf--p graf-after--p">The 75% complete pattern hides in that gap between unit tests (components work) and integration tests (components connect).</p><p name="f3d0" id="f3d0" class="graf graf--p graf-after--p">Taking a moment to plan and work through all the integration points that must succceed for a workflow to deliver can help ensure that integration testing is comprehensive and tests the right things.</p><h3 name="0dfe" id="0dfe" class="graf graf--h3 graf-after--p">The scaffolding trap</h3><p name="c003" id="c003" class="graf graf--p graf-after--h3">The pattern often starts with good architectural thinking. “We should use Pydantic models for API contracts.” “We should have a proper DI pattern.” “We should have a design token system.” All correct.</p><p name="b91d" id="b91d" class="graf graf--p graf-after--p">The first implementation creates the infrastructure. Models defined, patterns established, systems built. This feels like progress. It is progress — the hard architectural work is done.</p><p name="0dcc" id="0dcc" class="graf graf--p graf-after--p">But then the implementation stops at scaffolding. The infrastructure exists but isn’t fully integrated. Maybe time pressure happened. Maybe the builder moved to something else. Maybe they assumed someone else would handle the wiring.</p><p name="23fa" id="23fa" class="graf graf--p graf-after--p">The result: scaffolding that looks done but doesn’t function. The appearance of completion without the reality of integration.</p><h3 name="dd33" id="dd33" class="graf graf--h3 graf-after--p">How to detect 75% completion</h3><p name="0566" id="0566" class="graf graf--p graf-after--h3">Look for these signals.</p><p name="e2b5" id="e2b5" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Files exist but aren’t imported</strong> — utility created but never used, helper functions defined but not called, libraries added but not integrated.</p><p name="7828" id="7828" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Patterns defined but not applied</strong> — architecture documented, examples shown, but actual code doesn’t follow the pattern.</p><p name="7d70" id="7d70" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Configuration exists but isn’t referenced </strong>— settings defined, environment variables listed, but the code reads from different sources.</p><p name="36ac" id="36ac" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Tests pass but features don’t work </strong>— unit tests green, integration tests absent, users report failures.</p><p name="cac6" id="cac6" class="graf graf--p graf-after--p">The consistent signal: existence without integration. The thing is there but not connected.</p><h3 name="4478" id="4478" class="graf graf--h3 graf-after--p">How to prevent it</h3><p name="002a" id="002a" class="graf graf--p graf-after--h3">Make integration explicit. Don’t consider infrastructure complete until it’s integrated. The API models aren’t done when they’re defined — they’re done when the endpoints use them.</p><p name="3aa5" id="3aa5" class="graf graf--p graf-after--p">Test integration, not just components. Write tests that verify connections, not just correctness. Does the endpoint actually use the model? Does the function actually use the generator? Does the template actually link the stylesheet?</p><p name="5e4e" id="5e4e" class="graf graf--p graf-after--p">Complete before moving on. Finish integration before starting new infrastructure. Don’t leave wiring for later — later often means never.</p><p name="e7f1" id="e7f1" class="graf graf--p graf-after--p">Checklists for follow-through. After building infrastructure, check: What needs to reference this? Have all those references been updated? Have I tested the integration?</p><p name="63ad" id="63ad" class="graf graf--p graf-after--p">Manual testing and eyeballing the work yourself are also critical. It’s almost a guarantee that if I don’t look at it and just accept that the tests we ran are passing, that I will miss a gap.</p><h3 name="d371" id="d371" class="graf graf--h3 graf-after--p">Today’s lesson</h3><p name="7298" id="7298" class="graf graf--p graf-after--h3">Three bugs, same pattern, same evening. Each one revealed infrastructure that existed but wasn’t integrated. Fixing them wasn’t intellectually difficult — the architecture was sound. But finding them required patient investigation because they all looked complete.</p><p name="a420" id="a420" class="graf graf--p graf-after--p">The API models existed. The DI pattern existed. The design tokens existed. They just weren’t connected to the places that needed them. The scaffolding was there. The wiring wasn’t.</p><p name="05c8" id="05c8" class="graf graf--p graf-after--p">This is the anti-pattern of sophisticated development: we build elegant infrastructure and forget the mundane work of wiring it together. We solve the interesting architectural problems and skip the boring integration work.</p><p name="95d0" id="95d0" class="graf graf--p graf-after--p">I think we can chalk up this AI behavior, by the way, to the phenomenon I’ve taken to calling, “I learned it from <em class="markup--em markup--p-em">you</em>, Dad!” Why don’t LLMs always finish the job? Yes, it’s because they have no larger context and are just following limited orders, etc., but it’s also because they are trained on a semantic corpus generated by human beings who themselves… don’t always finish the job.</p><p name="37ae" id="37ae" class="graf graf--p graf-after--p">I mean just look at my kitchen. Stuff in the sink, cabinet doors not fully closed, crumbs on the bread board. Guilty as charged!</p><h3 name="0fb2" id="0fb2" class="graf graf--h3 graf-after--p">The completion discipline, revisited</h3><p name="011c" id="011c" class="graf graf--p graf-after--h3">Infrastructure isn’t complete when it exists. It’s complete when it’s integrated.</p><p name="0528" id="0528" class="graf graf--p graf-after--p">Models aren’t complete when they’re defined. They’re complete when endpoints use them.</p><p name="5f29" id="5f29" class="graf graf--p graf-after--p">Patterns aren’t complete when they’re implemented. They’re complete when code follows them.</p><p name="8a91" id="8a91" class="graf graf--p graf-after--p">Systems aren’t complete when they’re built. They’re complete when templates reference them.</p><p name="b41f" id="b41f" class="graf graf--p graf-after--p">The hard work is architecture. The discipline is follow-through. Both matter. The 75% complete pattern happens when we do the hard work and skip the discipline.</p><p name="27bd" id="27bd" class="graf graf--p graf-after--p">This day taught us to value both. Not just building elegant infrastructure, but completing the mundane work of wiring it together. Not just defining patterns, but ensuring they’re applied. Not just creating systems, but verifying they’re referenced.</p><p name="9fd5" id="9fd5" class="graf graf--p graf-after--p graf--trailing">The satisfaction isn’t in building the infrastructure.
