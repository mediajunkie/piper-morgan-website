1b:T1818,<p><em>December 22, 2025</em></p>
<p>It was 8:06 AM on a Monday in December. I asked the Lead Developer to implement Query #2 in our list of canonical queries (dynamic capabilities, the ability for users to ask "what can you do?").
</p>
<p>The response came back in nine minutes: "Implementation already exists."
</p>
<p>Not "I'll build it." Not "here's the plan." The code was already there, written months ago, fully functional. We just didn't know it.
</p>
<h2>A recurring pattern</h2>
<p>Thus December 22nd became an archaeology day. We had set out to implement 20 canonical queries. What we found instead was that many were already built:
</p>
<ul><li><strong>Query #2</strong> (dynamic capabilities): Complete implementation in Issue #493. Code worked. Tests were missing. Issue was still marked open.</li></ul>
<ul><li><strong>Query #16</strong> (GitHub issue creation): Fully functional. Never tracked.</li></ul>
<ul><li><strong>Query #18</strong> (list projects): Working perfectly. Completely forgotten.</li></ul>
<ul><li><strong>Queries #7, #8, #14</strong>: Implementations existed but lacked documentation.</li></ul>
<p>By the end of the day, we'd "implemented" 14 queries. But 4 of them were really <em>discoveries</em>—code that existed and worked but had fallen out of organizational memory.
</p>
<p>This is the <strong>75% completion pattern</strong> applied to knowledge, not code. The work was done. The tracking was abandoned.
</p>
<h2>Why this happens with AI agents</h2>
<p>The pattern has a predictable cause: <strong>building is more interesting than closing.</strong>
</p>
<p>When you're deep in implementation, you're solving problems, making things work, feeling productive. When the code works, there's a dopamine hit. You move to the next challenge.
</p>
<p>But updating the issue tracker? Writing the tests that prove it works? Closing the loop in the documentation? That's administrative. That's boring. That's "I'll do it later."
</p>
<p>Later never comes. The code sits there, working perfectly, invisible to everyone including your future self.
</p>
<p>Sounds like people problems, right? Yes, they learned it from us.
</p>
<h2>The ongoing cost</h2>
<p>That day, we had to spend time investigating how to implement features that already existed. We wrote gameplans for code that was already written. We deployed agents to build things that were already built.
</p>
<p>That's pure waste. But it's not even the worst cost.
</p>
<p>The worse cost is <strong>confidence erosion</strong>. When you don't know what you have, you can't trust your own system. Every new feature request triggers investigation instead of decision-making. "Can we do X?" becomes a research project instead of a yes/no answer.
</p>
<p>And there's a third cost: <strong>architectural confusion</strong>. If you don't know Query #2 is already implemented, you might implement it again—differently. Now you have two implementations of the same thing, probably with subtle inconsistencies. Future debugging becomes archaeology.
</p>
<h2>One fix we tried</h2>
<p>After December 22nd, we instituted a simple rule: <strong>every implementation gets a test, and every test gets an issue closure.</strong>
</p>
<p>Not "every implementation gets documented." Documentation can lag. But tests are executable documentation. If there's a test that passes, there's evidence the feature exists. If there's an issue closure, there's a record it was completed.
</p>
<p>The audit process became:
</p>
<ol>
<li>Before implementing anything, search for existing implementations</li>
<li>Check both the codebase AND the issue tracker</li>
<li>If code exists without tests, add tests first</li>
<li>If code exists without issue closure, close the issue with evidence</li>
<li>Only then move to new implementation</li>
</ol>
<p>This sounds bureaucratic. It's actually faster. The investigation phase that day took 15 minutes per query. Implementation for queries that actually needed building took 30-60 minutes. Finding existing code saved 15-45 minutes per discovery.
</p>
<h2>The broader lesson</h2>
<p>Since that day we have implemented auditing more broadly into most of our processes. This isn't unique to AI-assisted development, but AI easily makes it worse. When you can build things quickly, you build more things. When you build more things, you forget more things.
</p>
<p>The solution isn't to slow down building. It's to <strong>make closure as frictionless as creation</strong>.
</p>
<p>Some practices that help:
</p>
<p><strong>Test as you go, not after.</strong> Tests are searchable proof that something exists. Code without tests is code that will be forgotten.</p>
<p><strong>Close issues immediately.</strong> Not "when I have time." Not "in the weekly cleanup." The moment the code works, close the issue with evidence.</p>
<p><strong>Search before you build.</strong> Before writing a gameplan, grep the codebase. Before creating an issue, search existing issues. The five minutes of searching can save hours of duplicate work.</p>
<p><strong>Treat "already done" as success.</strong> When you discover existing code, that's a win—you just saved implementation time. Don't feel disappointed that you didn't get to build it. Feel relieved that past-you already did.</p>
<hr>
<h2>This one's meta-pattern</h2>
<p>I'm noticing something about how projects evolve. We think of software development as linear: plan, build, ship. But real projects are archaeological sites. They have layers. Things get built and buried. Features exist in strata, some documented, some not.
</p>
<p>Your job as a developer—or as someone coordinating AI developers—isn't just to build new things. It's to know what you have. To maintain an accurate map of the terrain. To remember.
</p>
<p>The alternative is to keep rediscovering your own work, forever.
</p>
