1b:T2d42,<figure><img alt="A robot mechanic shows its human partner that a clock radio has working parts in a broken whole" src="https://cdn-images-1.medium.com/max/1024/1*1Os82Mwq78eLm0Huyb8JtQ.png" /><figcaption>“Each piece is OK — it’s the connections!”</figcaption></figure><p><em>December 1–4</em></p><p>The tests were passing. The features worked. Version 0.8.2 shipped on Tuesday with authentication, error recovery, the whole setup wizard. Clean deploys, no rollbacks. Everything looked good.</p><p>Then I tried to create a list.</p><p>This is the story of what happened next — not a quick fix but a patient debugging marathon that revealed three separate layers of problems, all hiding behind a single symptom. It’s about what integration testing actually means, and why “the tests pass” is not the same as “users can use it.”</p><h3>When alpha testing becomes truth-telling</h3><p>Monday and Tuesday were productive. Auth infrastructure completed, five issues closed, new roles launching. My Principal Product Manager had their inaugural session, creating our first Product Decision Record. The Chief Architect was analyzing feedback from Ted Nadeau, who’d independently arrived at terminology compatible with our object model. Everything felt like forward motion.</p><p>Wednesday afternoon, after getting Michelle onboarded, I resumed my own alpha testing. Seven bugs found. Two critical, one high-priority, four polish issues. The critical ones weren’t subtle: chat didn’t work, create buttons failed silently. Things we’d tested, things that had passed tests, things that <em>should</em> have worked — didn’t work when a real user tried to use them.</p><p>The immediate issues were clear. Twenty-seven fetch calls across ten templates were missing credentials: &#39;include&#39; - the browser was sending cookies but our JavaScript wasn&#39;t asking for them. The standup endpoint had four separate mismatches: wrong path, wrong method, wrong request format, wrong response parsing. All fixable. All fixed by Wednesday evening.</p><p>Thursday morning, I pulled the fixes to the alpha laptop. I tried again. The chat still returned 401 errors. The standup still failed. Create buttons still didn’t work.</p><p>That’s when we discovered this wasn’t about fetch calls or endpoints. This was about layers.</p><h3>Layer one: the routes never read cookies</h3><p>The Wednesday fixes had added credentials: &#39;include&#39; to all the templates. The cookies were being sent. But Thursday morning&#39;s debugging revealed the problem: the route dependencies weren&#39;t reading them.</p><p>Our authentication middleware worked fine — it extracted cookies, validated tokens, great. But every endpoint using Depends(get_current_user) only checked the Authorization header. The dependency functions had HTTPBearer logic but no cookie fallback. The middleware extracted the token from the cookie and then... the routes ignored it because they were only looking at headers.</p><p>Sixty-plus endpoints affected. All technically correct in isolation. All failing when integrated through browser usage.</p><p>The fix took about an hour. Add request: Request parameters to the dependency functions, check request.cookies.get(&quot;auth_token&quot;) as fallback. Simple once you see it. Impossible to see until you&#39;re actually testing in a browser with cookies instead of in unit tests with mocked auth.</p><h3>Layer two: components misused</h3><p>Midday Thursday. I’d fixed the cookie issue, verified it locally, pushed to production. I pulled again. Different error: “Failed to load” messages, warning icons appearing on create buttons, confirmation dialogs showing up for form actions.</p><p>I’d been about to implement a quick fix — just swap out the dialog component for something simpler. But I stopped and asked myself what the broader pattern was. Not “how do I make this error go away” but “what does this error reveal about how we’re building?”</p><p>The pattern was component reuse without considering context. We had a confirmation dialog designed for destructive actions — Delete, Reset, things that should make you think twice. We’d reused it for forms because it was convenient. Create a list, upload a file, share a resource — all using a component designed for warnings.</p><p>The component was working as designed. The design was being used inappropriately.</p><p>This was where the Time Lord Doctrine came in handy. The Lead Developer had “felt pressure to rush” because “the PM was blocked.” But that’s not what priority means. Priority signals what to work on next. It doesn’t dictate how to work on it. The pace should remain deliberate, craft-focused. When you feel cross-pressured between speed and thoroughness, that’s the signal to stop and discuss.</p><p>We implemented a dialog mode system. Confirmation mode for destructive actions (red button, warning icon). Form mode for creation actions (blue button, no icon). Same component, context-appropriate behavior. It took longer than a quick fix would have. It solved the actual problem instead of masking symptoms.</p><h3>Layer three: the swiss cheese</h3><p>Evening Thursday. I tried again. “Failed to load lists.” “Failed to create list: Unknown error.”</p><p>Three hours of debugging revealed not one bug but three separate issues, all presenting as the same symptom:</p><p><strong>The API contract mismatch</strong>: Frontend sending JSON request bodies, backend expecting query parameters. Result: 422 Unprocessable Entity, which the error handler translated to “Unknown error.” We implemented this test-first — wrote failing tests, added Pydantic models, verified all four endpoints (lists, todos, projects, files) now matched their contracts.</p><p><strong>The dependency injection pattern</strong>: All the endpoints expected request.state.db to be set by middleware. The middleware was never created. This is why the dependency injection pattern had never worked - we&#39;d scaffolded the expectation but not the implementation. The fix changed from expecting middleware to using async generators with proper session management.</p><p><strong>The CSS design tokens</strong>: We’d defined all our colors and spacing in tokens.css. We just never linked it in five of our templates. The UI was invisibly broken - buttons and forms rendering but with no styling. Classic &quot;75% complete&quot; - the infrastructure existed, the integration didn&#39;t.</p><p>At 10:19 PM, after fixing all three issues, I deployed and tested one more time. I created a list. “List created successfully” appeared. The list showed up in the UI. It worked. Actually worked. For a real user, in a real browser, doing the thing the feature was supposed to enable.</p><p>That’s when we knew we were done.</p><h3>What integration testing actually tests</h3><p>Unit tests verify that components work in isolation. Integration tests verify they work together. But there’s a subtler layer: they verify that the <em>way</em> things work together matches how users will actually use them.</p><p>Our auth system passed all seventeen unit tests. Our feature endpoints passed their tests. But we’d never tested: log in with cookies in a browser, then try to use a feature. That gap — between “components work” and “users can use components” — is where these three layers hid.</p><p>The cookie auth issue existed because we tested auth and features separately. The dialog misuse existed because we tested components in isolation from user context. The swiss cheese bugs existed because we tested that pieces were built, not that they were fully integrated.</p><p>This isn’t an argument against unit tests. Unit tests caught real bugs and prevented regressions throughout this work. But unit tests passing is necessary, not sufficient. The discipline is to mark something done when a user can use it, not when the code is written.</p><h3>The patient investigation model</h3><p>Wednesday’s fixes took a few hours. Thursday’s took seventeen. Not because Thursday’s bugs were harder, but because Thursday required patient layer-by-layer investigation instead of quick fixes.</p><p>When I first reported the Thursday failures, each fix prompted the question: what’s the broader pattern? The cookie issue revealed that route dependencies needed to be cookie-aware. The dialog issue revealed component misuse. The swiss cheese revealed three separate “75% complete” patterns.</p><p>This is what the methodology enables. Not speed — if anything, this approach is slower. But it prevents the cycle where quick fixes create technical debt that slows everything later. You pay for thoroughness now or you pay for rework later. We chose to pay now.</p><h3>The work that happened in parallel</h3><p>While the debugging marathon unfolded, other work continued. The Chief Architect created our first Wardley map, revealing that our moat is coherence — Genesis components that work together. The Chief of Staff reorganized our workstreams from seven to six, recognizing that role proliferation needed systematic coordination. My Communications Director received an expansion brief, acknowledging earned growth into strategic responsibility.</p><p>None of this work would have been possible if we’d been fighting fires from premature releases. The patient debugging approach freed other agents to do strategic work because they trusted the integration work was being done thoroughly.</p><p>Ted Nadeau’s feedback kept arriving, independently validating our architecture. Sam Zimmerman’s ethics model kept refining. The security audit came back clean. Good work compounds when the foundation is solid.</p><h3>What we learned about “done”</h3><p>Four days. Building, releasing, testing, debugging. Tests passed on Tuesday. Real usage failed on Wednesday. Thursday revealed three layers. Thursday evening, it actually worked.</p><p>The lesson isn’t “test more” — we were testing. The lesson is “test differently.” Test the way users will use it. Test integration, not just components. Test in browsers with cookies, not just in units with mocked auth. Test that pieces connect, not just that pieces exist.</p><p>And when users find problems — because they will — investigate patiently. Each fix is an opportunity to understand broader patterns. Quick fixes mask symptoms. Thorough investigation reveals systemic issues that improve everything after.</p><p>The tests passing is not the end. Users being able to use what you built is the end. Everything looked good.</p><p name="3119" id="3119" class="graf graf--p graf-after--p">Then I tried to create a list.</p><p name="3e8d" id="3e8d" class="graf graf--p graf-after--p">This is the story of what happened next — not a quick fix but a patient debugging marathon that revealed three separate layers of problems, all hiding behind a single symptom. It’s about what integration testing actually means, and why “the tests pass” is not the same as “users can use it.”</p><h3 name="3565" id="3565" class="graf graf--h3 graf-after--p">When alpha testing becomes truth-telling</h3><p name="74c5" id="74c5" class="graf graf--p graf-after--h3">Monday and Tuesday were productive. Auth infrastructure completed, five issues closed, new roles launching. My Principal Product Manager had their inaugural session, creating our first Product Decision Record. The Chief Architect was analyzing feedback from Ted Nadeau, who’d independently arrived at terminology compatible with our object model. Everything felt like forward motion.</p><p name="4993" id="4993" class="graf graf--p graf-after--p">Wednesday afternoon, after getting Michelle onboarded, I resumed my own alpha testing. Seven bugs found. Two critical, one high-priority, four polish issues. The critical ones weren’t subtle: chat didn’t work, create buttons failed silently. Things we’d tested, things that had passed tests, things that <em class="markup--em markup--p-em">should</em> have worked — didn’t work when a real user tried to use them.</p><p name="4aaa" id="4aaa" class="graf graf--p graf-after--p">The immediate issues were clear. Twenty-seven fetch calls across ten templates were missing <code class="markup--code markup--p-code">credentials: &#39;include&#39;</code> - the browser was sending cookies but our JavaScript wasn&#39;t asking for them. The standup endpoint had four separate mismatches: wrong path, wrong method, wrong request format, wrong response parsing. All fixable. All fixed by Wednesday evening.</p><p name="72e9" id="72e9" class="graf graf--p graf-after--p">Thursday morning, I pulled the fixes to the alpha laptop. I tried again. The chat still returned 401 errors. The standup still failed. Create buttons still didn’t work.</p><p name="471a" id="471a" class="graf graf--p graf-after--p">That’s when we discovered this wasn’t about fetch calls or endpoints. This was about layers.</p><h3 name="144e" id="144e" class="graf graf--h3 graf-after--p">Layer one: the routes never read cookies</h3><p name="2fc7" id="2fc7" class="graf graf--p graf-after--h3">The Wednesday fixes had added <code class="markup--code markup--p-code">credentials: &#39;include&#39;</code> to all the templates. The cookies were being sent. But Thursday morning&#39;s debugging revealed the problem: the route dependencies weren&#39;t reading them.</p><p name="0100" id="0100" class="graf graf--p graf-after--p">Our authentication middleware worked fine — it extracted cookies, validated tokens, great. But every endpoint using <code class="markup--code markup--p-code">Depends(get_current_user)</code> only checked the Authorization header. The dependency functions had HTTPBearer logic but no cookie fallback. The middleware extracted the token from the cookie and then... the routes ignored it because they were only looking at headers.</p><p name="9df6" id="9df6" class="graf graf--p graf-after--p">Sixty-plus endpoints affected. All technically correct in isolation. All failing when integrated through browser usage.</p><p name="6369" id="6369" class="graf graf--p graf-after--p">The fix took about an hour. Add <code class="markup--code markup--p-code">request: Request</code> parameters to the dependency functions, check <code class="markup--code markup--p-code">request.cookies.get(&quot;auth_token&quot;)</code> as fallback. Simple once you see it. Impossible to see until you&#39;re actually testing in a browser with cookies instead of in unit tests with mocked auth.</p><h3 name="550a" id="550a" class="graf graf--h3 graf-after--p">Layer two: components misused</h3><p name="28c4" id="28c4" class="graf graf--p graf-after--h3">Midday Thursday. I’d fixed the cookie issue, verified it locally, pushed to production. I pulled again. Different error: “Failed to load” messages, warning icons appearing on create buttons, confirmation dialogs showing up for form actions.</p><p name="e184" id="e184" class="graf graf--p graf-after--p">I’d been about to implement a quick fix — just swap out the dialog component for something simpler. But I stopped and asked myself what the broader pattern was. Not “how do I make this error go away” but “what does this error reveal about how we’re building?”</p><p name="3aca" id="3aca" class="graf graf--p graf-after--p">The pattern was component reuse without considering context. We had a confirmation dialog designed for destructive actions — Delete, Reset, things that should make you think twice. We’d reused it for forms because it was convenient. Create a list, upload a file, share a resource — all using a component designed for warnings.</p><p name="c04a" id="c04a" class="graf graf--p graf-after--p">The component was working as designed. The design was being used inappropriately.</p><p name="1988" id="1988" class="graf graf--p graf-after--p">This was where the Time Lord Doctrine came in handy. The Lead Developer had “felt pressure to rush” because “the PM was blocked.” But that’s not what priority means. Priority signals what to work on next. It doesn’t dictate how to work on it. The pace should remain deliberate, craft-focused. When you feel cross-pressured between speed and thoroughness, that’s the signal to stop and discuss.</p><p name="75a4" id="75a4" class="graf graf--p graf-after--p">We implemented a dialog mode system. Confirmation mode for destructive actions (red button, warning icon). Form mode for creation actions (blue button, no icon). Same component, context-appropriate behavior. It took longer than a quick fix would have. It solved the actual problem instead of masking symptoms.</p><h3 name="f7d9" id="f7d9" class="graf graf--h3 graf-after--p">Layer three: the swiss cheese</h3><p name="f9cf" id="f9cf" class="graf graf--p graf-after--h3">Evening Thursday. I tried again. “Failed to load lists.” “Failed to create list: Unknown error.”</p><p name="9a7c" id="9a7c" class="graf graf--p graf-after--p">Three hours of debugging revealed not one bug but three separate issues, all presenting as the same symptom:</p><p name="8d68" id="8d68" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">The API contract mismatch</strong>: Frontend sending JSON request bodies, backend expecting query parameters. Result: 422 Unprocessable Entity, which the error handler translated to “Unknown error.” We implemented this test-first — wrote failing tests, added Pydantic models, verified all four endpoints (lists, todos, projects, files) now matched their contracts.</p><p name="7d5c" id="7d5c" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">The dependency injection pattern</strong>: All the endpoints expected <code class="markup--code markup--p-code">request.state.db</code> to be set by middleware. The middleware was never created. This is why the dependency injection pattern had never worked - we&#39;d scaffolded the expectation but not the implementation. The fix changed from expecting middleware to using async generators with proper session management.</p><p name="a216" id="a216" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">The CSS design tokens</strong>: We’d defined all our colors and spacing in <code class="markup--code markup--p-code">tokens.css</code>. We just never linked it in five of our templates. The UI was invisibly broken - buttons and forms rendering but with no styling. Classic &quot;75% complete&quot; - the infrastructure existed, the integration didn&#39;t.</p><p name="036e" id="036e" class="graf graf--p graf-after--p">At 10:19 PM, after fixing all three issues, I deployed and tested one more time. I created a list. “List created successfully” appeared. The list showed up in the UI. It worked. Actually worked. For a real user, in a real browser, doing the thing the feature was supposed to enable.</p><p name="c771" id="c771" class="graf graf--p graf-after--p">That’s when we knew we were done.</p><h3 name="75c1" id="75c1" class="graf graf--h3 graf-after--p">What integration testing actually tests</h3><p name="86f2" id="86f2" class="graf graf--p graf-after--h3">Unit tests verify that components work in isolation. Integration tests verify they work together. But there’s a subtler layer: they verify that the <em class="markup--em markup--p-em">way</em> things work together matches how users will actually use them.</p><p name="2506" id="2506" class="graf graf--p graf-after--p">Our auth system passed all seventeen unit tests. Our feature endpoints passed their tests. But we’d never tested: log in with cookies in a browser, then try to use a feature. That gap — between “components work” and “users can use components” — is where these three layers hid.</p><p name="bdb6" id="bdb6" class="graf graf--p graf-after--p">The cookie auth issue existed because we tested auth and features separately. The dialog misuse existed because we tested components in isolation from user context. The swiss cheese bugs existed because we tested that pieces were built, not that they were fully integrated.</p><p name="710f" id="710f" class="graf graf--p graf-after--p">This isn’t an argument against unit tests. Unit tests caught real bugs and prevented regressions throughout this work. But unit tests passing is necessary, not sufficient. The discipline is to mark something done when a user can use it, not when the code is written.</p><h3 name="b083" id="b083" class="graf graf--h3 graf-after--p">The patient investigation model</h3><p name="a219" id="a219" class="graf graf--p graf-after--h3">Wednesday’s fixes took a few hours. Thursday’s took seventeen. Not because Thursday’s bugs were harder, but because Thursday required patient layer-by-layer investigation instead of quick fixes.</p><p name="97fc" id="97fc" class="graf graf--p graf-after--p">When I first reported the Thursday failures, each fix prompted the question: what’s the broader pattern? The cookie issue revealed that route dependencies needed to be cookie-aware. The dialog issue revealed component misuse. The swiss cheese revealed three separate “75% complete” patterns.</p><p name="6695" id="6695" class="graf graf--p graf-after--p">This is what the methodology enables. Not speed — if anything, this approach is slower. But it prevents the cycle where quick fixes create technical debt that slows everything later. You pay for thoroughness now or you pay for rework later. We chose to pay now.</p><h3 name="c041" id="c041" class="graf graf--h3 graf-after--p">The work that happened in parallel</h3><p name="867b" id="867b" class="graf graf--p graf-after--h3">While the debugging marathon unfolded, other work continued. The Chief Architect created our first Wardley map, revealing that our moat is coherence — Genesis components that work together. The Chief of Staff reorganized our workstreams from seven to six, recognizing that role proliferation needed systematic coordination. My Communications Director received an expansion brief, acknowledging earned growth into strategic responsibility.</p><p name="1d99" id="1d99" class="graf graf--p graf-after--p">None of this work would have been possible if we’d been fighting fires from premature releases. The patient debugging approach freed other agents to do strategic work because they trusted the integration work was being done thoroughly.</p><p name="8dc2" id="8dc2" class="graf graf--p graf-after--p">Ted Nadeau’s feedback kept arriving, independently validating our architecture. Sam Zimmerman’s ethics model kept refining. The security audit came back clean. Good work compounds when the foundation is solid.</p><h3 name="e470" id="e470" class="graf graf--h3 graf-after--p">What we learned about “done”</h3><p name="5202" id="5202" class="graf graf--p graf-after--h3">Four days. Building, releasing, testing, debugging. Tests passed on Tuesday. Real usage failed on Wednesday. Thursday revealed three layers. Thursday evening, it actually worked.</p><p name="bc4e" id="bc4e" class="graf graf--p graf-after--p">The lesson isn’t “test more” — we were testing. The lesson is “test differently.” Test the way users will use it. Test integration, not just components. Test in browsers with cookies, not just in units with mocked auth. Test that pieces connect, not just that pieces exist.</p><p name="e8aa" id="e8aa" class="graf graf--p graf-after--p">And when users find problems — because they will — investigate patiently. Each fix is an opportunity to understand broader patterns. Quick fixes mask symptoms. Thorough investigation reveals systemic issues that improve everything after.</p><p name="969e" id="969e" class="graf graf--p graf-after--p graf--trailing">The tests passing is not the end. Users being able to use what you built is the end.
