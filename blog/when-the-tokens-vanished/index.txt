The OAuth flow completed successfully. Tokens were stored. The system reported success at every step.</p><p>But nobody could actually use Slack through Piper. Every attempt failed silently. The connection existed on paper but not in practice.</p><p>It took a systematic audit to find out why.</p><h3>The audit that found nothing (and then everything)</h3><p>February 25 started with a security review. After the fresh-account testing revealed authentication bugs in the calendar adapter, we decided to audit all our keychain usage. The keychain stores OAuth tokens and other sensitive credentials, and we’d discovered that some code was using inconsistent key patterns — hardcoded strings instead of user-scoped keys.</p><p>The Lead Developer started with a simple question: how many places in the codebase access the keychain, and are they all using the correct scoping pattern?</p><p>The answer was worse than expected. Fifteen sites across five categories were using non-scoped keys. But the Slack integration stood out for a different reason.</p><p>The audit found two lines of code that told a complete story:</p><p><strong>Storing the token:</strong></p><pre>keychain.store(&quot;slack_oauth_{username}&quot;, token)</pre><p><strong>Retrieving the token:</strong></p><pre>token = keychain.get(f&quot;slack_oauth_{username}&quot;)</pre><p>Do you see it?</p><h3>The f-string that wasn’t</h3><p>In Python, f-strings are a way to embed variables in text. When you write f&quot;hello_{name}&quot;, Python substitutes the value of name into the string. If name is “alice”, you get “hello_alice”.</p><p>But f-strings only work if you include the f prefix. Without it, &quot;hello_{name}&quot; is just a literal string containing the characters {, n, a, m, e, }.</p><p>The storage line was missing the f prefix. Every token was being stored under the literal key &quot;slack_oauth_{username}&quot; — the actual characters, curly braces and all.</p><p>The retrieval line had the f prefix. It was looking for &quot;slack_oauth_alice&quot; or &quot;slack_oauth_bob&quot; — the properly formatted, user-scoped key.</p><p>The keys never matched. Every token stored was immediately unretrievable.</p><h3>How this stayed hidden</h3><p>The bug had been in production for months. How did nobody notice?</p><p>First: the OAuth flow appeared to work. Users clicked “Connect Slack,” went through the OAuth dance, got redirected back with a success message. The token was stored successfully — just under the wrong key. Nothing failed. Nothing threw an error. The user saw “Connected” and assumed it was.</p><p>Second: the failure was silent. When Piper tried to use Slack features, the token retrieval returned null. The code handled null gracefully — it just didn’t do the Slack operation. No error message, no notification. The feature quietly didn’t work.</p><p>Third: testing didn’t catch it. The test fixtures for Slack used mock keychain operations that didn’t actually go through the store-then-retrieve cycle. The store was mocked. The retrieve was mocked. The mismatch between them was invisible.</p><p>Fourth: developers didn’t use it. The Slack integration was there, but it wasn’t a core workflow. Calendar was more heavily used. Projects were more heavily used. Slack was a secondary feature that people connected and then… didn’t really use, because it didn’t work, and they assumed that was just how it was.</p><p>The bug persisted because every system that could have caught it was designed to verify components, not journeys. The OAuth flow worked. The token storage worked. The token retrieval worked. The fact that storage and retrieval were using different keys — that lived in the gap between verifications.</p><h3>The audit cascade</h3><p>Finding the bug was only the beginning. The Lead Developer’s response was methodical:</p><p><strong>Step 1: Scope the problem.</strong> How many keychain sites had similar issues? The audit found 15 total across five categories: OAuth tokens, API keys, user preferences, session data, and integration credentials.</p><p><strong>Step 2: Categorize and prioritize.</strong> Some non-scoped keys were benign (system-wide configuration that genuinely shouldn’t be user-scoped). Others were serious (anything involving authentication or user data). The Slack bug was the most severe — actively breaking functionality — but there were others that could have caused problems.</p><p><strong>Step 3: Fix systematically.</strong> Rather than playing whack-a-mole with individual bugs, the Lead Developer created a pattern: every keychain access would use a consistent, user-scoped key format. Then applied that pattern everywhere.</p><p><strong>Step 4: Prevent recurrence.</strong> A CI guard was added — a grep check that fails the build if it detects keychain access patterns without user scoping. The bug that hid for months could now never be reintroduced.</p><p><strong>Step 5: Test the seams.</strong> Twenty-five new tests were added, specifically testing the store-then-retrieve cycle for each integration. Not mocked storage and mocked retrieval — actual round trips through the real keychain layer.</p><p>The whole fix took a few hours. The bug had been hiding for months.</p><h3>What the f-string teaches</h3><p>There’s a category of bug that exists not because code is wrong, but because two pieces of correct code don’t match. The storage line was valid Python. The retrieval line was valid Python. Each worked as written. Together, they created a gap.</p><p>These bugs are invisible to most verification approaches:</p><ul><li><strong>Unit tests</strong> verify each function in isolation. Both the store and retrieve functions worked correctly — they just weren’t called with matching arguments.</li><li><strong>Integration tests</strong> often mock the components they’re integrating with. The mock didn’t reproduce the mismatch.</li><li><strong>Code review</strong> reads code linearly. The storage was in one file, the retrieval in another. Nobody was looking at both simultaneously and comparing the string formats.</li><li><strong>User testing</strong> showed a successful connection. The failure only manifested later, in a different context, as an absence rather than an error.</li></ul><p>The bug lived in the seam between two systems that never talked to each other directly. Storage didn’t know about retrieval. Retrieval didn’t know about storage. Each was tested in isolation. The gap was systematically excluded from verification.</p><h3>Systematic finding, systematic prevention</h3><p>The f-string bug could have been found by accident — someone eventually would have noticed that Slack didn’t work and dug into why. But accidental discovery is slow and unreliable. The bug persisted for months precisely because accidental discovery hadn’t happened yet.</p><p>What found it was systematic audit. Not “look for bugs” but “examine every place this pattern occurs and verify consistency.” The audit didn’t know the f-string bug existed. It was looking for a category of issue (non-scoped keychain access) and found this specific instance along the way.</p><p>This is the difference between debugging and auditing. Debugging is reactive — something broke, find out why. Auditing is proactive — nothing appears broken, verify that it isn’t.</p><p>The CI guard is the same principle applied to prevention. It doesn’t know about f-string bugs specifically. It knows about a pattern that’s dangerous (keychain access without user scoping) and flags all instances. Future developers don’t need to know this bug’s history. They just need to follow the pattern, and if they don’t, the build fails.</p><p>Systematic approaches find bugs that hide in plain sight. They catch the mismatches that each component is too local to see. They turn “we got lucky someone noticed” into “we would have caught this automatically.”</p><h3>The tokens return</h3><p>After the fix, we did something we should have done months earlier: actually tested the Slack integration end to end. OAuth flow, token storage, token retrieval, actual Slack operation.</p><p>It worked.</p><p>The tokens that had been vanishing into a mislabeled bucket were now stored where they could be found. The feature that had been silently broken was now quietly functional.</p><p>Nobody celebrated. The fix was too small, the bug too embarrassing, the months of it hiding too uncomfortable. But there was satisfaction in the systematic approach — in knowing that the audit would have found this bug whether or not we were looking for it, and that the CI guard would prevent it from ever returning.</p><h3>The question the audit answered</h3><p>Before the audit, I would have said the Slack integration worked. It went through OAuth successfully. It stored tokens. It had test coverage.</p><p>After the audit, I knew the Slack integration worked. The tokens were stored correctly. They could be retrieved. The round trip was verified.</p><p>The difference is the gap between “appears to work” and “verified to work.” Most of what we believe about our systems falls into the first category. We saw it succeed once. We assume it still does. We trust our tests to tell us if something breaks.</p><p>But tests only catch what they’re designed to catch. The f-string bug was designed (accidentally, structurally) to slip past. Not maliciously — through the ordinary, structural blind spots of how we verify software.</p><p>Systematic audits are how you find the bugs that were designed (accidentally, structurally) to remain hidden. They’re not glamorous. They don’t produce new features. But they’re the difference between thinking your OAuth works and knowing it does.</p><p><em>Next on Building Piper Morgan: Four Voices, One Spec — a research agent, a UX lead, a product manager, and graf-after--figure"><em class="markup--em markup--p-em">February 25</em></p><p name="ad7d" id="ad7d" class="graf graf--p graf-after--p">The Slack integration had been “working” for months.</p><p name="2e41" id="2e41" class="graf graf--p graf-after--p">Users could connect their Slack accounts. The OAuth flow completed successfully. Tokens were stored. The system reported success at every step.</p><p name="d55e" id="d55e" class="graf graf--p graf-after--p">But nobody could actually use Slack through Piper. Every attempt failed silently. The connection existed on paper but not in practice.</p><p name="7ea2" id="7ea2" class="graf graf--p graf-after--p">It took a systematic audit to find out why.</p><h3 name="0c92" id="0c92" class="graf graf--h3 graf-after--p">The audit that found nothing (and then everything)</h3><p name="bf4a" id="bf4a" class="graf graf--p graf-after--h3">February 25 started with a security review. After the fresh-account testing revealed authentication bugs in the calendar adapter, we decided to audit all our keychain usage. The keychain stores OAuth tokens and other sensitive credentials, and we’d discovered that some code was using inconsistent key patterns — hardcoded strings instead of user-scoped keys.</p><p name="6348" id="6348" class="graf graf--p graf-after--p">The Lead Developer started with a simple question: how many places in the codebase access the keychain, and are they all using the correct scoping pattern?</p><p name="04a8" id="04a8" class="graf graf--p graf-after--p">The answer was worse than expected. Fifteen sites across five categories were using non-scoped keys. But the Slack integration stood out for a different reason.</p><p name="42d9" id="42d9" class="graf graf--p graf-after--p">The audit found two lines of code that told a complete story:</p><p name="e0ed" id="e0ed" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Storing the token:</strong></p><pre data-code-block-mode="2" spellcheck="false" data-code-block-lang="python" name="1ff2" id="1ff2" class="graf graf--pre graf-after--p graf--preV2"><span class="pre--content">keychain.store(<span class="hljs-string">&quot;slack_oauth_{username}&quot;</span>, token)</span></pre><p name="9dcd" id="9dcd" class="graf graf--p graf-after--pre"><strong class="markup--strong markup--p-strong">Retrieving the token:</strong></p><pre data-code-block-mode="2" spellcheck="false" data-code-block-lang="python" name="3338" id="3338" class="graf graf--pre graf-after--p graf--preV2"><span class="pre--content">token = keychain.get(<span class="hljs-string">f&quot;slack_oauth_<span class="hljs-subst">{username}</span>&quot;</span>)</span></pre><p name="1b5c" id="1b5c" class="graf graf--p graf-after--pre">Do you see it?</p><h3 name="8e44" id="8e44" class="graf graf--h3 graf-after--p">The f-string that wasn’t</h3><p name="452f" id="452f" class="graf graf--p graf-after--h3">In Python, f-strings are a way to embed variables in text. When you write <code class="markup--code markup--p-code">f&quot;hello_{name}&quot;</code>, Python substitutes the value of <code class="markup--code markup--p-code">name</code> into the string. If <code class="markup--code markup--p-code">name</code> is “alice”, you get “hello_alice”.</p><p name="dc06" id="dc06" class="graf graf--p graf-after--p">But f-strings only work if you include the <code class="markup--code markup--p-code">f</code> prefix. Without it, <code class="markup--code markup--p-code">&quot;hello_{name}&quot;</code> is just a literal string containing the characters <code class="markup--code markup--p-code">{</code>, <code class="markup--code markup--p-code">n</code>, <code class="markup--code markup--p-code">a</code>, <code class="markup--code markup--p-code">m</code>, <code class="markup--code markup--p-code">e</code>, <code class="markup--code markup--p-code">}</code>.</p><p name="4d2b" id="4d2b" class="graf graf--p graf-after--p">The storage line was missing the <code class="markup--code markup--p-code">f</code> prefix. Every token was being stored under the literal key <code class="markup--code markup--p-code">&quot;slack_oauth_{username}&quot;</code> — the actual characters, curly braces and all.</p><p name="e67a" id="e67a" class="graf graf--p graf-after--p">The retrieval line had the <code class="markup--code markup--p-code">f</code> prefix. It was looking for <code class="markup--code markup--p-code">&quot;slack_oauth_alice&quot;</code> or <code class="markup--code markup--p-code">&quot;slack_oauth_bob&quot;</code> — the properly formatted, user-scoped key.</p><p name="321a" id="321a" class="graf graf--p graf-after--p">The keys never matched. Every token stored was immediately unretrievable.</p><h3 name="48ee" id="48ee" class="graf graf--h3 graf-after--p">How this stayed hidden</h3><p name="dfb1" id="dfb1" class="graf graf--p graf-after--h3">The bug had been in production for months. How did nobody notice?</p><p name="a201" id="a201" class="graf graf--p graf-after--p">First: the OAuth flow appeared to work. Users clicked “Connect Slack,” went through the OAuth dance, got redirected back with a success message. The token was stored successfully — just under the wrong key. Nothing failed. Nothing threw an error. The user saw “Connected” and assumed it was.</p><p name="db49" id="db49" class="graf graf--p graf-after--p">Second: the failure was silent. When Piper tried to use Slack features, the token retrieval returned null. The code handled null gracefully — it just didn’t do the Slack operation. No error message, no notification. The feature quietly didn’t work.</p><p name="ff37" id="ff37" class="graf graf--p graf-after--p">Third: testing didn’t catch it. The test fixtures for Slack used mock keychain operations that didn’t actually go through the store-then-retrieve cycle. The store was mocked. The retrieve was mocked. The mismatch between them was invisible.</p><p name="6c80" id="6c80" class="graf graf--p graf-after--p">Fourth: developers didn’t use it. The Slack integration was there, but it wasn’t a core workflow. Calendar was more heavily used. Projects were more heavily used. Slack was a secondary feature that people connected and then… didn’t really use, because it didn’t work, and they assumed that was just how it was.</p><p name="900c" id="900c" class="graf graf--p graf-after--p">The bug persisted because every system that could have caught it was designed to verify components, not journeys. The OAuth flow worked. The token storage worked. The token retrieval worked. The fact that storage and retrieval were using different keys — that lived in the gap between verifications.</p><h3 name="1027" id="1027" class="graf graf--h3 graf-after--p">The audit cascade</h3><p name="25d7" id="25d7" class="graf graf--p graf-after--h3">Finding the bug was only the beginning. The Lead Developer’s response was methodical:</p><p name="78bc" id="78bc" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Step 1: Scope the problem.</strong> How many keychain sites had similar issues? The audit found 15 total across five categories: OAuth tokens, API keys, user preferences, session data, and integration credentials.</p><p name="a00e" id="a00e" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Step 2: Categorize and prioritize.</strong> Some non-scoped keys were benign (system-wide configuration that genuinely shouldn’t be user-scoped). Others were serious (anything involving authentication or user data). The Slack bug was the most severe — actively breaking functionality — but there were others that could have caused problems.</p><p name="8a48" id="8a48" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Step 3: Fix systematically.</strong> Rather than playing whack-a-mole with individual bugs, the Lead Developer created a pattern: every keychain access would use a consistent, user-scoped key format. Then applied that pattern everywhere.</p><p name="b905" id="b905" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Step 4: Prevent recurrence.</strong> A CI guard was added — a grep check that fails the build if it detects keychain access patterns without user scoping. The bug that hid for months could now never be reintroduced.</p><p name="4477" id="4477" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Step 5: Test the seams.</strong> Twenty-five new tests were added, specifically testing the store-then-retrieve cycle for each integration. Not mocked storage and mocked retrieval — actual round trips through the real keychain layer.</p><p name="ea9d" id="ea9d" class="graf graf--p graf-after--p">The whole fix took a few hours. The bug had been hiding for months.</p><h3 name="2b96" id="2b96" class="graf graf--h3 graf-after--p">What the f-string teaches</h3><p name="26d5" id="26d5" class="graf graf--p graf-after--h3">There’s a category of bug that exists not because code is wrong, but because two pieces of correct code don’t match. The storage line was valid Python. The retrieval line was valid Python. Each worked as written. Together, they created a gap.</p><p name="d8a7" id="d8a7" class="graf graf--p graf-after--p">These bugs are invisible to most verification approaches:</p><ul class="postList"><li name="408f" id="408f" class="graf graf--li graf-after--p"><strong class="markup--strong markup--li-strong">Unit tests</strong> verify each function in isolation. Both the store and retrieve functions worked correctly — they just weren’t called with matching arguments.</li><li name="5262" id="5262" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Integration tests</strong> often mock the components they’re integrating with. The mock didn’t reproduce the mismatch.</li><li name="7852" id="7852" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Code review</strong> reads code linearly. The storage was in one file, the retrieval in another. Nobody was looking at both simultaneously and comparing the string formats.</li><li name="b57c" id="b57c" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">User testing</strong> showed a successful connection. The failure only manifested later, in a different context, as an absence rather than an error.</li></ul><p name="c7f7" id="c7f7" class="graf graf--p graf-after--li">The bug lived in the seam between two systems that never talked to each other directly. Storage didn’t know about retrieval. Retrieval didn’t know about storage. Each was tested in isolation. The gap was systematically excluded from verification.</p><h3 name="3bb7" id="3bb7" class="graf graf--h3 graf-after--p">Systematic finding, systematic prevention</h3><p name="90cb" id="90cb" class="graf graf--p graf-after--h3">The f-string bug could have been found by accident — someone eventually would have noticed that Slack didn’t work and dug into why. But accidental discovery is slow and unreliable. The bug persisted for months precisely because accidental discovery hadn’t happened yet.</p><p name="6aeb" id="6aeb" class="graf graf--p graf-after--p">What found it was systematic audit. Not “look for bugs” but “examine every place this pattern occurs and verify consistency.” The audit didn’t know the f-string bug existed. It was looking for a category of issue (non-scoped keychain access) and found this specific instance along the way.</p><p name="4064" id="4064" class="graf graf--p graf-after--p">This is the difference between debugging and auditing. Debugging is reactive — something broke, find out why. Auditing is proactive — nothing appears broken, verify that it isn’t.</p><p name="6a73" id="6a73" class="graf graf--p graf-after--p">The CI guard is the same principle applied to prevention. It doesn’t know about f-string bugs specifically. It knows about a pattern that’s dangerous (keychain access without user scoping) and flags all instances. Future developers don’t need to know this bug’s history. They just need to follow the pattern, and if they don’t, the build fails.</p><p name="10dd" id="10dd" class="graf graf--p graf-after--p">Systematic approaches find bugs that hide in plain sight. They catch the mismatches that each component is too local to see. They turn “we got lucky someone noticed” into “we would have caught this automatically.”</p><h3 name="f620" id="f620" class="graf graf--h3 graf-after--p">The tokens return</h3><p name="a07c" id="a07c" class="graf graf--p graf-after--h3">After the fix, we did something we should have done months earlier: actually tested the Slack integration end to end. OAuth flow, token storage, token retrieval, actual Slack operation.</p><p name="4b03" id="4b03" class="graf graf--p graf-after--p">It worked.</p><p name="9b0e" id="9b0e" class="graf graf--p graf-after--p">The tokens that had been vanishing into a mislabeled bucket were now stored where they could be found. The feature that had been silently broken was now quietly functional.</p><p name="3148" id="3148" class="graf graf--p graf-after--p">Nobody celebrated. The fix was too small, the bug too embarrassing, the months of it hiding too uncomfortable. But there was satisfaction in the systematic approach — in knowing that the audit would have found this bug whether or not we were looking for it, and that the CI guard would prevent it from ever returning.</p><h3 name="df04" id="df04" class="graf graf--h3 graf-after--p">The question the audit answered</h3><p name="6fad" id="6fad" class="graf graf--p graf-after--h3">Before the audit, I would have said the Slack integration worked. It went through OAuth successfully. It stored tokens. It had test coverage.</p><p name="a221" id="a221" class="graf graf--p graf-after--p">After the audit, I knew the Slack integration worked. The tokens were stored correctly. They could be retrieved. The round trip was verified.</p><p name="161a" id="161a" class="graf graf--p graf-after--p">The difference is the gap between “appears to work” and “verified to work.” Most of what we believe about our systems falls into the first category. We saw it succeed once. We assume it still does. We trust our tests to tell us if something breaks.</p><p name="9e36" id="9e36" class="graf graf--p graf-after--p">But tests only catch what they’re designed to catch. The f-string bug was designed (accidentally, structurally) to slip past. Not maliciously — through the ordinary, structural blind spots of how we verify software.</p><p name="8034" id="8034" class="graf graf--p graf-after--p graf--trailing">Systematic audits are how you find the bugs that were designed (accidentally, structurally) to remain hidden. They’re not glamorous. They don’t produce new features.
