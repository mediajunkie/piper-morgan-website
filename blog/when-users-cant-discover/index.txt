Fresh install complete, setup wizard working (FK bugs from last week now fixed). Time to try actual PM workflows.</p><p>First test: “What services do you offer?”</p><p>The response came back generic. Helpful, polite, but not what I’d asked for. No menu of capabilities, no feature overview, just a conversational “I can help with various tasks” non-answer.</p><p>Second test: “Help me setup my projects.”</p><p>Piper diverted immediately to GitHub issue creation. Wrong handler entirely. I’d asked for setup guidance, got issue tracking instead.</p><p>Two simple questions. Both should have worked. Neither did.</p><h3>The investigation</h3><p>The pattern was clear: Piper assumed I knew what to ask for. Ask about “GitHub issues” specifically, you’d get GitHub help. Ask about “projects” generically, the system would guess — often wrong.</p><p>We’d built a command-oriented interface. Users needed to know the right commands, the right phrasing, the specific features available. Discovery wasn’t supported. Exploration wasn’t encouraged.</p><p>This is a classic expert user trap. When you know the system intimately, you forget that new users need to <em>find</em> features before they can <em>use</em> them.</p><p>Lead Developer ran Five Whys analysis overnight:</p><p><strong>Why did “menu of services” fail?</strong><br>No pattern matched IDENTITY queries about capabilities.</p><p><strong>Why no pattern for capability discovery?</strong><br>Pre-classifier focused on execution (GitHub, Slack, Calendar) not exploration.</p><p><strong>Why focus on execution?</strong><br>System design assumed users know what they want.</p><p><strong>Why that assumption?</strong><br>Built from developer perspective — we knew every feature.</p><p><strong>Why didn’t alpha testing catch this earlier?</strong><br>We were the alpha testers. We knew what to ask.</p><p>Root cause: architectural gap. The system had no concept of “discovery intent.” No way to handle “show me what you can do.” No bridge between plugin capabilities and user exploration.</p><p>None of this surprised me. I knew we hadn’t really addresses this gap. The awareness had been nagging at me for some time but only alpha testing brought it to the surface.</p><p>The overnight implementation added 17 new patterns — “what services,” “what can you help with,” “show me capabilities,” “help setup,” “getting started.” All the discovery phrases users would naturally try.</p><p>But patterns alone weren’t the solution. They were symptoms treatment. The real issue ran deeper.</p><h3>The architectural shift</h3><p>Sunday morning, Lead Developer created Epic #488: “Discovery-Oriented Intent Architecture.”</p><p>The epic framed it clearly: We’d built a system where users need to know what to ask for. We needed a system where users could explore what’s available.</p><p>The difference:</p><ul><li><strong>Command-oriented</strong>: “Create a GitHub issue about X” (requires knowing GitHub integration exists)</li><li><strong>Discovery-oriented</strong>: “What can you do?” → “I can help with GitHub, Slack, Calendar…” → “Tell me about GitHub” → capabilities explained</li></ul><p>This wasn’t just adding patterns. It was reframing how Piper handles uncertainty and exploration.</p><p>The proposed architecture:</p><ol><li>Add DISCOVERY as an intent category (alongside QUERY, EXECUTION, STATUS)</li><li>Route discovery requests to dedicated handler</li><li>Handler bridges to PluginRegistry metadata</li><li>Return structured capability menu</li><li>Support progressive disclosure (overview → details → usage)</li></ol><p>Phase 1 could ship with pattern matching. Long-term would integrate with the UX 2.0 modeled consciousness work — but that’s post-alpha. For now: make discovery <em>possible</em>.</p><p>Meanwhile, canonical query work ramped up. Query #7 (historical retrospective), Query #14 (project-specific status) — the queries that power morning standups and daily check-ins. Implementation work that would feed into discovery.</p><h3>Discovery becomes real</h3><p>When we were finally ready, Lead Developer coordinated twelve Code agents in parallel. Each agent got a precise prompt: specific query, specific patterns, specific formatters, specific tests.</p><p>The queries that got implemented weren’t just features getting checked off. They were validation that the discovery architecture worked:</p><p><strong>Identity queries</strong>: “What can you do?” now returns dynamic capabilities from PluginRegistry. “Are you working properly?” runs actual health checks. “How do I get help?” provides real onboarding guidance.</p><p><strong>Temporal queries</strong>: “What did we accomplish yesterday?” “When did we last work on this?” “How long have we been working on this?” All the time-based questions PMs naturally ask.</p><p><strong>Spatial queries</strong>: “What projects are we working on?” “Which project should I focus on?” “Show me the project landscape.” The workspace navigation questions.</p><p><strong>Capability queries</strong>: “Can you create a GitHub issue?” “What integrations do you have?” The meta-questions about what’s possible.</p><p>By end of day: users could ask exploration questions and get real answers. Not generic responses, not “I can help with various tasks” — actual enumeration of capabilities, actual context about what’s available.</p><p>The interesting discovery: we’d already implemented many of these queries. Query #2 (dynamic capabilities) — code existed, just untracked. Query #16 (GitHub issue creation) — working, just undocumented. Query #18 (list projects) — functional, just missing from test matrix.</p><p>The 75% completion pattern striking again. Code written, functionality working, tracking abandoned. The systematic audit revealed what was actually there versus what we thought was missing.</p><h3>Parallel momentum</h3><p>While canonical queries were getting documented and tested, a different blocker resolved.</p><p>The mobile PoC had been stuck since December 7th. Expo Go’s SDK 54 bundled incompatible native module versions — Worklets 0.5.1 native vs. 0.7.1 JavaScript. No App Store update available to fix it.</p><p>Solution deferred that week: install Xcode, build native instead of relying on Expo Go’s pre-bundled modules.</p><p>Two weeks of infrastructure work later — FK bugs, intent classification, canonical queries — development circles back to mobile. Xcode installed. Native build path available.</p><p>npx expo run:ios</p><p>All native modules compiled. Fresh native code matching JavaScript expectations exactly. iPhone simulator launched. Entity-based gesture mapping fully functional — swipe right to complete tasks, swipe left to defer, haptic feedback, intent toasts, all working.</p><p>Seven-day blocker bypassed by building native instead of debugging Expo Go’s bundled limitations.</p><p>This is a skunkworks project. It doesn’t block anything. But it will be a crucial input into UX research and understanding what the the holistic UX model should feel like for a mobile user.</p><p>The parallel progress validated something important: sustainable work doesn’t mean single-threaded work. It means systematic work that can pause, resume, and move between contexts without losing thread.</p><h3>What discovery enables</h3><p>Four days. One architectural gap identified, one epic created, one implementation sprint, one mobile breakthrough.</p><p>But the real change was conceptual, not technical.</p><p>Before: Piper waited for users to know what to ask. “Create a GitHub issue” worked. “What can you do” didn’t.</p><p>After: Piper supports exploration. Neither did.</p><h3 name="9d0b" id="9d0b" class="graf graf--h3 graf-after--p">The investigation</h3><p name="5df5" id="5df5" class="graf graf--p graf-after--h3">The pattern was clear: Piper assumed I knew what to ask for. Ask about “GitHub issues” specifically, you’d get GitHub help. Ask about “projects” generically, the system would guess — often wrong.</p><p name="1789" id="1789" class="graf graf--p graf-after--p">We’d built a command-oriented interface. Users needed to know the right commands, the right phrasing, the specific features available. Discovery wasn’t supported. Exploration wasn’t encouraged.</p><p name="9167" id="9167" class="graf graf--p graf-after--p">This is a classic expert user trap. When you know the system intimately, you forget that new users need to <em class="markup--em markup--p-em">find</em> features before they can <em class="markup--em markup--p-em">use</em> them.</p><p name="67e5" id="67e5" class="graf graf--p graf-after--p">Lead Developer ran Five Whys analysis overnight:</p><p name="8d21" id="8d21" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why did “menu of services” fail?</strong><br>No pattern matched IDENTITY queries about capabilities.</p><p name="4823" id="4823" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why no pattern for capability discovery?</strong><br>Pre-classifier focused on execution (GitHub, Slack, Calendar) not exploration.</p><p name="3e3c" id="3e3c" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why focus on execution?</strong><br>System design assumed users know what they want.</p><p name="8471" id="8471" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why that assumption?</strong><br>Built from developer perspective — we knew every feature.</p><p name="9f59" id="9f59" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Why didn’t alpha testing catch this earlier?</strong><br>We were the alpha testers. We knew what to ask.</p><p name="57e8" id="57e8" class="graf graf--p graf-after--p">Root cause: architectural gap. The system had no concept of “discovery intent.” No way to handle “show me what you can do.” No bridge between plugin capabilities and user exploration.</p><p name="3169" id="3169" class="graf graf--p graf-after--p">None of this surprised me. I knew we hadn’t really addresses this gap. The awareness had been nagging at me for some time but only alpha testing brought it to the surface.</p><p name="64ee" id="64ee" class="graf graf--p graf-after--p">The overnight implementation added 17 new patterns — “what services,” “what can you help with,” “show me capabilities,” “help setup,” “getting started.” All the discovery phrases users would naturally try.</p><p name="5721" id="5721" class="graf graf--p graf-after--p">But patterns alone weren’t the solution. They were symptoms treatment. The real issue ran deeper.</p><h3 name="7a33" id="7a33" class="graf graf--h3 graf-after--p">The architectural shift</h3><p name="6492" id="6492" class="graf graf--p graf-after--h3">Sunday morning, Lead Developer created Epic #488: “Discovery-Oriented Intent Architecture.”</p><p name="8bf3" id="8bf3" class="graf graf--p graf-after--p">The epic framed it clearly: We’d built a system where users need to know what to ask for. We needed a system where users could explore what’s available.</p><p name="8d26" id="8d26" class="graf graf--p graf-after--p">The difference:</p><ul class="postList"><li name="a745" id="a745" class="graf graf--li graf-after--p"><strong class="markup--strong markup--li-strong">Command-oriented</strong>: “Create a GitHub issue about X” (requires knowing GitHub integration exists)</li><li name="28a5" id="28a5" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Discovery-oriented</strong>: “What can you do?” → “I can help with GitHub, Slack, Calendar…” → “Tell me about GitHub” → capabilities explained</li></ul><p name="53f1" id="53f1" class="graf graf--p graf-after--li">This wasn’t just adding patterns. It was reframing how Piper handles uncertainty and exploration.</p><p name="26d3" id="26d3" class="graf graf--p graf-after--p">The proposed architecture:</p><ol class="postList"><li name="790f" id="790f" class="graf graf--li graf-after--p">Add DISCOVERY as an intent category (alongside QUERY, EXECUTION, STATUS)</li><li name="413d" id="413d" class="graf graf--li graf-after--li">Route discovery requests to dedicated handler</li><li name="4de3" id="4de3" class="graf graf--li graf-after--li">Handler bridges to PluginRegistry metadata</li><li name="68ef" id="68ef" class="graf graf--li graf-after--li">Return structured capability menu</li><li name="fb23" id="fb23" class="graf graf--li graf-after--li">Support progressive disclosure (overview → details → usage)</li></ol><p name="7438" id="7438" class="graf graf--p graf-after--li">Phase 1 could ship with pattern matching. Long-term would integrate with the UX 2.0 modeled consciousness work — but that’s post-alpha. For now: make discovery <em class="markup--em markup--p-em">possible</em>.</p><p name="f809" id="f809" class="graf graf--p graf-after--p">Meanwhile, canonical query work ramped up. Query #7 (historical retrospective), Query #14 (project-specific status) — the queries that power morning standups and daily check-ins. Implementation work that would feed into discovery.</p><h3 name="a947" id="a947" class="graf graf--h3 graf-after--p">Discovery becomes real</h3><p name="6681" id="6681" class="graf graf--p graf-after--h3">When we were finally ready, Lead Developer coordinated twelve Code agents in parallel. Each agent got a precise prompt: specific query, specific patterns, specific formatters, specific tests.</p><p name="098c" id="098c" class="graf graf--p graf-after--p">The queries that got implemented weren’t just features getting checked off. They were validation that the discovery architecture worked:</p><p name="a0ee" id="a0ee" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Identity queries</strong>: “What can you do?” now returns dynamic capabilities from PluginRegistry. “Are you working properly?” runs actual health checks. “How do I get help?” provides real onboarding guidance.</p><p name="1954" id="1954" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Temporal queries</strong>: “What did we accomplish yesterday?” “When did we last work on this?” “How long have we been working on this?” All the time-based questions PMs naturally ask.</p><p name="c4c5" id="c4c5" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Spatial queries</strong>: “What projects are we working on?” “Which project should I focus on?” “Show me the project landscape.” The workspace navigation questions.</p><p name="5d3e" id="5d3e" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Capability queries</strong>: “Can you create a GitHub issue?” “What integrations do you have?” The meta-questions about what’s possible.</p><p name="b677" id="b677" class="graf graf--p graf-after--p">By end of day: users could ask exploration questions and get real answers. Not generic responses, not “I can help with various tasks” — actual enumeration of capabilities, actual context about what’s available.</p><p name="cb47" id="cb47" class="graf graf--p graf-after--p">The interesting discovery: we’d already implemented many of these queries. Query #2 (dynamic capabilities) — code existed, just untracked. Query #16 (GitHub issue creation) — working, just undocumented. Query #18 (list projects) — functional, just missing from test matrix.</p><p name="d98a" id="d98a" class="graf graf--p graf-after--p">The 75% completion pattern striking again. Code written, functionality working, tracking abandoned. The systematic audit revealed what was actually there versus what we thought was missing.</p><h3 name="cd7e" id="cd7e" class="graf graf--h3 graf-after--p">Parallel momentum</h3><p name="1bc1" id="1bc1" class="graf graf--p graf-after--h3">While canonical queries were getting documented and tested, a different blocker resolved.</p><p name="29ed" id="29ed" class="graf graf--p graf-after--p">The mobile PoC had been stuck since December 7th. Expo Go’s SDK 54 bundled incompatible native module versions — Worklets 0.5.1 native vs. 0.7.1 JavaScript. No App Store update available to fix it.</p><p name="e774" id="e774" class="graf graf--p graf-after--p">Solution deferred that week: install Xcode, build native instead of relying on Expo Go’s pre-bundled modules.</p><p name="103b" id="103b" class="graf graf--p graf-after--p">Two weeks of infrastructure work later — FK bugs, intent classification, canonical queries — development circles back to mobile. Xcode installed. Native build path available.</p><p name="542e" id="542e" class="graf graf--p graf-after--p"><code class="markup--code markup--p-code">npx expo run:ios</code></p><p name="8a1c" id="8a1c" class="graf graf--p graf-after--p">All native modules compiled. Fresh native code matching JavaScript expectations exactly. iPhone simulator launched. Entity-based gesture mapping fully functional — swipe right to complete tasks, swipe left to defer, haptic feedback, intent toasts, all working.</p><p name="a191" id="a191" class="graf graf--p graf-after--p">Seven-day blocker bypassed by building native instead of debugging Expo Go’s bundled limitations.</p><p name="e090" id="e090" class="graf graf--p graf-after--p">This is a skunkworks project. It doesn’t block anything. But it will be a crucial input into UX research and understanding what the the holistic UX model should feel like for a mobile user.</p><p name="05cf" id="05cf" class="graf graf--p graf-after--p">The parallel progress validated something important: sustainable work doesn’t mean single-threaded work. It means systematic work that can pause, resume, and move between contexts without losing thread.</p><h3 name="5868" id="5868" class="graf graf--h3 graf-after--p">What discovery enables</h3><p name="a4fd" id="a4fd" class="graf graf--p graf-after--h3">Four days. One architectural gap identified, one epic created, one implementation sprint, one mobile breakthrough.</p><p name="37de" id="37de" class="graf graf--p graf-after--p">But the real change was conceptual, not technical.</p><p name="dc42" id="dc42" class="graf graf--p graf-after--p">Before: Piper waited for users to know what to ask. “Create a GitHub issue” worked. “What can you do” didn’t.</p><p name="f760" id="f760" class="graf graf--p graf-after--p">After: Piper supports exploration. New users can ask “what services do you offer” and get an answer. They can ask “help me get started” and get guidance. They can ask “what makes you different” and get Piper’s differentiation story.</p><p name="e724" id="e724" class="graf graf--p graf-after--p">The technical implementation — 17 patterns, 31 tests, 12 queries, proper capability enumeration — proved the architecture shift worked. But the real validation would come from actual alpha testers trying to explore what Piper could do.</p><p name="5d43" id="5d43" class="graf graf--p graf-after--p">Discovery isn’t just a feature category. It’s a philosophical shift from assuming user knowledge to supporting user learning. From command-oriented to exploration-friendly.</p><p name="fc67" id="fc67" class="graf graf--p graf-after--p">We’d built a system that expected expertise. We needed a system that enabled learning.
