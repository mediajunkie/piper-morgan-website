Two weeks since the last full development session. In between: album release party on the 12th, weekend recovery, catching up on life outside the project.</p><p>I opened my laptop expecting to spend the morning getting reoriented. We validate API inputs, we validate business logic, but we don’t validate that our database schemas match our ORM models.</p><p>That’s the kind of architectural thinking that only happens when you’re not in crisis mode. The prerequisites were clear. The known issues were tracked. We were ready.</p><h3>When users reveal what tests miss</h3><p>Tuesday evening, December 17th. Huge red error appeared.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/730/1*MVhLM0LtaR5AQePGB1UogA.png" /><figcaption>Validation error</figcaption></figure><p>Foreign key violation in the user_api_keys table. The kind of thing that makes you question whether anything actually works.</p><p>Lead Developer session: 54 minutes to catalogue four bugs from my screenshots, trace root causes, and implement fixes.</p><p>The primary bug turned out to be a temporal paradox. The setup wizard’s validate_api_key() function created a temporary user_id and tried to store it in the database before the user actually existed. PostgreSQL, being strict about foreign keys, rejected the operation.</p><p>Fix: Add a store parameter to store_user_key(). Documentation of the pattern so future developers understand the issue.</p><p>Not just fixing the bug — preventing the next one.</p><h3>The pattern repeats</h3><p>Wednesday afternoon, December 18th. When you have space to ask “what class of problems does this symptom represent?”</p><p name="7d70" id="7d70" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">I felt real relief when I realized that my project’s continuity didn’t evaporate if I took my eye off the ball for a few days!</em></p><p name="eb93" id="eb93" class="graf graf--p graf-after--p">Meanwhile, the Programmer was assessing alpha readiness. v0.8.2 had shipped December 11th — GUI setup wizard, smart routing, 602 smoke tests, stable core functionality. The question: ready to bring in more external testers?</p><p name="a64d" id="a64d" class="graf graf--p graf-after--p">Assessment: HIGH. The kind of database-level failure that should never reach users. The kind of thing that makes you question whether anything actually works.</p><p name="136f" id="136f" class="graf graf--p graf-after--p">Lead Developer session: 54 minutes to catalogue four bugs from my screenshots, trace root causes, and implement fixes.</p><p name="6c95" id="6c95" class="graf graf--p graf-after--p">The primary bug turned out to be a temporal paradox. The setup wizard’s <code class="markup--code markup--p-code">validate_api_key()</code> function created a temporary user_id and tried to store it in the database before the user actually existed. PostgreSQL, being strict about foreign keys, rejected the operation.</p><p name="5d38" id="5d38" class="graf graf--p graf-after--p">Fix: Add a <code class="markup--code markup--p-code">store</code> parameter to <code class="markup--code markup--p-code">store_user_key()</code>. When <code class="markup--code markup--p-code">store=False</code>, the function validates but doesn&#39;t write to the database. Only store keys after the user exists.</p><p name="284c" id="284c" class="graf graf--p graf-after--p">Three other bugs cascaded from that single root cause: LLM clients not initialized (because API keys never stored), JavaScript errors in chat (because LLM responses were malformed), all resolved once the primary fix landed.</p><p name="590c" id="590c" class="graf graf--p graf-after--p">But here’s what made this interesting: we also added defensive programming. Documentation of the pattern so future developers understand the issue.</p><p name="1a60" id="1a60" class="graf graf--p graf-after--p">Not just fixing the bug — preventing the next one.</p><h3 name="2274" id="2274" class="graf graf--h3 graf-after--p">The pattern repeats</h3><p name="47e6" id="47e6" class="graf graf--p graf-after--h3">Wednesday afternoon, December 18th. I tested again after the Tuesday fix.</p><p name="c0b2" id="c0b2" class="graf graf--p graf-after--p">Same error. Different table.</p><p name="1bec" id="1bec" class="graf graf--p graf-after--p">This time: foreign key violation in learned_patterns. Same temporal bug pattern, different location. When user doesn’t exist (fresh install), skip learning and log it. When user exists, proceed normally.</p><p name="869c" id="869c" class="graf graf--p graf-after--p">Two FK violations in two days, same root cause: operations executing before their prerequisites exist. Not one bug — a systematic architecture issue. We’d built temporal assumptions into the setup flow.</p><p name="9c96" id="9c96" class="graf graf--p graf-after--p">The fix for the second instance was faster than the first because we’d established the pattern. We understood the class of problem. We knew what solution worked.</p><h3 name="e432" id="e432" class="graf graf--h3 graf-after--p">What sustainability feels like</h3><p name="a729" id="a729" class="graf graf--p graf-after--h3">Three days of work.
