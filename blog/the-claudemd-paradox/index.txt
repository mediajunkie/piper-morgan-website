1b:T18e7,<figure><img alt="A robot calmly reads an “Emergency Procedures” binder beside an open filing cabinet while smoke and sparks rise nearby and an untouched fire alarm hangs on the wall." src="https://cdn-images-1.medium.com/max/1024/1*bTI4Yq205NNQgvnIYwiUQw.png" /><figcaption>“By the book…”</figcaption></figure><p><em>January 22–23, 2026</em></p><p>We made CLAUDE.md 87% smaller. Then we lost 12 hours of work logs. Then we fixed it the next morning.</p><p>That’s the shape of January 22nd and 23rd — the days we learned that instructions meant to help AI agents can backfire spectacularly when they get too detailed.</p><h3>The problem</h3><p>CLAUDE.md had grown to 1,257 lines. This is the file that tells AI agents how to behave in our codebase — what protocols to follow, how to handle various situations, where to find things.</p><p>Research suggests AI agents follow instructions better when those instructions are under 300 lines. We were four times over that limit. The docs agent proposed a “Tier 3 refactor” — an 87% reduction that would move detailed protocols to external files, leaving only essential triggers in the main document.</p><p>I approved it. The refactor was clean, well-organized, and principled. CLAUDE.md went from 1,257 lines to 157 lines.</p><p>What could possibly go wrong?</p><h3>The failure</h3><p>Here’s what we didn’t anticipate: after a context compaction (when Claude’s context window fills up and gets summarized), the agent loses access to everything except what’s explicitly in the current context.</p><p>The old CLAUDE.md had a post-compaction protocol embedded directly in the file. “After compaction, check your session log. If it doesn’t exist, create it.”</p><p>The new CLAUDE.md referenced that protocol in an external file. After compaction, agents didn’t load the external file. They didn’t know to check their logs. They didn’t know logs existed.</p><p>For the next 12 hours, substantial work happened — 17 issues closed, hundreds of tests written — with no session logs maintained. The agents were productive. They just weren’t documenting anything.</p><h3>The discovery</h3><p>The next morning, the docs agent started creating the January 22 omnibus log and noticed something wrong. Where were the session logs from yesterday afternoon and evening?</p><p>Forensic investigation revealed the gap. File timestamps showed when work happened. Git commits showed what was done. But the real-time session logs that capture context, decisions, and reasoning? Gone.</p><p>We reconstructed what we could from file timestamps and git history. But reconstruction is archaeology, not journalism. You lose the “why” and keep only the “what.”</p><h3>The fix</h3><p>The fix was almost embarrassingly simple: put the critical protocol back into CLAUDE.md.</p><p>Not the whole 1,257 lines. Just the parts that absolutely must survive context compaction. About 70 lines total — the post-compaction protocol, session log requirements, and a few other essentials.</p><p>CLAUDE.md went from 157 lines to 230 lines. Still 82% smaller than the original, but now containing everything an agent needs to maintain continuity across context boundaries.</p><p>Win!</p><h3>The paradox</h3><p>Here’s what makes this interesting: the refactor was correct by every reasonable standard. Shorter is better for AI instructions. External references reduce duplication. Clean organization aids comprehension.</p><p>But it broke a hidden assumption: some instructions must survive context boundaries, and external references don’t.</p><p>The paradox is that our attempt to make instructions more followable made them less followable. The 1,257-line file was bloated and hard to parse, but it worked. The 157-line file was elegant and well-organized, but it failed at a critical moment.</p><h3>The lesson</h3><p>We’ve since formalized this as a principle: protocols that must survive compaction must be inline, but they must also be simple.</p><p>The old 1,257-line file survived compaction because everything was inline. But agents probably weren’t following most of it — too much to process. The new 157-line file was followable but didn’t survive compaction.</p><p>The sweet spot is minimal inline protocols (the triggers) with detailed external references (the procedures). The trigger survives compaction and tells the agent what to do. The agent then loads the detailed procedure when needed.</p><p>It’s like the difference between a fire alarm and a fire evacuation plan. The alarm needs to be simple and loud — just “CHECK YOUR LOGS.” The detailed procedure can live elsewhere, consulted when the alarm goes off.</p><h3>The meta-lesson</h3><p>This incident revealed something about working with AI agents that I hadn’t fully appreciated: they have a “memory horizon” that humans don’t.</p><p>When I approve a refactor, I remember the reasoning behind my approval even if the document changes. An AI agent doesn’t. After compaction, it only knows what’s in front of it. If the instructions it needs aren’t there, they might as well not exist.</p><p>Building systems with AI requires thinking about these boundaries — not just what the agent can do, but what it can remember, and what happens when it forgets.</p><p><em>Next up in the Building Piper Morgan narrative, “The MVP Rubric” from January 26, but first we’ll flash back to November for an insight piece tomorow, “The Completion Discipline” on the ongoing struggle to get AIs to finish their work when the difference between “mostly done” and “actually complete” turned out to be a methodology problem, not a motivation problem.</em></p><p><em>Have you ever had a well-intentioned improvement backfire in unexpected ways? Still 82% smaller than the original, but now containing everything an agent needs to maintain continuity across context boundaries.</p><p name="a41f" id="a41f" class="graf graf--p graf-after--p">Win!</p><h3 name="85dc" id="85dc" class="graf graf--h3 graf-after--p">The paradox</h3><p name="12bb" id="12bb" class="graf graf--p graf-after--h3">Here’s what makes this interesting: the refactor was correct by every reasonable standard. Shorter is better for AI instructions. External references reduce duplication. Clean organization aids comprehension.</p><p name="9d5f" id="9d5f" class="graf graf--p graf-after--p">But it broke a hidden assumption: some instructions must survive context boundaries, and external references don’t.</p><p name="8155" id="8155" class="graf graf--p graf-after--p">The paradox is that our attempt to make instructions more followable made them less followable. The 1,257-line file was bloated and hard to parse, but it worked. The 157-line file was elegant and well-organized, but it failed at a critical moment.</p><h3 name="e386" id="e386" class="graf graf--h3 graf-after--p">The lesson</h3><p name="7780" id="7780" class="graf graf--p graf-after--h3">We’ve since formalized this as a principle: protocols that must survive compaction must be inline, but they must also be simple.</p><p name="8798" id="8798" class="graf graf--p graf-after--p">The old 1,257-line file survived compaction because everything was inline. But agents probably weren’t following most of it — too much to process. The new 157-line file was followable but didn’t survive compaction.</p><p name="be13" id="be13" class="graf graf--p graf-after--p">The sweet spot is minimal inline protocols (the triggers) with detailed external references (the procedures). The trigger survives compaction and tells the agent what to do. The agent then loads the detailed procedure when needed.</p><p name="c876" id="c876" class="graf graf--p graf-after--p">It’s like the difference between a fire alarm and a fire evacuation plan. The alarm needs to be simple and loud — just “CHECK YOUR LOGS.” The detailed procedure can live elsewhere, consulted when the alarm goes off.</p><h3 name="f0ae" id="f0ae" class="graf graf--h3 graf-after--p">The meta-lesson</h3><p name="c350" id="c350" class="graf graf--p graf-after--h3">This incident revealed something about working with AI agents that I hadn’t fully appreciated: they have a “memory horizon” that humans don’t.</p><p name="6dc6" id="6dc6" class="graf graf--p graf-after--p">When I approve a refactor, I remember the reasoning behind my approval even if the document changes.
