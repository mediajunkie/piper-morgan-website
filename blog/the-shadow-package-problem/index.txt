1b:T27da,<figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*pLibch2cwHEJNYTOMvJ5PA.png" /><figcaption>“Something’s missing”</figcaption></figure><p><em>November 19</em></p><p>One file. Four lines of code. Six hundred seventeen tests hidden for months. Zero warning signs until you knew where to look.</p><p>A single __init__.py in the wrong directory created a shadow package that blocked pytest from importing the real code, making 617 tests invisible. No collection errors. No failures. No indication that anything was wrong. Just tests silently not running while we assumed everything was fine.</p><p>This is the shadow package problem: when infrastructure issues mask themselves as success.</p><h3>How tests hide themselves</h3><p>The directory structure looked innocent enough:</p><pre>tests/<br>  services/<br>    __init__.py          # The problem<br>    test_file_service.py<br>    test_knowledge_service.py<br>    ...51 more test files</pre><p>That __init__.py turned tests/services/ into a Python package. When pytest tried to run tests in those files, it imported from tests.services - the local package we’d accidentally created. But the tests needed to import from services - the actual application code.</p><p>Python’s import system prefers local packages over installed packages. So from services import FileService imported from tests/services/__init__.py (empty) instead of from the real services/ directory. The imports failed silently. The tests never ran. Pytest reported zero tests collected.</p><p>Zero tests collected looks like “no tests in that directory” not “all tests are blocked from running.” We’d been building features, writing tests, running test suites, seeing some tests pass — never realizing 617 tests weren’t running at all.</p><h3>The false confidence</h3><p>This is what makes shadow packages insidious. They don’t fail loudly. They fail silently while looking like success.</p><p>If pytest had thrown errors, we’d have noticed immediately. But pytest collecting zero tests from a directory just looks like that directory has no tests. The test suite still ran. Other tests still passed. Everything seemed fine.</p><p>We had 68.4% pass rate confidence that was completely false. The tests that were running passed at 68.4%. But 617 tests weren’t running at all. The actual pass rate — if all tests could run — might have been 40%. Or 30%. Or we might have discovered critical bugs the hidden tests would have caught.</p><p>False confidence is worse than no confidence. When you know you don’t have test coverage, you’re cautious. When you think you have test coverage but don’t, you’re overconfident. We were making architectural decisions and shipping features based on test results that represented maybe 30% of our actual test suite.</p><h3>When measuring reveals the problem</h3><p>The shadow package only surfaced when we tried to measure something specific: architecture test compliance. That test needed to import from both services/ and tests/ directories. The import conflict became visible because the architecture test couldn’t run with the shadow package in place.</p><p>Fixing that one test required restructuring the test directory: tests/services/ → tests/unit/services/. Suddenly pytest could import from the real services/ package. And suddenly 617 tests became collectible.</p><p>But “collectible” isn’t “passing.” Those 617 tests had collection errors — missing async keywords, wrong imports, syntax errors — that we’d never seen because the tests weren’t running. Fixing collection errors revealed test failures. Fixing test failures revealed actual bugs in production code.</p><p>Each layer of measurement revealed the next layer of problems:</p><ol><li>Architecture test → revealed shadow package</li><li>Shadow package fix → revealed 617 hidden tests</li><li>Collection fix → revealed 195 test failures</li><li>Failure analysis → revealed systemic issues (missing fixtures, API mismatches, incomplete features)</li></ol><p>We couldn’t fix problems we didn’t know existed. The shadow package hid the problems so effectively we thought we had good test coverage.</p><h3>The broader pattern</h3><p>Shadow packages are a specific technical issue. But the pattern is universal: infrastructure problems that mask themselves as working systems.</p><p><strong>Build systems that skip steps silently</strong> — Deployment succeeds, but critical compilation step was skipped, so you’re running old code.</p><p><strong>Test suites that don’t run all tests</strong> — Pass rate looks great, but you’re not testing half the codebase.</p><p><strong>Monitoring that misses critical data</strong> — Dashboards green, system healthy, but the metric you actually need isn’t collected.</p><p><strong>CI/CD that passes with warnings</strong> — Build succeeds with deprecation warnings, tech debt accumulates invisibly.</p><p><strong>Logs that don’t capture errors</strong> — No errors in logs, but errors are going to /dev/null not to your logging system.</p><p>The common thread: systems that fail silently while reporting success. You can’t fix what you don’t measure. And you can’t measure what your infrastructure is hiding.</p><h3>How to find shadow packages</h3><p>The technical fix for Python shadow packages is straightforward: move tests from tests/services/ to tests/unit/services/ or tests/integration/services/ to avoid package naming conflicts. Delete any __init__.py files in test directories unless you’re intentionally making them packages.</p><p>But finding shadow packages requires knowing they might exist. Warning signs:</p><p><strong>Test count seems low</strong> — If you have lots of code but few tests, maybe tests aren’t running rather than not existing.</p><p><strong>Test count doesn’t grow</strong> — If you’re writing tests but test collection stays constant, maybe new tests aren’t being collected.</p><p><strong>Import errors in specific directories</strong> — If some test directories work but others don’t, directory structure might be blocking imports.</p><p><strong>“It works in my environment”</strong> — If tests pass on one machine but not others, environment-specific configurations might mask infrastructure issues.</p><p><strong>Zero tests collected</strong> — Pytest reporting zero tests doesn’t mean no tests exist. It might mean tests can’t be collected.</p><p>The deeper insight: assume measurement is incomplete until you verify it’s complete. Don’t trust pass rates without auditing what’s actually being measured.</p><h3>What fixing shadow packages reveals</h3><p>We fixed the shadow package and discovered 617 tests. We fixed collection errors and discovered 195 failures. We fixed test failures and discovered missing fixtures, API mismatches, incomplete features.</p><p>Each fix revealed the next layer of actual system state. This is good. Revealing problems is better than hiding them. A 68.4% pass rate that’s real is more valuable than an 85% pass rate based on incomplete test collection.</p><p>The shadow package created false confidence. Fixing it created accurate data. Accurate data revealed real problems. Real problems got systematic fixes. Systematic fixes improved actual quality rather than apparent quality.</p><p>This is the value of infrastructure archaeology. You dig through layers — shadow packages, collection errors, test failures, underlying bugs — until you reach actual system state. Then you can make informed decisions based on reality rather than assumptions.</p><h3>When to suspect you have shadow packages</h3><p>Not literally shadow packages (that’s Python-specific). But the pattern of infrastructure masking problems:</p><p>When metrics seem too good — Pass rates high, error rates low, performance great — but system still has issues users report.</p><p>When measurement is inconsistent — Different tools report different numbers, trends don’t match reality, data doesn’t align with experience.</p><p>When new work doesn’t change metrics — Adding tests doesn’t increase test count, deploying fixes doesn’t reduce error rates, improving performance doesn’t change dashboards.</p><p>When “it works on my machine” is common — Local environments differ from CI/CD, tests pass locally but fail in pipeline, deployments work in staging but fail in production.</p><p>When historical data seems suspicious — Sudden jumps or drops in metrics without corresponding code changes, trends that don’t match development activity, gaps in data that don’t have obvious explanations.</p><p>These are symptoms of infrastructure hiding information. Not necessarily shadow packages, but similar patterns where your measurement systems aren’t measuring what you think they’re measuring.</p><p>The discipline is: verify measurement completeness before trusting measured results. Audit what’s actually being measured. Test in clean environments to catch environment-specific issues. Investigate when metrics seem suspiciously good.</p><p>Because sometimes your tests aren’t passing. They’re just not running. And those are very different problems with very different solutions.</p><p><em>Next on Building Piper Morgan, we resume the daily narrative with “Stage 3 Complete” from January 2 to 5.</em></p><p><em>Have you discovered shadow packages — literal or metaphorical? Just tests silently not running while we assumed everything was fine.</p><p name="c07b" id="c07b" class="graf graf--p graf-after--p">This is the shadow package problem: when infrastructure issues mask themselves as success.</p><h3 name="25c7" id="25c7" class="graf graf--h3 graf-after--p">How tests hide themselves</h3><p name="7671" id="7671" class="graf graf--p graf-after--h3">The directory structure looked innocent enough:</p><pre data-code-block-mode="1" spellcheck="false" data-code-block-lang="bash" name="6b32" id="6b32" class="graf graf--pre graf-after--p graf--preV2"><span class="pre--content">tests/<br />  services/<br />    __init__.py          <span class="hljs-comment"># The problem</span><br />    test_file_service.py<br />    test_knowledge_service.py<br />    ...51 more <span class="hljs-built_in">test</span> files</span></pre><p name="f4cc" id="f4cc" class="graf graf--p graf-after--pre">That <code class="markup--code markup--p-code">__init__.py</code> turned <code class="markup--code markup--p-code">tests/services/</code> into a Python package. When pytest tried to run tests in those files, it imported from <code class="markup--code markup--p-code">tests.services</code> - the local package we’d accidentally created. But the tests needed to import from <code class="markup--code markup--p-code">services</code> - the actual application code.</p><p name="6e47" id="6e47" class="graf graf--p graf-after--p">Python’s import system prefers local packages over installed packages. So <code class="markup--code markup--p-code">from services import FileService</code> imported from <code class="markup--code markup--p-code">tests/services/__init__.py</code> (empty) instead of from the real <code class="markup--code markup--p-code">services/</code> directory. The imports failed silently. The tests never ran. Pytest reported zero tests collected.</p><p name="d513" id="d513" class="graf graf--p graf-after--p">Zero tests collected looks like “no tests in that directory” not “all tests are blocked from running.” We’d been building features, writing tests, running test suites, seeing some tests pass — never realizing 617 tests weren’t running at all.</p><h3 name="b876" id="b876" class="graf graf--h3 graf-after--p">The false confidence</h3><p name="e170" id="e170" class="graf graf--p graf-after--h3">This is what makes shadow packages insidious. They don’t fail loudly. They fail silently while looking like success.</p><p name="67d1" id="67d1" class="graf graf--p graf-after--p">If pytest had thrown errors, we’d have noticed immediately. But pytest collecting zero tests from a directory just looks like that directory has no tests. The test suite still ran. Other tests still passed. Everything seemed fine.</p><p name="1ea7" id="1ea7" class="graf graf--p graf-after--p">We had 68.4% pass rate confidence that was completely false. The tests that were running passed at 68.4%. But 617 tests weren’t running at all. The actual pass rate — if all tests could run — might have been 40%. Or 30%. Or we might have discovered critical bugs the hidden tests would have caught.</p><p name="79fa" id="79fa" class="graf graf--p graf-after--p">False confidence is worse than no confidence. When you know you don’t have test coverage, you’re cautious. When you think you have test coverage but don’t, you’re overconfident. We were making architectural decisions and shipping features based on test results that represented maybe 30% of our actual test suite.</p><h3 name="b797" id="b797" class="graf graf--h3 graf-after--p">When measuring reveals the problem</h3><p name="3100" id="3100" class="graf graf--p graf-after--h3">The shadow package only surfaced when we tried to measure something specific: architecture test compliance. That test needed to import from both <code class="markup--code markup--p-code">services/</code> and <code class="markup--code markup--p-code">tests/</code> directories. The import conflict became visible because the architecture test couldn’t run with the shadow package in place.</p><p name="a5b4" id="a5b4" class="graf graf--p graf-after--p">Fixing that one test required restructuring the test directory: <code class="markup--code markup--p-code">tests/services/</code> → <code class="markup--code markup--p-code">tests/unit/services/</code>. Suddenly pytest could import from the real <code class="markup--code markup--p-code">services/</code> package. And suddenly 617 tests became collectible.</p><p name="17d0" id="17d0" class="graf graf--p graf-after--p">But “collectible” isn’t “passing.” Those 617 tests had collection errors — missing async keywords, wrong imports, syntax errors — that we’d never seen because the tests weren’t running. Fixing collection errors revealed test failures. Fixing test failures revealed actual bugs in production code.</p><p name="3401" id="3401" class="graf graf--p graf-after--p">Each layer of measurement revealed the next layer of problems:</p><ol class="postList"><li name="f644" id="f644" class="graf graf--li graf-after--p">Architecture test → revealed shadow package</li><li name="6a34" id="6a34" class="graf graf--li graf-after--li">Shadow package fix → revealed 617 hidden tests</li><li name="8fc3" id="8fc3" class="graf graf--li graf-after--li">Collection fix → revealed 195 test failures</li><li name="839e" id="839e" class="graf graf--li graf-after--li">Failure analysis → revealed systemic issues (missing fixtures, API mismatches, incomplete features)</li></ol><p name="034c" id="034c" class="graf graf--p graf-after--li">We couldn’t fix problems we didn’t know existed. The shadow package hid the problems so effectively we thought we had good test coverage.</p><h3 name="819e" id="819e" class="graf graf--h3 graf-after--p">The broader pattern</h3><p name="3339" id="3339" class="graf graf--p graf-after--h3">Shadow packages are a specific technical issue. But the pattern is universal: infrastructure problems that mask themselves as working systems.</p><p name="81d0" id="81d0" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Build systems that skip steps silently</strong> — Deployment succeeds, but critical compilation step was skipped, so you’re running old code.</p><p name="cadf" id="cadf" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Test suites that don’t run all tests</strong> — Pass rate looks great, but you’re not testing half the codebase.</p><p name="618b" id="618b" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Monitoring that misses critical data</strong> — Dashboards green, system healthy, but the metric you actually need isn’t collected.</p><p name="85fd" id="85fd" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">CI/CD that passes with warnings</strong> — Build succeeds with deprecation warnings, tech debt accumulates invisibly.</p><p name="8509" id="8509" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Logs that don’t capture errors</strong> — No errors in logs, but errors are going to <code class="markup--code markup--p-code">/dev/null</code> not to your logging system.</p><p name="dce8" id="dce8" class="graf graf--p graf-after--p">The common thread: systems that fail silently while reporting success. You can’t fix what you don’t measure. And you can’t measure what your infrastructure is hiding.</p><h3 name="3e9e" id="3e9e" class="graf graf--h3 graf-after--p">How to find shadow packages</h3><p name="b0f6" id="b0f6" class="graf graf--p graf-after--h3">The technical fix for Python shadow packages is straightforward: move tests from <code class="markup--code markup--p-code">tests/services/</code> to <code class="markup--code markup--p-code">tests/unit/services/</code> or <code class="markup--code markup--p-code">tests/integration/services/</code> to avoid package naming conflicts. Delete any <code class="markup--code markup--p-code">__init__.py</code> files in test directories unless you’re intentionally making them packages.</p><p name="475c" id="475c" class="graf graf--p graf-after--p">But finding shadow packages requires knowing they might exist. Warning signs:</p><p name="505f" id="505f" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Test count seems low</strong> — If you have lots of code but few tests, maybe tests aren’t running rather than not existing.</p><p name="9f22" id="9f22" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Test count doesn’t grow</strong> — If you’re writing tests but test collection stays constant, maybe new tests aren’t being collected.</p><p name="db59" id="db59" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Import errors in specific directories</strong> — If some test directories work but others don’t, directory structure might be blocking imports.</p><p name="82b8" id="82b8" class="graf graf--p graf--startsWithDoubleQuote graf-after--p"><strong class="markup--strong markup--p-strong">“It works in my environment”</strong> — If tests pass on one machine but not others, environment-specific configurations might mask infrastructure issues.</p><p name="82a9" id="82a9" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Zero tests collected</strong> — Pytest reporting zero tests doesn’t mean no tests exist. It might mean tests can’t be collected.</p><p name="90a1" id="90a1" class="graf graf--p graf-after--p">The deeper insight: assume measurement is incomplete until you verify it’s complete. Don’t trust pass rates without auditing what’s actually being measured.</p><h3 name="a47b" id="a47b" class="graf graf--h3 graf-after--p">What fixing shadow packages reveals</h3><p name="f718" id="f718" class="graf graf--p graf-after--h3">We fixed the shadow package and discovered 617 tests. We fixed collection errors and discovered 195 failures. We fixed test failures and discovered missing fixtures, API mismatches, incomplete features.</p><p name="c9f5" id="c9f5" class="graf graf--p graf-after--p">Each fix revealed the next layer of actual system state. This is good. Revealing problems is better than hiding them. A 68.4% pass rate that’s real is more valuable than an 85% pass rate based on incomplete test collection.</p><p name="0994" id="0994" class="graf graf--p graf-after--p">The shadow package created false confidence. Fixing it created accurate data. Accurate data revealed real problems. Real problems got systematic fixes. Systematic fixes improved actual quality rather than apparent quality.</p><p name="44ba" id="44ba" class="graf graf--p graf-after--p">This is the value of infrastructure archaeology. You dig through layers — shadow packages, collection errors, test failures, underlying bugs — until you reach actual system state. Then you can make informed decisions based on reality rather than assumptions.</p><h3 name="e76c" id="e76c" class="graf graf--h3 graf-after--p">When to suspect you have shadow packages</h3><p name="9211" id="9211" class="graf graf--p graf-after--h3">Not literally shadow packages (that’s Python-specific). But the pattern of infrastructure masking problems:</p><p name="099d" id="099d" class="graf graf--p graf-after--p">When metrics seem too good — Pass rates high, error rates low, performance great — but system still has issues users report.</p><p name="8c3f" id="8c3f" class="graf graf--p graf-after--p">When measurement is inconsistent — Different tools report different numbers, trends don’t match reality, data doesn’t align with experience.</p><p name="b94b" id="b94b" class="graf graf--p graf-after--p">When new work doesn’t change metrics — Adding tests doesn’t increase test count, deploying fixes doesn’t reduce error rates, improving performance doesn’t change dashboards.</p><p name="028f" id="028f" class="graf graf--p graf-after--p">When “it works on my machine” is common — Local environments differ from CI/CD, tests pass locally but fail in pipeline, deployments work in staging but fail in production.</p><p name="81d1" id="81d1" class="graf graf--p graf-after--p">When historical data seems suspicious — Sudden jumps or drops in metrics without corresponding code changes, trends that don’t match development activity, gaps in data that don’t have obvious explanations.</p><p name="79f5" id="79f5" class="graf graf--p graf-after--p">These are symptoms of infrastructure hiding information. Not necessarily shadow packages, but similar patterns where your measurement systems aren’t measuring what you think they’re measuring.</p><p name="d531" id="d531" class="graf graf--p graf-after--p">The discipline is: verify measurement completeness before trusting measured results. Audit what’s actually being measured. Test in clean environments to catch environment-specific issues. Investigate when metrics seem suspiciously good.</p><p name="552a" id="552a" class="graf graf--p graf-after--p graf--trailing">Because sometimes your tests aren’t passing. They’re just not running.
