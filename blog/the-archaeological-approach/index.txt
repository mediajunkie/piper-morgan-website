1b:T3ab7,<figure><img alt="A robot shows its human roommate that they already own a vacuum cleaner" src="https://cdn-images-1.medium.com/max/1024/1*SapxrAkrnillGYL_gacUaw.png" /><figcaption>“We already have one”</figcaption></figure><p>I’ve spoken in this series before about a pattern we’ve adopted in building Piper Morgan, something I call “Phase −1 infrastructure verification,” and I’d like to unpack it a bit more.</p><p>It may seem like optional overhead, but it’s not. My LLM friends want to express it in terms of ROI with all sorts of time estimates and I take that all with a grain of salt but the substantive value is undeniable. This type of investigation consistently saves more time and energy than it costs. It’s methodology that often turns “we need to build this” into simply “we just need to wire this up!”</p><p>Here are four recent examples from the past few months, though the practice started in late July or early August if I recall (I can always check my logs if anyone cares).</p><h3>Testing file upload</h3><p>First alpha user testing. I’m trying to use file upload. Doesn’t work. Error messages suggest files aren’t uploading at all.</p><p>The diagnosis in my head: File upload functionality doesn’t exist yet. We haven’t built it. This will be significant work.</p><p>Chief Architect investigates: “Let me check what file infrastructure exists.”</p><p>15 minutes later: Files work perfectly. Uploads succeed. Storage functions. The problem isn’t missing infrastructure. It’s routing assumption. Incorrect URL paths.</p><p>The fix: Update one configuration file. 10 minutes.</p><p>What we assumed: Build file upload system (8–12 hours)</p><p>What was actually needed: Fix routing path (10 minutes)</p><p>Investigation time: 15 minutes</p><p>Don’t assume functionality is missing. Investigate what exists. The problem might be simpler than it looks.</p><p>But one occurrence doesn’t establish a pattern.The pattern becomes apparent when it happens repeatedly.</p><h3>Document processing, a few days later</h3><p>Saturday morning. Historic sprint day. Four P0 blockers to resolve. One of them: Document processing integration.</p><p>The symptom: Document uploads work but processing doesn’t trigger. Seems like document processing service doesn’t exist.</p><p>Code Agent begins Phase −1 investigation: “Let me verify what document infrastructure exists before planning implementation.”</p><p>20 minutes later: DocumentService discovered. Complete document processing infrastructure. Text extraction. Metadata parsing. Storage integration. Comprehensive tests.</p><p>Status: 75% complete. Database layer works. Service layer works. Just not wired to web routes or chat handlers.</p><p>The task revision: Not “build DocumentService” (estimated 8–12 hours). But “wire existing DocumentService to web tier” (actual 2 hours).</p><p>Investigation time: 20 minutes</p><p>Time saved: 6–10 hours</p><h3>The unfinished “todo” feature</h3><p>Sunday afternoon. Planning Monday’s P1 work. Issue #285: Todo system completion.</p><p>The GitHub issue description: “Todo functionality never finished or wired up.”</p><p>Traditional approach: Estimate building complete todo CRUD system. 8–12 hours. Database models, repositories, services, web routes, chat handlers. Full vertical implementation.</p><p>But Saturday taught lesson. Sunday applies it systematically.</p><p>Code Agent Phase −1 investigation: “Before estimating development time, let me verify what todo infrastructure already exists.”</p><p>30 minutes later: Todo infrastructure discovered 75% complete.</p><p>Found:</p><pre>✅ Database Models: TodoListDB, TodoDB<br>✅ Repositories: TodoRepository, TodoListRepository, TodoManagementRepository<br>✅ Services: TodoKnowledgeService<br>✅ API Layer: TodoCreateRequest, TodoUpdateRequest, TodoResponse<br>✅ Domain Models: Todo, TodoList<br>✅ Tests: test_todo_management_api.py EXISTS!<br>✅ Documentation: PM-081-todo-api-documentation.md in archive<br><br>❌ Missing:<br>   - Not wired to web routes<br>   - Not connected to chat interface<br>   - Not integrated with intent handlers</pre><p>The task revision: Not “build todo system” (8–12 hours). But “wire existing infrastructure” (4–6 hours).</p><p>Investigation time: 30 minutes</p><p>Time saved: 4–6 hours</p><p>Three times in one week. Pattern re-confirmed. Time to document the methodology, if we hadn’t already.</p><h3>Humanizing error messages</h3><p>Monday morning. P1 sprint execution. Three issues on deck. One of them: Issue #283, error message humanization.</p><p>The symptom: Technical error messages break conversational UX. Need to humanize errors.</p><p>The assumption: Build error humanization infrastructure from scratch.</p><p>Cursor Agent begins Phase −1 investigation: “Checking what error handling infrastructure exists before planning implementation.”</p><p>15 minutes later: Complete existing infrastructure discovered.</p><p>Found:</p><ul><li>UserFriendlyErrorService (300+ lines, comprehensive)</li><li>ActionHumanizer (160+ lines)</li><li>EnhancedErrorMiddleware (180+ lines)</li></ul><p>Status: Fully built. Just not mounted in web/app.py.</p><p>The task revision: Not “build error humanization” (6–8 hours). But “mount existing middleware” (30 minutes).</p><p>Investigation time: 15 minutes</p><p>Time saved: 5.5–7.5 hours</p><p>Four days. Four investigations. Four discoveries. Four massive time savings.</p><p>The pattern isn’t luck. It’s methodology. It’s now called “Phase −1 infrastructure verification” and it’s mandatory before any implementation work begins.</p><h3>Why the 75% pattern exists</h3><p>The discoveries aren’t random. There’s reason infrastructure consistently appears 75% complete:</p><p>Single-developer good architecture:</p><ul><li>Build database/service layers with proper separation of concerns</li><li>Infrastructure created for immediate needs</li><li>Layers advance independently (deep before wide)</li><li>Integration layers deferred (not blocking current work)</li></ul><p>The progression:</p><ol><li>Need todo functionality for something specific</li><li>Build database models (25% of total work)</li><li>Build repositories (15% more)</li><li>Build services (15% more)</li><li>Build domain logic (20% more)</li><li>Total: 75% complete</li><li>Missing: Web routes (10%) + Chat handlers (10%) + UI integration (5%) = 25%</li></ol><p>Why it appears “missing”:</p><ul><li>From user perspective: Feature not in UI = doesn’t exist</li><li>From code perspective: Feature in database/service = nearly complete</li><li>Gap: Integration layers not built yet</li></ul><p>This isn’t as sloppy as it seems at time. In many ways, it’s a natural evolution when building with good practices. Database layer gets attention first (data modeling is hard). Service layer gets built second (business logic is complex). Integration layer gets deferred (seems trivial compared to foundation).</p><p>Then months later: “We need todo functionality.” Assumption: Doesn’t exist. Investigation reveals: 75% exists, just not wired.</p><h3>What makes investigation effective</h3><p>Not all investigation finds infrastructure. These four discoveries worked because investigation was systematic:</p><h3>1. Don’t assume based on symptoms</h3><p>File upload fails → doesn’t mean file infrastructure missing Document processing missing → doesn’t mean DocumentService nonexistent Todos don’t work → doesn’t mean todo system unbuilt Errors not humanized → doesn’t mean humanization infrastructure absent</p><p>Symptoms show user-visible gaps. Investigation reveals actual gaps.</p><h3>2. Check all layers systematically</h3><p>For each feature, investigate:</p><ul><li>Database models (lowest layer)</li><li>Repositories (data access)</li><li>Services (business logic)</li><li>Domain models (core entities)</li><li>API layer (external interface)</li><li>Web routes (HTTP endpoints)</li><li>Chat handlers (conversational interface)</li><li>UI integration (user-facing)</li></ul><p>Missing at top layers doesn’t mean missing at bottom layers.</p><h3>3. Use proper tools</h3><p>Serena MCP for symbol search: Find classes, methods, references Git history for archaeology: When was it built? Why deferred? Test suites for validation: Does it work? How complete? Documentation for context: What was intended?</p><p>Grep searches find surface mentions. Symbol search finds actual implementations.</p><h3>4. Invest 15–30 minutes minimum</h3><p>Quick glance isn’t investigation. Real investigation takes time:</p><ul><li>Search multiple layers</li><li>Read actual code</li><li>Check tests</li><li>Verify completeness</li><li>Assess integration needs</li></ul><h3>When investigation finds nothing</h3><p>Important note: Investigation doesn’t always find existing infrastructure.</p><p>Example: Monday’s action mapping issue. Investigation confirmed: No ActionMapper exists. Classifier outputs don’t match handler names. Mapping layer genuinely missing.</p><p>Result: Build it from scratch. 2 hours. No discovery, no time savings.</p><p>The value: Even when infrastructure doesn’t exist, investigation provides clarity:</p><ul><li>Confirmed scope (know what needs building)</li><li>Validated approach (no alternative already exists)</li><li>Identified dependencies (what integrates with what)</li><li>Clear implementation path (no surprises during development)</li></ul><p>Failed discovery isn’t wasted time. It’s validated scope. You build confidently knowing you’re not duplicating existing work.</p><p>The ROI on “found nothing” investigations: Can’t quantify time savings. But can quantify risk reduction. Building wrong thing because you didn’t check existing code: Expensive mistake. Spending 30 minutes to confirm scope: Cheap insurance.</p><h3>The methodology integration</h3><p>We had this practice for a while but as of September, Phase −1 investigation is now standard practice in gameplan template v9.0:</p><p>Phase −1: Infrastructure Verification</p><ul><li>Objective: Verify what infrastructure already exists before planning implementation</li><li>Method: Systematic investigation across all layers (database, services, routes, handlers)</li><li>Output: Existing infrastructure map + gap analysis</li><li>Duration: 15–30 minutes minimum</li><li>Decision: Wire existing infrastructure OR build from scratch</li><li>Stop condition: Clear understanding of what exists vs what needs building</li></ul><p>This phase is mandatory. Not optional. Not “if you have time.” But required step before Phase 0 begins.</p><p>The template also includes:</p><ul><li>Proper tools specification (Serena for symbol search)</li><li>Layer-by-layer checklist (database through UI)</li><li>Discovery documentation requirements</li><li>ROI tracking (investigation time vs time saved)</li></ul><h3>What this means for other projects</h3><p>The archaeological approach applies beyond Piper Morgan:</p><p>Any project with:</p><ul><li>Good architectural practices (separation of concerns)</li><li>Multiple layers (database, services, API, UI)</li><li>Incremental development (features built over time)</li><li>Single or small team (knowledge not perfectly distributed)</li></ul><p>Will likely have:</p><ul><li>Infrastructure more complete than visible</li><li>Features built but not integrated</li><li>Functionality at lower layers not surfaced</li><li>“Missing” features that actually exist</li></ul><p>The discipline:</p><ol><li>Symptom appears (feature seems missing)</li><li>Resist assumption (don’t assume missing = unbuilt)</li><li>Investigate systematically (15–30 minutes, all layers)</li><li>Discover existing work (often 75% complete)</li><li>Revise approach (wire vs build)</li><li>Save massive time (8–27x ROI)</li></ol><p>The pattern isn’t Piper Morgan-specific. It’s consequence of good architecture + incremental development + imperfect knowledge.</p><h3>The counter-intuitive insight</h3><p>Traditional project management: “Don’t investigate, start building. Investigation is overhead.”</p><p>Archaeological approach: “Always investigate before building. Investigation saves multiples of its cost.”</p><p>The counter-intuitive part: Slowing down to investigate makes overall development faster.</p><p>30 minutes investigation feels slow compared to starting immediately. It’s about clarity before action.</p><p>And when investigation consistently provides 18–27:1 ROI on time invested, the methodology justifies itself empirically.</p><p><em>This is part of the Building Piper Morgan methodology series, exploring systematic approaches to AI-assisted development.</em></p><p><em>Do you practice systematic investigation before implementation?
