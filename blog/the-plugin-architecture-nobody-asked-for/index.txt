If that sounds like over-engineering, let me explain why it’s not completely ridiculous.</p><h3>The setup</h3><p>GREAT-3A — our third major epic in the plugin architecture sequence — started with what seemed like a clear mission: extract our four integrations (Slack, GitHub, Notion, Calendar) into plugins. The gameplan assumed we’d need to pull apart embedded code and restructure everything around a new plugin interface.</p><p>Then we actually looked at the code.</p><p>Main.py, which the documentation claimed was a bloated 1,107 lines, turned out to be 141 lines of clean microservice orchestration. The integration routers we thought were scattered across the codebase were exactly where they should be, in services/integrations/. We didn&#39;t need extraction. We needed <em>wrapping</em>.</p><p>This is where methodology becomes infrastructure.</p><h3>When four things reveal a pattern</h3><p>Our config pattern analysis told the real story. We had four integrations. Three different approaches to configuration:</p><ul><li><strong>Slack</strong>: Clean service injection with a dedicated SlackConfigService</li><li><strong>GitHub</strong>: Had a config service but the router wasn’t using it</li><li><strong>Notion</strong>: No config service at all — just reading environment variables directly</li><li><strong>Calendar</strong>: Same as Notion, grabbing credentials straight from the environment</li></ul><p>Pattern compliance? <strong>25%</strong> (one of four doing it right).</p><p>Have you ever discovered your team has been solving the same problem three different ways? You know that moment when you realize nobody talked to each other about the approach before plunging in?</p><p>The question wasn’t “should we build a plugin system?” The question was: “We’re about to standardize these four things anyway — what’s the marginal cost of making it <em>systematic</em>?”</p><h3>The config compliance sprint</h3><p>Here’s where the careful methodology meets reality. We tackled config standardization one integration at a time, with our test suite becoming both validator and teacher.</p><p><strong>Phase 1B: Notion</strong> (30 minutes estimated, 23 minutes actual) Created NotionConfigService following the Slack pattern exactly. Not &quot;inspired by&quot; or &quot;similar to&quot;—we literally used Slack as a template. One integration at a time. Compliance: 50%.</p><p><strong>Phase 1C: GitHub</strong> (30 minutes estimated, 15 minutes actual)<br> The existing GitHubConfigService was already complete. We just needed to wire it to the router. Update the constructor signature, add the parameter, done. Compliance: 75%.</p><p><strong>Phase 1D: Calendar</strong> (60–90 minutes estimated, 24 minutes actual) Created CalendarConfigService, updated the adapter, verified the integration. Our test suite immediately validated everything. Compliance: <strong>100%</strong>.</p><p>From 25% to 100% in a single day. Zero regressions. 38 config compliance tests passing.</p><h3>The plugin wrapper pattern</h3><p>Once the config services were standardized, the plugin wrappers became almost trivial. Each one implements the same PiperPlugin interface with six required methods:</p><pre>class NotionPlugin(PiperPlugin):<br>    def get_metadata(self) -&gt; PluginMetadata:<br>        return PluginMetadata(<br>            name=&quot;notion&quot;,<br>            version=&quot;1.0.0&quot;,<br>            description=&quot;Notion workspace integration&quot;,<br>            capabilities=[&quot;routes&quot;, &quot;mcp&quot;]<br>        )<br>    <br>    def get_router(self) -&gt; Optional[APIRouter]:<br>        # Returns FastAPI router with status endpoint<br>        <br>    def is_configured(self) -&gt; bool:<br>        return self.config_service.is_configured()<br>        <br>    async def initialize(self) -&gt; None:<br>        # Startup logic<br>        <br>    async def shutdown(self) -&gt; None:<br>        # Cleanup logic<br>        <br>    def get_status(self) -&gt; Dict[str, Any]:<br>        # Health reporting</pre><p>The wrappers don’t replace the integration routers — they <em>coordinate</em> them. The router does the work, the plugin wrapper provides lifecycle management and registration.</p><p>Auto-registration happens via module import:</p><p>python</p><pre># At module level<br>_notion_plugin = NotionPlugin()<br>get_plugin_registry().register(_notion_plugin)</pre><p>Import the module, the plugin registers itself. No explicit registration calls scattered through startup code.</p><h3>Why this isn’t over-engineering</h3><p>Let me address the obvious question: why build plugin infrastructure for exactly four plugins?</p><p>Because we were doing the work anyway.</p><p>The config standardization? That was fixing refactoring artifacts from earlier domain-driven design work. We needed to do it regardless of plugins. The interface definition? That clarified the contract all integrations needed to follow. The registry? That replaced ad-hoc router mounting with systematic lifecycle management.</p><p>The marginal cost of making it a proper plugin system was essentially:</p><ul><li>Define the interface (265 lines)</li><li>Create the registry (266 lines)</li><li>Write four thin wrappers (417 lines total)</li><li>Build the test suite (126 lines)</li></ul><p>About 1,000 lines of infrastructure code. In return:</p><p><strong>The fifth integration becomes trivial.</strong> Not “easier” — trivial. Implement six methods, import the module, done. The test suite validates interface compliance automatically. The registry handles lifecycle. The router mounts itself.</p><p><strong>Zero breaking changes.</strong> All existing functionality preserved. 72/72 tests passing. Config compliance at 100%.</p><p><strong>Documentation through structure.</strong> The plugin interface <em>is</em> the documentation. Every plugin implements the same contract, follows the same patterns, reports status the same way.</p><p>Production-ready as an integration hub. Piper Morgan will be able to easily plug in alternative ticket-tracking tools, chat apps, calendars, and team wikis, among other services, all by extending this plug-in architecture.</p><p>This is what “Time Lord Philosophy” means in practice — taking the time to do it right because you’re doing it anyway, and that investment makes everything afterward easier.</p><h3>The multi-agent coordination moment</h3><p>Worth noting: this wasn’t solo work. Two AI coding agents (Code and Cursor) were working in parallel across different phases, consistently finishing within minutes of each other. Because the methodology created clear boundaries, when Phase 1C finishes, Phase 1D can start — regardless of which agent is handling which. I enjoy watching the photo finishes!</p><p>The Lead Developer’s post-session satisfaction assessment guessed I found the day “energizing” rather than exhausting. Low cognitive load from systematic approach, watching the methodology manifest in practice, clear progression feeling productive. It was correct.</p><p>That’s the feedback loop: methodology reduces overhead, which creates space for noticing patterns, which improves methodology.</p><h3>What this means for you</h3><p>You probably don’t need a plugin system. Not today.</p><p>But if you find yourself with three or four things that do similar work in different ways, and you’re about to standardize them anyway — that’s the moment. We needed <em class="markup--em markup--p-em">wrapping</em>.</p><p name="f0b2" id="f0b2" class="graf graf--p graf-after--p">This is where methodology becomes infrastructure.</p><h3 name="58d6" id="58d6" class="graf graf--h3 graf-after--p">When four things reveal a pattern</h3><p name="850f" id="850f" class="graf graf--p graf-after--h3">Our config pattern analysis told the real story. We had four integrations. Three different approaches to configuration:</p><ul class="postList"><li name="67ae" id="67ae" class="graf graf--li graf-after--p"><strong class="markup--strong markup--li-strong">Slack</strong>: Clean service injection with a dedicated <code class="markup--code markup--li-code">SlackConfigService</code></li><li name="025a" id="025a" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">GitHub</strong>: Had a config service but the router wasn’t using it</li><li name="9083" id="9083" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Notion</strong>: No config service at all — just reading environment variables directly</li><li name="aa96" id="aa96" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Calendar</strong>: Same as Notion, grabbing credentials straight from the environment</li></ul><p name="bf4c" id="bf4c" class="graf graf--p graf-after--li">Pattern compliance? <strong class="markup--strong markup--p-strong">25%</strong> (one of four doing it right).</p><p name="b855" id="b855" class="graf graf--p graf-after--p">Have you ever discovered your team has been solving the same problem three different ways? You know that moment when you realize nobody talked to each other about the approach before plunging in?</p><p name="c462" id="c462" class="graf graf--p graf-after--p">The question wasn’t “should we build a plugin system?” The question was: “We’re about to standardize these four things anyway — what’s the marginal cost of making it <em class="markup--em markup--p-em">systematic</em>?”</p><h3 name="5805" id="5805" class="graf graf--h3 graf-after--p">The config compliance sprint</h3><p name="6065" id="6065" class="graf graf--p graf-after--h3">Here’s where the careful methodology meets reality. We tackled config standardization one integration at a time, with our test suite becoming both validator and teacher.</p><p name="be28" id="be28" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Phase 1B: Notion</strong> (30 minutes estimated, 23 minutes actual) Created <code class="markup--code markup--p-code">NotionConfigService</code> following the Slack pattern exactly. Not &quot;inspired by&quot; or &quot;similar to&quot;—we literally used Slack as a template. One integration at a time. Compliance: 50%.</p><p name="6c36" id="6c36" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Phase 1C: GitHub</strong> (30 minutes estimated, 15 minutes actual)<br> The existing <code class="markup--code markup--p-code">GitHubConfigService</code> was already complete. We just needed to wire it to the router. Update the constructor signature, add the parameter, done. Compliance: 75%.</p><p name="909c" id="909c" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Phase 1D: Calendar</strong> (60–90 minutes estimated, 24 minutes actual) Created <code class="markup--code markup--p-code">CalendarConfigService</code>, updated the adapter, verified the integration. Our test suite immediately validated everything. Compliance: <strong class="markup--strong markup--p-strong">100%</strong>.</p><p name="9ca7" id="9ca7" class="graf graf--p graf-after--p">From 25% to 100% in a single day. Zero regressions. 38 config compliance tests passing.</p><h3 name="f75a" id="f75a" class="graf graf--h3 graf-after--p">The plugin wrapper pattern</h3><p name="7742" id="7742" class="graf graf--p graf-after--h3">Once the config services were standardized, the plugin wrappers became almost trivial. That was fixing refactoring artifacts from earlier domain-driven design work. We needed to do it regardless of plugins. The interface definition? That clarified the contract all integrations needed to follow. The registry? That replaced ad-hoc router mounting with systematic lifecycle management.</p><p name="e8ea" id="e8ea" class="graf graf--p graf-after--p">The marginal cost of making it a proper plugin system was essentially:</p><ul class="postList"><li name="d624" id="d624" class="graf graf--li graf-after--p">Define the interface (265 lines)</li><li name="e45d" id="e45d" class="graf graf--li graf-after--li">Create the registry (266 lines)</li><li name="48f4" id="48f4" class="graf graf--li graf-after--li">Write four thin wrappers (417 lines total)</li><li name="2bbf" id="2bbf" class="graf graf--li graf-after--li">Build the test suite (126 lines)</li></ul><p name="e9c3" id="e9c3" class="graf graf--p graf-after--li">About 1,000 lines of infrastructure code. In return:</p><p name="998e" id="998e" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">The fifth integration becomes trivial.</strong> Not “easier” — trivial. Implement six methods, import the module, done. The test suite validates interface compliance automatically. The registry handles lifecycle. The router mounts itself.</p><p name="57ed" id="57ed" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Zero breaking changes.</strong> All existing functionality preserved. 72/72 tests passing. Config compliance at 100%.</p><p name="9c48" id="9c48" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Documentation through structure.</strong> The plugin interface <em class="markup--em markup--p-em">is</em> the documentation. Every plugin implements the same contract, follows the same patterns, reports status the same way.</p><p name="902b" id="902b" class="graf graf--p graf-after--p">Production-ready as an integration hub. Piper Morgan will be able to easily plug in alternative ticket-tracking tools, chat apps, calendars, and team wikis, among other services, all by extending this plug-in architecture.</p><p name="206c" id="206c" class="graf graf--p graf-after--p">This is what “Time Lord Philosophy” means in practice — taking the time to do it right because you’re doing it anyway, and that investment makes everything afterward easier.</p><h3 name="a5df" id="a5df" class="graf graf--h3 graf-after--p">The multi-agent coordination moment</h3><p name="6aa6" id="6aa6" class="graf graf--p graf-after--h3">Worth noting: this wasn’t solo work. Two AI coding agents (Code and Cursor) were working in parallel across different phases, consistently finishing within minutes of each other. Because the methodology created clear boundaries, when Phase 1C finishes, Phase 1D can start — regardless of which agent is handling which. I enjoy watching the photo finishes!</p><p name="4f5c" id="4f5c" class="graf graf--p graf-after--p">The Lead Developer’s post-session satisfaction assessment guessed I found the day “energizing” rather than exhausting. Low cognitive load from systematic approach, watching the methodology manifest in practice, clear progression feeling productive. It was correct.</p><p name="ea48" id="ea48" class="graf graf--p graf-after--p">That’s the feedback loop: methodology reduces overhead, which creates space for noticing patterns, which improves methodology.</p><h3 name="9e83" id="9e83" class="graf graf--h3 graf-after--p">What this means for you</h3><p name="080d" id="080d" class="graf graf--p graf-after--h3">You probably don’t need a plugin system. Not today.</p><p name="3ec2" id="3ec2" class="graf graf--p graf-after--p">But if you find yourself with three or four things that do similar work in different ways, and you’re about to standardize them anyway — that’s the moment. The marginal cost of systematization when you’re already touching every integration is surprisingly low.</p><p name="4601" id="4601" class="graf graf--p graf-after--p">The questions to ask:</p><ul class="postList"><li name="906c" id="906c" class="graf graf--li graf-after--p">Are we doing this work regardless? (Config standardization, interface clarification, lifecycle management)</li><li name="f1bc" id="f1bc" class="graf graf--li graf-after--li">What’s the marginal cost of making it systematic?</li><li name="010c" id="010c" class="graf graf--li graf-after--li">Does this create infrastructure for future work or just wrap current work?</li></ul><p name="c49b" id="c49b" class="graf graf--p graf-after--li">For us, the answers were: yes, minimal, and creates infrastructure.</p><p name="ceff" id="ceff" class="graf graf--p graf-after--p graf--trailing">Your mileage will vary. But don’t assume “plugin system” automatically means over-engineering.
