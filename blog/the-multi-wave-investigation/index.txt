1b:T1538,<p><em>December 25, 2025</em></p>
<p>We had a problem: 44 new canonical queries to investigate. Each one needed infrastructure assessment, blocker identification, implementation complexity rating. Traditional approach: investigate them one by one, maybe three or four per session.</p>
<p>At that pace, the reconnaissance alone would take two weeks.</p>
<p>Instead, we deployed 13 subagents across 4 waves in a single 90-minute session. All 44 queries investigated. Complete infrastructure map. Priority matrix created.</p>
<h1>The wave structure</h1>
<p><strong>Wave 1: Core Infrastructure Audit</strong> (3 agents)</p>
<ul>
<li>Intent Router Auditor: 8 intent categories mapped, coverage percentages calculated</li>
<li>Integration Inventory: 6 integration routers catalogued, 200+ methods found</li>
<li>Repository Method Scan: 97 methods across 6 repositories analyzed</li>
</ul>
<p>These three agents established the baseline. What existed? What was complete? Where were the foundations solid?</p>
<p><strong>Wave 2: Category Deep Dives</strong> (4 agents)</p>
<ul>
<li>GitHub Operations: 8 queries mapped, complexity breakdown (2 LOW, 4 MEDIUM, 2 HIGH)</li>
<li>Slack Operations: 5 queries mapped, infrastructure complete except LLM summarization</li>
<li>Calendar Operations: 7 queries mapped, critical OAuth scope constraint identified</li>
<li>Todo Operations: 4 queries mapped, critical blocker found (hardcoded user_id)</li>
</ul>
<p>These four agents worked in parallel, each focused on a specific integration category. They didn't need to coordinate—they were examining independent infrastructure.</p>
<p><strong>Wave 3: Gap Analysis</strong> (synthesis)</p>
<ul>
<li>Priority matrix created: Phase A (5 LOW), Phase B (11 MEDIUM), Phase C (8 HIGH), Phase D (12 blocked)</li>
<li>Critical blockers documented with priority levels (P0, P1, P2)</li>
</ul>
<p>Wave 3 synthesized the category deep dives into actionable prioritization.</p>
<p><strong>Wave 4: Remaining Categories</strong> (6 agents)</p>
<ul>
<li>Help/Capability System, Activity/Audit Logging, Notion Write Operations</li>
<li>Notification Infrastructure, Conversation Serialization, File Upload</li>
<li>Each assessed independently, blockers identified, readiness scored</li>
</ul>
<h1>Why it worked just fine</h1>
<p>The key insight: investigation parallelizes well. Analysis doesn't require shared state. Each agent can examine a portion of the codebase without coordinating with others.</p>
<p>Implementation is different. Two agents trying to modify the same file create merge conflicts. Two agents implementing related features need to coordinate on interfaces.</p>
<p>But two agents reading different parts of the codebase? They can run simultaneously without collision.</p>
<h1>A taxonomy for blockers</h1>
<p>Wave 3 produced something useful beyond the priority matrix: a blocker taxonomy.</p>
<ul>
<li><strong>P0</strong> (blocks all): Problems that prevent any query from working. User_id hardcoded as "default" was P0—it broke multi-user isolation across everything.</li>
<li><strong>P1</strong> (blocks category): Problems that prevent a category from working. Calendar OAuth readonly scope was P1—it blocked all scheduling features.</li>
<li><strong>P2</strong> (nice to have): Problems that degrade experience but don't block core functionality. Missing LLM summarization in Slack was P2.</li>
</ul>
<p>This taxonomy separated complexity from criticality. A query could be easy to implement (LOW complexity) but blocked by a P0 issue. Another could be hard to implement (HIGH complexity) but have no blockers.</p>
<p>The priority matrix combined both dimensions.</p>
<h1>When to use this model</h1>
<p>Multi-wave investigation works when:</p>
<ul>
<li>The problem space can be partitioned cleanly (categories, modules, time periods)</li>
<li>Each partition can be analyzed independently</li>
<li>Synthesis can happen after parallel analysis completes</li>
<li>You need breadth more than depth initially</li>
</ul>
<p>It doesn't work when:</p>
<ul>
<li>Agents need to coordinate during investigation</li>
<li>Analysis requires iterative refinement based on early findings</li>
<li>The partitions have hidden dependencies</li>
</ul>
<p>Deploying multiple agents to fan out and research a problem has since become a routine part of my work habits, something Claude agents do very comfortably using their subagent capabilities, and clearly part of where this sort of tooling stack is headed in the future, as a matter of course.</p>
<h1>Seeing from all angles at once</h1>
<p>90 minutes of parallel investigation saved an estimated 10 days of sequential work (at least in trad-human terms) and, more importantly, helped develop a more complete picture.</p>
<p>Sequential investigation would have answered questions as they arose. Parallel investigation answered questions we hadn't thought to ask yet. The OAuth scope constraint. The hardcoded user_id. The missing router methods.</p>
<p>Sometimes you work from a checklist when you know what you're looking for, but all of these discoveries emerged from the comprehensive coverage achievable with this multi-directional approach.</p>
