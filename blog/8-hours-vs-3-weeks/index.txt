It took approximately 8 hours.</p><p>Not 8 hours of heroic coding. Not 8 hours of cutting corners. Not 8 hours that would create 8 weeks of technical debt. Eight hours of systematic execution, finishing 45% ahead of even the optimistic estimates.</p><p>This isn’t a story about working faster. It’s a story about preparation front-loading the hard work until execution becomes almost mechanical.</p><h3>The estimate wasn’t wrong</h3><p>Sprint S1 — the security foundation sprint — was scoped at 81 hours total. RBAC alone was estimated at 20–24 hours. Encryption at rest another 24–30 hours. Supporting work for the remainder.</p><p>These weren’t pessimistic estimates. They were based on reasonable assumptions:</p><ul><li>Phase 1.1 (database schema): 4–6 hours</li><li>Phase 1.2 (service layer): 8–12 hours</li><li>Phase 1.3 (endpoint protection): 4–6 hours</li><li>Phase 1.4 (shared resources): 4–6 hours</li><li>Phase 2 (role-based permissions): 6–8 hours</li><li>Phase 3 (admin and testing): 4–6 hours</li></ul><p>Each estimate assumed some discovery work, some problem-solving, some iteration. Normal software development.</p><p>What the estimates didn’t account for: we’d already done most of the discovery work.</p><h3>Phase 0 was the real work</h3><p>The security sprint had a Phase 0: audit and planning. Six comprehensive reports created before any implementation began:</p><ol><li>Phase -1 verification (infrastructure assessment)</li><li>Clarifications research (recommendations)</li><li>API endpoint catalog (56 endpoints mapped)</li><li>Service methods inventory (47 methods identified)</li><li>Risk assessment (P0 and P1 blockers categorized)</li><li>Phase 0 completion summary</li></ol><p>This audit took time — multiple sessions, deep investigation, comprehensive documentation. It didn’t feel like progress. No code was written. No tests passed. No features shipped.</p><p>But when execution day arrived, there was nothing left to figure out.</p><p>The endpoint catalog meant Phase 1.3 knew exactly which endpoints needed protection before writing any code. The service methods inventory meant Phase 1.2 had a checklist, not a discovery process. The risk assessment meant priorities were already set.</p><p>Phase 0 wasn’t overhead. It was the work. Everything after was mechanical application of decisions already made.</p><h3>Architecture decisions as force multipliers</h3><p>ADR-044 decided the fundamental approach: Lightweight RBAC using JSONB columns rather than traditional role and permission tables.</p><p>This decision was made days before implementation. The Chief Architect evaluated options, documented trade-offs, recommended an approach, got approval. His assessment was blunt: “Traditional RBAC would be architectural astronauting at our current scale. Building for 10,000 users when we have &lt;100 is how projects die from complexity before reaching users.”</p><p>The traditional approach — role tables, permission tables, junction tables — would have taken 2–3 weeks. The lightweight approach took 5 hours. Both met our security requirements. Only one let us ship on time.</p><p>When Phase 2 implementation began, nobody debated whether to use JSONB or separate tables. That question was answered.</p><p>The implementation followed directly from the decision:</p><pre>JSONB upgrade: [&quot;uuid&quot;] → [{&quot;user_id&quot;: &quot;uuid&quot;, &quot;role&quot;: &quot;viewer&quot;}]<br>Applied to lists and todos tables<br>Existing shares default to &quot;viewer&quot;</pre><p>Clear architectural decisions become implementation specifications. They’re the ones whose preparation makes implementation almost trivial.</p><h3>When this doesn’t work</h3><p>Front-loading works when:</p><ul><li>Requirements are reasonably stable</li><li>Patterns apply across multiple implementations</li><li>Architecture decisions are tractable before coding</li><li>The team can maintain context across preparation and execution</li></ul><p>Front-loading fails when:</p><ul><li>Requirements will change significantly during implementation</li><li>Each feature is genuinely unique (no patterns to establish)</li><li>You need to build to learn what the architecture should be</li><li>Long gaps between preparation and execution lose context</li></ul><p>The SEC-RBAC epic was a good candidate for front-loading: security requirements were clear, patterns applied across all services, architecture options were well-understood, and execution followed preparation within days.</p><p>Exploratory work — figuring out what to build rather than building a known thing — might need a different approach. When “done” is explicit, progress is measurable and scope is contained.</p><p><strong>Invest in coordination infrastructure</strong>. It was the predictable result of systematic preparation making execution mechanical.</p><p>Three weeks of work, done in 8 hours. Not by working faster. By doing less work during execution — because the work had already been done during preparation.</p><p><em>Next on Building Piper Morgan: Grammar as Decision Tool — on how the meaning of what you are building can help you figure out what exactly to build and how to build it.</em></p><p><em>Have you experienced preparation dramatically accelerating execution? It took approximately 8 hours.</p><p name="2a02" id="2a02" class="graf graf--p graf-after--p">Not 8 hours of heroic coding. Not 8 hours of cutting corners. Not 8 hours that would create 8 weeks of technical debt. Eight hours of systematic execution, finishing 45% ahead of even the optimistic estimates.</p><p name="0a3f" id="0a3f" class="graf graf--p graf-after--p">This isn’t a story about working faster. It’s a story about preparation front-loading the hard work until execution becomes almost mechanical.</p><h3 name="a2af" id="a2af" class="graf graf--h3 graf-after--p">The estimate wasn’t wrong</h3><p name="d9ce" id="d9ce" class="graf graf--p graf-after--h3">Sprint S1 — the security foundation sprint — was scoped at 81 hours total. RBAC alone was estimated at 20–24 hours. Encryption at rest another 24–30 hours. Supporting work for the remainder.</p><p name="a55e" id="a55e" class="graf graf--p graf-after--p">These weren’t pessimistic estimates. They were based on reasonable assumptions:</p><ul class="postList"><li name="6e04" id="6e04" class="graf graf--li graf-after--p">Phase 1.1 (database schema): 4–6 hours</li><li name="0802" id="0802" class="graf graf--li graf-after--li">Phase 1.2 (service layer): 8–12 hours</li><li name="cab2" id="cab2" class="graf graf--li graf-after--li">Phase 1.3 (endpoint protection): 4–6 hours</li><li name="6a4e" id="6a4e" class="graf graf--li graf-after--li">Phase 1.4 (shared resources): 4–6 hours</li><li name="18f0" id="18f0" class="graf graf--li graf-after--li">Phase 2 (role-based permissions): 6–8 hours</li><li name="e23f" id="e23f" class="graf graf--li graf-after--li">Phase 3 (admin and testing): 4–6 hours</li></ul><p name="2f40" id="2f40" class="graf graf--p graf-after--li">Each estimate assumed some discovery work, some problem-solving, some iteration. Normal software development.</p><p name="c5b0" id="c5b0" class="graf graf--p graf-after--p">What the estimates didn’t account for: we’d already done most of the discovery work.</p><h3 name="85cd" id="85cd" class="graf graf--h3 graf-after--p">Phase 0 was the real work</h3><p name="97d4" id="97d4" class="graf graf--p graf-after--h3">The security sprint had a Phase 0: audit and planning. Six comprehensive reports created before any implementation began:</p><ol class="postList"><li name="953b" id="953b" class="graf graf--li graf-after--p">Phase -1 verification (infrastructure assessment)</li><li name="8982" id="8982" class="graf graf--li graf-after--li">Clarifications research (recommendations)</li><li name="3161" id="3161" class="graf graf--li graf-after--li">API endpoint catalog (56 endpoints mapped)</li><li name="d28d" id="d28d" class="graf graf--li graf-after--li">Service methods inventory (47 methods identified)</li><li name="0752" id="0752" class="graf graf--li graf-after--li">Risk assessment (P0 and P1 blockers categorized)</li><li name="cfae" id="cfae" class="graf graf--li graf-after--li">Phase 0 completion summary</li></ol><p name="5201" id="5201" class="graf graf--p graf-after--li">This audit took time — multiple sessions, deep investigation, comprehensive documentation. It didn’t feel like progress. No code was written. No tests passed. No features shipped.</p><p name="4f52" id="4f52" class="graf graf--p graf-after--p">But when execution day arrived, there was nothing left to figure out.</p><p name="0f45" id="0f45" class="graf graf--p graf-after--p">The endpoint catalog meant Phase 1.3 knew exactly which endpoints needed protection before writing any code. The service methods inventory meant Phase 1.2 had a checklist, not a discovery process. The risk assessment meant priorities were already set.</p><p name="98c4" id="98c4" class="graf graf--p graf-after--p">Phase 0 wasn’t overhead. It was the work. Everything after was mechanical application of decisions already made.</p><h3 name="99d8" id="99d8" class="graf graf--h3 graf-after--p">Architecture decisions as force multipliers</h3><p name="d491" id="d491" class="graf graf--p graf-after--h3">ADR-044 decided the fundamental approach: Lightweight RBAC using JSONB columns rather than traditional role and permission tables.</p><p name="480f" id="480f" class="graf graf--p graf-after--p">This decision was made days before implementation. The Chief Architect evaluated options, documented trade-offs, recommended an approach, got approval. His assessment was blunt: “Traditional RBAC would be architectural astronauting at our current scale. Building for 10,000 users when we have &lt;100 is how projects die from complexity before reaching users.”</p><p name="70f6" id="70f6" class="graf graf--p graf-after--p">The traditional approach — role tables, permission tables, junction tables — would have taken 2–3 weeks. The lightweight approach took 5 hours. Both met our security requirements. Only one let us ship on time.</p><p name="9d45" id="9d45" class="graf graf--p graf-after--p">When Phase 2 implementation began, nobody debated whether to use JSONB or separate tables. That question was answered.</p><p name="ac7f" id="ac7f" class="graf graf--p graf-after--p">The implementation followed directly from the decision:</p><pre data-code-block-mode="1" spellcheck="false" data-code-block-lang="css" name="08f2" id="08f2" class="graf graf--pre graf-after--p graf--preV2"><span class="pre--content">JSONB upgrade: [<span class="hljs-string">&quot;uuid&quot;</span>] → [{&quot;user_id&quot;: <span class="hljs-string">&quot;uuid&quot;</span>, <span class="hljs-string">&quot;role&quot;</span>: <span class="hljs-string">&quot;viewer&quot;</span>}]<br />Applied <span class="hljs-selector-tag">to</span> lists and todos tables<br />Existing shares default <span class="hljs-selector-tag">to</span> &quot;viewer&quot;</span></pre><p name="c711" id="c711" class="graf graf--p graf-after--pre">Clear architectural decisions become implementation specifications. The team doesn’t stop to think; they execute the specification.</p><p name="5f1f" id="5f1f" class="graf graf--p graf-after--p">Every architectural decision made before implementation day was time saved during implementation day. ADR-044 probably saved 2–4 hours of implementation-time debate and iteration.</p><h3 name="db15" id="db15" class="graf graf--h3 graf-after--p">Patterns established, patterns reused</h3><p name="44eb" id="44eb" class="graf graf--p graf-after--h3">The first service to get ownership validation was KnowledgeGraphService — 20 methods across service and repository layers. That implementation established the pattern:</p><ul class="postList"><li name="1ebc" id="1ebc" class="graf graf--li graf-after--p">Optional <code class="markup--code markup--li-code">owner_id</code> parameter</li><li name="0340" id="0340" class="graf graf--li graf-after--li">Conditional filtering when provided</li><li name="1e7e" id="1e7e" class="graf graf--li graf-after--li">404 responses for non-owned resources (prevents information leakage)</li><li name="b4eb" id="b4eb" class="graf graf--li graf-after--li">Tests validating ownership boundaries</li></ul><p name="931b" id="931b" class="graf graf--p graf-after--li">Once the pattern was established, applying it to ProjectRepository (7 methods), FileRepository (14 methods), and six other services was mechanical. Same pattern, different services. The second service took half the time of the first. The sixth took a quarter.</p><p name="fd58" id="fd58" class="graf graf--p graf-after--p">Pattern reuse isn’t just code efficiency. It’s cognitive efficiency. The agent doesn’t need to think about <em class="markup--em markup--p-em">how</em> to implement ownership validation. It just applies the established pattern to new services.</p><p name="7d31" id="7d31" class="graf graf--p graf-after--p">This is why Phase 1.2 — estimated at 8–12 hours for 67+ methods — completed so quickly. The pattern was figured out once and applied many times.</p><h3 name="7e29" id="7e29" class="graf graf--h3 graf-after--p">Completion matrices as execution guides</h3><p name="519a" id="519a" class="graf graf--p graf-after--h3">Each phase had a completion matrix: explicit lists of what needed to be done, with clear done/not-done states.</p><p name="8c82" id="8c82" class="graf graf--p graf-after--p">Phase 1.2 completion matrix listed every service and every method:</p><ul class="postList"><li name="70b1" id="70b1" class="graf graf--li graf-after--p">FileRepository: 14 methods (done/not done for each)</li><li name="78e9" id="78e9" class="graf graf--li graf-after--li">KnowledgeGraphService: 12 methods</li><li name="79e5" id="79e5" class="graf graf--li graf-after--li">ProjectRepository: 7 methods</li><li name="b25b" id="b25b" class="graf graf--li graf-after--li">UniversalListRepository: 11 methods</li><li name="b8dd" id="b8dd" class="graf graf--li graf-after--li">… and so on</li></ul><p name="a9c3" id="a9c3" class="graf graf--p graf-after--li">The agent didn’t need to discover what work remained. The matrix showed it. Progress was measurable. Scope was explicit. Nothing was ambiguous.</p><p name="3448" id="3448" class="graf graf--p graf-after--p">When the completion matrix says “67 methods across 9 services,” the agent knows exactly what success looks like. No scope creep. No “one more thing.” Just: execute the matrix.</p><p name="82f3" id="82f3" class="graf graf--p graf-after--p">Phase 2’s permission matrix was similarly explicit:</p><figure name="f978" id="f978" class="graf graf--figure graf-after--p"><img class="graf-image" data-image-id="1*Q7STdniZCnzliCEbrVNaSg.png" data-width="944" data-height="438" src="https://cdn-images-1.medium.com/max/800/1*Q7STdniZCnzliCEbrVNaSg.png"></figure><p name="2833" id="2833" class="graf graf--p graf-after--figure">Twenty-four test cases fell directly out of this matrix: 4 roles × 6 operations. The security audit doesn’t ship features. ADRs don’t pass tests. Completion matrices aren’t code.</p><p name="b129" id="b129" class="graf graf--p graf-after--p">But if you measure “time from project start to feature complete,” preparation is obviously investment. Every hour of Phase 0 saved multiple hours during Phases 1–3. The return on preparation was 3–5x.</p><p name="64cf" id="64cf" class="graf graf--p graf-after--p">The teams that ship fastest aren’t the ones that start implementing fastest. Sometimes you need to start implementing to discover what the architecture should be.</p><h3 name="52c0" id="52c0" class="graf graf--h3 graf-after--p">The transfer</h3><p name="ca34" id="ca34" class="graf graf--p graf-after--h3">Here’s what transfers to other projects:</p><p name="2340" id="2340" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Audit before building</strong>. Map the landscape. Catalog what exists. Identify what needs to change. This investment pays off in execution clarity.</p><p name="624a" id="624a" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Decide architecture separately from implementation</strong>. ADRs, design docs, technical specifications — whatever format works. The point is: don’t debate during implementation. Debate before. Document the decision. Execute the decision.</p><p name="6958" id="6958" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Establish patterns explicitly</strong>. The first implementation of a pattern is learning. Document what you learned. Make subsequent implementations mechanical application, not repeated discovery.</p><p name="e523" id="e523" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Create explicit completion criteria</strong>. Matrices, checklists, acceptance criteria — whatever makes scope unambiguous. When “done” is explicit, progress is measurable and scope is contained.</p><p name="0f28" id="0f28" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Invest in coordination infrastructure</strong>.
