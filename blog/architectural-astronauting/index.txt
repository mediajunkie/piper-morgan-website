1b:T2796,<figure><img alt="A friendly blobby AI creature builds an enormous rocket launch tower while a person by a small garage ladder looks puzzled, highlighting extreme overengineering versus a simple solution." src="https://cdn-images-1.medium.com/max/1024/1*zPle-LeyoT1JLt814M7p-g.png" /><figcaption>“Isn’t that kind of overkill?”</figcaption></figure><p><em>November 22</em></p><p>The security epic needed role-based access control. The traditional approach was well-documented: role tables, permission tables, role-permission junction tables, user-role assignments. Clean relational design. Proper normalization. Industry standard.</p><p>It would have taken 2–3 weeks.</p><p>My Chief Architect’s assessment: “Traditional RBAC would be <strong>architectural astronauting</strong> at our current scale. Building for 10,000 users when we have &lt;100 is how projects die from complexity before reaching users.”</p><p>We built lightweight RBAC in 5 hours instead. It shipped. It works. And it can be refactored when — if — we ever need the traditional approach.</p><h3>The astronauting pattern</h3><p>Architectural astronauting is building for a future that may never arrive.</p><p>It looks responsible. “We should build this properly from the start.” “We don’t want to accumulate technical debt.” “It’s harder to refactor later.” All true statements that lead to false conclusions.</p><p>The false conclusion: therefore, build the sophisticated version now.</p><p>The problem: sophisticated architectures have costs that start immediately, while their benefits arrive only if specific futures materialize. You pay the complexity tax today for options you might never exercise.</p><p>I got a first-hand lesson in that at CloudOn, when we opted to build for Android while still securing our iOS market. Twice as much reach with minimal extra effort, we thought. Instead we instantly doubled the complexity on the project with no clear guarantee of a payoff (and, in fact, the Android market never came close to our iOS userbase, it turned out.)</p><p>Traditional RBAC for our alpha would have meant:</p><ul><li>2–3 weeks of implementation instead of 5 hours</li><li>6+ new database tables instead of 2 JSONB columns</li><li>Complex query joins instead of single indexed lookups</li><li>Ongoing maintenance burden for features we don’t need</li></ul><p>All to serve fewer than 100 users who don’t need role hierarchies, organizational permissions, or granular admin controls.</p><h3>The scale question</h3><p>The right architecture depends on scale. This seems obvious but is often ignored.</p><p>Scale Right Approach &lt;1,000 users Lightweight (JSONB, simple checks) 1,000–10,000 Evaluate based on actual needs &gt;10,000 Traditional (if role complexity warrants)</p><p>We have fewer than 100 users. Traditional RBAC is designed for enterprises with thousands. Building enterprise architecture for a product that hasn’t proven product-market fit is how projects die.</p><p>Not from technical failure. From complexity that slows everything else down while waiting for scale that never arrives.</p><p>The astronauting temptation is strongest when you <em>can</em> see the sophisticated path clearly. “I know how to build traditional RBAC. It’s the proper way. Let me do it properly.”</p><p>Knowing how to build something doesn’t mean you should build it now.</p><h3>What lightweight actually meant</h3><p>Our lightweight RBAC uses JSONB columns:</p><pre>Before: shared_with = [&quot;user-uuid-1&quot;, &quot;user-uuid-2&quot;]<br>After:  shared_with = [{&quot;user_id&quot;: &quot;uuid&quot;, &quot;role&quot;: &quot;viewer&quot;}, ...]</pre><p>That’s it. Role information embedded in the sharing structure. GIN index for fast lookups. Single query to check permissions.</p><p>Performance: 10–20ms per query. Traditional approach: 30–50ms (joins across multiple tables).</p><p>The “proper” architecture is actually slower for our use case. The lightweight approach isn’t a compromise — it’s genuinely better at our scale.</p><p>Modern companies know this. Stripe, Notion, Linear all use JSONB for permissions at significant scale. The pattern is proven. We’re not being clever; we’re following established practice.</p><h3>The refactoring trigger list</h3><p>Lightweight doesn’t mean permanent. We defined explicit triggers for when to refactor:</p><ul><li>User base exceeds 1,000 active users</li><li>Need role hierarchies (teams, organizations)</li><li>Granular admin permissions required</li><li>JSONB query performance degrades beyond 50ms</li><li>Enterprise customer requires traditional RBAC</li><li>Security audit mandates relational approach</li></ul><p>None of these are true today. When any becomes true, we refactor. Until then, the lightweight approach serves perfectly.</p><p>This is the key insight: you can build simple now AND have a plan for sophisticated later. The triggers make the decision explicit. You’re not ignoring future needs; you’re deferring investment until those needs are real.</p><p>Oh, and the trigger has to actually fire! We do that by scheduling regular staggered code (and doc) audits that look for those triggers.</p><h3>The velocity argument</h3><p>Those weeks spent on traditional RBAC would have been weeks not spent on:</p><ul><li>Frontend permission awareness</li><li>Alpha documentation</li><li>Navigation fixes</li><li>User onboarding</li><li>The features that actually determine whether anyone uses the product</li></ul><p>Every hour spent on premature sophistication is an hour stolen from proving the product matters.</p><p>Our first alpha tester arrived that week. If we’d started traditional RBAC, we’d still be implementing database migrations. Instead, she was using working software with working permissions.</p><p>Sophistication that prevents shipping is sophistication that prevents learning. And learning — from real users using real software — is how products survive.</p><h3>When sophistication is right</h3><p>Architectural astronauting is a failure mode, not a universal warning against sophistication.</p><p>Build sophisticated when:</p><ul><li>Current scale demands it (not projected scale)</li><li>Simpler approaches have proven inadequate</li><li>The complexity serves users, not architectural aesthetics</li><li>You can afford the timeline impact</li></ul><p>Build simple when:</p><ul><li>Current scale doesn’t demand sophistication</li><li>Simpler approaches might be adequate (you don’t know yet)</li><li>Time to learning matters more than architectural elegance</li><li>Explicit refactoring triggers can guide future decisions</li></ul><p>The question isn’t “what’s the best architecture?” It’s “what’s the best architecture for where we are now, with a path to where we might need to go?”</p><p>(For various definitions and sizes of now, so your mileage may vary!)</p><h3>A necessary distinction</h3><p>This isn’t about avoiding infrastructure investment entirely.</p><p>We’ve since learned what we call the Assembly Assumption: features hide their wiring requirements until you try to connect them. You’ll discover infrastructure gaps when components need to talk to each other. That infrastructure is necessary — it serves the product you’re building now.</p><p>The difference: build for <em>connection</em>, defer for <em>scale</em>. The wiring that lets your current features work together is essential. The architecture designed for millions of users you don’t have is astronauting.</p><p>One responds to present reality. The other anticipates a future that your present choices will determine.</p><h3>The death pattern</h3><p>Here’s how architectural astronauting kills projects:</p><ol><li>Team identifies future need (scale, enterprise features, complex permissions)</li><li>Team builds for that future now</li><li>Implementation takes weeks/months longer than simple approach</li><li>Product ships later (or never)</li><li>Users never materialize because product shipped too late</li><li>Sophisticated architecture serves no one</li></ol><p>The sophistication was correct for a future that never arrived because the sophistication itself prevented that future from arriving.</p><p>This is the cruelest irony: building for scale can prevent you from reaching scale.</p><h3>The Chief Architect’s call</h3><p>My Chief Architect approved lightweight RBAC with clear reasoning:</p><blockquote><em>Meets all security requirements. Appropriate for current and medium-term scale. Modern, proven pattern. Superior performance without caching. Enables alpha launch on schedule.</em></blockquote><p>And the key condition: document the approach clearly, define refactoring triggers, monitor as scale grows.</p><p>This is what good architectural judgment looks like. Not “always build sophisticated” or “always build simple.” Build appropriate. Know what appropriate means for your current situation. Have a plan for when the situation changes.</p><p>The astronauts who build for 10,000 users when they have 100 often end up with 0 users because they never shipped.</p><p>We shipped. That week, our first alpha tester logged in. The permissions worked. The architecture served its purpose.</p><p>That’s not a compromise. That’s success.</p><p><em>Next on Building Piper Morgan: Accepting Architectural Limits — when 67% complete is actually complete.</em></p><p><em>Have you seen projects die from premature sophistication? And it can be refactored when — if — we ever need the traditional approach.</p><h3 name="8947" id="8947" class="graf graf--h3 graf-after--p">The astronauting pattern</h3><p name="bcfb" id="bcfb" class="graf graf--p graf-after--h3">Architectural astronauting is building for a future that may never arrive.</p><p name="afe4" id="afe4" class="graf graf--p graf-after--p">It looks responsible. “We should build this properly from the start.” “We don’t want to accumulate technical debt.” “It’s harder to refactor later.” All true statements that lead to false conclusions.</p><p name="c2da" id="c2da" class="graf graf--p graf-after--p">The false conclusion: therefore, build the sophisticated version now.</p><p name="d4fa" id="d4fa" class="graf graf--p graf-after--p">The problem: sophisticated architectures have costs that start immediately, while their benefits arrive only if specific futures materialize. You pay the complexity tax today for options you might never exercise.</p><p name="949f" id="949f" class="graf graf--p graf-after--p">I got a first-hand lesson in that at CloudOn, when we opted to build for Android while still securing our iOS market. Twice as much reach with minimal extra effort, we thought. Instead we instantly doubled the complexity on the project with no clear guarantee of a payoff (and, in fact, the Android market never came close to our iOS userbase, it turned out.)</p><p name="12d5" id="12d5" class="graf graf--p graf-after--p">Traditional RBAC for our alpha would have meant:</p><ul class="postList"><li name="b4aa" id="b4aa" class="graf graf--li graf-after--p">2–3 weeks of implementation instead of 5 hours</li><li name="058a" id="058a" class="graf graf--li graf-after--li">6+ new database tables instead of 2 JSONB columns</li><li name="16b1" id="16b1" class="graf graf--li graf-after--li">Complex query joins instead of single indexed lookups</li><li name="9d51" id="9d51" class="graf graf--li graf-after--li">Ongoing maintenance burden for features we don’t need</li></ul><p name="41aa" id="41aa" class="graf graf--p graf-after--li">All to serve fewer than 100 users who don’t need role hierarchies, organizational permissions, or granular admin controls.</p><h3 name="09e0" id="09e0" class="graf graf--h3 graf-after--p">The scale question</h3><p name="645b" id="645b" class="graf graf--p graf-after--h3">The right architecture depends on scale. This seems obvious but is often ignored.</p><p name="8e67" id="8e67" class="graf graf--p graf-after--p">Scale Right Approach &lt;1,000 users Lightweight (JSONB, simple checks) 1,000–10,000 Evaluate based on actual needs &gt;10,000 Traditional (if role complexity warrants)</p><p name="6f86" id="6f86" class="graf graf--p graf-after--p">We have fewer than 100 users. Traditional RBAC is designed for enterprises with thousands. Building enterprise architecture for a product that hasn’t proven product-market fit is how projects die.</p><p name="05e1" id="05e1" class="graf graf--p graf-after--p">Not from technical failure. From complexity that slows everything else down while waiting for scale that never arrives.</p><p name="a1c8" id="a1c8" class="graf graf--p graf-after--p">The astronauting temptation is strongest when you <em class="markup--em markup--p-em">can</em> see the sophisticated path clearly. “I know how to build traditional RBAC. It’s the proper way. Let me do it properly.”</p><p name="1640" id="1640" class="graf graf--p graf-after--p">Knowing how to build something doesn’t mean you should build it now.</p><h3 name="c471" id="c471" class="graf graf--h3 graf-after--p">What lightweight actually meant</h3><p name="3d08" id="3d08" class="graf graf--p graf-after--h3">Our lightweight RBAC uses JSONB columns:</p><pre data-code-block-mode="2" spellcheck="false" data-code-block-lang="json" name="1757" id="1757" class="graf graf--pre graf-after--p graf--preV2"><span class="pre--content">Before<span class="hljs-punctuation">:</span> shared_with = <span class="hljs-punctuation">[</span><span class="hljs-string">&quot;user-uuid-1&quot;</span><span class="hljs-punctuation">,</span> <span class="hljs-string">&quot;user-uuid-2&quot;</span><span class="hljs-punctuation">]</span><br />After<span class="hljs-punctuation">:</span>  shared_with = <span class="hljs-punctuation">[</span><span class="hljs-punctuation">{</span><span class="hljs-attr">&quot;user_id&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-string">&quot;uuid&quot;</span><span class="hljs-punctuation">,</span> <span class="hljs-attr">&quot;role&quot;</span><span class="hljs-punctuation">:</span> <span class="hljs-string">&quot;viewer&quot;</span><span class="hljs-punctuation">}</span><span class="hljs-punctuation">,</span> ...<span class="hljs-punctuation">]</span></span></pre><p name="fd2c" id="fd2c" class="graf graf--p graf-after--pre">That’s it. Role information embedded in the sharing structure. GIN index for fast lookups. Single query to check permissions.</p><p name="5ce9" id="5ce9" class="graf graf--p graf-after--p">Performance: 10–20ms per query. Traditional approach: 30–50ms (joins across multiple tables).</p><p name="0150" id="0150" class="graf graf--p graf-after--p">The “proper” architecture is actually slower for our use case. The lightweight approach isn’t a compromise — it’s genuinely better at our scale.</p><p name="1ccc" id="1ccc" class="graf graf--p graf-after--p">Modern companies know this. Stripe, Notion, Linear all use JSONB for permissions at significant scale. The pattern is proven. We’re not being clever; we’re following established practice.</p><h3 name="54c2" id="54c2" class="graf graf--h3 graf-after--p">The refactoring trigger list</h3><p name="5082" id="5082" class="graf graf--p graf-after--h3">Lightweight doesn’t mean permanent. We defined explicit triggers for when to refactor:</p><ul class="postList"><li name="2334" id="2334" class="graf graf--li graf-after--p">User base exceeds 1,000 active users</li><li name="6f34" id="6f34" class="graf graf--li graf-after--li">Need role hierarchies (teams, organizations)</li><li name="f28a" id="f28a" class="graf graf--li graf-after--li">Granular admin permissions required</li><li name="78f3" id="78f3" class="graf graf--li graf-after--li">JSONB query performance degrades beyond 50ms</li><li name="ccf0" id="ccf0" class="graf graf--li graf-after--li">Enterprise customer requires traditional RBAC</li><li name="b94d" id="b94d" class="graf graf--li graf-after--li">Security audit mandates relational approach</li></ul><p name="a3ba" id="a3ba" class="graf graf--p graf-after--li">None of these are true today. When any becomes true, we refactor. Until then, the lightweight approach serves perfectly.</p><p name="8f86" id="8f86" class="graf graf--p graf-after--p">This is the key insight: you can build simple now AND have a plan for sophisticated later. The triggers make the decision explicit. You’re not ignoring future needs; you’re deferring investment until those needs are real.</p><p name="f666" id="f666" class="graf graf--p graf-after--p">Oh, and the trigger has to actually fire! We do that by scheduling regular staggered code (and doc) audits that look for those triggers.</p><h3 name="a57f" id="a57f" class="graf graf--h3 graf-after--p">The velocity argument</h3><p name="dc82" id="dc82" class="graf graf--p graf-after--h3">Those weeks spent on traditional RBAC would have been weeks not spent on:</p><ul class="postList"><li name="c435" id="c435" class="graf graf--li graf-after--p">Frontend permission awareness</li><li name="8778" id="8778" class="graf graf--li graf-after--li">Alpha documentation</li><li name="04e3" id="04e3" class="graf graf--li graf-after--li">Navigation fixes</li><li name="85ae" id="85ae" class="graf graf--li graf-after--li">User onboarding</li><li name="5578" id="5578" class="graf graf--li graf-after--li">The features that actually determine whether anyone uses the product</li></ul><p name="6c06" id="6c06" class="graf graf--p graf-after--li">Every hour spent on premature sophistication is an hour stolen from proving the product matters.</p><p name="c90c" id="c90c" class="graf graf--p graf-after--p">Our first alpha tester arrived that week. If we’d started traditional RBAC, we’d still be implementing database migrations. Instead, she was using working software with working permissions.</p><p name="2ce6" id="2ce6" class="graf graf--p graf-after--p">Sophistication that prevents shipping is sophistication that prevents learning. And learning — from real users using real software — is how products survive.</p><h3 name="ae2f" id="ae2f" class="graf graf--h3 graf-after--p">When sophistication is right</h3><p name="3470" id="3470" class="graf graf--p graf-after--h3">Architectural astronauting is a failure mode, not a universal warning against sophistication.</p><p name="5700" id="5700" class="graf graf--p graf-after--p">Build sophisticated when:</p><ul class="postList"><li name="eabb" id="eabb" class="graf graf--li graf-after--p">Current scale demands it (not projected scale)</li><li name="a723" id="a723" class="graf graf--li graf-after--li">Simpler approaches have proven inadequate</li><li name="b0e4" id="b0e4" class="graf graf--li graf-after--li">The complexity serves users, not architectural aesthetics</li><li name="3036" id="3036" class="graf graf--li graf-after--li">You can afford the timeline impact</li></ul><p name="ca5a" id="ca5a" class="graf graf--p graf-after--li">Build simple when:</p><ul class="postList"><li name="807e" id="807e" class="graf graf--li graf-after--p">Current scale doesn’t demand sophistication</li><li name="4011" id="4011" class="graf graf--li graf-after--li">Simpler approaches might be adequate (you don’t know yet)</li><li name="1089" id="1089" class="graf graf--li graf-after--li">Time to learning matters more than architectural elegance</li><li name="0b01" id="0b01" class="graf graf--li graf-after--li">Explicit refactoring triggers can guide future decisions</li></ul><p name="0f2f" id="0f2f" class="graf graf--p graf-after--li">The question isn’t “what’s the best architecture?” It’s “what’s the best architecture for where we are now, with a path to where we might need to go?”</p><p name="59ad" id="59ad" class="graf graf--p graf-after--p">(For various definitions and sizes of now, so your mileage may vary!)</p><h3 name="ee97" id="ee97" class="graf graf--h3 graf-after--p">A necessary distinction</h3><p name="05e3" id="05e3" class="graf graf--p graf-after--h3">This isn’t about avoiding infrastructure investment entirely.</p><p name="1c95" id="1c95" class="graf graf--p graf-after--p">We’ve since learned what we call the Assembly Assumption: features hide their wiring requirements until you try to connect them. You’ll discover infrastructure gaps when components need to talk to each other. That infrastructure is necessary — it serves the product you’re building now.</p><p name="ee7f" id="ee7f" class="graf graf--p graf-after--p">The difference: build for <em class="markup--em markup--p-em">connection</em>, defer for <em class="markup--em markup--p-em">scale</em>. The wiring that lets your current features work together is essential. The architecture designed for millions of users you don’t have is astronauting.</p><p name="37b7" id="37b7" class="graf graf--p graf-after--p">One responds to present reality. The other anticipates a future that your present choices will determine.</p><h3 name="14e6" id="14e6" class="graf graf--h3 graf-after--p">The death pattern</h3><p name="8226" id="8226" class="graf graf--p graf-after--h3">Here’s how architectural astronauting kills projects:</p><ol class="postList"><li name="9832" id="9832" class="graf graf--li graf-after--p">Team identifies future need (scale, enterprise features, complex permissions)</li><li name="8f92" id="8f92" class="graf graf--li graf-after--li">Team builds for that future now</li><li name="1ca6" id="1ca6" class="graf graf--li graf-after--li">Implementation takes weeks/months longer than simple approach</li><li name="79fe" id="79fe" class="graf graf--li graf-after--li">Product ships later (or never)</li><li name="af3f" id="af3f" class="graf graf--li graf-after--li">Users never materialize because product shipped too late</li><li name="fe31" id="fe31" class="graf graf--li graf-after--li">Sophisticated architecture serves no one</li></ol><p name="8ad6" id="8ad6" class="graf graf--p graf-after--li">The sophistication was correct for a future that never arrived because the sophistication itself prevented that future from arriving.</p><p name="c2d5" id="c2d5" class="graf graf--p graf-after--p">This is the cruelest irony: building for scale can prevent you from reaching scale.</p><h3 name="e8b7" id="e8b7" class="graf graf--h3 graf-after--p">The Chief Architect’s call</h3><p name="c760" id="c760" class="graf graf--p graf-after--h3">My Chief Architect approved lightweight RBAC with clear reasoning:</p><blockquote name="d2ae" id="d2ae" class="graf graf--blockquote graf-after--p"><em class="markup--em markup--blockquote-em">Meets all security requirements. Appropriate for current and medium-term scale. Modern, proven pattern. Superior performance without caching. Enables alpha launch on schedule.</em></blockquote><p name="1097" id="1097" class="graf graf--p graf-after--blockquote">And the key condition: document the approach clearly, define refactoring triggers, monitor as scale grows.</p><p name="a497" id="a497" class="graf graf--p graf-after--p">This is what good architectural judgment looks like. Not “always build sophisticated” or “always build simple.” Build appropriate. Know what appropriate means for your current situation. Have a plan for when the situation changes.</p><p name="2d21" id="2d21" class="graf graf--p graf-after--p">The astronauts who build for 10,000 users when they have 100 often end up with 0 users because they never shipped.</p><p name="6938" id="6938" class="graf graf--p graf-after--p">We shipped. That week, our first alpha tester logged in. The permissions worked. The architecture served its purpose.</p><p name="b32c" id="b32c" class="graf graf--p graf-after--p graf--trailing">That’s not a compromise.
