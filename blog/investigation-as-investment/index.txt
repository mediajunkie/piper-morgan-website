Actual fix time: 5 minutes. That’s not a typo. Five minutes. Three bugs. All fixed.</p><p>It wasn’t a matter of coding faster. It came from thirty minutes of systematic investigation before touching any code. Root cause analysis that revealed the bugs were simpler than they appeared. .</p><p>This is investigation as investment: spending time to understand before spending time to implement.</p><h3>The speedup</h3><p>Saturday was frantic. First alpha tester arriving Monday. Fourteen navigation issues identified during a full system walkthrough. The pressure to just start fixing things was immense.</p><p>Instead, we investigated first.</p><p>Phase 1 took thirty minutes. The goal wasn’t to fix anything — just to understand what was actually broken. Three high-priority issues examined: logout not working, lists creation failing, todos creation failing.</p><p>What the investigation revealed:</p><p><strong>Issue #14 (Logout)</strong>: The frontend called /api/v1/auth/logout. The backend expected /auth/logout. Path mismatch. Five-minute fix once you know what&#39;s wrong.</p><p><strong>Issue #6 (Lists)</strong>: The UI tried to POST to /api/v1/lists. That endpoint didn&#39;t exist. The UI was built; the endpoint wasn&#39;t wired up. Five-minute fix once you know what&#39;s missing.</p><p><strong>Issue #7 (Todos)</strong>: Identical pattern to Issue #6. POST endpoint missing. Same fix pattern.</p><p>Without investigation, each bug looks like “feature doesn’t work.” That could mean anything — authentication issues, database problems, validation failures, race conditions. The estimate of 35 minutes assumed some debugging, some experimentation, some wrong turns.</p><p>With investigation, each bug has a specific diagnosis. Path mismatch. Missing endpoint. Missing endpoint. Implementation becomes mechanical: change the path, add the endpoint, add the endpoint.</p><p><em>This is “measure twice, cut once” on steroids.</em></p><p>Thirty minutes of investigation. Five minutes of implementation.</p><h3>That ol’ “75% complete” pattern</h3><p>Investigation revealed something broader than individual bugs. Many features were 75–90% complete.</p><p>The Lists UI existed. The buttons existed. The forms existed. The styling existed. The only thing missing: the POST endpoint to actually create a list.</p><p>This pattern repeated across the codebase. Elements existed but wiring was incomplete. Frontend built but backend missing. Backend ready but frontend not connected. Features that looked broken were actually almost finished.</p><p>Without investigation, you might rebuild the whole feature. “Lists aren’t working” becomes “let’s implement lists.” Hours of work recreating what already exists.</p><p>With investigation, you find the gap. “Lists aren’t working because POST /api/v1/lists doesn’t exist.” Ten minutes to add the endpoint. Feature works.</p><p>Investigation isn’t just finding bugs. It’s finding what’s already built. In a complex codebase — especially one built by multiple agents across many sessions — significant work might exist that no one remembers building. Investigation surfaces that work.</p><h3>Investigation vs implementation time</h3><p>Here’s how the Saturday numbers broke down:</p><p>The three high-priority bugs took 30 minutes to investigate and just 5 minutes to implement — a 7x speedup. Three medium-priority bugs followed a similar pattern: 20 minutes of investigation, 15 minutes of implementation, roughly 3x faster than diving straight in. The Frontend RBAC work (Option B) was estimated at 6–7 hours but completed in 54 minutes; investigation had moved the discovery phase earlier, yielding a 6–8x improvement. Even alpha documentation benefited: 15 minutes of investigation followed by 40 minutes of writing, about 1.8x faster than starting cold.</p><p>The pattern: investigation time is rarely wasted. Even when it doesn’t dramatically accelerate implementation, it prevents wrong turns and wasted effort.</p><p>When investigation revealed Issue #8 (Files UI) would require 90–120 minutes — significantly more than other bugs — we made an informed decision to defer it. Backend was ready; frontend work could wait. Without investigation, we might have started the fix, discovered the scope mid-implementation, and lost time to the false start.</p><p>Investigation enables better decisions about what to work on, not just how to work on it.</p><h3>What investigation actually looks like</h3><p>Effective investigation isn’t “look at the code until inspiration strikes.” It’s systematic analysis with specific questions:</p><p><strong>What’s the user-visible symptom?</strong> Logout button does nothing. Lists creation fails. User menu doesn’t appear.</p><p><strong>What’s the expected behavior?</strong> Logout should clear session and redirect to login. Lists creation should add a new list to the database. User menu should appear when logged in.</p><p><strong>Where does the behavior break?</strong> Frontend calling wrong endpoint? Backend missing the endpoint? Database operation failing? Authentication not working?</p><p><strong>What evidence exists?</strong> Network requests in browser console. Server logs. Database state. Error messages.</p><p><strong>What’s the minimum fix?</strong> Change endpoint path. Add missing route. Fix authentication extraction.</p><p>Each question narrows the problem space. By the time you’re ready to implement, you’re not fixing “logout doesn’t work.” You’re fixing “frontend calls /api/v1/auth/logout but backend expects /auth/logout.”</p><p>The discipline is: don’t touch code until you can state the specific problem. If you can’t articulate exactly what’s broken, you don’t understand enough to fix it efficiently.</p><h3>When investigation doesn’t pay off</h3><p>Investigation has diminishing returns in some contexts:</p><p><strong>Obvious problems</strong>: If the error message says “ModuleNotFoundError: No module named ‘foo’,” you don’t need investigation. Install foo.</p><p><strong>Exploratory work</strong>: If you’re building something new, investigation has nothing to find. You need to implement to discover what the architecture should be.</p><p><strong>Time-critical emergencies</strong>: If production is down, start with likely fixes. Post-incident, investigate properly.</p><p><strong>Truly novel bugs</strong>: If no one has seen this kind of bug before, investigation might not surface patterns. You might need to experiment.</p><p>The speed boost happened because the bugs were standard web application issues in an established codebase. Path mismatches and missing endpoints have well-understood fixes. Investigation surfaced the specific instances.</p><p>For a bug that’s genuinely novel — a race condition in async code, a memory leak under specific conditions, a security vulnerability with subtle triggers — investigation might take longer and yield less dramatic speedups. The payoff is different for different problem types.</p><h3>Investigation as documentation</h3><p>Investigation produces more than faster fixes. It produces understanding.</p><p>The thirty-minute investigation on Saturday created a classification system:</p><ul><li>Type A bugs (simple path or config fixes) take about 5 minutes.</li><li>Type B bugs (missing endpoint implementation) run 45–60 minutes.</li><li>Type C bugs (logic errors in existing code) fall in the 15–30 minute range.</li><li>Type D bugs (feature gaps requiring new development) need 90–120 minutes.</li></ul><p>Future bugs could be classified using this system. When a new bug arrives, the question becomes “is this Type A, B, C, or D?” The classification implies the fix approach and time estimate.</p><p>Investigation isn’t just solving today’s problem. It’s building understanding that accelerates tomorrow’s problem-solving.</p><h3>The investment mindset</h3><p>The temptation under deadline pressure is to start fixing immediately. “Michelle arrives Monday. We don’t have time to investigate. Just fix things.”</p><p>This is false economy. Fixing things without understanding wastes more time than it saves. The 35-minute estimate assumed some wrong turns. Investigation eliminated the wrong turns.</p><p>The investment mindset asks: “What understanding would make this fix trivial?” Then invests time in acquiring that understanding.</p><p>Sometimes the answer is “just look at the error message.” Sometimes it’s “trace the request through the full stack.” Sometimes it’s “read the original design document.” The investment varies. The principle doesn’t.</p><p>Understanding first. Implementation second.</p><h3>What transfers</h3><p>The speed gain isn’t guaranteed for every bug. But the principle applies broadly:</p><p><strong>Diagnose before treating</strong>. Don’t start fixing until you can state specifically what’s broken. If you can’t articulate the problem, you don’t understand enough to fix it efficiently.</p><p><strong>Look for existing work</strong>. In complex codebases, features are often partially implemented. Investigation surfaces what exists before you rebuild it.</p><p><strong>Classify problems</strong>. Investigation produces categories. Categories enable pattern recognition. Pattern recognition accelerates future investigations.</p><p><strong>Make investigation explicit</strong>. Allocate investigation time in estimates. Track investigation separately from implementation. Make the investment visible so it can be managed.</p><p><strong>Trust the investment</strong>. Under deadline pressure, investigation feels like delay. It’s not. It’s compression. Thirty minutes of investigation that saves thirty minutes of wrong turns isn’t cost — it’s break-even. Thirty minutes that enables five-minute fixes is 7x return.</p><p>The 30-minute investigation yielded a 5-minute implementation. That’s not luck. That’s the predictable return on understanding before acting.</p><p><em>Next on Building Piper Morgan: Analysis as Acceleration, Not Delay, from December 8. Actual fix time: 5 minutes. That’s not a typo. Five minutes. Three bugs. All fixed.</p><p name="3f77" id="3f77" class="graf graf--p graf-after--p">It wasn’t a matter of coding faster. It came from thirty minutes of systematic investigation before touching any code. Root cause analysis that revealed the bugs were simpler than they appeared. .</p><p name="2134" id="2134" class="graf graf--p graf-after--p">This is investigation as investment: spending time to understand before spending time to implement.</p><h3 name="7770" id="7770" class="graf graf--h3 graf-after--p">The speedup</h3><p name="bc58" id="bc58" class="graf graf--p graf-after--h3">Saturday was frantic. First alpha tester arriving Monday. Fourteen navigation issues identified during a full system walkthrough. The pressure to just start fixing things was immense.</p><p name="af19" id="af19" class="graf graf--p graf-after--p">Instead, we investigated first.</p><p name="2f98" id="2f98" class="graf graf--p graf-after--p">Phase 1 took thirty minutes. The goal wasn’t to fix anything — just to understand what was actually broken. Three high-priority issues examined: logout not working, lists creation failing, todos creation failing.</p><p name="7c9e" id="7c9e" class="graf graf--p graf-after--p">What the investigation revealed:</p><p name="c4d9" id="c4d9" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Issue #14 (Logout)</strong>: The frontend called <code class="markup--code markup--p-code">/api/v1/auth/logout</code>. The backend expected <code class="markup--code markup--p-code">/auth/logout</code>. Path mismatch. Five-minute fix once you know what&#39;s wrong.</p><p name="b857" id="b857" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Issue #6 (Lists)</strong>: The UI tried to POST to <code class="markup--code markup--p-code">/api/v1/lists</code>. That endpoint didn&#39;t exist. The UI was built; the endpoint wasn&#39;t wired up. Five-minute fix once you know what&#39;s missing.</p><p name="588e" id="588e" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Issue #7 (Todos)</strong>: Identical pattern to Issue #6. POST endpoint missing. Same fix pattern.</p><p name="0aaf" id="0aaf" class="graf graf--p graf-after--p">Without investigation, each bug looks like “feature doesn’t work.” That could mean anything — authentication issues, database problems, validation failures, race conditions. The estimate of 35 minutes assumed some debugging, some experimentation, some wrong turns.</p><p name="16c5" id="16c5" class="graf graf--p graf-after--p">With investigation, each bug has a specific diagnosis. Path mismatch. Missing endpoint. Missing endpoint. Implementation becomes mechanical: change the path, add the endpoint, add the endpoint.</p><p name="aa4b" id="aa4b" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">This is “measure twice, cut once” on steroids.</em></p><p name="27df" id="27df" class="graf graf--p graf-after--p">Thirty minutes of investigation. Five minutes of implementation.</p><h3 name="e5dc" id="e5dc" class="graf graf--h3 graf-after--p">That ol’ “75% complete” pattern</h3><p name="a8ee" id="a8ee" class="graf graf--p graf-after--h3">Investigation revealed something broader than individual bugs. Many features were 75–90% complete.</p><p name="4d96" id="4d96" class="graf graf--p graf-after--p">The Lists UI existed. The buttons existed. The forms existed. The styling existed. The only thing missing: the POST endpoint to actually create a list.</p><p name="4a4b" id="4a4b" class="graf graf--p graf-after--p">This pattern repeated across the codebase. Elements existed but wiring was incomplete. Frontend built but backend missing. Backend ready but frontend not connected. Features that looked broken were actually almost finished.</p><p name="42a2" id="42a2" class="graf graf--p graf-after--p">Without investigation, you might rebuild the whole feature. “Lists aren’t working” becomes “let’s implement lists.” Hours of work recreating what already exists.</p><p name="18ab" id="18ab" class="graf graf--p graf-after--p">With investigation, you find the gap. “Lists aren’t working because POST /api/v1/lists doesn’t exist.” Ten minutes to add the endpoint. Feature works.</p><p name="100b" id="100b" class="graf graf--p graf-after--p">Investigation isn’t just finding bugs. It’s finding what’s already built. In a complex codebase — especially one built by multiple agents across many sessions — significant work might exist that no one remembers building. Investigation surfaces that work.</p><h3 name="a460" id="a460" class="graf graf--h3 graf-after--p">Investigation vs implementation time</h3><p name="edde" id="edde" class="graf graf--p graf-after--h3">Here’s how the Saturday numbers broke down:</p><p name="5bbf" id="5bbf" class="graf graf--p graf-after--p">The three high-priority bugs took 30 minutes to investigate and just 5 minutes to implement — a 7x speedup. Three medium-priority bugs followed a similar pattern: 20 minutes of investigation, 15 minutes of implementation, roughly 3x faster than diving straight in. The Frontend RBAC work (Option B) was estimated at 6–7 hours but completed in 54 minutes; investigation had moved the discovery phase earlier, yielding a 6–8x improvement. Even alpha documentation benefited: 15 minutes of investigation followed by 40 minutes of writing, about 1.8x faster than starting cold.</p><p name="c414" id="c414" class="graf graf--p graf-after--p">The pattern: investigation time is rarely wasted. Even when it doesn’t dramatically accelerate implementation, it prevents wrong turns and wasted effort.</p><p name="57ff" id="57ff" class="graf graf--p graf-after--p">When investigation revealed Issue #8 (Files UI) would require 90–120 minutes — significantly more than other bugs — we made an informed decision to defer it. Backend was ready; frontend work could wait. Without investigation, we might have started the fix, discovered the scope mid-implementation, and lost time to the false start.</p><p name="d57d" id="d57d" class="graf graf--p graf-after--p">Investigation enables better decisions about what to work on, not just how to work on it.</p><h3 name="7613" id="7613" class="graf graf--h3 graf-after--p">What investigation actually looks like</h3><p name="5929" id="5929" class="graf graf--p graf-after--h3">Effective investigation isn’t “look at the code until inspiration strikes.” It’s systematic analysis with specific questions:</p><p name="2454" id="2454" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">What’s the user-visible symptom?</strong> Logout button does nothing. Lists creation fails. User menu doesn’t appear.</p><p name="b551" id="b551" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">What’s the expected behavior?</strong> Logout should clear session and redirect to login. Lists creation should add a new list to the database. User menu should appear when logged in.</p><p name="1a5d" id="1a5d" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Where does the behavior break?</strong> Frontend calling wrong endpoint? Backend missing the endpoint? Database operation failing? Authentication not working?</p><p name="c1b3" id="c1b3" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">What evidence exists?</strong> Network requests in browser console. Server logs. Database state. Error messages.</p><p name="4571" id="4571" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">What’s the minimum fix?</strong> Change endpoint path. Add missing route. Fix authentication extraction.</p><p name="6a04" id="6a04" class="graf graf--p graf-after--p">Each question narrows the problem space. By the time you’re ready to implement, you’re not fixing “logout doesn’t work.” You’re fixing “frontend calls /api/v1/auth/logout but backend expects /auth/logout.”</p><p name="bba0" id="bba0" class="graf graf--p graf-after--p">The discipline is: don’t touch code until you can state the specific problem. If you can’t articulate exactly what’s broken, you don’t understand enough to fix it efficiently.</p><h3 name="ec3d" id="ec3d" class="graf graf--h3 graf-after--p">When investigation doesn’t pay off</h3><p name="a484" id="a484" class="graf graf--p graf-after--h3">Investigation has diminishing returns in some contexts:</p><p name="6022" id="6022" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Obvious problems</strong>: If the error message says “ModuleNotFoundError: No module named ‘foo’,” you don’t need investigation. Install foo.</p><p name="12f9" id="12f9" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Exploratory work</strong>: If you’re building something new, investigation has nothing to find. You need to implement to discover what the architecture should be.</p><p name="c56e" id="c56e" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Time-critical emergencies</strong>: If production is down, start with likely fixes. Post-incident, investigate properly.</p><p name="2c5e" id="2c5e" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Truly novel bugs</strong>: If no one has seen this kind of bug before, investigation might not surface patterns. You might need to experiment.</p><p name="5ff4" id="5ff4" class="graf graf--p graf-after--p">The speed boost happened because the bugs were standard web application issues in an established codebase. Path mismatches and missing endpoints have well-understood fixes. Investigation surfaced the specific instances.</p><p name="3aa7" id="3aa7" class="graf graf--p graf-after--p">For a bug that’s genuinely novel — a race condition in async code, a memory leak under specific conditions, a security vulnerability with subtle triggers — investigation might take longer and yield less dramatic speedups. The payoff is different for different problem types.</p><h3 name="b92e" id="b92e" class="graf graf--h3 graf-after--p">Investigation as documentation</h3><p name="6e28" id="6e28" class="graf graf--p graf-after--h3">Investigation produces more than faster fixes. It produces understanding.</p><p name="bf99" id="bf99" class="graf graf--p graf-after--p">The thirty-minute investigation on Saturday created a classification system:</p><ul class="postList"><li name="c40e" id="c40e" class="graf graf--li graf-after--p">Type A bugs (simple path or config fixes) take about 5 minutes.</li><li name="f463" id="f463" class="graf graf--li graf-after--li">Type B bugs (missing endpoint implementation) run 45–60 minutes.</li><li name="85e7" id="85e7" class="graf graf--li graf-after--li">Type C bugs (logic errors in existing code) fall in the 15–30 minute range.</li><li name="9863" id="9863" class="graf graf--li graf-after--li">Type D bugs (feature gaps requiring new development) need 90–120 minutes.</li></ul><p name="da1a" id="da1a" class="graf graf--p graf-after--li">Future bugs could be classified using this system. When a new bug arrives, the question becomes “is this Type A, B, C, or D?” The classification implies the fix approach and time estimate.</p><p name="a1ed" id="a1ed" class="graf graf--p graf-after--p">Investigation isn’t just solving today’s problem. It’s building understanding that accelerates tomorrow’s problem-solving.</p><h3 name="c1d0" id="c1d0" class="graf graf--h3 graf-after--p">The investment mindset</h3><p name="c2bb" id="c2bb" class="graf graf--p graf-after--h3">The temptation under deadline pressure is to start fixing immediately. “Michelle arrives Monday. We don’t have time to investigate. Just fix things.”</p><p name="606a" id="606a" class="graf graf--p graf-after--p">This is false economy. Fixing things without understanding wastes more time than it saves. The 35-minute estimate assumed some wrong turns. Investigation eliminated the wrong turns.</p><p name="dd9c" id="dd9c" class="graf graf--p graf-after--p">The investment mindset asks: “What understanding would make this fix trivial?” Then invests time in acquiring that understanding.</p><p name="9486" id="9486" class="graf graf--p graf-after--p">Sometimes the answer is “just look at the error message.” Sometimes it’s “trace the request through the full stack.” Sometimes it’s “read the original design document.” The investment varies. The principle doesn’t.</p><p name="3308" id="3308" class="graf graf--p graf-after--p">Understanding first. Implementation second.</p><h3 name="66c9" id="66c9" class="graf graf--h3 graf-after--p">What transfers</h3><p name="cce2" id="cce2" class="graf graf--p graf-after--h3">The speed gain isn’t guaranteed for every bug. But the principle applies broadly:</p><p name="904c" id="904c" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Diagnose before treating</strong>. Don’t start fixing until you can state specifically what’s broken. If you can’t articulate the problem, you don’t understand enough to fix it efficiently.</p><p name="27b4" id="27b4" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Look for existing work</strong>. In complex codebases, features are often partially implemented. Investigation surfaces what exists before you rebuild it.</p><p name="8e59" id="8e59" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Classify problems</strong>. Investigation produces categories. Categories enable pattern recognition. Pattern recognition accelerates future investigations.</p><p name="5432" id="5432" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Make investigation explicit</strong>. Allocate investigation time in estimates. Track investigation separately from implementation. Make the investment visible so it can be managed.</p><p name="ffa1" id="ffa1" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Trust the investment</strong>. Under deadline pressure, investigation feels like delay. It’s not. It’s compression. Thirty minutes of investigation that saves thirty minutes of wrong turns isn’t cost — it’s break-even. Thirty minutes that enables five-minute fixes is 7x return.</p><p name="59fd" id="59fd" class="graf graf--p graf-after--p graf--trailing">The 30-minute investigation yielded a 5-minute implementation.
