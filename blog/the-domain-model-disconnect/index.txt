1b:T1c85,<figure><img alt="A builder team (person and robot) ralize the front of the house is done but the back is missing." src="https://cdn-images-1.medium.com/max/1024/1*M2YwMevHjc48AxhXBPGYVg.png" /><figcaption>“I thought we were done?”</figcaption></figure><p><em>January 12, 2026</em></p><p>It was release day. January 12th, v0.8.4 of Piper Morgan was live. We’d just completed Sprint B1 — six days of work, 23 issues closed, a new calendar integration, improved intent classification, and a cleaner codebase.</p><p>Then one of our alpha testers (me) reported a bug.</p><p>“The standup says I have no projects. But I just finished onboarding my portfolio.”</p><h3>Portfolio onboarding</h3><p>Piper Morgan has a portfolio onboarding flow. You tell the assistant about your projects — what you’re working on, what matters, what’s blocked. This context helps the Morning Standup feature give you relevant suggestions.</p><p>The onboarding worked perfectly. You could add projects. They were saved to the database. If you asked “what projects do I have?” the system would tell you. Everything passed our tests.</p><p>But when you asked for a standup? “I don’t see any projects in your portfolio yet.”</p><h3>The investigation</h3><p>The Lead Developer dug in at 9:30 PM. Here’s what they found:</p><p><strong>Portfolio onboarding</strong> stored projects in the projects table using ProjectRepository. This worked correctly. The data was there.</p><p><strong>Morning Standup</strong> asked UserContextService what projects the user had. UserContextService checked two places: the user’s preferences and the PIPER.md configuration file.</p><p>It never checked the database.</p><p>The storage worked. The retrieval worked. They just weren’t connected to each other.</p><h3>The anti-pattern</h3><p>We’ve started calling this the <strong>75% Trap</strong>. It’s a failure mode that seems to emerge specifically from AI-assisted development.</p><p>Here’s how it happens: You ask an AI to build Feature A. It builds Feature A beautifully — clean code, good tests, proper error handling. You ask it to build Feature B. Same thing. Beautiful work.</p><p>But Feature A stores data that Feature B needs to consume. And unless someone explicitly connects them, they won’t be connected. The AI built exactly what you asked for. You just didn’t ask for the connection (or you did but it forgot.)</p><p>This is different from traditional bugs. Traditional bugs are things that don’t work. The 75% Trap is things that <em>do</em> work — just not <em>together</em>.</p><p>Our tests passed because we tested each piece in isolation. Portfolio onboarding: ✓ stores projects. UserContextService: ✓ returns user context. Neither test knew the other existed.</p><h3>Why AI makes this worse</h3><p>Human developers carry mental models across features. A developer who built the portfolio onboarding would probably remember, three weeks later when building the standup feature, “oh right, projects are in that table I created.”</p><p>AI assistants don’t have that continuity. Each conversation, each feature, each file is somewhat isolated. They’re excellent at building the thing you’re looking at. They’re less reliable at remembering the thing you built last month.</p><p>This isn’t a criticism — it’s a characteristic. Understanding it lets you design around it.</p><h3>The simple fix</h3><p>The fix itself was simple. One line of real change: make UserContextService query the database directly instead of only checking preferences.</p><p>The investigation took longer than the fix. That’s usually how it goes with 75% Trap bugs — the problem isn’t that the code is hard to write, it’s that the <em>disconnection</em> is hard to see.</p><p>By 10:30 PM, the fix was committed. By the next morning, our tester confirmed: standup now sees their projects.</p><h3>What we learned</h3><p><strong>1. Test the user journey, not just the components.</strong></p><p>Our unit tests were green. Our integration tests were green. But we didn’t have a test that said “onboard a portfolio, then request a standup, and verify the standup knows about the portfolio.” That’s the test that would have caught this.</p><p><strong>2. AI-assisted development needs explicit connection reviews.</strong></p><p>When you build a new feature, ask: “What other features need to know about this?” When you build something that consumes data, ask: “Where does this data actually come from?” These questions feel obvious. They’re easy to skip when the code looks so complete.</p><p><strong>3. The house can look finished from the front.</strong></p><p>Our alpha tester saw a beautiful front porch — portfolio onboarding that worked, a standup feature that ran. They couldn’t see that the back wall was missing.</p><p>The 75% Trap isn’t about incomplete work. It’s about work that <em>looks</em> complete because each piece functions correctly in isolation. The incompleteness is in the connections, not the components.</p><h3>The broader lesson</h3><p>We’ve been building Piper Morgan for seven months now, mostly with AI assistance. The 75% Trap is probably our most common failure mode.</p><p>It shows up as:</p><ul><li>Features that store data nothing reads</li><li>APIs that exist but aren’t called</li><li>Configuration options that aren’t wired to behavior</li><li>Error handlers that catch exceptions but don’t surface them to users</li></ul><p>Every time, the individual pieces work. Every time, we passed our tests. Every time, the actual user experience was broken.</p><p>The fix isn’t to stop using AI. The fix is to understand that AI excels at building components and struggles with connections. So you build the connections into your process:</p><ul><li>Explicit “integration review” steps in your workflow</li><li>End-to-end tests that trace real user journeys</li><li>The question “what consumes this?” asked at every data storage point</li><li>The question “where does this come from?” asked at every data retrieval point</li></ul><p>We’re getting better at catching these. Bug #582 was found and fixed within twelve hours of release. Six months ago, it might have lingered for weeks.</p><p>But we’re not fooling ourselves. There’s probably another 75% Trap bug hiding in the codebase right now, waiting for a user to walk around to the back of the house.</p><p><em>Next on Building Piper Morgan: “The Audit Cascade” on a strict technique I’m adopting that is helping to tame the 75% antipattern.</em></p><p><em>Have you found unfinished gaps in work your AI assistant told you was all done and ready to ship. They just weren’t connected to each other.</p><h3 name="9776" id="9776" class="graf graf--h3 graf-after--p">The anti-pattern</h3><p name="9448" id="9448" class="graf graf--p graf-after--h3">We’ve started calling this the <strong class="markup--strong markup--p-strong">75% Trap</strong>. It’s a failure mode that seems to emerge specifically from AI-assisted development.</p><p name="7488" id="7488" class="graf graf--p graf-after--p">Here’s how it happens: You ask an AI to build Feature A. It builds Feature A beautifully — clean code, good tests, proper error handling. You ask it to build Feature B. Same thing. Beautiful work.</p><p name="ad6c" id="ad6c" class="graf graf--p graf-after--p">But Feature A stores data that Feature B needs to consume. And unless someone explicitly connects them, they won’t be connected. The AI built exactly what you asked for. You just didn’t ask for the connection (or you did but it forgot.)</p><p name="9101" id="9101" class="graf graf--p graf-after--p">This is different from traditional bugs. Traditional bugs are things that don’t work. The 75% Trap is things that <em class="markup--em markup--p-em">do</em> work — just not <em class="markup--em markup--p-em">together</em>.</p><p name="1722" id="1722" class="graf graf--p graf-after--p">Our tests passed because we tested each piece in isolation. Portfolio onboarding: ✓ stores projects. UserContextService: ✓ returns user context. Neither test knew the other existed.</p><h3 name="e7d6" id="e7d6" class="graf graf--h3 graf-after--p">Why AI makes this worse</h3><p name="a1bb" id="a1bb" class="graf graf--p graf-after--h3">Human developers carry mental models across features. A developer who built the portfolio onboarding would probably remember, three weeks later when building the standup feature, “oh right, projects are in that table I created.”</p><p name="2cb1" id="2cb1" class="graf graf--p graf-after--p">AI assistants don’t have that continuity. Each conversation, each feature, each file is somewhat isolated. They’re excellent at building the thing you’re looking at. They’re less reliable at remembering the thing you built last month.</p><p name="7b0b" id="7b0b" class="graf graf--p graf-after--p">This isn’t a criticism — it’s a characteristic. Understanding it lets you design around it.</p><h3 name="0e61" id="0e61" class="graf graf--h3 graf-after--p">The simple fix</h3><p name="a8ff" id="a8ff" class="graf graf--p graf-after--h3">The fix itself was simple. One line of real change: make <code class="markup--code markup--p-code">UserContextService</code> query the database directly instead of only checking preferences.</p><p name="e38a" id="e38a" class="graf graf--p graf-after--p">The investigation took longer than the fix. That’s usually how it goes with 75% Trap bugs — the problem isn’t that the code is hard to write, it’s that the <em class="markup--em markup--p-em">disconnection</em> is hard to see.</p><p name="5615" id="5615" class="graf graf--p graf-after--p">By 10:30 PM, the fix was committed. By the next morning, our tester confirmed: standup now sees their projects.</p><h3 name="fad8" id="fad8" class="graf graf--h3 graf-after--p">What we learned</h3><p name="3da9" id="3da9" class="graf graf--p graf-after--h3"><strong class="markup--strong markup--p-strong">1. Test the user journey, not just the components.</strong></p><p name="a2fe" id="a2fe" class="graf graf--p graf-after--p">Our unit tests were green. Our integration tests were green. But we didn’t have a test that said “onboard a portfolio, then request a standup, and verify the standup knows about the portfolio.” That’s the test that would have caught this.</p><p name="2c59" id="2c59" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">2. AI-assisted development needs explicit connection reviews.</strong></p><p name="cc3d" id="cc3d" class="graf graf--p graf-after--p">When you build a new feature, ask: “What other features need to know about this?” When you build something that consumes data, ask: “Where does this data actually come from?” These questions feel obvious. They’re easy to skip when the code looks so complete.</p><p name="c91b" id="c91b" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">3. The house can look finished from the front.</strong></p><p name="22e6" id="22e6" class="graf graf--p graf-after--p">Our alpha tester saw a beautiful front porch — portfolio onboarding that worked, a standup feature that ran. They couldn’t see that the back wall was missing.</p><p name="099e" id="099e" class="graf graf--p graf-after--p">The 75% Trap isn’t about incomplete work. It’s about work that <em class="markup--em markup--p-em">looks</em> complete because each piece functions correctly in isolation. The incompleteness is in the connections, not the components.</p><h3 name="50c0" id="50c0" class="graf graf--h3 graf-after--p">The broader lesson</h3><p name="e1ee" id="e1ee" class="graf graf--p graf-after--h3">We’ve been building Piper Morgan for seven months now, mostly with AI assistance. The 75% Trap is probably our most common failure mode.</p><p name="bec0" id="bec0" class="graf graf--p graf-after--p">It shows up as:</p><ul class="postList"><li name="ce5a" id="ce5a" class="graf graf--li graf-after--p">Features that store data nothing reads</li><li name="e95b" id="e95b" class="graf graf--li graf-after--li">APIs that exist but aren’t called</li><li name="045c" id="045c" class="graf graf--li graf-after--li">Configuration options that aren’t wired to behavior</li><li name="a0f7" id="a0f7" class="graf graf--li graf-after--li">Error handlers that catch exceptions but don’t surface them to users</li></ul><p name="efc3" id="efc3" class="graf graf--p graf-after--li">Every time, the individual pieces work. Every time, we passed our tests. Every time, the actual user experience was broken.</p><p name="ac2d" id="ac2d" class="graf graf--p graf-after--p">The fix isn’t to stop using AI. The fix is to understand that AI excels at building components and struggles with connections. So you build the connections into your process:</p><ul class="postList"><li name="2a31" id="2a31" class="graf graf--li graf-after--p">Explicit “integration review” steps in your workflow</li><li name="fd8e" id="fd8e" class="graf graf--li graf-after--li">End-to-end tests that trace real user journeys</li><li name="81fb" id="81fb" class="graf graf--li graf-after--li">The question “what consumes this?” asked at every data storage point</li><li name="170f" id="170f" class="graf graf--li graf-after--li">The question “where does this come from?” asked at every data retrieval point</li></ul><p name="5115" id="5115" class="graf graf--p graf-after--li">We’re getting better at catching these. Bug #582 was found and fixed within twelve hours of release. Six months ago, it might have lingered for weeks.</p><p name="7e77" id="7e77" class="graf graf--p graf-after--p graf--trailing">But we’re not fooling ourselves.
