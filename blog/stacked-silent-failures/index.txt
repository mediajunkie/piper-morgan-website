1a:T1ea1,<p><em>May 8–9, 2026</em></p>
<p>A few weeks ago an automated quality test on my project's main system came back six points below its prior baseline. The system runs a hundred scripted user inputs against the production code, scores each response on a rubric (relevance, competence, tone), and produces an aggregate quality number. The number had been hovering around 72%. The new number was 65.6%.</p>
<p>The engineering role filed an investigation. The working hypothesis was the obvious one. <em>The model has started fabricating responses.</em> The last few weeks' worth of architectural changes — there had been a lot — had presumably introduced a regression somewhere in the response-generation pipeline. The auto-fail rule on the rubric (any single dimension scoring zero triggers a FAIL) had caught ten cases that looked like fabrications.</p>
<p>The investigation took an afternoon. By the time the report landed, the hypothesis was largely refuted, and the actual finding was more interesting than the hypothesis.</p>
<p>The six-point drop wasn't one regression. It was <em>three</em> silent failures, stacked, each of which would have looked like a non-problem in isolation.</p>
<p>The first failure was in the judge. The scoring rubric had drifted toward over-weighting a specific dimension — <em>user-context-specificity</em>, how closely the response referenced the specific user's specific recent activity. The drift had been gradual and the judge had been doing it for at least the prior several weeks. If you looked at the judge in isolation, nothing was broken. It was scoring along a slightly different axis than the team had thought it was scoring on.</p>
<p>The second failure was in the auto-fail rule. <em>Any dimension at zero produces a FAIL on the case.</em> The rule had been there for months. It was a critical guard — when responses are unambiguously bad on any axis, you want them flagged regardless of how good the other axes look. But in combination with the judge's drift toward user-context-specificity, the auto-fail rule started flipping borderline cases to FAIL whenever the response was generic-but-correct rather than user-specific-and-correct. The rule was correctly applied. The rule's interaction with the drifting judge was the problem.</p>
<p>The third failure was in the test fixture. The canonical test user had a database populated by prior runs of the test itself. Several test cases mutate state — <em>add a todo</em>, <em>create a project</em>, <em>log an entry</em> — and the test infrastructure had never cleaned up the mutations between runs. By May, the canonical user had fifteen real todos accumulated in the database. One of the failing test cases asked Piper to recap the user's recent activity. The response was a confident recap of the polluted state. From the outside it looked exactly like a fabrication. From the inside it was an accurate report of a contaminated reference.</p>
<p>Three failures, each silent in isolation. Together they produced one observed quality regression that none of them caused individually.</p>
<h1>Why this kept hiding</h1>
<p>Each of the three failures had been latent for weeks. None of them screamed.</p>
<p>The judge's drift was silent because judges don't usually report what they're scoring along — they report a number. The number had a trendline. The trendline didn't show the drift, because the drift was in <em>what the number meant,</em> not in the number itself.</p>
<p>The auto-fail rule's amplification was silent because the rule was operating correctly per its own spec. <em>Any dim at zero → FAIL</em> doesn't have a wrong-answer state. It does what it says. The fact that it was now triggering on cases the team would not have called failures was a feature of the <em>composition</em> of the rule with the drifting judge, not of either piece alone.</p>
<p>The fixture pollution was silent because the database was working. Reads returned the data. Writes persisted. No error. No exception. The contamination only became legible when someone went looking for <em>why this test produced a confident-sounding wrong-shape answer.</em></p>
<p>The hypothesis the team started with — <em>the LLM is fabricating</em> — wasn't wrong about the surface symptom. It was wrong about what kind of failure surface it was. <em>Fabrication</em> names a single cause. The actual failure surface had three causes stacked, none of which was fabrication.</p>
<h1>Why "find the root cause" is sometimes the wrong move</h1>
<p>Most debugging discipline is built around finding <em>the</em> root cause. <em>Five Whys</em> digs from symptom to cause by asking why the layer above happened. <em>Fishbone diagrams</em> organize possible causes into categories so you can pick the most likely branch. Both techniques implicitly assume that there's <em>one</em> cause down at the bottom of the dig, and the discipline is to keep going until you find it.</p>
<p>That's right when there's one cause. It's exactly wrong when there are several, each contributing partly, none individually screaming.</p>
<p>The diagnostic move that worked on the quality-regression case wasn't <em>dig deeper.</em> It was <em>enumerate breadth.</em> Instead of pursuing the LLM-fabrication hypothesis to its limit, the engineering role catalogued all of the failed cases and asked, for each, <em>what could be contributing to this not just looking right.</em> The list of contributing factors converged to three. Each factor was a reasonable explanation for part of the failure surface. None alone explained it.</p>
<p>Once the team saw the stack, the fixes came in three pieces. Reset the fixture. Recalibrate the judge. Tighten the auto-fail rule. Each fix was small. The aggregate fix moved the quality number from 65.6% back to 68.9%, above the prior baseline. The system wasn't broken. The measurement infrastructure had drifted into a configuration that <em>produced</em> the appearance of breakage.</p>
<h1>Recognizing the shape</h1>
<p>If you're debugging something and you've found <em>a</em> cause but the fix doesn't move the metric as much as the surface symptom suggested it should, the shape is probably stacked silent failures.</p>
<p>Other signals:</p>
<ul><li>The surface symptom is suspiciously dramatic but the diagnostic feels thin.</li><li>You find a real bug, fix it, and the metric improves a little but not as much as you expected.</li><li>Multiple plausible explanations are sitting in your investigation notes, and you've been treating them as alternatives rather than as possibly simultaneous.</li><li>The system was working a few weeks ago and "nothing big changed."</li></ul>
<p>When you see those signals, the right discipline isn't to dig further on the cause you've already found. It's to <em>enumerate everything that could be contributing</em>, set each against the failure surface, and ask which combination explains what you're seeing. If the answer is <em>all three,</em> fix all three.</p>
<p>The deeper observation is that complex systems fail more often by composition than by single-cause. The vocabulary of root cause has been useful and will keep being useful, but it tends to bias the diagnostic toward depth when sometimes the right axis is breadth. A second discipline — <em>find all the contributing latent failures, including the ones that have been silent for weeks</em> — is the one that catches the stacked case.</p>
