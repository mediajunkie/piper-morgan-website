1b:T21a2,<figure><img alt="Two agents — a human and a robot — stand upstream at a fork in a river, each calmly choosing a different channel marked as claimed, while the water flows smoothly apart with no collisions downstream." src="https://cdn-images-1.medium.com/max/1024/1*nHZbKFADfo_89yMnz5PgNA.png" /><figcaption>“I’ll take this one!”</figcaption></figure><p><em>November 29</em></p><p>Two agents claimed different prompts from the coordination queue and worked in parallel. No conflicts. No merge issues. No stepping on each other’s work. When they finished, both deliverables integrated cleanly.</p><p>This seemed like a success story about the coordination system. It was, but not the way I first thought.</p><p>The system didn’t resolve conflicts between the agents. It prevented conflicts from occurring. These sound similar but they’re fundamentally different problems with different solutions.</p><h3>The distinction</h3><p><strong>Upstream coordination</strong> prevents two agents from working on the same thing. It happens before work begins. Claim a task, others see it’s claimed, they work on something else.</p><p><strong>Conflict resolution</strong> merges work when two agents have produced incompatible outputs. It happens after work is done. Compare the outputs, identify conflicts, decide which to keep.</p><p>Most coordination systems conflate these. Version control does both — branches prevent some conflicts, merge resolution handles others. Meeting scheduling does both — calendars prevent double-booking, rescheduling handles overlaps.</p><p>But they’re different problems requiring different mechanisms. Upstream coordination is about visibility and claiming. Conflict resolution is about comparison and decision-making.</p><h3>What the queue actually provides</h3><p>Our coordination queue is purely upstream. An agent claims a prompt. The manifest updates. Other agents see it’s claimed. They choose something else.</p><p>No comparison of outputs. No merge logic. No decision about whose work takes precedence. If two agents somehow claimed the same prompt and produced different deliverables, the system has no mechanism to resolve that.</p><p>This seems like a limitation. It’s actually a feature.</p><p>Conflict resolution for AI agent outputs is genuinely hard. How do you compare two different architectural proposals? Two different code implementations? Two different analyses of the same problem? There’s no diff tool for conceptual work.</p><p>By solving only upstream coordination, the queue stays simple. Claiming and visibility. No complicated merge logic that might fail in edge cases. The simpler system is more reliable precisely because it attempts less.</p><h3>Why upstream is usually enough</h3><p>The dirty secret: most coordination problems are upstream problems disguised as conflict problems.</p><p>Two agents don’t conflict because they work on the same thing and produce incompatible results. They conflict because nobody told them not to work on the same thing. The conflict happens downstream but the failure was upstream.</p><p>If you catch it upstream — clear claiming, visible work allocation, explicit coordination — the downstream conflict never materializes. You don’t need merge logic if you never have incompatible outputs.</p><p>This is why the parallel execution worked. Both agents could see what was claimed. Neither chose something already in progress. The coordination was complete before the work began.</p><h3>What this doesn’t solve</h3><p>The queue doesn’t solve:</p><p><strong>Dependency conflicts.</strong> Agent A’s work assumes X, Agent B’s work assumes not-X. Both complete their prompts successfully, but the outputs contradict each other. Upstream coordination can’t detect conceptual dependencies — it only knows about explicit task claims.</p><p><strong>Scope creep.</strong> Agent A claims Prompt 1, then while working discovers that Prompt 2 is related. Does work on both. Agent B claims Prompt 2 not knowing A has already touched it. Upstream coordination only knows what was claimed, not what was touched.</p><p><strong>Quality conflicts.</strong> Both agents produce valid work, but one is better than the other. Upstream coordination doesn’t compare quality — it just prevents duplication.</p><p>These are real limitations. They require either human judgment (review the outputs, choose between them) or more sophisticated coordination (dependency tracking, scope detection, quality assessment). The queue provides none of that.</p><p>But knowing what it doesn’t solve is valuable. You don’t expect the queue to handle conceptual dependencies. You add that capability elsewhere — or accept that it’s a human responsibility.</p><h3>The PM insight</h3><p>When testing the coordination queue, I noted: “File reservation solves upstream coordination, not conflict resolution.”</p><p>This framing helped me stop expecting things the queue wouldn’t provide. It coordinates work allocation. It doesn’t coordinate work content. Agents won’t duplicate effort, but they might produce incompatible results.</p><p>That’s okay. It’s the responsibility allocation I wanted. The queue handles who works on what. I handle whether the outputs cohere. Division of labor matches division of capability.</p><p>Human judgment for content coherence. Mechanical coordination for task allocation. Each system does what it’s good at.</p><h3>Building the right thing</h3><p>If I’d built conflict resolution into the queue, it would be:</p><ul><li>More complex (comparison logic, decision trees, escalation paths)</li><li>More fragile (edge cases in merge logic are infinite)</li><li>More presumptuous (assuming the system knows which output is better)</li><li>Slower to build and harder to maintain</li></ul><p>Instead I built only upstream coordination:</p><ul><li>Simple (claim and visibility)</li><li>Robust (binary state: claimed or available)</li><li>Modest (doesn’t pretend to judge quality)</li><li>Quick to build and easy to maintain</li></ul><p>The parallel agents validated this design. They didn’t need conflict resolution because upstream coordination prevented conflicts. The sophisticated mechanism I might have built would have been unused — and would have introduced bugs for problems that never arose.</p><p><em>Half the battle in these complex systems is realizing what not to build!</em></p><h3>Implications for multi-agent systems</h3><p>Multi-agent coordination is an emerging problem. Lots of people building systems where multiple AI agents collaborate. Most of the conversation focuses on sophisticated mechanisms: consensus protocols, negotiation frameworks, conflict resolution engines.</p><p>Maybe we’re overengineering. Maybe most multi-agent coordination is upstream — make task allocation visible, let agents claim before working, prevent conflicts rather than resolve them.</p><p>The hard problems remain hard. Conceptual dependencies. Quality assessment. Coherence checking. But those might be human problems wearing technical disguises. No mechanism resolves whether two architectural proposals are compatible — that requires judgment about architecture.</p><p>Upstream coordination is mechanical. Conflict resolution is often judgmental. Build the mechanical part. Accept responsibility for the rest.</p><p>The queue taught me this distinction. Not by handling conflicts elegantly, but by not having any conflicts to handle. Prevention, it turns out, is not just better than cure — it’s a different kind of solution entirely.</p><p><em>Next on Building Piper Morgan, we resume the narrative with The Cathedral in Winter, from February 10 to 13, a period when my flu got bad but the project kept building itself.</em></p><p><em>How do you distinguish upstream coordination from conflict resolution in your systems? There’s no diff tool for conceptual work.</p><p name="b661" id="b661" class="graf graf--p graf-after--p">By solving only upstream coordination, the queue stays simple. Claiming and visibility. No complicated merge logic that might fail in edge cases. The simpler system is more reliable precisely because it attempts less.</p><h3 name="bc0d" id="bc0d" class="graf graf--h3 graf-after--p">Why upstream is usually enough</h3><p name="7478" id="7478" class="graf graf--p graf-after--h3">The dirty secret: most coordination problems are upstream problems disguised as conflict problems.</p><p name="8e72" id="8e72" class="graf graf--p graf-after--p">Two agents don’t conflict because they work on the same thing and produce incompatible results. They conflict because nobody told them not to work on the same thing. The conflict happens downstream but the failure was upstream.</p><p name="6883" id="6883" class="graf graf--p graf-after--p">If you catch it upstream — clear claiming, visible work allocation, explicit coordination — the downstream conflict never materializes. You don’t need merge logic if you never have incompatible outputs.</p><p name="f88b" id="f88b" class="graf graf--p graf-after--p">This is why the parallel execution worked. Both agents could see what was claimed. Neither chose something already in progress. The coordination was complete before the work began.</p><h3 name="9190" id="9190" class="graf graf--h3 graf-after--p">What this doesn’t solve</h3><p name="a756" id="a756" class="graf graf--p graf-after--h3">The queue doesn’t solve:</p><p name="9a6b" id="9a6b" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Dependency conflicts.</strong> Agent A’s work assumes X, Agent B’s work assumes not-X. Both complete their prompts successfully, but the outputs contradict each other. Upstream coordination can’t detect conceptual dependencies — it only knows about explicit task claims.</p><p name="a95c" id="a95c" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Scope creep.</strong> Agent A claims Prompt 1, then while working discovers that Prompt 2 is related. Does work on both. Agent B claims Prompt 2 not knowing A has already touched it. Upstream coordination only knows what was claimed, not what was touched.</p><p name="ad80" id="ad80" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">Quality conflicts.</strong> Both agents produce valid work, but one is better than the other. Upstream coordination doesn’t compare quality — it just prevents duplication.</p><p name="de84" id="de84" class="graf graf--p graf-after--p">These are real limitations. They require either human judgment (review the outputs, choose between them) or more sophisticated coordination (dependency tracking, scope detection, quality assessment). The queue provides none of that.</p><p name="7008" id="7008" class="graf graf--p graf-after--p">But knowing what it doesn’t solve is valuable. You don’t expect the queue to handle conceptual dependencies. You add that capability elsewhere — or accept that it’s a human responsibility.</p><h3 name="f1fa" id="f1fa" class="graf graf--h3 graf-after--p">The PM insight</h3><p name="c113" id="c113" class="graf graf--p graf-after--h3">When testing the coordination queue, I noted: “File reservation solves upstream coordination, not conflict resolution.”</p><p name="ea19" id="ea19" class="graf graf--p graf-after--p">This framing helped me stop expecting things the queue wouldn’t provide. It coordinates work allocation. It doesn’t coordinate work content. Agents won’t duplicate effort, but they might produce incompatible results.</p><p name="aacd" id="aacd" class="graf graf--p graf-after--p">That’s okay. It’s the responsibility allocation I wanted. The queue handles who works on what. I handle whether the outputs cohere. Division of labor matches division of capability.</p><p name="edbc" id="edbc" class="graf graf--p graf-after--p">Human judgment for content coherence. Mechanical coordination for task allocation. Each system does what it’s good at.</p><h3 name="681b" id="681b" class="graf graf--h3 graf-after--p">Building the right thing</h3><p name="0b0d" id="0b0d" class="graf graf--p graf-after--h3">If I’d built conflict resolution into the queue, it would be:</p><ul class="postList"><li name="eb7b" id="eb7b" class="graf graf--li graf-after--p">More complex (comparison logic, decision trees, escalation paths)</li><li name="d8c8" id="d8c8" class="graf graf--li graf-after--li">More fragile (edge cases in merge logic are infinite)</li><li name="536c" id="536c" class="graf graf--li graf-after--li">More presumptuous (assuming the system knows which output is better)</li><li name="eeef" id="eeef" class="graf graf--li graf-after--li">Slower to build and harder to maintain</li></ul><p name="f4c5" id="f4c5" class="graf graf--p graf-after--li">Instead I built only upstream coordination:</p><ul class="postList"><li name="b091" id="b091" class="graf graf--li graf-after--p">Simple (claim and visibility)</li><li name="b9b9" id="b9b9" class="graf graf--li graf-after--li">Robust (binary state: claimed or available)</li><li name="3f4d" id="3f4d" class="graf graf--li graf-after--li">Modest (doesn’t pretend to judge quality)</li><li name="832d" id="832d" class="graf graf--li graf-after--li">Quick to build and easy to maintain</li></ul><p name="5db8" id="5db8" class="graf graf--p graf-after--li">The parallel agents validated this design. They didn’t need conflict resolution because upstream coordination prevented conflicts. The sophisticated mechanism I might have built would have been unused — and would have introduced bugs for problems that never arose.</p><p name="8fd0" id="8fd0" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">Half the battle in these complex systems is realizing what not to build!</em></p><h3 name="eee1" id="eee1" class="graf graf--h3 graf-after--p">Implications for multi-agent systems</h3><p name="bc8e" id="bc8e" class="graf graf--p graf-after--h3">Multi-agent coordination is an emerging problem. Lots of people building systems where multiple AI agents collaborate. Most of the conversation focuses on sophisticated mechanisms: consensus protocols, negotiation frameworks, conflict resolution engines.</p><p name="4135" id="4135" class="graf graf--p graf-after--p">Maybe we’re overengineering. Maybe most multi-agent coordination is upstream — make task allocation visible, let agents claim before working, prevent conflicts rather than resolve them.</p><p name="2886" id="2886" class="graf graf--p graf-after--p">The hard problems remain hard. Conceptual dependencies. Quality assessment. Coherence checking. But those might be human problems wearing technical disguises. No mechanism resolves whether two architectural proposals are compatible — that requires judgment about architecture.</p><p name="5d14" id="5d14" class="graf graf--p graf-after--p">Upstream coordination is mechanical. Conflict resolution is often judgmental. Build the mechanical part. Accept responsibility for the rest.</p><p name="6bbb" id="6bbb" class="graf graf--p graf-after--p graf--trailing">The queue taught me this distinction. Not by handling conflicts elegantly, but by not having any conflicts to handle.
